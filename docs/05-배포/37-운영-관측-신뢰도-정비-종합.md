# 37. 운영 관측 신뢰도 정비 종합

## 문서 목적

운영 관측 체계 도입 이후, 실제 운영과 검증 흐름에서 관측 도구를 `믿고 볼 수 있는 상태`로 다듬은 결과를 한 문서로 정리한다.  
이번 문서는 새 도구를 추가하는 데 목적이 있지 않고, 이미 붙어 있는 `Grafana / Loki / Prometheus / Alertmanager / Sentry`를 실제 배포와 smoke 검증 기준으로 다시 정리한 결과를 요약하는 데 목적이 있다.

## 1. 작업 배경

기존 운영 관측 체계는 다음 요소를 이미 갖추고 있었다.

- Prometheus
- Grafana
- Loki / Promtail
- Alertmanager
- Sentry
- `requestId`
- smoke / `k6` 스크립트

하지만 실제 운영 기준으로는 다음 한계가 남아 있었다.

- Grafana 대시보드가 운영 목적별로 충분히 분리돼 있지 않았다.
- smoke 1건이나 짧은 검증 흐름이 대시보드에서 바로 해석되기 어려웠다.
- alert rule은 있었지만 제품 API 기준으로 더 좁힐 필요가 있었다.
- Sentry는 예외를 보기는 했지만, 배포 / 요청 / 인증 문맥을 충분히 담고 있지 않았다.
- prod 배포 후 `health -> smoke -> Grafana -> Loki -> Sentry` 흐름을 실전적으로 다시 확인할 필요가 있었다.

결국 이번 단계에서는 새 관측 도구를 붙이는 대신, 운영 관측 체계의 `신뢰도`와 `가독성`을 높이는 데 집중했다.

## 2. 최종 관측 구조

이번 단계 이후 기준 관측 구조는 다음과 같이 본다.

```text
배포 / smoke / k6 입력
-> Prometheus 메트릭
-> Grafana 대시보드
-> Loki 로그
-> Sentry 예외 문맥
-> Alertmanager 알림
```

역할은 다음처럼 정리된다.

- `Overview`
  - 추세, 부하, 지연, 에러율 확인
- `Logs`
  - app / nginx 로그 분리 확인
- `Deploy / Smoke`
  - 배포 직후와 smoke 직후의 성공/실패 흔적 확인
- Sentry
  - 앱 버그와 실제 5xx 중심 예외 추적

즉 이번 단계의 핵심은 `메트릭 -> 로그 -> 예외`의 역할 구분을 더 명확하게 만든 점이다.

## 3. 핵심 적용 사항

### 3.1 Grafana 대시보드 재구성

기존 overview는 메트릭과 로그 역할이 충분히 분리되지 않았고, 운영 판단용으로 보기엔 구조가 거칠었다.  
이번 단계에서는 역할 기준으로 대시보드를 나눴다.

- `Click Checker Overview`
  - `API Request Mix`
  - `Ingest / Analytics RPS`
  - `Ingest / Analytics p95 / p99`
  - `4xx / 5xx ratio`
  - `DB pool`, `DB active connections`, `heap`
- `Click Checker Logs`
  - app 로그
  - nginx access 로그
  - error / warning focus
- `Click Checker Deploy / Smoke`
  - 최근 ingest 성공
  - 최근 analytics 성공
  - 최근 API 5xx
  - smoke 로그 흔적
  - color별 availability / public request

이 과정에서 Loki 로그 쿼리 문법 오류가 있던 nginx 패널도 함께 정리했다.

### 3.2 Alert 기준 정리

Prometheus alert rule은 그대로 두지 않고, 제품 API 기준으로 더 좁혔다.

- `5xx ratio`
  - 전체 요청이 아니라 `/api/...` 중심으로 해석
- `p95 latency`
  - `ingest`, `analytics` 중심으로 해석

즉 `health`, `/`, 스캐너성 요청이 제품 API alert 해석을 어지럽히는 비중을 줄였다.

### 3.3 Sentry 요청 문맥 보강

이번 단계에서는 Sentry를 단순 예외 저장소가 아니라, 운영 추적 도구로 한 단계 더 사용하도록 정리했다.

기본 문맥:

- `requestId`
- `httpMethod`
- `requestPath`
- `authType=anonymous`

JWT 인증 성공 시:

- `authType=jwt`
- `accountId`

API Key 인증 성공 시:

- `authType=api-key`
- `organizationId`
- `apiKeyKid`

또한 Sentry에는 다음 문맥도 함께 보이도록 정리했다.

- `appColor`
- `release`

### 3.4 Sentry 잡음 제거

운영 기준으로 봇과 잘못된 클라이언트 요청이 만드는 프레임워크성 4xx는 더 이상 Sentry에 올리지 않도록 정리했다.

제외 대상:

- `NoResourceFoundException`
- `HttpRequestMethodNotSupportedException`
- `HttpMediaTypeNotAcceptableException`
- `HttpMediaTypeNotSupportedException`

즉 이번 단계 이후 Sentry는 앱 버그와 실제 5xx성 문제를 더 잘 드러내는 방향으로 정리됐다.

### 3.5 release 전달 경로 정리

Sentry에서 배포 회귀를 볼 수 있도록 release 전달 경로도 추가했다.

- `GitHub Actions`
  - image tag = `GITHUB_SHA`
- `CodeDeploy AfterInstall`
  - `APP_IMAGE`에서 tag를 잘라 `SENTRY_RELEASE` 생성
- `docker-compose.prod.yml`
  - prod app env에 `SENTRY_RELEASE` 전달

즉 배포 이미지 태그가 Sentry release 문맥으로 이어지게 됐다.

### 3.6 smoke / `k6` 검증 흐름 정리

`deploy-smoke.sh`는 현재 공식 흐름 기준으로 정리됐다.

```text
signup
-> admin organization create
-> apiKey 수령
-> event ingest
-> aggregate query
```

또한:

- smoke payload에 `source=deploy-smoke`, `runId` 반영
- 선택적으로만 임시 API key 파일 export 가능
- `k6` read smoke의 시간 파라미터를 현재 API 계약에 맞게 보정

### 3.7 Sentry 실제 이벤트 검증 경로 추가

이번 단계에서는 Sentry 문맥이 실제로 붙는지 확인하기 위해 internal-only smoke 경로를 추가해 1회 검증할 수 있게 했다.

- `POST /internal/sentry/smoke`
- app 포트(`127.0.0.1:8081` / `8082`) 직접 호출 기준
- 의도적으로 `500` 예외를 발생시켜 Sentry 이벤트 생성

이 경로를 이용해 다음 문맥이 실제로 붙는지 확인했다.

- `release`
- `appColor`
- `requestId`
- `authType`
- `requestPath`

## 4. 실제 검증 결과

### 4.1 local 검증

local에서는 다음을 확인했다.

- 최신 `bootJar` 기준 app 재빌드
- `docker-compose.yml + docker-compose.local.yml` 조합 기동
- `deploy smoke` 성공
  - `signup`
  - `organization create`
  - `event ingest`
  - `aggregate query`
- 짧은 `k6` write burst 성공
- 짧은 `k6` read burst 성공

이 과정에서 확인한 점:

- 단건 smoke는 `Overview`보다 `Deploy / Smoke`, `Logs`에서 더 잘 보인다.
- `Overview`는 단건보다는 추세 / burst 해석에 더 적합하다.

### 4.2 prod 검증

prod에서는 다음을 확인했다.

- CodeDeploy 배포 성공
- observability 재기동 / health 검증 스크립트 기준 정상 동작
- prod 배포 중 readiness 실패 원인을 실제로 추적
  - 원인: `JWT_SECRET` 누락
  - 조치:
    - 운영 `.env`에 `JWT_SECRET` 추가
    - `docker-compose.prod.yml`에 `JWT_SECRET` 전달 추가
    - 배포 오케스트레이터에서 `.env` 필수값 검증 추가
- Grafana nginx 패널 parse error 원인 확인
  - 원인: Loki 쿼리의 `or { ... }` 문법 오류
  - 조치: regex filter 기반 LogQL로 수정
- internal sentry smoke 1회 호출로 실제 Sentry 이벤트 문맥 확인
  - `release=<이번 배포 sha>`
  - `appColor=<active color>`
  - `requestId`
  - `authType=anonymous`
  - `requestPath=/internal/sentry/smoke`

즉 이번 단계는 단순 설정 추가가 아니라, 실제 운영 검증 중 드러난 배포/관측 연결 문제를 함께 다듬는 과정이기도 했다.

## 5. 확인된 운영 기준

현재 기준 운영 관측 판단은 다음과 같이 정리된다.

- 단건 smoke 여부는 `Deploy / Smoke`에서 먼저 본다.
- 추세와 부하 흐름은 `Overview`에서 본다.
- app / nginx 로그는 `Logs`에서 분리해서 본다.
- 예외는 Sentry에서 본다.
- 배포 직후에는 `health -> smoke -> Grafana -> Loki -> Sentry` 순서로 확인한다.
- Sentry에서는 프레임워크성 4xx보다 앱 버그와 실제 5xx를 중심으로 본다.
- release는 image tag 기준으로 연결한다.

## 6. 이번 단계에서 남긴 기준

- observability 도구는 단순 연결보다, 실제 운영에서 해석 가능한 상태인지가 더 중요하다.
- 단건 smoke 확인과 추세 확인은 같은 대시보드에 몰지 않는 편이 더 낫다.
- `requestId`, `appColor`, `release`, 인증 문맥이 있어야 Sentry가 운영 도구로 의미를 갖는다.
- prod `.env` 필수값과 compose env 전달 경로는 따로 확인해야 한다.
- 배포 검증용 스크립트와 대시보드 반응은 실제로 한 번 연결해서 보는 것이 중요하다.

## 7. 후속 보정 후보

이번 단계에서 즉시 닫지 않고 남겨둔 항목은 다음과 같다.

- Alert 실제 발송 체인 반복 검증
- `Smoke Log Hits`를 로그 줄 수가 아니라 더 좁은 의미의 지표로 바꿀지 검토
- 성능 개선 단계에서 현재 `Overview`를 기준선 대시보드로 계속 사용할지 검토

## 8. 결론

이번 작업을 통해 운영 관측 체계는 “도구가 붙어 있는 상태”에서 “실제 운영 검증과 배포 확인에 쓸 수 있는 상태”로 한 단계 정리됐다.

특히 이번 단계의 핵심은 다음 세 가지다.

- 대시보드를 목적별로 분리해 해석 가능성을 높였다.
- Sentry에 요청 / 인증 / 배포 문맥을 보강하고 잡음을 줄였다.
- smoke와 짧은 `k6` 검증, 그리고 prod 배포 확인까지 연결해 실제 운영 기준을 다시 점검했다.

즉 이번 단계의 결론은 다음처럼 정리할 수 있다.

`운영 관측 체계가 이제 실제 배포와 smoke 검증 흐름에서 신뢰 가능한 수준으로 정리됐다.`
