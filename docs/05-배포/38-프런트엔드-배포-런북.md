# 38. 프런트엔드 배포 런북

## 문서 목적

기존 `CodeDeploy + ECR + backend blue/green + EC2 nginx` 운영 구조 위에 프런트엔드를 처음 올릴 때 따라갈 절차를 정리한다.  
이번 문서는 backend의 blue/green 전환 구조는 유지하고, frontend는 single container로 추가하는 기준 런북이다.

아직 이번 단계는 실제 운영 첫 적용 전이므로, 본 문서는 **첫 적용 기준 런북 초안** 성격을 가진다.  
실제 적용 중 발생한 문제와 수정 사항은 이후 트러블슈팅 문서에 누적하고, 최종 운영 기준은 종합 문서에서 다시 정리한다.

## 현재 전제

- 운영 도메인: `https://clickchecker.dev`
- 관리자 관측 경로: `https://grafana.clickchecker.dev`
- backend 배포 구조:
  - `GitHub Actions -> ECR -> S3 -> CodeDeploy -> EC2 -> blue/green`
  - `app-blue(8081)`, `app-green(8082)`
- 운영 nginx는 현재 `clickchecker.dev` 요청을 active backend로 프록시한다.
- frontend는 아직 운영 배포 대상에 포함되지 않았다.
- 이번 단계에서 frontend는 **single container**로 추가한다.
- 이번 단계에서 frontend health check 경로는 `/healthz`로 고정한다.

## 현재 구현 상태

- `frontend/Dockerfile` 추가 완료
- `frontend`의 `output: "standalone"` 적용 완료
- `frontend /healthz` route 추가 완료
- `docker-compose.prod.yml`의 `frontend` 서비스 추가 완료
- repo `nginx/click-checker.conf`의 frontend/backend 분기 구조 반영 완료
- CodeDeploy / deploy script에 frontend deploy, frontend recover, nginx apply 흐름 반영 완료
- 실제 운영 first apply는 아직 수행하지 않았다

## 목표

- 운영 nginx가 same-origin 기준으로 frontend와 backend를 경로별로 분기한다.
- backend는 기존처럼 blue/green으로 유지한다.
- frontend는 단일 컨테이너로 기동하고, nginx가 `/_next/**`, 일반 page route, `/healthz`를 frontend로 전달한다.
- `/api/**`, `/actuator/health`, `/actuator/health/readiness`는 active backend로 전달한다.
- frontend 배포 실패가 backend active color 전환 로직을 오염시키지 않게 한다.

## 이번 런북의 범위

- frontend 운영 이미지 준비
- prod compose에 frontend 서비스 추가
- nginx path routing 수정
- frontend health check(`/healthz`) 확인
- frontend only / backend only / both 변경 시 적용 순서 정리
- 운영 검증 기준 정리

## 고정 운영 원칙

### 1. backend는 계속 blue/green이다

- `app-blue`, `app-green` 전환 구조는 유지한다.
- frontend 추가 때문에 backend 전환 구조를 단순화하지 않는다.

### 2. frontend는 single container다

- frontend까지 blue/green으로 확장하지 않는다.
- 현재 단계에서는 운영 복잡도 감소를 우선한다.
- 대신 frontend 재기동 시 짧은 공백이 생길 수 있다는 trade-off를 감수한다.

### 3. 외부 도메인은 하나로 유지한다

- 사용자는 계속 `https://clickchecker.dev` 하나로 접근한다.
- nginx가 same-origin 기준으로 frontend와 backend를 path routing 한다.

## 적용 순서

### 1. frontend 운영 이미지 준비

- `frontend`를 Next standalone 기준으로 빌드한다.
- 운영 Dockerfile을 기준으로 image를 만든다.
- image는 backend와 별도 이름으로 push한다.

현재 구현:

- `frontend/Dockerfile`
- `frontend/.dockerignore`
- `frontend/next.config.ts`
- `frontend/src/app/healthz/route.ts`

확인 포인트:

- 컨테이너가 `3000`에서 정상 기동하는지
- `/healthz`가 200을 반환하는지

### 2. prod compose에 frontend 서비스 추가

예상 기준:

- 서비스 이름: `frontend`
- 포트 예시: `127.0.0.1:3001:3000`

현재 구현:

- `docker-compose.prod.yml`
- frontend healthcheck:
  - `wget -qO- http://localhost:3000/healthz | grep -q '"status":"ok"'`

원칙:

- 외부 전체 공개는 하지 않는다.
- nginx만 loopback 기준으로 frontend에 붙는다.

### 3. nginx routing 수정

원칙:

- backend 경로를 먼저 명시적으로 선언한다.
- 그 다음 frontend 정적 자산 경로를 선언한다.
- 마지막에 일반 page route를 frontend catch-all로 둔다.

최소 확인 경로:

- `/api/events`
- `/actuator/health`
- `/actuator/health/readiness`
- `/_next/static/...`
- `/healthz`
- `/guide`

현재 구현:

- `nginx/click-checker.conf`
- `scripts/deploy/apply-public-nginx-config.sh`

### 4. frontend health 확인

예시:

```bash
curl -i http://127.0.0.1:3001/healthz
```

기대 결과:

- `200`
- auth 가드, 데이터 fetch, 리다이렉트 없이 가벼운 응답

현재 구현:

- frontend image를 컨테이너로 띄워 `/healthz` 응답 확인 완료

### 5. nginx 검증 및 반영

예시:

```bash
sudo nginx -t
sudo systemctl reload nginx
```

확인 포인트:

- backend path가 여전히 active backend를 본다.
- frontend page route가 frontend 컨테이너를 본다.
- Grafana 서브도메인 경로는 영향이 없다.

### 6. public smoke 확인

확인 경로:

```bash
curl -I https://clickchecker.dev/healthz
curl -I https://clickchecker.dev/actuator/health
curl -I https://clickchecker.dev/actuator/health/readiness
```

추가 확인:

- `https://clickchecker.dev/guide`
- `https://clickchecker.dev/login`
- `https://clickchecker.dev/organizations`
- `https://clickchecker.dev/api/...`

## 배포 순서 기준

### 1. frontend만 변경된 경우

흐름:

```text
frontend image pull
-> frontend recreate
-> /healthz 확인
-> public frontend smoke
```

원칙:

- backend active color 전환 없음
- frontend 실패는 frontend 배포 실패로만 본다

### 2. backend만 변경된 경우

흐름:

```text
기존 backend blue/green 전환
-> readiness
-> direct/public smoke
-> old color stop
```

원칙:

- frontend 재기동 없음

### 3. frontend와 backend가 모두 변경된 경우

흐름:

```text
backend blue/green 전환
-> backend public smoke
-> frontend recreate
-> /healthz 확인
-> public frontend smoke
```

원칙:

- backend active color 전환 판단은 backend 기준으로 유지한다.
- frontend 실패가 backend rollback 판단을 직접 오염시키지 않게 분리한다.

## 롤백 기준

다음 상황이면 frontend 적용을 롤백하거나 이전 상태로 복구한다.

- `/healthz` 실패
- page route가 5xx 또는 blank 응답
- `/api/**`가 frontend로 잘못 라우팅됨
- backend health/readiness 경로가 frontend로 잘못 라우팅됨
- Grafana 또는 기존 관리자 경로가 예상치 않게 흔들림

## 롤백 절차

### frontend only 변경 롤백

1. 이전 frontend image 또는 이전 compose 상태로 되돌린다.
2. frontend recreate
3. `/healthz`와 public page route 재확인

### routing 변경 롤백

1. nginx 설정을 백업본으로 되돌린다.
2. `sudo nginx -t`
3. `sudo systemctl reload nginx`
4. `/healthz`, `/actuator/health`, `/api/**` 재확인

### both 변경 중 frontend 실패

1. backend active color는 그대로 유지한다.
2. frontend만 이전 이미지/이전 설정으로 되돌린다.
3. backend rollback은 backend health/readiness 문제가 있을 때만 별도로 판단한다.

## 주의사항

- frontend는 single container라서 재기동 순간 짧은 공백이 생길 수 있다.
- 현재 단계에서는 이 공백을 frontend blue/green 추가 복잡도보다 작은 비용으로 본다.
- frontend health check는 `/`가 아니라 `/healthz`로 고정한다.
- `/`는 auth 가드나 렌더링 변경 영향이 있을 수 있으므로 liveness 용도로 쓰지 않는다.
- nginx catch-all 앞에 backend 전용 location을 반드시 먼저 선언한다.
- frontend는 same-origin 기준으로 `/api`를 호출하므로 nginx routing 실수가 API 장애처럼 보일 수 있다.
