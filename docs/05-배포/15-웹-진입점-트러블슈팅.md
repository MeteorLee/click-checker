# 15. 웹 진입점 트러블슈팅

## 0. 목적
- 웹 진입점 구성 과정에서 실제로 발생한 문제와 해결 과정을 기록한다.
- 이후 도메인, Nginx, HTTPS를 다시 구성할 때 같은 실수를 줄인다.

---

## 1. EC2에 repo nginx 설정 파일이 없었던 문제

### 증상
- EC2에서 `~/click-checker/nginx/click-checker.conf`를 찾지 못했다.

### 원인
- 로컬에서 nginx 설정 파일을 만들었지만, 아직 `prod`까지 반영하지 않은 상태였다.
- 로컬 작업공간 파일과 EC2 배포 상태를 혼동했다.

### 해결
- nginx 설정 파일을 커밋한 뒤 `main -> prod` 반영
- EC2 배포 후 `~/click-checker/nginx/click-checker.conf` 존재 확인

---

## 2. `.dev` 도메인 접속 불가 문제

### 증상
- 브라우저에서 `clickchecker.dev` 접속 시 "사이트에 연결할 수 없음"
- EC2에서 `curl -i http://clickchecker.dev`도 멈춘 것처럼 보였다.

### 원인
- `.dev` 도메인은 브라우저에서 사실상 HTTPS 전용처럼 동작한다.
- 초기 상태에서는 `443` 포트와 인증서가 없어서 브라우저 접속이 실패했다.
- EC2 내부에서 자기 자신의 퍼블릭 도메인으로 접근하는 방식도 확인 경로로 적절하지 않았다.

### 해결
- 도메인 테스트는 `curl -i -H 'Host: clickchecker.dev' http://127.0.0.1`로 대체
- 이후 `certbot`으로 HTTPS 적용

---

## 3. certbot 첫 인증서 발급 실패

### 증상
- `sudo certbot --nginx -d clickchecker.dev` 실행 시 인증서 발급 실패
- 에러 메시지:
  - `Timeout during connect (likely firewall problem)`

### 원인
- EC2 보안 그룹에 `443`만 열어두고 `80`을 열지 않았다.
- Let's Encrypt는 첫 검증 단계에서 `HTTP 80`으로 도메인 소유 확인을 시도했다.

### 해결
- EC2 보안 그룹 인바운드에 `HTTP 80` 추가
- 이후 `sudo certbot --nginx -d clickchecker.dev` 재실행

---

## 4. Nginx는 정상인데 `/`가 500이던 문제

### 증상
- `curl -I http://localhost` 결과가 `500`
- `/actuator/health`는 `200`, `/actuator/prometheus`는 `403`

### 원인
- Nginx reverse proxy 자체는 정상 동작했다.
- 앱에 `/` 루트 응답이 없어서 `NoResourceFoundException`이 발생했다.

### 해결
- 앱에 루트 기본 응답 추가
- `GET /`에서 최소 JSON 응답 반환

---

## 5. certbot이 실제 nginx 설정을 수정해서 repo와 서버 설정이 달라진 문제

### 증상
- HTTPS 적용 후 EC2의 `/etc/nginx/sites-available/default`에는 `443`, 인증서 경로, `80 -> 443` 리다이렉트가 생겼다.
- repo의 `nginx/click-checker.conf`는 여전히 HTTP-only 상태였다.

### 원인
- `certbot --nginx`가 서버의 실제 nginx 설정 파일을 직접 수정했다.
- git 관리 중인 설정 파일은 자동으로 갱신되지 않는다.

### 해결
- EC2의 실제 nginx 설정을 확인
- 변경 내용을 `nginx/click-checker.conf`에 반영
- 이후에는 repo 파일이 현재 운영 진실과 일치하도록 유지

---

## 6. 정리
- 도메인/HTTPS 작업은 로컬 파일, 배포 상태, 서버 실제 설정 파일을 분리해서 봐야 한다.
- `.dev` 도메인은 HTTPS 적용 전에는 브라우저 확인이 불편하다.
- certbot은 `443`뿐 아니라 `80` 접근도 필요하다.
- `certbot --nginx` 사용 후에는 서버 실제 설정과 repo 설정을 반드시 다시 동기화해야 한다.
