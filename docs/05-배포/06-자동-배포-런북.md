# Click-Checker 자동 배포 v1 런북 (v1.2)

## 0. 목적
- prod 브랜치 push 이후 자동 배포를 수행한다.
- 배포 실패 시 자동 롤백을 수행한다.
- 브랜치별 CI 강도를 분리한다 (develop: 빠름 / main: 강함 / prod: 배포).

---

# 1. 전체 구조 요약

- develop: 빠른 CI (H2 기반)
- main: 강한 CI (PostgreSQL 통합 테스트 포함)
- prod: 배포 + EC2 스모크 + 실패 시 롤백

배포 방식(v1): EC2에서 `git pull` + `docker compose up -d --build` (A 방식)

---

# 2. 브랜치별 CI 정책

## develop
- build + H2(ci) 테스트만 (빠르게)
- 기능 개발/실험 브랜치로 사용한다.
- 빠른 CI 통과 시에만 상위 브랜치로 머지한다.

## main
- develop CI 포함 + PostgreSQL 통합 테스트(느리지만 강하게)
- 릴리즈 후보 브랜치로 사용한다.
- 강한 CI(H2 + PostgreSQL 호환) 통과 시에만 머지한다.

## prod
- deploy 실행 + EC2 스모크로 최종 확인
- (v1) prod 단계에서는 build/test를 다시 돌리지 않는다. (main에서 강하게 검증했다고 가정)
- 배포 전용 브랜치로 사용한다.
- main에서 검증된 코드만 머지한다.

---

# 3. 배포 성공 기준 (prod)

다음 조건을 모두 만족해야 성공:

1. `GET /actuator/health` → 200 + "UP"
2. 이벤트 저장 API 1건 성공 (POST 1회)
3. 집계 조회 API 1건 성공 (GET 1회)

---

# 4. 스모크 실행 위치/명령 고정 (중요)

스모크는 **EC2 서버 내부**에서 실행한다.  
(서비스 접근은 `localhost` 기준)

---

## 4.1 Health 확인

```bash
curl -fsS http://localhost:8080/actuator/health
```

기대:

- HTTP 200
- body에 `"UP"` 포함

---

## 4.2 이벤트 저장 1건 (WRITE)

### 1) Organization 생성 (이름 유니크)

```bash
TS=$(date +"%Y%m%d%H%M%S")

ORG_JSON=$(curl -fsS -X POST http://localhost:8080/api/organizations \
  -H "Content-Type: application/json" \
  -d "{\"name\":\"deploy-smoke-org-${TS}\"}")
```

### 2) ORG_ID 추출 (jq 사용)

```bash
ORG_ID=$(echo "${ORG_JSON}" | jq -r '.id')

test -n "${ORG_ID}" && test "${ORG_ID}" != "null"
echo "ORG_ID=${ORG_ID}"
```

### 3) 이벤트 저장

```bash
curl -fsS -X POST http://localhost:8080/api/events \
  -H "Content-Type: application/json" \
  -d "{
    \"organizationId\": ${ORG_ID},
    \"externalUserId\": \"deploy-smoke-user\",
    \"eventType\": \"click\",
    \"path\": \"/__test/deploy-smoke/${TS}\",
    \"occurredAt\": \"2026-03-02T00:00:00\",
    \"payload\": \"{}\"
  }"
```

기대:

- HTTP 2xx 응답

---

## 4.3 집계 조회 1건 (READ)

```bash
curl -fsS "http://localhost:8080/api/v1/events/analytics/aggregates/paths?organizationId=${ORG_ID}&from=2020-01-01T00:00:00&to=2030-01-01T00:00:00&top=5"
```

기대:

- HTTP 200

# 5. 롤백 정책 (v1)

## 5.1 롤백 조건

- 배포 직후 health 실패
- 배포 직후 스모크 실패

---

## 5.2 롤백 방식 (reset 대신 checkout)

배포 직전 커밋 SHA를 `PREV_SHA`로 저장한다.

실패 시:

```bash
git checkout -f <PREV_SHA>

docker compose -f docker-compose.yml -f docker-compose.prod.yml up -d --build

curl -fsS http://localhost:8080/actuator/health
```

---

## 주의 사항

- DB 스키마 변경이 포함되면 코드만 롤백해도 정상 동작이 보장되지 않을 수 있다.
- v1 단계에서는 파괴적인 스키마 변경을 지양한다.
