# 26. 배포 실전 검증 기록

## 문서 목적

배포 실전 검증 과정에서 실제로 실행한 baseline, 배포 중 검증, 결과 수치, 해석을 기록한다.  
이번 문서는 절차를 설명하는 런북이 아니라, 어떤 조건에서 무엇을 실행했고 어떤 수치가 나왔는지 남기는 데 목적이 있다.

## 1. 검증 범위

- 대상 환경: `prod`
- 검증 대상:
  - 지속 read 요청 baseline
  - 지속 read 요청 중 배포 전환
  - 지속 write 요청 baseline
  - 지속 write 요청 중 배포 전환
- 관찰 기준:
  - `http_req_failed`
  - timeout
  - `p95`
  - `p99`
  - `RPS`
  - `5xx`

```bash
  -e TOP='5' \
```


### 4.3 `k6` 결과

- total requests: `5984`
- `http_req_failed`: `0.00%`
- checks failed: `0`
- status `200` 유지
- `http_req_duration.avg`: `35.78ms`
- `http_req_duration.p95`: `31.14ms`
- `http_req_duration.p99`: `167.4ms`
- max observed duration: `3.63s`
- observed RPS: `19.075661/s`
- dropped iterations: `17`

### 4.4 Grafana 관찰 결과

- `RPS`
  - 평소 약 `19.3` 수준을 유지했다.
  - `02:18` 부근부터 하락하기 시작해 `02:18:45` 무렵에는 절반 수준까지 떨어졌다.
  - 이후 다시 회복해 `02:19:15` 무렵에는 거의 원래 수준으로 돌아왔다.
- `p95`
  - `02:18:45` 부근에 상승했다가 서서히 내려왔다.
- `p99`
  - `02:18:45` 부근에 소폭 상승했다.
  - `02:20:15` 부근에 더 큰 상승이 보였는데, `k6` 종료 직후 구간 영향 가능성이 있다.
- `5xx`
  - 실질적으로 `0`으로 해석했다.
  - 그래프에서 잠깐 비어 보이는 구간이 있었지만, `k6` 결과와 함께 보면 5xx 발생으로 보이지는 않았다.

### 4.5 해석

- 배포 중에도 `http_req_failed`, `5xx`, timeout 없이 요청이 유지됐다.
- 전환 시점 근처에서 `RPS`, `p95`, `p99`는 일시적으로 흔들렸지만 다시 회복했다.
- `p99`는 baseline(5m)보다 높아졌지만 임계치(`1000ms`) 안에 있었고, 실패 응답으로 이어지지 않았다.
- 이번 read 실전 검증 기준으로는 현재 배포 구조가 무중단 성질을 대체로 만족한다고 볼 수 있다.

### 6.3 `k6` 결과

- total requests: `5996`
- `http_req_failed`: `5.57%`
- checks failed: `668`
- status `200` 비율: 약 `94%`
- failed requests: `334`
- `response has id` 실패: `334`

- `http_req_duration.p95`: `75.93ms`
- `http_req_duration.p99`: `252.94ms`

- observed RPS: `19.094587/s`

### 6.4 해석

- write baseline과 달리, 배포 중 write 검증에서는 실제 실패가 발생했다.
- `http_req_failed`가 `5.57%`로 임계치를 넘었고, 약 `334건`의 요청이 정상 응답(`200 + id`)을 받지 못했다.
- `p95`, `p99`도 baseline보다 상승했다.
- 이번 결과는 write 경로 기준 실전 검증 실패로 판단하며, 원인 분석이 추가로 필요하다.

## 7. 현재 메모

- 검증 준비와 `k6` 실행 위치는 `local`로 고정한다.
- EC2는 검증 스크립트 실행 위치가 아니라 배포 대상 서버로 본다.
- seed 결과 파일에는 plain `apiKey`가 들어가므로 보관과 노출 범위를 주의한다.
