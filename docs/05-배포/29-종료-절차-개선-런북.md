# 29. 종료 절차 개선 런북

## 문서 목적

배포 실전 검증 이후 확인된 write 경로 종료 취약점을 바탕으로, old color 종료 절차를 어떻게 개선하고 어떤 순서로 재검증할지 정리한다.  
이번 문서는 전체 배포 구조를 바꾸는 것이 아니라, 현재 Blue/Green 종료 흐름 안에서 old color stop 시점을 더 안전하게 다루는 기준을 남기는 데 목적이 있다.

## 현재 전제

- 운영 배포 구조는 `GitHub Actions -> ECR -> S3 -> CodeDeploy -> EC2 -> Blue/Green` 기준이다.
- 운영 서버는 단일 EC2 + Nginx + `app-blue/app-green` 구조다.
- 현재 보호 장치는 이미 적용돼 있다.
  - `server.shutdown=graceful`
  - `spring.lifecycle.timeout-per-shutdown-phase`
  - `stop_grace_period`
  - `OLD_COLOR_DRAIN_SECONDS`
- write 배포 검증에서는 old color 종료 타이밍에 따라 upstream reset이 발생한 적이 있다.
- 이번 단계에서는 종료 책임과 종료 시점을 더 설명 가능하게 정리하는 데 집중한다.
- 새 color 검증은 readiness만으로 충분하지 않고, 핵심 API direct smoke가 별도로 필요하다.
- 현재 구현 기준으로는 다음 준비가 반영돼 있다.
  - `deploy-prod-orchestrator.sh`가 메인 배포 엔진 역할을 한다.
  - `deploy-smoke.sh`가 direct/public smoke를 담당한다.
  - `deploy-drain.sh`가 old color draining 시작, 상태 조회, 종료를 담당한다.
  - 앱 내부에는 `TrafficState`, `ActiveRequestTracker`, `/internal/drain/start`, `/internal/drain/status`가 추가돼 있다.
  - 최근 실제 배포에서는 `direct smoke -> nginx 전환 -> public smoke -> internal drain -> old color stop` 흐름이 `d-6HPC1ST45`에서 정상 완료됐다.

## 이번 런북의 범위

- 새 color direct smoke 추가 여부 검토
- old color 종료 책임 위치 재확인
- 종료 시점 변경 시도
- 종료 시점 변경 후 write 재검증
- 실패 시 로그 확인 기준 정리

## 목표

- 새 color를 Nginx 전환 전에 직접 검증할 수 있게 한다.
- old color 종료 절차를 현재보다 더 명확한 순서로 설명할 수 있게 한다.
- 같은 write 배포 검증 조건에서 실패가 재발하는지 확인한다.
- 실제로 안정적이었던 종료 흐름을 기준선으로 다시 잡는다.

## 사전 체크리스트

- 현재 운영 활성 색상 확인
- `blue-green-prod-lib.sh`, `deploy-prod-orchestrator.sh`의 종료 흐름 확인
- `clickchecker.dev` health / readiness 정상 확인
- 새 color direct smoke에 사용할 API와 성공 기준 정리
- write 검증용 API Key와 `k6` 실행 명령 준비
- Nginx error/access 로그 확인 경로 준비
- 배포 시각 기록 기준 준비

## 적용 순서

### 1. 기준선 확인

- 기존 write 실패 원인이 old color 종료 타이밍과 연결돼 있는지 다시 확인한다.
- `OLD_COLOR_DRAIN_SECONDS` 추가 후 통과했던 종료 흐름을 현재 기준선으로 본다.

### 2. 종료 흐름 변경

- readiness 이후 direct smoke를 어디에 넣을지 정한다.
- 종료 책임을 어디서 가지는지 정한다.
- 필요하면 switch 스크립트와 deploy 스크립트의 역할을 조정한다.
- 변경 전후 흐름을 문장으로 설명할 수 있어야 한다.

권장 흐름:

```text
새 color 기동
-> readiness polling
-> 새 color direct smoke
-> 실패 시 전환 중단, old color 유지
-> 성공 시 Nginx 전환
-> 공개 경로 최종 확인
-> old color draining 유지
-> active request 정리 확인
-> old color 종료
```

현재 반영 흐름:

```text
deploy-prod-orchestrator.sh
-> target color 기동
-> readiness polling
-> deploy-smoke.sh direct 실행
-> 성공 시 nginx 전환
-> 공개 경로 health / public smoke 확인
-> deploy-drain.sh 호출
-> old color /internal/drain/start
-> /internal/drain/status polling
-> old color 종료
```

직접 smoke 원칙:

- readiness와 direct smoke는 같은 의미로 보지 않는다.
- readiness는 “요청 받을 준비”, direct smoke는 “핵심 기능 최소 검증”이다.
- direct smoke는 너무 무겁게 만들지 않고, 핵심 API 1~2개만 확인한다.

### 3. write 재검증

흐름:

```text
k6 write 시작
-> 배포 시작
-> 배포 성공 시각 기록
-> k6 종료
-> Nginx / app 로그 확인
-> 기존 결과와 비교
```

### 4. 결과 기록

- 어떤 종료 흐름을 시도했는지 기록
- `http_req_failed`, `p95`, `p99` 기록
- Nginx upstream reset 재발 여부 기록
- 유지할 흐름과 폐기할 흐름 기록

## 성공 기준

- 새 color direct smoke가 전환 전에 통과하거나, 실패 시 전환 없이 중단된다.
- 같은 write 배포 검증 조건에서 `http_req_failed = 0`에 가깝게 유지된다.
- Nginx error log에서 기존 upstream reset 패턴이 재현되지 않는다.
- 종료 순서를 현재보다 더 명확하게 설명할 수 있다.

## 실패 기준

- direct smoke가 readiness 이후에도 안정적으로 통과하지 못한다.
- write 검증에서 `http_req_failed`가 다시 발생한다.
- Nginx error log에 `connection reset by peer`, `upstream prematurely closed connection`이 재발한다.
- 종료 책임이나 종료 시점이 다시 모호해진다.

## 실패 시 확인 순서

1. `k6` summary에서 실패율 확인
2. 배포 시각 기준으로 Nginx error log 확인
3. `app-blue`, `app-green` 시작/종료 시각 확인
4. direct smoke 실패인지, 전환 후 공개 경로 실패인지, drain 시작 실패인지 먼저 구분
5. `/internal/drain/start`, `/internal/drain/status` 호출 결과와 app 로그 예외를 함께 확인
6. 어떤 upstream(`8081`/`8082`)에서 끊겼는지 확인
7. 유지할 종료 흐름과 폐기할 종료 흐름 정리

## 다음 단계

- 실제로 유지할 종료 흐름이 정해지면 운영 스크립트 기준으로 고정한다.
- 필요하면 실행기록과 트러블슈팅 문서에 결과를 반영한다.
