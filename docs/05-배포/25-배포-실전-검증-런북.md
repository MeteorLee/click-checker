# 25. 배포 실전 검증 런북

## 문서 목적

현재 운영 배포 구조에서 지속 요청이 들어오는 상황을 인위적으로 만든 뒤, 실제 전환 중 오류 없이 배포가 수행되는지 확인하는 기준을 정리한다.  
이번 단계에서는 새 배포 구조를 만드는 것이 아니라, 이미 적용된 `CodeDeploy + ECR + Blue/Green + readiness + graceful shutdown` 구성이 실제 요청 흐름에서도 안정적인지 검증하는 것을 목표로 한다.

## 현재 전제

- 운영 배포 구조는 `GitHub Actions -> ECR -> S3 -> CodeDeploy -> EC2 -> Blue/Green` 기준이다.
- 운영 서버는 단일 EC2 + Nginx + `app-blue/app-green` 구조다.
- readiness, graceful shutdown, Blue/Green 전환, CodeDeploy hook은 이미 적용된 상태다.
- 이번 단계에서는 실사용자 트래픽 대신 `k6`로 지속 요청 상황을 재현한다.
- 이번 검증은 읽기 시나리오와 쓰기 시나리오를 분리해서 수행한다.
- 검증 준비와 `k6` 실행은 `local` 기준으로 수행한다.
- EC2는 검증 스크립트 실행 위치가 아니라 배포 대상 서버로 본다.

## 이번 런북의 범위

- `k6` 지속 요청 시나리오 실행 기준 정리
- 배포 전환 중 관찰할 지표 정의
- 배포 시각 기록 기준 고정
- 읽기 검증 / 쓰기 검증 순서 정리
- 실패 판단 기준과 중단 기준 정리

## 목표

- 지속 요청이 흐르는 상태에서 배포를 수행한다.
- 전환 중 `5xx`, timeout, `http_req_failed`가 발생하지 않는지 확인한다.
- `p95`, `p99`, `RPS`를 기준선과 비교할 수 있게 기록한다.
- 전환 시각과 대상 버전, 실행 명령, 결과를 문서로 남길 수 있게 한다.

## 사전 체크리스트

- 현재 운영 활성 색상 확인
- `clickchecker.dev`, `/actuator/health`, `/actuator/health/readiness` 정상 확인
- CodeDeploy 최근 배포 성공 상태 확인
- seed 실행 위치와 `k6` 실행 위치가 `local`인지 확인
- `k6` 시나리오가 현재 API 계약과 맞는지 확인
- 읽기 검증과 쓰기 검증에 사용할 API Key 확인
- 쓰기 검증 시 테스트 식별 규칙 확인
  - 고정 path prefix
  - run id
  - 테스트용 event type 또는 payload 식별자
- Grafana에서 확인할 패널과 시간 범위 준비
- 결과 저장 위치 준비
  - 실행 명령
  - 대상 SHA / 이미지 태그
  - `k6` summary
  - 전환 시각

## 적용 순서

### 1. 계획 기준 확인

기준 문서:

```text
docs/00-계획/11-배포-실전-검증-계획.md
```

확인 포인트:

- 이번 단계가 구조 변경이 아니라 운영 검증 단계인지
- 읽기 / 쓰기 검증을 분리해서 수행하는지
- 성공 기준이 숫자로 고정돼 있는지

### 2. 읽기 검증 기준 준비

읽기 검증에서는 먼저 조회 요청만 지속적으로 발생시킨다.

기준:

- `k6` 읽기 시나리오 실행
- 대상 API Key 지정
- 조회 시간 범위와 파라미터 고정
- 배포 전 기준선 지표를 먼저 본다

확인 포인트:

- `RPS`
- `p95`
- `p99`
- `http_req_failed`
- status code 분포

### 3. 쓰기 검증 기준 준비

쓰기 검증은 읽기 검증이 끝난 뒤 별도로 수행한다.

기준:

- 테스트 데이터는 식별 가능해야 한다.
- 테스트 요청은 운영 데이터와 구분 가능한 path / payload / run id를 사용한다.
- 가능하면 테스트용 API Key를 사용한다.

주의:

- 읽기와 쓰기를 한 번에 섞어 시작하지 않는다.
- 이번 단계에서는 원인 분리가 더 중요하다.

### 4. 전환 시각 기록 기준 고정

최소 기록 시각:

- `T0`: `k6` 시작 시각
- `T1`: 새 색상 컨테이너 기동 시작
- `T2`: readiness `UP` 확인 시각
- `T3`: Nginx upstream 전환 시각
- `T4`: 공개 경로 재검증 완료 시각
- `T5`: 이전 색상 종료 시각

원칙:

- 지표 해석은 이 시각 기준으로 본다.
- 전환 시각 기록 없이 그래프만 보고 결론 내리지 않는다.

### 5. 읽기 검증 실행

흐름:

```text
k6 읽기 요청 시작
-> 기준선 확인
-> 배포 전환 수행
-> 전환 시각 기록
-> k6 종료
-> Grafana / k6 결과 정리
```

완료 기준:

- `http_req_failed = 0`
- timeout `0건`
- `5xx ratio = 0%`
- 전환 이후 공개 경로와 health / readiness 정상

### 6. 쓰기 검증 실행

흐름:

```text
k6 쓰기 요청 시작
-> 기준선 확인
-> 배포 전환 수행
-> 전환 시각 기록
-> k6 종료
-> 테스트 데이터 반영 여부 확인
-> Grafana / k6 결과 정리
```

추가 확인 포인트:

- 쓰기 실패 응답 유무
- 테스트 데이터가 의도한 형태로 저장 / 조회되는지
- 전환 중 write path 에서 timeout 또는 5xx가 없는지

### 7. 결과 기록

반드시 남길 항목:

- 검증 일시
- 실행 위치(`local`)
- 대상 환경
- 대상 SHA 또는 이미지 태그
- 실행한 `k6` 명령
- 읽기 / 쓰기 구분
- `T0 ~ T5`
- `k6` summary
- Grafana에서 확인한 `RPS`, `p95`, `p99`, `5xx`
- 이상 징후 유무
- 다음 확인 필요 항목

## 성공 기준

- `k6` 실행 중 `http_req_failed = 0`
- 배포 전환 중 timeout `0건`
- 배포 전환 중 `5xx ratio = 0%`
- `p95`, `p99`는 전환 시점에 일시 변동이 있더라도 짧은 시간 내 기준선 근처로 회복
- 전환 후 공개 경로 응답, health, readiness가 모두 정상
- 읽기 검증과 쓰기 검증 결과가 각각 분리되어 기록됨

## 실패 기준

다음 상황이면 이번 검증은 실패 또는 추가 분석 필요 상태로 본다.

- `http_req_failed > 0`
- timeout 발생
- `5xx` 발생
- 전환 후 readiness 또는 공개 경로 검증 실패
- `p95`, `p99`가 전환 후에도 비정상적으로 유지
- 쓰기 검증에서 테스트 데이터 정합성을 설명할 수 없음

## 실패 시 확인 순서

1. `k6` summary에서 실패 유형 확인
2. 배포 시각 기준으로 `app-blue`, `app-green` 로그 확인
3. Nginx access / error 로그 확인
4. readiness 응답 상태 확인
5. Sentry 예외 여부 확인
6. requestId 기반 추적 가능 여부 확인
7. 이전 색상 종료 시점이 너무 빨랐는지 검토

## 전환 후 확인 항목

- 읽기 검증 결과가 남아 있다.
- 쓰기 검증 결과가 남아 있다.
- 전환 시각과 대상 버전이 남아 있다.
- `k6` 결과와 Grafana 지표를 함께 비교할 수 있다.
- 현재 배포 구조가 지속 요청 상태에서도 설명 가능한 형태로 검증됐다.

## 다음 단계

- 실제 검증 중 발생한 문제는 트러블슈팅 문서에 누적한다.
- 읽기 / 쓰기 검증이 모두 끝나면 종합 문서에서 최종 운영 검증 기준을 정리한다.
- 필요하면 이후 혼합 시나리오 또는 더 긴 검증 시간을 별도 단계로 확장한다.
