# 27. 배포 실전 검증 트러블슈팅

## 문서 목적

배포 실전 검증을 준비하고 실행 기준을 정리하는 과정에서 실제로 발생한 문제와 해결 흐름을 기록한다.  
이번 문서는 배포 자체 장애만이 아니라, 관측 경로와 검증 도구를 현재 운영 구조에 맞추는 과정에서 생긴 혼선을 줄이는 데 목적이 있다.

## 1. `k6` 컨테이너 실행 시 `API_KEY`를 못 읽은 문제

### 상황

- 로컬에서 `docker run grafana/k6 run k6/smoke-read.js`를 실행했는데 반복적으로 다음 오류가 발생했다.
  - `API_KEY environment variable is required`

### 원인

- 로컬 쉘에 `API_KEY=...`를 붙였더라도, Docker 컨테이너는 해당 값을 자동 상속하지 않는다.
- `k6` 컨테이너 안으로 환경변수를 넘기지 않은 상태였다.

### 대응

- `docker run`에 `-e API_KEY=...`, `-e BASE_URL=...`, `-e FROM=...`, `-e TO=...`, `-e TOP=...` 형태로 환경변수를 명시적으로 전달했다.

### 정리

- `k6`를 컨테이너로 실행할 때는 필요한 환경변수를 컨테이너 안으로 직접 넘겨야 한다.
- 이후 실전 검증 명령은 `docker run -e ... grafana/k6 ...` 형태로 고정한다.

## 2. Grafana가 열려도 바로 지표를 볼 수 없었던 문제

### 상황

- `https://grafana.clickchecker.dev` 접속은 가능했지만,
- 대시보드가 없어서 바로 `RPS`, `p95`, `p99`, `5xx`를 확인할 수 없었다.

### 원인

- Grafana는 이미 외부 제한 공개로 열려 있었지만,
- 실전 검증용 대시보드나 데이터 소스가 미리 고정돼 있지 않았다.

### 대응

- 먼저 Grafana에서 Prometheus 데이터 소스를 수동으로 연결했다.
- URL은 compose 네트워크 기준으로 `http://prometheus:9090`을 사용했다.
- 대시보드부터 만들지 않고, `Explore`에서 필요한 쿼리 4개를 먼저 확인하는 방식으로 진행했다.

### 정리

- 실전 검증 1회차에서는 대시보드보다 `Explore`가 더 빠르고 단순하다.
- 반복 검증이 늘어나면 이후 대시보드로 고정하는 것이 자연스럽다.

## 3. Prometheus가 예전 `app:8080`을 계속 scrape하고 있던 문제

### 상황

- Grafana Explore에서 `up`을 조회했더니
  - `up{instance="app:8080", job="click-checker-app"} = 0`
  만 보였다.
- `RPS`, `p95`, `p99`, `5xx` 쿼리도 의미 있는 값이 잘 나오지 않았다.

### 원인

- 운영 앱 구조는 이미 `app-blue(8081)` / `app-green(8082)`인데,
- `prometheus/prometheus.yml`은 아직 예전 단일 앱 대상인 `app:8080`을 scrape하고 있었다.

### 대응

- `prometheus/prometheus.yml`의 scrape target을
  - `app-blue:8081`
  - `app-green:8082`
  로 수정했다.

### 정리

- Blue/Green 구조로 전환한 뒤에는 관측 대상도 현재 서비스 이름 기준으로 함께 바꿔야 한다.
- 앱 구조를 바꿨는데 Prometheus scrape 대상이 예전 기준이면 Grafana는 사실상 쓸 수 없다.

## 4. Prometheus 설정은 바뀌었지만 새 target이 바로 안 보인 문제

### 상황

- 운영 서버의 `~/click-checker/prometheus/prometheus.yml`은 이미 `app-blue`, `app-green`으로 바뀌어 있었다.
- 그런데 Grafana Explore의 `up`에는 계속 `app:8080`만 보였다.

### 원인

- Prometheus 컨테이너가 새 설정 파일을 다시 읽지 않은 상태였다.
- 파일 변경만으로는 실행 중인 Prometheus 프로세스에 즉시 반영되지 않았다.

### 대응

- 운영 서버에서 `click-checker-prometheus` 컨테이너를 재시작했다.
- 그 뒤 `up`을 다시 확인해
  - `app-blue:8081 = 1`
  - `app-green:8082 = 0`
  - `app:8080 = 0`
  상태를 확인했다.

### 정리

- scrape target 변경 후에는 Prometheus가 실제로 새 설정을 다시 읽었는지 반드시 확인해야 한다.
- `up` 쿼리는 이 확인에 가장 직접적이다.

## 5. 배포 중 `write` 요청이 old color 종료 시점에 끊긴 문제

### 상황

- `write` 배포 실전 검증에서 `http_req_failed = 5.57%`가 발생했다.
- 실패 구간은 대략 `2026-03-13 02:52:20 ~ 02:52:30 KST`였다.
- 같은 시간대 운영 서버의 Nginx 에러 로그에는 다음 흔적이 반복적으로 남았다.
  - `recv() failed (104: Connection reset by peer) while reading response header from upstream`
  - `upstream prematurely closed connection while reading response header from upstream`
  - 대상 upstream: `http://127.0.0.1:8081/api/events`
  - 요청: `POST /api/events`

### 관찰 및 해석

- Nginx 에러 로그 기준으로 실패 요청은 `app-green`이 아니라 old color였던 `app-blue(8081)`로 전달되고 있었다.
- 로그 정황상 old color 종료 시점과 in-flight `POST /api/events` 요청 처리 시점이 겹치면서, upstream 연결이 응답 헤더 전 종료된 것으로 보인다.
- 따라서 이번 실패는 write API 자체의 비즈니스 오류라기보다, 배포 중 old color 종료 타이밍에 따른 요청 처리 중단 가능성으로 해석하는 것이 타당하다.
- 다만 정확한 내부 처리 상태와 종료 시점은 앱 로그를 함께 대조해 추가 확인할 필요가 있다.

### 대응

- 이번 `write` 실전 검증 결과는 실패로 기록했다.
- 실패 원인 분석에서는 앱 비즈니스 로직보다 프록시/upstream 계층의 종료 흔적을 우선 근거로 삼았다.
- 다음 점검 대상으로 `blue-green-prod-switch.sh`, `deploy-prod-blue-green.sh`의 old color 종료 절차를 잡았다.
- 1차 조치로 `blue-green-prod-switch.sh`에 `OLD_COLOR_DRAIN_SECONDS`를 추가하고, public verify 뒤 old color stop 전에 drain 대기 시간을 두도록 수정했다.
- 현재 조치는 write 실패 재현 여부를 다시 확인하기 위한 1차 수정이며, 필요하면 이후 종료 유예 정책을 추가로 조정한다.
- 같은 조건으로 `write` 배포 실전 검증을 다시 수행했고, 재검증에서는 `http_req_failed = 0.00%`로 실패가 재발하지 않았다.

### 정리

- `read` 검증은 통과했지만, `write` 검증은 old color 종료 타이밍에 따라 실패할 수 있음을 확인했다.
- 특히 write 요청은 처리 중 연결 종료의 영향을 직접 받기 쉬워, 배포 전환 안정성 검증에서 별도로 확인할 필요가 있다.
- 1차 조치와 재검증 결과를 함께 보면, 이번 문제는 old color 즉시 종료보다는 전환 후 drain 시간이 부족했던 쪽에 더 가깝게 보인다.
- 이후에도 같은 유형의 upstream reset이 재발하는지 추가 관찰하면서, 필요하면 종료 유예 정책을 더 보강한다.

## 현재까지 정리된 기준

- 실전 검증 준비와 `k6` 실행은 `local`로 고정한다.
- Grafana는 `https://grafana.clickchecker.dev`에서 확인한다.
- Prometheus는 외부 공개하지 않고 Grafana 뒤에서만 사용한다.
- 현재 운영 관측 대상은 `app-blue:8081`, `app-green:8082`다.
- 실전 검증 최소 지표는 `RPS`, `p95`, `p99`, `5xx`다.
