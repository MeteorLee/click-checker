# 32. 종료 절차 개선 종합

## 문서 목적

종료 절차 개선 작업의 배경, 시행착오, 구조 변경, 재검증 결과를 한 문서로 정리한다.

## 1. 개선 배경

배포 실전 검증 단계에서 `read` 경로는 통과했지만 `write` 경로는 배포 중 `http_req_failed`가 발생했다.  
특히 `POST /api/events` 요청에서 Nginx `upstream prematurely closed connection`, `connection reset by peer`가 반복되어, old color 종료 시점과 in-flight write 요청 보호를 더 구조적으로 정리할 필요가 확인됐다.

## 2. 초기 개선과 한계

처음에는 `old color 종료를 더 늦추는 방식`으로 접근했다.  
즉 전환 이후 old color를 더 오래 살려두고 smoke 이후 종료하는 흐름을 시도했지만, 이 방식은 `draining 상태를 제어`한 것이 아니라 `종료 시점을 단순히 지연`한 것에 가까웠다.

실제 재검증에서는 여전히 `127.0.0.1:8081` upstream 기준 `POST /api/events` 요청에서 reset/close 문제가 재현됐다.  
이 단계에서 정리한 판단은 다음과 같다.

- old color를 단순히 오래 살려두는 것만으로는 충분하지 않았다.
- 전환 전 새 color 검증과 전환 후 old color draining을 분리해서 설계해야 했다.

## 3. 구조 재정리

이후 종료 절차는 다음 방향으로 재정리했다.

- 전환 전:
  - target color 기동
  - readiness 확인
  - target color direct smoke 수행
- 전환 후:
  - Nginx upstream 전환
  - public health / public smoke 확인
  - old color를 `draining` 상태로 전환
  - active request 정리 확인 후 old color stop

이를 위해 배포 구조도 함께 정리했다.

- [deploy-prod-orchestrator.sh](../../scripts/deploy/deploy-prod-orchestrator.sh)
  - 메인 배포 엔진
- [deploy-smoke.sh](../../scripts/deploy/deploy-smoke.sh)
  - direct/public smoke 전담
- [deploy-drain.sh](../../scripts/deploy/deploy-drain.sh)
  - drain start/status polling/stop 전담
- [blue-green-prod-lib.sh](../../scripts/deploy/blue-green-prod-lib.sh)
  - prod 배포 공통 helper

앱 내부에도 draining 상태를 표현하는 최소 구조를 추가했다.

- `ActiveRequestTracker`
- `ActiveRequestTrackingFilter`
- `TrafficState`, `TrafficStateManager`
- `/internal/drain/start`
- `/internal/drain/status`

또한 `/internal/*`는 외부 공개 경로에서 차단하고, 배포 스크립트가 직접 포트(`127.0.0.1:8081/8082`)로만 접근하도록 정리했다.

## 4. 시행착오와 정리된 원인

### 4.1 새 smoke/drain 스크립트 실행 권한 누락

새로 분리한 `deploy-smoke.sh`, `deploy-drain.sh`에 실행 권한이 빠져 첫 배포에서 `Permission denied`가 발생했다.  
이는 `after-install.sh`에서 `require_file`과 `chmod +x` 대상을 보강해 해결했다.

### 4.2 `/internal/drain/start` content negotiation 문제

다음 배포에서는 `POST /internal/drain/start` 호출이 500으로 실패했다.  
로그를 대조해 보니 old color가 기능을 모르는 상태가 아니라, `HttpMediaTypeNotAcceptableException`으로 응답 직렬화 단계에서 문제가 났다.

이에 `deploy-drain.sh`의 내부 호출에 `Accept: application/json`을 추가했고, 이후 `drain/start`와 `drain/status`가 정상 동작했다.

### 4.3 active color 조기 recreate 문제

가장 중요한 원인은 이후 write 재검증에서 드러났다.

문제 배포:
- `d-HCPCES0FH`

관찰 결과:
- active color=`blue`, target color=`green`
- 배포 초반 `starting infra services`
- 직후 `Container click-checker-app-blue-1 Recreate`
- 아직 Nginx는 blue를 보고 있는 시점에서 `POST /api/events` 요청이 `127.0.0.1:8081` upstream 기준 reset/close

즉 실패는 `drain` 단계보다 훨씬 앞에서 발생했다.  
원인은 배포 중 infra 기동 단계와 `prometheus -> app-blue` 의존 관계가 active app을 암묵적으로 재생성한 데 있었다.

정리된 판단:
- 문제의 핵심은 `old color stop`이 아니라, 배포 시작 직후 serving 중이던 active app이 건드려진 점이었다.
- 배포 중에는 공통 infra를 다시 올리지 않는 것이 더 안전하다.

## 5. 최종 적용 구조

최종적으로는 다음 두 가지를 반영했다.

1. [docker-compose.prod.yml](../../docker-compose.prod.yml)
- `prometheus`의 `app-blue` 의존 override 제거

2. [deploy-prod-orchestrator.sh](../../scripts/deploy/deploy-prod-orchestrator.sh)
- 배포 중 infra 기동 단계 제거
- active color는 전환 전까지 untouched
- target color만 기동 후 direct smoke -> switch -> public smoke -> internal drain -> stop

현재 기준 배포 흐름은 아래와 같다.

1. active / target color 계산
2. target color 기동
3. readiness polling
4. target color direct smoke
5. Nginx 전환
6. public health / public smoke
7. old color `drain/start`
8. old color `drain/status` polling
9. old color stop

## 6. 재검증 결과

최종 재검증 배포:
- `d-0UT8MMU45`

시각:
- `T0/T1 = 10:06` `k6 write 시작 / 배포 시작`
- `T2 = 10:09:19` `배포 성공`
- `T3 = 10:11 전후` `k6 종료`

`k6` 결과:
- `http_req_failed = 0.00%`
- `http_reqs = 6001`
- `p95 = 44.26ms`
- `p99 = 112.69ms`

배포 로그 기준 확인:
- `active color=green`
- `target color=blue`
- `starting target app-blue`
- direct smoke 통과
- `switching nginx to blue`
- public smoke 통과
- `start draining green`
- `state=DRAINING activeRequests=0`
- `stopping app-green`
- `completed successfully`

중요한 점:
- 이전 실패 배포와 달리 `starting infra services`가 더 이상 나오지 않았다.
- active app 조기 recreate 문제도 재현되지 않았다.
- target app recreate는 정상적인 동작 범위로 확인됐다.

## 7. 현재 운영 기준

현재 종료 절차 개선 작업 기준 운영 판단은 다음과 같다.

- 배포 중 공통 infra는 건드리지 않는다.
- 전환 전 direct smoke를 통과한 target color만 공개 트래픽을 받는다.
- 전환 후 old color는 `draining` 상태로 전환하고, active request 확인 후 종료한다.
- 내부 drain endpoint는 외부 공개 경로에서 차단하고, 배포 스크립트가 직접 포트로만 호출한다.

## 8. 결론

이번 종료 절차 개선 작업의 결론은 다음과 같다.

- 단순한 종료 지연만으로는 write 경로를 보호하기 어려웠다.
- 전환 전 direct smoke, 전환 후 internal drain 구조로 배포 흐름을 재정리해야 했다.
- 이후에도 첫 시도에서는 권한 문제, content negotiation 문제, active app 조기 recreate 문제가 순서대로 드러났다.
- 특히 마지막 문제의 원인은 배포 중 infra 기동과 `prometheus` 의존 관계가 active app을 건드린 데 있었다.
- 해당 원인을 제거한 뒤, write 재검증은 `http_req_failed = 0.00%`로 통과했다.

즉 현재 기준 종료 절차 개선은 다음 결론으로 정리할 수 있다.

`target color만 준비하고, direct smoke 후 전환하며, old color는 internal drain으로 정리하는 구조가 현재 단일 EC2 Blue/Green 배포 기준에 가장 잘 맞는다.`
