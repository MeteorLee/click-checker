# 36. 운영 관측 신뢰도 정비 런북

## 문서 목적

이미 도입된 운영 관측 체계를 기준으로, 실제 운영에서 `믿고 볼 수 있는 상태`로 다듬기 위한 확인 절차를 정리한다.  
이번 단계에서는 새 관측 도구를 추가하는 것이 아니라, 기존 `Grafana / Loki / Prometheus / Alertmanager / Sentry` 구성이 배포 이후와 smoke 실행 이후에도 일관되게 해석되는지 검증하는 것을 목표로 한다.

## 현재 전제

- 운영 배포 구조는 `GitHub Actions -> ECR -> S3 -> CodeDeploy -> EC2 -> Blue/Green` 기준이다.
- 운영 서버는 단일 EC2 + Nginx + `app-blue/app-green` 구조다.
- 관측 스택은 이미 운영에 배치되어 있다.
  - Prometheus
  - Loki
  - Promtail
  - Alertmanager
  - Grafana
  - Grafana renderer
  - Sentry
- 이번 단계에서는 관측 스택의 도입 자체가 아니라, 다음 항목을 실제 운영 기준으로 재점검한다.
  - 대시보드 역할 분리
  - alert 기준 정리
  - Sentry 요청 문맥 보강
  - smoke / 짧은 `k6` 검증과 대시보드 반응 연결
- 이번 검증은 `local`과 `prod`를 모두 사용하되, 역할을 분리해서 본다.
  - `local`
    - smoke / 짧은 `k6` 실행과 대시보드 반응 확인
  - `prod`
    - 실제 배포 후 health / smoke / observability 반영 확인

## 이번 런북의 범위

- Grafana 운영 대시보드 재구성 이후 확인 순서 정리
- observability 재기동 / 검증 스크립트 실행 기준 정리
- Grafana renderer와 scripted PNG 캡처 반영 기준 정리
- smoke / 짧은 `k6` 실행 기준 정리
- Sentry 요청 문맥과 release 확인 기준 정리
- prod 배포 후 최소 운영 확인 흐름 정리
- EC2 관측 스택에서 renderer 포함 여부 확인

## 목표

- `Overview`, `Logs`, `Deploy / Smoke` 대시보드가 역할대로 동작하는지 확인한다.
- smoke와 짧은 `k6` 요청이 대시보드와 로그에 실제로 반영되는지 확인한다.
- alert rule이 제품 API 기준으로 해석 가능한 상태인지 확인한다.
- Sentry에서 `requestId`, `appColor`, `release`, 인증 문맥을 따라갈 수 있게 한다.
- prod 배포 이후 `health -> smoke -> Grafana -> Loki -> Sentry` 흐름을 실제로 확인한다.

## 사전 체크리스트

- 현재 운영 배포가 성공 상태인지 확인한다.
- 운영 서버 `.env`와 `.env.codedeploy`가 최신 상태인지 확인한다.
- `JWT_SECRET`, `DB_URL`, `DB_USERNAME`, `DB_PASSWORD`, `API_KEY_PEPPER`가 운영 `.env`에 있는지 확인한다.
- local 검증 시 `docker-compose.yml + docker-compose.local.yml`를 함께 사용하고 있는지 확인한다.
- local Docker image가 최신 `bootJar` 기준으로 다시 빌드됐는지 확인한다.
- Grafana에 신규 대시보드 3개가 provisioning 되는지 확인한다.
- 이후 smoke / `k6` 검증에 사용할 API Key 확보 경로를 확인한다.

## 적용 순서

### 1. 현재 기준 다시 읽기

기준 문서:

```text
docs/05-배포/33-운영-관측-체계-런북.md
docs/05-배포/35-운영-관측-체계-종합.md
```

확인 포인트:

- 이번 단계가 observability 도구 추가가 아니라 해석 기준 정비 단계인지
- 기존 메트릭 / 로그 / 예외 / 알림 흐름을 그대로 유지하면서 신뢰도와 가독성을 올리는 단계인지

### 2. local 관측 기준 정리

local에서는 다음 흐름을 먼저 확인한다.

```text
bootJar
-> Docker app 재빌드
-> deploy smoke
-> Grafana / Loki 확인
-> 짧은 k6 write/read 실행
-> Overview 반응 확인
```

확인 포인트:

- `docker-compose.local.yml` 오버레이를 같이 사용하고 있는지
- `deploy-smoke.sh`가 현재 공식 흐름(`signup -> admin organization create -> apiKey -> event -> aggregate`)을 따라가는지
- smoke 후 `Logs`, `Deploy / Smoke` 대시보드에 흔적이 남는지
- `k6` 짧은 burst 후 `Overview` 대시보드가 실제로 반응하는지

### 3. Grafana 대시보드 확인

현재 기준 대시보드는 다음 세 가지다.

- `Click Checker Overview`
- `Click Checker Logs`
- `Click Checker Deploy / Smoke`

확인 기준:

- `Overview`
  - 추세와 부하 흐름 확인용
  - `RPS`, `p95/p99`, `4xx/5xx`, `DB pool`, `heap`
- `Logs`
  - app 로그와 nginx 로그를 분리해서 확인
- `Deploy / Smoke`
  - 최근 smoke 성공 / API 5xx / smoke 로그 흔적 확인

주의:

- `Overview`는 단건 smoke 확인보다 추세 확인에 가깝다.
- 단건 smoke 여부는 `Deploy / Smoke`와 `Logs`에서 먼저 본다.

### 4. observability 재기동 / 검증

운영 서버 기준으로는 아래 스크립트를 사용한다.

- `scripts/observability/deploy-observability-prod.sh`
  - observability 스택 최초 기동
- `scripts/observability/restart-observability-prod.sh`
  - observability 서비스 재시작
- `scripts/observability/validate-observability-prod.sh`
  - Prometheus / Loki / Alertmanager / Grafana readiness 확인

원칙:

- observability 설정 변경 후에는 Grafana 또는 전체 observability 재기동 여부를 먼저 판단한다.
- prod에서는 `--env-file .env --env-file .env.codedeploy` 기준으로 compose가 실행된다는 점을 전제로 본다.
- 현재 기준 observability compose에는 `renderer`가 포함되며, `deploy/restart/validate` 스크립트도 renderer를 같이 다룬다.
- `validate-observability-prod.sh`는 renderer 컨테이너 내부의 `http://localhost:8081/metrics`를 확인한다.

### 4.1 Grafana renderer 반영 기준

이번 단계 이후 Grafana는 외부 renderer 서비스를 사용한다.

- `docker-compose.yml`
  - `renderer` 서비스 추가
  - `GF_RENDERING_SERVER_URL`
  - `GF_RENDERING_CALLBACK_URL`
  - `GF_RENDERING_RENDERER_TOKEN`
- `prometheus.prod.yml`
  - `click-checker-renderer` scrape job 추가
- `scripts/observability/*`
  - `renderer` 포함 기동 / 재시작 / 검증

운영 확인 포인트:

- `renderer` 컨테이너가 기동한다.
- Grafana 로그에 remote renderer 버전이 출력된다.
- `validate-observability-prod.sh`가 renderer까지 통과한다.

### 4.2 scripted PNG 캡처 기준

성능 단계의 `W1` run은 기본적으로 `capture.mode=scripted`를 사용한다.

- `scripts/perf/w1/run-local.sh` 또는 후속 perf run 스크립트 종료 후
- `scripts/perf/common/capture-grafana-render.sh`가 renderer API를 호출해
- 산출물 디렉터리에 PNG를 저장한다.

예상 산출물:

- `dashboard-overview.png`
- `ingest-rps.png`
- `ingest-latency.png`
- `http-error-ratio.png`
- `db-pool.png`
- `db-connections.png`
- `jvm-heap-by-app.png`

원칙:

- scripted capture가 있으면 수동 Grafana 캡처는 필수가 아니다.
- 캡처 실패 시 `meta.json.capture.status`로 상태를 남긴다.

### 5. Sentry 확인 기준

이번 단계에서 Sentry는 다음 문맥을 우선 본다.

- `requestId`
- `appColor`
- `release`
- `authType`
- `accountId`
- `organizationId`
- `apiKeyKid`

또한 다음 예외는 Sentry 잡음으로 보지 않는다.

- `NoResourceFoundException`
- `HttpRequestMethodNotSupportedException`
- `HttpMediaTypeNotAcceptableException`
- `HttpMediaTypeNotSupportedException`

원칙:

- 봇 / 스캐너 / 잘못된 클라이언트가 만드는 프레임워크성 4xx는 Sentry에서 과감히 제외한다.
- Sentry는 앱 버그와 실제 5xx성 문제를 보는 도구로 유지한다.
- 실제 Sentry 문맥 검증이 필요하면 `internal sentry smoke`를 1회 호출해 확인한다.

확인 절차:

```text
active color 확인
-> 127.0.0.1:8081 또는 8082 로 /internal/sentry/smoke 직접 호출
-> 500 응답 확인
-> Sentry 최신 이벤트에서 release / appColor / requestId / authType / requestPath 확인
```

확인 포인트:

- `requestPath=/internal/sentry/smoke`
- `release=<이번 배포 sha>`
- `appColor=<blue|green>`
- `requestId`
- `authType=anonymous`
- `httpMethod=POST`

### 6. prod 배포 후 최소 확인

prod 배포 후에는 최소한 다음 순서로 확인한다.

```text
CodeDeploy success
-> public health
-> deploy smoke
-> Grafana 3개 대시보드
-> Loki app/nginx 로그
-> Sentry release / appColor / requestId 문맥
```

추가 확인 포인트:

- target color readiness 정상
- direct smoke / public smoke 통과
- `Deploy / Smoke` 패널에서 최근 성공 흔적 확인
- Sentry에서 이번 배포의 `release`가 실제로 붙는지 확인
- 필요 시 internal sentry smoke 1회로 실제 예외 이벤트 문맥까지 확인

## 성공 기준

- local smoke와 짧은 `k6` 검증이 모두 현재 API 계약 기준으로 정상 동작한다.
- `Overview`, `Logs`, `Deploy / Smoke` 대시보드가 역할대로 구분되어 동작한다.
- observability 재기동과 readiness 검증 스크립트가 운영 기준으로 정상 동작한다.
- alert 기준이 제품 API 중심으로 해석된다.
- Sentry에서 `requestId`, `appColor`, `release`, 인증 문맥을 확인할 수 있다.
- prod 배포 후 `health -> smoke -> Grafana -> Loki -> Sentry` 흐름을 실제로 확인할 수 있다.

## 실패 기준

- smoke는 성공했지만 대시보드 / Loki / Sentry 중 하나에서 흔적을 해석할 수 없다.
- Grafana 패널이 parse error 또는 datasource mismatch 상태로 남아 있다.
- observability 재기동 후 readiness 검증이 실패한다.
- Sentry에 여전히 프레임워크성 4xx 잡음이 반복된다.
- prod 배포 후 `release` 또는 `appColor` 문맥이 실제 예외 이벤트에 반영되지 않는다.

## 실패 시 확인 순서

1. observability 스택 서비스 상태 확인
2. Grafana provisioning / datasource 상태 확인
3. Loki 로그 수집 상태 확인
4. smoke 스크립트 결과 확인
5. `k6` 실행 파라미터와 시간 범위 확인
6. Sentry 이벤트 문맥 확인
7. prod `.env`와 `.env.codedeploy` 전달 상태 확인
