# 34. 운영 관측 체계 트러블슈팅

## 문서 목적

운영 관측 체계를 붙이는 과정에서 실제로 발생한 혼선과, 현재 단계에서 무엇을 기준선으로 보고 무엇을 후속 보정 대상으로 남길지 정리한다.  
이번 문서는 Loki / Promtail / Alertmanager / Grafana provisioning을 처음 연결하는 과정에서 드러난 로그 수집 문제를 기록하는 데 목적이 있다.

## 1. Promtail -> Loki에서 `entry too far behind`가 반복된 문제

### 상황

- Loki, Promtail을 붙인 뒤 `promtail` 로그에 아래 오류가 반복적으로 나타났다.

```text
server returned HTTP status 400 Bad Request (400): entry with timestamp ... ignored, reason: 'entry too far behind'
```

- Loki 로그에도 비슷하게 아래 오류가 확인됐다.

```text
write operation failed ... entry too far behind
```

- 처음에는 "Promtail 로그 수집 자체가 실패한 것인가?"를 먼저 의심했다.

### 원인

- 실제 기동 상태와 로그를 함께 확인해보니 다음 사실이 같이 보였다.
  - `docker compose config`는 정상 통과했다.
  - Promtail은 Docker target과 nginx file target을 정상 등록하고 있었다.
  - app 컨테이너에서는 새 요청마다 `requestId`, `color:default` 로그가 계속 생성됐다.
  - Loki에는 `app="app"`, `job="click-checker"`, `source="docker"` 라벨이 붙은 app 로그 stream flush 기록이 남았다.
- 즉 현재 시점 로그가 전혀 안 들어가는 상태는 아니었고, 문제는 "로그 수집 전체 실패"보다 "일부 배치가 거절되는 현상"으로 보는 쪽이 더 타당했다.
- 이후 로그를 더 대조한 결과, 현재 단계에서의 판단은 다음과 같이 정리됐다.
  - Promtail이 기존 Docker 로그 일부를 다시 읽으면서 과거 timestamp 로그를 Loki로 보냈다.
  - Loki는 현재 시점보다 너무 오래된 로그를 ingest 대상으로 받아들이지 않아 해당 배치를 거절했다.
  - 특히 기존 컨테이너 이력이나 이전 실행 시점의 Docker 로그가 남아 있는 상태에서 초기 기동 시 이런 현상이 발생할 수 있다.

즉 핵심 원인은 `현재 애플리케이션 로그 생성 실패`가 아니라, `초기 기동 시 과거 로그 재적재 시도`에 더 가까웠다.

### 대응

- `promtail/promtail-config.yml`에서 Click Checker 앱 컨테이너만 남기도록 `keep` relabel 필터를 추가했다.
  - 다른 Docker 컨테이너 로그가 섞이는 범위를 줄인다.
  - observability 검증 대상을 `app`, `app-blue`, `app-green`으로 제한한다.
- `positions.filename`을 `/tmp/positions.yaml`에서 `/var/lib/promtail/positions.yaml`로 변경했다.
- `docker-compose.yml`에서 Promtail 전용 볼륨을 연결했다.
  - Promtail 재기동 후 읽은 위치 정보가 유지되게 한다.
  - 매번 처음부터 과거 로그를 다시 훑는 상황을 줄인다.

### 정리

- 조치 후 Promtail은 재기동 후 정상 startup 로그를 남겼다.
- app 로그는 계속 생성됐고, Loki에는 실제 app 로그 stream flush 기록이 남았다.
- 다만 `entry too far behind` 로그가 완전히 사라지지는 않았다.

현재 시점 해석은 다음과 같다.

- 최소 작동선은 확보됐다.
- 현재부터의 app 로그 수집이 완전히 막힌 상태는 아니다.
- 남아 있는 경고는 과거 로그 찌꺼기 또는 초기 재수집 구간과 관련된 품질 이슈에 더 가깝다.

## 2. Grafana에서 datasource UID mismatch가 발생한 문제

### 상황

- Grafana 대시보드를 열었을 때 패널 전체에 빨간 느낌표가 보였다.
- 특히 로그 패널에서는 `Datasource click-checker-loki was not found`가 나타났다.
- API로 확인해보니 provisioning으로 넣은 데이터소스 외에, 예전에 UI에서 만든 `prometheus` 데이터소스가 따로 남아 있었다.

### 원인

- 대시보드 JSON은 `click-checker-prometheus`, `click-checker-loki` uid를 직접 참조하고 있었다.
- 하지만 Grafana 내부 저장소에는 이전 수동 데이터소스 상태가 남아 있었고, 이 상태가 provisioning으로 생성된 데이터소스와 섞여 있었다.
- 즉 현재 문제는 대시보드 JSON 자체가 아니라, 로컬 Grafana 상태에 남아 있던 기존 데이터소스 흔적과 provisioning 기준이 충돌한 데 있었다.

### 대응

- 로컬 Grafana 컨테이너를 중지하고 제거했다.
- Grafana 데이터 볼륨을 삭제한 뒤 다시 기동했다.
- 이후 Grafana API를 다시 확인해보니, 다음 두 데이터소스만 남았다.
  - `click-checker-prometheus`
  - `click-checker-loki`
- `Click Checker Overview` 대시보드도 같은 uid 기준으로 정상 조회됐다.

### 정리

- Grafana provisioning 자체는 정상 동작하고 있었다.
- 문제는 provisioning 실패가 아니라, 로컬 Grafana 내부 저장 상태가 남아 있던 쪽에 더 가까웠다.
- 현재 기준으로는 로컬에서 provisioning 기준을 다시 맞출 때, 기존 수동 상태가 꼬여 있으면 볼륨 초기화가 더 빠르고 명확한 해결책이 될 수 있다.

## 3. prod validate 스크립트가 호스트 포트 비공개 정책과 충돌한 문제

### 상황

- prod에서 observability 서비스를 기동한 뒤 `validate-observability-prod.sh`를 실행했을 때, 처음에는 Prometheus `127.0.0.1:9090` 접근에서 실패했다.
- 하지만 같은 시점 `docker compose ps`를 보면 `prometheus`, `loki`, `alertmanager` 컨테이너는 실제로 떠 있었다.
- 즉 "서비스가 안 떴다"보다 "검증 방식이 현재 노출 정책과 맞지 않는다"는 쪽을 먼저 의심해야 하는 상황이었다.

### 원인

- prod compose 기준으로 `prometheus`, `loki`, `alertmanager`는 호스트 포트를 외부나 loopback에 publish하지 않았다.
- 반면 초기 `validate-observability-prod.sh`는 `127.0.0.1:9090`, `127.0.0.1:3100`, `127.0.0.1:9093`를 직접 호출하는 방식이었다.
- 즉 현재 운영 정책은 "포트를 닫아둔다"였는데, validate 스크립트는 "호스트에서 직접 접근 가능하다"를 전제로 작성되어 있었다.

### 대응

- prod 포트를 새로 열지 않기로 했다.
- 대신 `validate-observability-prod.sh`를 수정해, 호스트 loopback이 아니라 `docker compose exec`로 각 컨테이너 내부에서 health/ready를 확인하도록 바꿨다.
- 이후 prod에서 다시 검증했을 때 다음 흐름이 통과했다.
  - `prometheus`
  - `loki`
  - `alertmanager`
  - `grafana`

### 정리

- 이번 문제는 서비스 기동 실패가 아니라, validate 스크립트가 현재 운영 노출 정책과 어긋난 데 있었다.
- 현재 기준으로는 observability 서비스 포트를 굳이 host loopback에 열지 않고, 내부 exec 기반 검증을 유지하는 쪽이 더 자연스럽다.

## 4. Grafana 패널 해석이 바로 되지 않았던 문제

### 상황

- prod Grafana에서 대시보드를 확인했을 때 `JVM Heap Used` 패널은 blue / green 각 1줄이 아니라 여러 줄로 보였다.
- `DB Pool Usage` 패널은 거의 움직이지 않아 메트릭이 잘못된 것처럼 보였다.
- 처음에는 대시보드 쿼리나 Prometheus 메트릭 자체가 잘못 연결된 것인가를 먼저 의심했다.

### 원인

- `JVM Heap Used` 패널은 `jvm_memory_used_bytes{area="heap"}`를 그대로 그리고 있어서, app별 합계가 아니라 heap 세부 영역(Eden / Survivor / Old)이 각각 따로 보이고 있었다.
- 기존 범례는 `{{app}} heap used`처럼 동일한 이름만 반복돼, 실제로는 여러 시계열인데도 구분이 잘 되지 않았다.
- `DB Pool Usage`는 비율 패널이었고, 실제 active connection 값이 거의 `0`이라 변화가 작았다.
- 즉 두 문제 모두 "메트릭이 없다"기보다, "패널 표현이 처음 보는 사람에게 바로 읽히지 않는다"는 쪽에 더 가까웠다.

### 대응

- `JVM Heap Used` 기존 패널은 유지하되, 범례에 heap 영역 이름이 드러나도록 조정했다.
- 별도 `JVM Heap Used by App` 패널을 추가해 app별 heap 합계도 함께 보이게 했다.
- `DB Pool Usage` 기존 비율 패널은 유지하되, 별도 `DB Active Connections` 패널을 추가해 raw active 값을 함께 보이게 했다.

### 정리

- 기존 패널을 지우는 대신, 운영 판단용 요약 패널과 설명용 상세 패널을 같이 두는 쪽이 더 적절했다.
- 이 방식은 포트폴리오 관점에서도 "지표를 단순히 붙였다"보다 "어떻게 해석해야 하는지까지 정리했다"는 설명이 가능하다.

## 5. Sentry에 스캐너 404 잡음이 계속 쌓이던 문제

### 상황

- Sentry에서 `NoResourceFoundException`이 14일 기준 수백 건 이상 쌓이고 있었다.
- 대표 경로는 아래와 같았다.
  - `/wp-admin/setup-config.php`
  - `/wordpress/wp-admin/setup-config.php`
  - `/.git/config`
  - `local.php.old`
  - `parameters.php.save`
  - `*.php`, `*.bak`, `*.old`, `*.dist`
- 처음에는 이 예외가 실제 앱 문제인지, 아니면 인터넷에 노출된 서버가 받는 일반적인 잡음인지 먼저 구분할 필요가 있었다.

### 원인

- 경로 패턴을 보면 대부분 워드프레스, Git 노출, PHP 설정 파일, Mautic 설정 파일을 탐색하는 자동 스캐너 요청에 가까웠다.
- 즉 이 예외는 앱 비즈니스 로직 오류보다, 외부 봇이 존재하지 않는 경로를 반복 탐색하면서 Spring이 `NoResourceFoundException`으로 표현한 경우가 대부분이었다.
- 현재 프로젝트는 실사용자가 없으므로, 내가 직접 만든 요청이 아닌 이런 경로는 사실상 운영 잡음으로 보는 것이 타당했다.

### 대응

- `GlobalExceptionHandler`에서 `NoResourceFoundException`은 별도 분기하도록 정리했다.
- 그중 아래와 같은 명백한 스캐너 패턴 경로는 Sentry 전송 대상에서 제외했다.
  - `/wp-admin`
  - `/wordpress/`
  - `/.git/`
  - `.php`
  - `.bak`, `.old`, `.save`, `.dist`, `.sample`, `~`
- 정상적인 다른 예외는 계속 Sentry로 전송되도록 유지했다.

### 정리

- 이번 문제의 핵심은 "앱 예외를 줄인다"보다 "관측 신호 대 잡음비를 높인다"는 데 있었다.
- 현재 기준으로는 명백한 스캐너 404 경로만 Sentry에서 제외하고, 실제 앱 예외는 그대로 유지하는 쪽이 더 적절하다.

## 현재까지 정리된 기준

- 이번 문제는 `critical blocker`로 보지 않는다.
- observability 스택 전체가 실패한 것은 아니다.
- "현재부터의 로그 수집이 된다"는 최소 작동선은 확보됐다.
- 다만 운영형 품질 관점에서는 후속 보정이 필요하다.

현재 시점에서 열어둔 후속 후보는 아래와 같다.

- 오래된 Docker 로그 재적재를 더 줄일 수 있는 Promtail 설정 검토
- Loki ingest 허용 범위 조정 필요성 검토
- 초기 기동 직후 발생하는 과거 로그 경고를 어디까지 허용할지 운영 기준 정리
- 장기적으로는 Promtail deprecated/EOL 상태를 고려해 Grafana Alloy 전환 검토
- Grafana provisioning 사용 시 기존 수동 데이터소스 상태 정리 기준
- observability validate 스크립트의 내부 exec 기준 유지 여부
- 패널 해석용 요약/상세 구성 기준
- Sentry에서 명백한 스캐너 404 경로를 어디까지 제외할지 기준

즉 이번 단계 기준으로는 `트러블슈팅 기록 + 후속 보정 후보 등록`까지를 우선 처리하고, 이후 Grafana / Loki 조회 및 Alertmanager 검증을 이어가는 흐름이 적절하다.
