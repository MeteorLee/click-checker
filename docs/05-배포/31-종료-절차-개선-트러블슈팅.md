# 31. 종료 절차 개선 트러블슈팅

## 문서 목적

종료 절차 개선을 시도하는 과정에서 실제로 발생한 혼선과, 어떤 흐름을 버리고 어떤 흐름을 기준선으로 유지할지 정리한다.  
이번 문서는 종료 책임과 종료 시점 조정에서 생긴 시행착오를 기록하는 데 목적이 있다.

## 1. smoke 후 old color 종료 흐름이 더 보수적으로 보였지만 실제로는 실패가 재현된 문제

### 상황

- write 요청 보호를 위해 old color를 smoke 성공 후에만 종료하는 흐름(안 2)을 시도했다.
- 기대한 흐름은 다음과 같았다.
  - switch 스크립트: 전환, public verify, drain
  - deploy 스크립트: smoke 성공 후 old color 종료
- 같은 write 검증 조건으로 다시 실행했지만 `http_req_failed = 5.50%`가 발생했다.

### 원인 후보

- 안 2는 겉으로 보기에는 “smoke 전까지 old color를 살려두는 더 보수적인 흐름”처럼 보였다.
- 하지만 실제 구현은 old color를 `serving -> draining -> stopped` 상태로 명확히 제어한 것이 아니라, old color 종료 시점만 뒤로 미룬 수준에 가까웠다.
- 즉 old color가 “새 요청은 안 받고 기존 요청만 정리하는 상태”로 충분히 관리됐다고 보기 어려웠고, smoke 전까지 살아 있는 것만으로는 write 안정성이 보장되지 않았다.
- 기대한 구조와 실제 구현의 차이를 나누면 다음과 같다.
  - 기대한 구조: 새 color 검증이 끝난 뒤 old color는 `draining` 상태로만 남고, 신규 요청은 차단된 채 기존 요청만 마무리한 뒤 종료
  - 실제 안 2 구현: old color 종료 시점을 smoke 뒤로 미뤘을 뿐, old color를 `draining` 상태로 분명하게 제어하지는 못함
- 즉 안 2는 “draining 기반 종료”라기보다 “종료 지연”에 더 가까웠다.
- 이 차이 때문에 old color 역할이 애매해졌다.
  - 새 color는 전환 전에 충분히 직접 검증되지 않았고
  - old color는 smoke 동안 계속 살아 있었지만 `serving`인지 `draining`인지 구조적으로 설명하기 어려웠다.
  - 결과적으로 “더 오래 살려둔 것”과 “기존 요청만 정리하게 둔 것”이 같은 의미가 아니었다.
- Nginx error log에는 다시 다음 패턴이 반복됐다.
  - `recv() failed (104: Connection reset by peer) while reading response header from upstream`
  - `upstream prematurely closed connection while reading response header from upstream`
- 실패 요청은 모두 `POST /api/events`였고 upstream은 계속 `http://127.0.0.1:8081/api/events`였다.
- `app-blue` 로그를 보면 `20:14:01 UTC` 기동 완료 후 `20:14:53 UTC`에 graceful shutdown이 시작됐다.
- 현재 정황만 보면, old color를 smoke 전까지 더 오래 살려두는 것이 write 안정성을 높여주지 못했고, `8081(app-blue)` 경계 구간이 여전히 취약했던 것으로 보인다.
- 결과적으로 안 2의 문제는 “old color를 너무 빨리 껐다”기보다, direct smoke를 전환 전에 충분히 검증하지 못한 상태에서 old color 역할만 애매하게 늘려둔 쪽에 더 가까웠다.
- 즉 핵심 실패 원인은 “smoke 전까지 old color를 살려두는 아이디어” 자체보다, 그 old color를 `draining` 상태로 구조적으로 관리하지 못한 구현에 있었다.

### 대응

- 안 2는 현재 프로젝트 기준선으로 채택하지 않았다.
- 종료 절차는 다시 안 1 기준으로 복귀했다.
  - `blue-green-prod-lib.sh`가 `전환 -> public verify -> drain -> old color stop`까지 담당
- 이후 추가 개선은 안 1을 유지한 상태에서 검토한다.
- 다음 개선 방향은 “smoke 뒤에 종료”가 아니라 아래 흐름으로 다시 잡는다.
  - 새 color readiness 확인
  - 전환 전 새 color direct smoke
  - direct smoke 실패 시 전환 중단
  - 성공 시 nginx 전환
  - 전환 후 old color는 `serving`이 아니라 `draining` 상태로만 유지
  - 가능한 범위에서 active request 정리 확인 후 종료

### 정리

- “더 보수적으로 보이는 종료 흐름”이 실제로 더 안정적이라는 보장은 없다.
- 종료 절차는 이론보다 실제 write 검증 결과를 기준으로 판단해야 한다.
- old color 종료 시점을 단순히 뒤로 미루는 것만으로는 구조적 해결이 되지 않는다.
- 안 2의 핵심 한계는 `smoke 후 종료` 자체보다, `종료 지연`과 `draining 제어`를 구분하지 못했다는 점이었다.
- 새 color 검증은 전환 전에 direct smoke로 당기고, 전환 후 old color는 `serving`이 아니라 `draining` 상태로만 정리하는 쪽이 더 설명 가능하고 자연스럽다.

## 현재까지 정리된 기준

- 종료 절차 개선의 비교 기준은 `OLD_COLOR_DRAIN_SECONDS` 추가 후 통과했던 안 1이다.
- 안 2(smoke 후 old color 종료)는 현재 프로젝트 기준으로는 폐기한다.
- 추가 개선은 안 1 기준을 유지한 상태에서 다시 검토한다.
- 다음 개선 방향은 `전환 전 새 color direct smoke + 전환 후 old color draining` 구조다.

## 2. 새 smoke / drain 스크립트가 첫 배포에서 실행 권한 부족으로 실패한 문제

### 상황

- `deploy-prod-orchestrator.sh` 기준으로 새 배포 흐름을 적용한 첫 배포에서 direct smoke 단계 진입 직후 실패했다.
- CodeDeploy 로그에는 다음과 같이 남았다.
  - `/home/ubuntu/click-checker/scripts/deploy-prod-orchestrator.sh: line 90: /home/ubuntu/click-checker/scripts/deploy-smoke.sh: Permission denied`

### 원인

- `appspec.yml`에 `scripts/**/*.sh` 권한 설정이 있어도, 실제 운영 배포에서는 새로 추가된 핵심 스크립트 실행 권한이 확실히 맞지 않았다.
- `after-install.sh`는 기존 오케스트레이터와 helper만 `chmod +x` 하고 있었고, 새로 추가된 `deploy-smoke.sh`, `deploy-drain.sh`는 대상에 포함하지 않았다.

### 대응

- `after-install.sh`에 다음을 추가했다.
  - `deploy-smoke.sh`, `deploy-drain.sh` 존재 확인
  - 두 스크립트에 대한 `chmod +x`

### 정리

- 배포 핵심 스크립트는 `appspec.yml` 권한 설정만 믿지 않고, `after-install.sh`에서 다시 명시적으로 실행 권한을 주는 편이 더 안전하다.

## 3. old color drain 시작 호출에서 500이 발생한 문제

### 상황

- direct smoke와 public smoke는 통과했지만, `deploy-drain.sh`가 old color에 `POST /internal/drain/start`를 호출하는 단계에서 `500`이 발생했다.
- `deploy-drain.sh` 출력은 다음과 같았다.
  - `curl: (22) The requested URL returned error: 500`

### 원인 후보

- 처음에는 old color가 이전 버전이라 `/internal/drain/start`를 모르는 호환성 문제를 의심했다.
- 하지만 old color(app-green) 로그를 확인해보니, endpoint 미지원보다는 `HttpMediaTypeNotAcceptableException`이 남아 있었다.
- 현재 정황상 drain 시작 로직 자체가 아니라 `/internal/drain/start` 응답의 content negotiation이 꼬인 쪽이 더 가깝다.

### 대응

- `deploy-drain.sh`의 `start` / `status` 호출에 `Accept: application/json`을 명시적으로 추가했다.

### 정리

- 새 internal endpoint를 도입할 때는 endpoint 존재 여부만이 아니라, 스크립트 요청 헤더와 응답 직렬화도 함께 맞춰야 한다.
- 이후 재배포(`d-6HPC1ST45`)에서는 `start draining green -> state=DRAINING activeRequests=0 -> stopping app-green`까지 정상 완료됐다.
- 즉 이번 문제는 drain 상태 모델 자체보다, 내부 호출의 응답 형식 협상이 맞지 않았던 쪽에 더 가까웠다.

## 4. 배포 초반 infra 단계가 active color를 recreate해서 write 요청이 끊긴 문제

### 상황

- direct smoke, public smoke, internal drain까지 연결한 뒤 write 부하 상태에서 다시 배포를 수행했지만 `http_req_failed = 5.43%`가 발생했다.
- 배포(`d-HCPCES0FH`) 자체는 성공했지만, `POST /api/events` write 요청이 다수 실패했다.

### 원인

- 같은 시간대 CodeDeploy 로그에는 다음 흐름이 남았다.
  - `active color=blue`
  - `target color=green`
  - `starting infra services`
  - `Container click-checker-app-blue-1 Recreate`
- 즉 target color를 준비하기 전에, infra 단계가 serving 중이던 active blue를 먼저 recreate했다.
- 같은 시간대 Nginx error log는 모두 `http://127.0.0.1:8081/api/events`에 대해
  - `connection reset by peer`
  - `upstream prematurely closed connection`
  를 반복했다.
- 원인을 다시 보면 배포 중 `prometheus`, `grafana`를 `up -d` 하는 단계와 `docker-compose.prod.yml`의 `prometheus -> app-blue depends_on`이 결합되면서 active color가 `blue`일 때 `app-blue`가 함께 재생성될 수 있는 구조였다.

### 대응

- `docker-compose.prod.yml`에서 `prometheus -> app-blue depends_on` override를 제거했다.
- `deploy-prod-orchestrator.sh`에서 배포 중 infra를 다시 기동하는 단계 자체를 제거했다.
- 즉 배포 중에는 공통 인프라를 건드리지 않고, target color 앱만 기동하도록 바꿨다.

### 정리

- 이번 실패의 직접 원인은 draining이나 internal drain이 아니라, 배포 초반 infra 단계가 active app을 조기 recreate한 데 있었다.
- 해당 수정 후 재검증 배포(`d-0UT8MMU45`)에서는
  - `starting infra services`가 사라졌고
  - active app 조기 recreate도 재현되지 않았으며
  - write 검증이 `http_req_failed = 0.00%`로 통과했다.
- 현재 기준으로는 `prometheus depends_on + 배포 중 infra 재기동`이 write 실패의 핵심 원인이었다고 봐도 무방하다.
