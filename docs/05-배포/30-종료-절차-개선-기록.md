# 30. 종료 절차 개선 기록

## 문서 목적

종료 절차 개선 단계에서 실제로 시도한 종료 흐름, 검증 결과, 로그 근거, 현재 판단을 기록한다.  
이번 문서는 “어떤 종료 시도를 했는지”와 “그 시도가 실제 write 검증에서 어땠는지”를 분리해서 남기는 데 목적이 있다.

## 1. 개선 범위

- 대상 환경: `prod`
- 대상 시나리오:
  - write 검증에 유리했던 종료 흐름 재검증
  - smoke 후 old color 종료 흐름(안 2) 시도
- 관찰 기준:
  - `http_req_failed`
  - `p95`
  - `p99`
  - Nginx upstream reset 재발 여부

## 2. 기준선

### 2.1 기준 흐름

- `blue-green-prod-lib.sh`가
  - 전환
  - public verify
  - drain
  - old color stop
  까지 담당하는 흐름을 기준선으로 본다.

### 2.2 기준 결과

- `OLD_COLOR_DRAIN_SECONDS` 추가 후 write 재검증에서는 `http_req_failed = 0.00%`였다.
- 현재 종료 절차 개선 단계에서는 이 결과를 비교 기준으로 사용한다.

## 3. smoke 후 old color 종료 흐름(안 2) 시도

### 3.1 실행 정보

- 실행일: `2026-03-13`
- 실행 위치: `local`
- 대상 환경: `prod`
- 대상 URL: `https://clickchecker.dev`
- 시나리오: `k6/smoke-write.js`
- 목표 RPS: `20 req/s`
- 실행 시간: `5m`
- 시각 기록:
  - `T0/T1`: `05:11` (`k6` 시작 / 배포 시작)
  - `T2`: `05:15` (배포 성공)
  - `T3`: `05:16` (`k6` 종료)

### 3.2 적용 내용

- `blue-green-prod-lib.sh`는 `SKIP_STOP_OLD=1`로 호출해 전환, public verify, drain까지만 담당했다.
- `deploy-prod-orchestrator.sh`가 smoke 성공 후 old color를 종료하도록 조정했다.
- 의도는 smoke 전까지 old color를 더 오래 유지해 보수적으로 운영하는 것이었다.

### 3.3 `k6` 결과

- total requests: `5995`
- `http_req_failed`: `5.50%`
- checks failed: `660`
- failed requests: `330`
- `http_req_duration.avg`: `41.02ms`
- `http_req_duration.p95`: `78.62ms`
- `http_req_duration.p99`: `328.51ms`
- max observed duration: `1.7s`
- observed RPS: `19.247035/s`

### 3.4 로그 관찰

- 같은 시간대 Nginx error log에는 다음 패턴이 다시 반복됐다.
  - `recv() failed (104: Connection reset by peer) while reading response header from upstream`
  - `upstream prematurely closed connection while reading response header from upstream`
- 실패 요청은 모두 `POST /api/events`였고, upstream은 계속 `http://127.0.0.1:8081/api/events`를 가리켰다.
- `app-blue` 로그를 보면:
  - `2026-03-12 20:14:01 UTC` 기동 완료
  - `2026-03-12 20:14:53 UTC` graceful shutdown 시작

### 3.5 해석

- smoke 전까지 old color를 유지하는 흐름이 write 검증 안정성을 더 높여주지는 못했다.
- 이번 실패는 old color를 늦게 끄는 것 자체보다, `8081(app-blue)` 경계 구간에서 upstream reset이 다시 재현된 쪽이 더 직접적인 문제로 보인다.
- 현재 프로젝트 기준으로는 안 2를 유지할 근거가 부족하다.

## 4. 현재 판단

- 종료 절차 개선 기준선은 다시 안 1로 둔다.
- 즉 실제 운영 경로는 다시 `switch -> public verify -> drain -> old color stop`을 기준으로 본다.
- 추가 개선은 안 1을 유지한 상태에서 검토한다.
- 다음 개선 방향은 “smoke 후 종료”가 아니라, `새 color direct smoke를 전환 전에 수행하고 old color는 전환 후 draining 상태로만 정리하는 구조`로 잡는다.

## 5. 현재 메모

- 이번 단계에서는 “더 보수적으로 보이는 흐름”이 실제로 더 안정적이라는 보장이 없음을 확인했다.
- 종료 절차 개선은 실제 검증 결과를 기준으로 선택해야 한다.
- 새 color 검증은 readiness만으로 충분하지 않으며, 핵심 API direct smoke가 별도로 필요하다.
