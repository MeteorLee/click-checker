# 28. 배포 실전 검증 종합

## 문서 목적

현재 운영 배포 구조에서 지속 요청이 들어오는 상황을 인위적으로 만든 뒤, 실제 전환 중 오류 없이 배포가 수행되는지 검증한 결과를 종합한다.  
이번 문서는 `CodeDeploy + ECR + Blue/Green + readiness + graceful shutdown` 구성이 `read` / `write` 요청 흐름에서도 어느 정도 안정적으로 동작하는지, 그리고 어떤 보완이 필요했는지를 정리하는 데 목적이 있다.

## 검증 범위

- 대상 환경: `prod`
- 실행 위치: `local`
- 검증 대상:
  - 지속 `read` 요청 baseline
  - 지속 `read` 요청 중 배포 전환
  - 지속 `write` 요청 baseline
  - 지속 `write` 요청 중 배포 전환
- 관찰 기준:
  - `http_req_failed`
  - timeout
  - `p95`
  - `p99`
  - `RPS`
  - `5xx`

## baseline 요약

### `read` baseline

- `2m` 초기 baseline
  - `http_req_failed = 0.00%`
  - `p95 = 25.93ms`
  - `p99 = 53.37ms`
- `5m` 실전 검증 baseline
  - `http_req_failed = 0.00%`
  - `p95 = 38.95ms`
  - `p99 = 83.98ms`

### `write` baseline

- `5m`
  - `http_req_failed = 0.00%`
  - `p95 = 30.98ms`
  - `p99 = 59.22ms`

정리:

- `read`, `write` 모두 baseline에서는 `20 req/s` 수준의 지속 요청을 안정적으로 처리했다.
- 이후 배포 중 검증은 이 baseline과 비교해 실패율과 지연 변화를 보는 방식으로 진행했다.

## 배포 중 `read` 검증 결과

### 결과 요약

- 시각 기록:
  - `T0/T1`: `02:15` (`k6` 시작 / 배포 시작)
  - `T2`: `02:18` (배포 성공)
  - `T3`: `02:20` (`k6` 종료)
- `http_req_failed = 0.00%`
- `p95 = 31.14ms`
- `p99 = 167.4ms`

### 관찰

- 전환 시점 근처에서 `RPS`, `p95`, `p99`는 일시적으로 흔들렸지만 다시 회복했다.
- `5xx`나 timeout으로 이어지지는 않았다.

### 정리

- 현재 배포 구조는 `read` 경로 기준으로는 무중단 성질을 대체로 만족한다고 볼 수 있다.

## 배포 중 `write` 검증 결과

### 1차 검증

- 시각 기록:
  - `T0/T1`: `02:50` (`k6` 시작 / 배포 시작)
  - `T2`: `02:53` (배포 성공)
  - `T3`: `02:55` (`k6` 종료)
- `http_req_failed = 5.57%`
- failed requests: `334`
- `p95 = 75.93ms`
- `p99 = 252.94ms`

### 관찰 및 해석

- baseline과 달리 배포 중 `write` 검증에서는 실제 실패가 발생했다.
- 운영 서버 Nginx 에러 로그에는 다음 흔적이 반복적으로 남았다.
  - `recv() failed (104: Connection reset by peer) while reading response header from upstream`
  - `upstream prematurely closed connection while reading response header from upstream`
  - upstream: `http://127.0.0.1:8081/api/events`
- 정황상 old color였던 `app-blue(8081)` 종료 시점과 in-flight `POST /api/events` 요청 처리 시점이 겹치면서 upstream 연결이 응답 헤더 전 종료된 것으로 해석했다.

### 1차 조치

- `blue-green-prod-lib.sh`에 `OLD_COLOR_DRAIN_SECONDS=15`를 추가했다.
- public verify 뒤 old color를 즉시 `stop`하지 않고, drain 대기 시간을 둔 뒤 종료하도록 수정했다.

### 재검증

- 시각 기록:
  - `T0/T1`: `03:35` (`k6` 시작 / 배포 시작)
  - `T2`: `03:38` (배포 성공)
  - `T3`: `03:40` (`k6` 종료)
- `http_req_failed = 0.00%`
- `p95 = 49.37ms`
- `p99 = 151.35ms`

### 정리

- drain 대기 추가 후에는 `write` 경로도 배포 중 실패 없이 유지됐다.
- 이번 재검증 기준으로는 `write` 실패 원인이 old color 종료 타이밍과 더 가깝다고 보는 것이 타당하다.
- 다만 현재 조치는 1차 수정이므로, 이후에도 같은 유형의 upstream reset이 재발하는지 계속 관찰할 필요가 있다.

## 확인된 운영 기준

- 실전 검증 준비와 `k6` 실행은 `local` 기준으로 수행한다.
- Grafana는 `https://grafana.clickchecker.dev`에서 확인한다.
- Prometheus는 외부 공개하지 않고 Grafana 뒤에서만 사용한다.
- 운영 관측 대상은 `app-blue:8081`, `app-green:8082`다.
- 종료 관련 현재 기준은 다음과 같다.
  - Spring graceful shutdown 활성화
  - `spring.lifecycle.timeout-per-shutdown-phase = 30s`
  - `stop_grace_period = 40s`
  - `OLD_COLOR_DRAIN_SECONDS = 15`

## 최종 결론

- 현재 운영 배포 구조는 `read` 경로 기준으로는 무중단 성질을 확인했다.
- `write` 경로는 초기 검증에서 old color 종료 타이밍 문제로 실패했지만, drain 대기 추가 후 재검증에서는 실패 없이 통과했다.
- 따라서 이번 단계 기준으로는 `CodeDeploy + Blue/Green + readiness + graceful shutdown` 구성이 지속 요청 상황에서도 실전적으로 동작함을 확인했다.
- 다만 `write` 경로는 종료 타이밍에 더 민감하게 드러났으므로, 이후에도 upstream reset 재발 여부와 종료 유예 정책을 함께 관찰하는 것이 적절하다.
