# 33. 운영 관측 체계 런북

## 문서 목적

현재 운영 배포 구조(단일 EC2 + Nginx + Blue/Green + RDS)를 기준으로, 관측 체계를 어떤 순서로 확장하고 어떤 기준으로 확인할지 정리한다.  
이번 문서는 새 도구를 많이 붙이는 것 자체보다, `메트릭 -> 로그 -> 예외 -> 알림` 흐름을 현재 프로젝트 구조 안에서 설명 가능하게 만드는 데 목적이 있다.

## 현재 전제

- 운영 배포 구조는 `GitHub Actions -> ECR -> S3 -> CodeDeploy -> EC2 -> Blue/Green` 기준이다.
- 운영 서버는 단일 EC2 + Nginx + `app-blue/app-green` 구조다.
- 현재까지 확보된 관측 기반은 다음과 같다.
  - `/actuator/prometheus`
  - Prometheus
  - Grafana 기본 패널
  - `requestId` 기반 로그 추적
  - Sentry 최소 모드
  - `k6` smoke script
- 현재 운영 문서에는 이미 다음 기준이 반영돼 있다.
  - `배포-운영-체크리스트.md`
  - `K6-스모크-런북.md`
  - `ci-품질선.md`
- 이번 단계에서는 다음 항목을 추가한다.
  - Loki
  - Promtail
  - Alertmanager
  - Prometheus rule 파일
  - Grafana provisioning

## 이번 런북의 범위

- 관측 스택 compose 구조 확장
- Prometheus alert rule 분리
- Grafana datasource / dashboard provisioning 추가
- Loki / Promtail 로그 수집 구조 추가
- nginx access/error 로그 수집 경로 반영
- `X-Request-Id` 전달 기준 점검
- 운영 최소 대시보드와 최소 알림 기준 정리

## 목표

- 현재 프로젝트에서 메트릭, 로그, 예외, 알림의 역할을 분리해서 설명할 수 있게 한다.
- Grafana에서 메트릭과 로그를 모두 조회할 수 있게 한다.
- Prometheus rule과 Alertmanager 전달 경로를 연결한다.
- nginx와 app 로그를 같은 관측 체계 안에서 볼 수 있게 한다.
- 배포 또는 smoke 실행 시 `지표 -> 로그 -> Sentry` 흐름이 실제로 이어지는지 확인한다.

## 사전 체크리스트

- `13-운영-관측-체계-계획.md` 내용 다시 확인
- 현재 `docker-compose`, `prometheus`, `grafana`, `nginx`, `logback` 설정 파일 위치 확인
- 현재 Prometheus scrape 대상 확인
- 현재 로그 출력 방식(stdout / nginx file log) 확인
- 운영/로컬에서 어떤 서비스가 어떤 포트로 떠 있는지 확인
- 이후 검증에 사용할 `k6` smoke 명령과 health endpoint 준비

## 적용 순서

### 1. 현재 기준 다시 읽기

- `배포-운영-체크리스트.md`에서 현재 알림 기준 초안과 배포 후 확인 항목을 다시 읽는다.
- `K6-스모크-런북.md`에서 direct smoke / public smoke 문맥을 다시 읽는다.
- `ci-품질선.md`에서 최소 알림 3종 초안을 다시 읽는다.

### 2. compose 구조 확장

- `docker-compose.yml`에서 현재 공통 observability 구성 위치를 확인한다.
- Loki, Promtail, Alertmanager를 어떤 서비스로 추가할지 정한다.
- 로컬 기준 포트 노출과 운영 기준 내부 연결을 분리해서 본다.

이 단계에서 확인할 것:

- Prometheus / Grafana와 새 서비스 간 연결 경로
- Promtail의 로그 수집 대상
  - app stdout
  - nginx access log
  - nginx error log

### 3. 설정 파일 분리

- Prometheus는 `scrape config`와 `alert rule`을 분리한다.
- Alertmanager는 최소 route / receiver 구조를 갖춘다.
- Grafana는 datasource provisioning과 dashboard provisioning을 코드로 관리한다.
- Loki / Promtail은 로컬과 운영 기준 모두 설명 가능한 수준으로 단순하게 유지한다.

권장 분리 대상:

- `prometheus/prometheus.yml`
- `prometheus/alerts.yml`
- `alertmanager/alertmanager.yml`
- `loki/loki-config.yml`
- `promtail/promtail-config.yml`
- `grafana/provisioning/datasources/*`
- `grafana/provisioning/dashboards/*`

### 4. 로그/요청 추적 기준 점검

- 앱 로그는 stdout 유지 여부를 먼저 확인한다.
- nginx는 access/error 로그 파일 수집 기준을 확인한다.
- `X-Request-Id`가 `nginx -> app -> 로그 -> Sentry` 경로에서 이어지는지 점검한다.
- 필요 시 `appColor`를 로그에서 구분 가능하게 보강한다.

원칙:

- 앱 로그는 파일보다 stdout 우선
- nginx 로그는 기존 file log 기준 유지
- request 추적 기준은 `X-Request-Id`

### 5. 대시보드 / 알림 기준 정리

- Grafana 최소 운영 대시보드를 정한다.
- Prometheus rule 기준을 파일로 고정한다.
- Alertmanager가 최소 1개 경로로 알림을 전달할 수 있게 한다.

최소 대시보드 후보:

- RPS
- p95 / p99
- 4xx / 5xx ratio
- DB pool usage
- JVM memory
- 최근 로그

최소 알림 후보:

- app down
- 5xx ratio 급증
- p95 급등
- DB pool 사용률 과다

### 6. 재검증

- 로컬 또는 운영형 smoke 기준으로 한 번 다시 확인한다.
- 각 도구에서 무엇이 보이는지 실제로 확인한다.
- 최소한 다음 흐름이 이어지는지 본다.

```text
입력(k6 또는 health/smoke)
-> Prometheus 메트릭 변화
-> Grafana 패널 확인
-> Loki 로그 확인
-> 필요 시 Sentry 예외 확인
-> Alertmanager 알림 확인
```

## 성공 기준

- compose 기준으로 observability 서비스가 정상 기동한다.
- Prometheus가 app 메트릭을 정상 수집한다.
- Grafana에서 Prometheus / Loki datasource가 모두 연결된다.
- Loki에서 app / nginx 로그 조회가 가능하다.
- `requestId` 기준 로그 추적이 가능하다.
- 최소 알림 규칙이 파일로 분리돼 있고 Prometheus가 이를 읽는다.
- Alertmanager가 최소 1개 전달 경로와 연결된다.
- 운영 기준으로 `지표 -> 로그 -> Sentry` 흐름을 설명할 수 있다.

## 실패 기준

- 새 서비스 중 하나라도 compose 기준으로 안정적으로 기동하지 못한다.
- Prometheus가 app 메트릭을 수집하지 못한다.
- Grafana datasource provisioning이 실패한다.
- Loki에서 app / nginx 로그가 조회되지 않는다.
- Alert rule은 있으나 Alertmanager 전달 경로가 실제로 연결되지 않는다.
- `requestId`가 nginx / app / 로그 / 예외 흐름에서 끊긴다.

## 실패 시 확인 순서

1. `docker compose ps`로 서비스 기동 상태 확인
2. Prometheus target 상태 확인
3. Grafana datasource / dashboard provisioning 상태 확인
4. Loki / Promtail 로그 수집 상태 확인
5. nginx access/error 로그 경로 확인
6. `X-Request-Id` 전달 여부 확인
7. Alertmanager route / receiver 설정 확인
8. 마지막으로 Sentry 이벤트 수집 여부 확인

## 추가 고려 사항

- Blue/Green color 전환을 더 직접적으로 보고 싶으면 이후 단계에서 `active color request ratio` 패널을 추가 검토한다.
- 다만 이번 단계에서는 필수 항목이 아니라, `Loki + nginx 로그 + Alertmanager`가 안정화된 뒤 여유가 있을 때 검토한다.
