# 35. 운영 관측 체계 종합

## 문서 목적

운영 관측 체계 작업의 배경, 최종 구조, 실제 검증 결과를 한 문서로 정리한다.  
이번 문서는 준비 문서나 트러블슈팅 문서와 다르게, 현재 Click-Checker 프로젝트에서 어떤 관측 구성이 운영 기준으로 자리 잡았는지 요약하는 데 목적이 있다.

## 1. 작업 배경

기존 프로젝트에는 다음과 같은 관측 기반이 이미 있었다.

- `/actuator/prometheus`
- Prometheus
- Grafana
- `requestId` 기반 로그 추적
- Sentry 최소 모드
- `k6` smoke script

하지만 이 상태만으로는 다음 한계가 있었다.

- Grafana 데이터소스와 대시보드가 수동 상태에 가까웠다.
- 로그 조회가 Grafana 안에서 하나의 흐름으로 묶여 있지 않았다.
- Prometheus alert rule과 Alertmanager 전달 구조가 코드 기준으로 정리돼 있지 않았다.
- 배포/장애 시 `지표 -> 로그 -> 예외` 흐름을 한 번에 설명하기 어려웠다.

결국 이번 단계에서는 단순히 도구를 더 붙이는 것이 아니라, 현재 단일 EC2 + Nginx + Blue/Green 구조 안에서 `메트릭 -> 로그 -> 예외 -> 알림` 흐름을 설명 가능한 관측 체계로 정리하는 것을 목표로 삼았다.

## 2. 최종 관측 구조

현재 기준 관측 구조는 다음과 같다.

```text
App / Nginx
-> Prometheus (metrics)
-> Loki (logs)
-> Grafana (dashboard / logs view)
-> Alertmanager (alerts)
-> Sentry (exceptions)
```

역할은 아래처럼 나뉜다.

- App
  - `/actuator/prometheus` 메트릭 노출
  - stdout 로그 출력
  - `requestId` + `app color` 로그 출력
- Nginx
  - access / error log 파일 기록
  - `X-Request-Id`를 app으로 전달
- Prometheus
  - app 메트릭 수집
  - alert rule 평가
- Loki
  - app / nginx 로그 저장
- Promtail
  - Docker app 로그, nginx file log 수집 후 Loki로 전달
- Grafana
  - Prometheus / Loki 데이터소스 조회
  - 운영 대시보드 표시
- Alertmanager
  - Prometheus alert 수신
- Sentry
  - 예외 추적

## 3. 핵심 적용 사항

### 3.1 compose 기준 observability 스택 확장

기존 `Prometheus + Grafana` 구성에 아래를 추가했다.

- Loki
- Promtail
- Alertmanager

또한 compose 구조는 다음 기준으로 정리했다.

- `docker-compose.yml`
  - 공통 observability 서비스 정의
- `docker-compose.local.yml`
  - 로컬 포트 노출
  - Docker 로그 수집용 Promtail 마운트
  - `prometheus.local.yml` 연결
- `docker-compose.prod.yml`
  - 운영 nginx 로그 수집용 Promtail 마운트
  - `prometheus.prod.yml` 연결

### 3.2 Prometheus local / prod 설정 분리

처음에는 Prometheus 설정 하나에 `app`, `app-blue`, `app-green`을 모두 넣었다.  
하지만 이 구조에서는 local 검증 중에도 존재하지 않는 `app-blue/app-green` 때문에 `AppDown` alert가 가짜로 firing했다.

이후 설정을 아래처럼 분리했다.

- `prometheus.local.yml`
  - `app:8080`
- `prometheus.prod.yml`
  - `app-blue:8081`
  - `app-green:8082`

정리:

- local 검증과 prod 운영 기준이 분리됐다.
- local false alert 문제가 사라졌다.

### 3.3 Alert rule 분리

Prometheus rule은 `prometheus/alerts.yml`로 분리했다.  
현재 기준 rule은 다음과 같다.

- `ClickCheckerNoScrapeTargets`
- `ClickCheckerAppDown`
- `ClickCheckerHigh5xxRatio`
- `ClickCheckerHighP95Latency`
- `ClickCheckerHighDbPoolUsage`

이 과정에서 다음 보정도 함께 반영했다.

- `NoScrapeTargets` 추가
- `5xx ratio`에 최소 트래픽 조건 추가
- `DB pool usage` 식 보정

### 3.4 Grafana provisioning 코드화

Grafana는 다음을 파일 기준으로 관리하게 정리했다.

- datasource provisioning
  - Prometheus
  - Loki
- dashboard provisioning
  - `Click Checker` 폴더
  - `Click Checker Overview` 대시보드

최소 운영 대시보드에는 다음 패널을 넣었다.

- `RPS by App`
- `Latency p95 / p99`
- `4xx / 5xx Ratio`
- `DB Pool Usage`
- `JVM Heap Used`
- `Recent Logs`

### 3.5 requestId / app color 추적 기준 보강

관측 흐름을 실제로 이어주기 위해 다음도 함께 보강했다.

- Nginx가 `X-Request-Id`를 app에 전달
- app 로그에 `requestId` 유지
- app 로그에 `APP_COLOR` 출력

즉 현재 로그 기준으로는 같은 요청을 다음처럼 추적할 수 있다.

- requestId
- app color
- log line
- 필요 시 Sentry 예외

## 4. 실제 검증 결과

이번 단계에서 실제로 확인한 결과는 다음과 같다.

### 4.1 Compose / 기동

- `docker compose config` 정상 통과
- Prometheus / Grafana / Loki / Alertmanager / Promtail / app 기동 확인

### 4.2 Prometheus / Alertmanager

- Prometheus가 alert rule을 정상 로드했다.
- Prometheus가 Alertmanager를 실제 alert target으로 등록했다.
- local false `AppDown` alert는 Prometheus local/prod 분리 후 사라졌다.

### 4.3 Loki / Promtail

- app 로그는 Loki로 실제 적재되는 것을 확인했다.
- Promtail은 Docker app target과 nginx file target을 등록했다.
- 다만 초기 기동 시 과거 Docker 로그 때문에 `entry too far behind` 경고가 발생했다.
- 이 문제는 별도 트러블슈팅 문서로 분리했다.

### 4.4 Grafana

- Prometheus / Loki 데이터소스 provisioning 확인
- `Click Checker Overview` 대시보드 provisioning 확인
- 로컬 Grafana에 남아 있던 기존 수동 데이터소스 때문에 datasource UID mismatch가 발생했지만, Grafana 로컬 상태 초기화 후 해결했다.

## 5. 이번 단계에서 확인된 운영 기준

현재 기준 운영 관측 판단은 다음과 같이 정리된다.

- 메트릭은 Prometheus가 본다.
- 로그는 Loki에서 본다.
- 대시보드와 로그 화면은 Grafana에서 본다.
- 예외는 Sentry에서 본다.
- 알림 판단은 Prometheus rule + Alertmanager 기준으로 간다.
- 요청 단위 추적 기준은 `X-Request-Id`다.
- Blue/Green 구분 기준은 `APP_COLOR` 로그 출력이다.

즉 장애나 배포 이상 징후가 발생했을 때 기본 흐름은 아래와 같다.

```text
지표 확인
-> 로그 확인
-> 필요 시 예외 확인
```

## 6. 이번 단계에서 남긴 기준

- observability 스택은 최소 작동선부터 확보하고, 이후 품질 보정으로 나아간다.
- local과 prod 관측 기준은 처음부터 분리하는 것이 더 안전하다.
- Grafana는 수동 UI 설정보다 provisioning 기준으로 관리하는 편이 재현성과 설명력 모두 좋다.
- 기존 수동 Grafana 상태가 남아 있으면 datasource UID mismatch가 발생할 수 있으므로, 로컬에서는 필요 시 상태 초기화가 더 빠를 수 있다.
- Promtail / Loki 초기 연결에서는 과거 로그 재적재 문제를 별도 품질 이슈로 볼 필요가 있다.

## 7. 후속 보정 후보

이번 단계에서 즉시 닫지 않고 남겨둔 항목은 다음과 같다.

- `p95`에서 `/actuator/*` 제외 여부 검토
- `active color request ratio` 패널 검토
- Loki retention / compactor / 장기 보존 정책 검토
- 초기 기동 시 과거 로그 경고 허용 범위 정리
- 장기적으로 Promtail deprecated/EOL 상태를 고려한 Grafana Alloy 전환 검토

## 8. 결론

이번 작업을 통해 Click-Checker의 운영 관측 체계는 `Prometheus + Grafana` 중심의 느슨한 상태에서, `Prometheus + Loki + Grafana + Alertmanager + Sentry`가 역할별로 정리된 구조로 발전했다.

또한 이번 단계의 핵심은 도구를 많이 붙인 것 자체보다, 아래 흐름을 실제 프로젝트 안에서 설명 가능하게 만들었다는 점이다.

`메트릭 -> 로그 -> 예외 -> 알림`

현재 기준으로는 최소 작동형 운영 관측 체계가 확보됐다고 볼 수 있으며, 이후 단계에서는 이 구조를 바탕으로 품질 보정과 운영 기준 정교화를 이어가는 것이 적절하다.
