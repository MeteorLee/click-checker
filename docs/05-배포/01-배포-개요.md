# 배포 전략 개요 (1.2)

## 1. 목적

Click-Checker는 단순 실행을 넘어서, **운영을 가정한 배포 구조 설계**를 목표로 한다.

배포는 다음 조건을 동시에 만족해야 한다:

- 기능이 정상 동작한다.
- 실패 시 안전하게 복구할 수 있다.
- 운영 환경에서 재현 가능하다.
- 점진적으로 자동화 및 확장이 가능하다.

이 문서는 수동 배포, 자동 배포, 향후 확장 전략을 하나의 흐름으로 정리한 상위 개요 문서이다.

---

## 2. 배포 진화 전략

배포는 한 번에 완성하지 않고 단계적으로 발전시킨다.

### 2.1 1단계 – 수동 배포

구성:

- EC2 환경 직접 구성
- build → test → deploy → health 확인
- 스모크 테스트 수행
- 실패 시 수동 롤백

목적:

- 배포 전 과정을 직접 경험
- 환경 차이로 인한 문제 이해
- 장애 발생 지점 파악
- 롤백 기준 수립

수동 배포는 자동화 이전에 배포 메커니즘을 충분히 이해하기 위한 단계이다.

---

### 2.2 2단계 – 자동 배포 (v1)

구성:

- 브랜치 전략 기반 배포
  - develop: 빠른 CI
  - main: PostgreSQL 포함 강한 CI
  - prod: 배포 전용 브랜치
- GitHub Actions 기반 배포 트리거
- EC2 내부 스모크 테스트 수행
- 실패 시 자동 롤백

목적:

- 수동 실수 제거
- 품질 게이트를 통한 배포 차단
- 배포 안정성 확보

자동 배포는 수동 배포 절차를 그대로 코드화한 형태로 설계한다.

---

### 2.3 3단계 – 확장 계획

향후 확장 방향:

- 무중단 배포 구조 실험
- Reverse Proxy 또는 ALB 도입
- API Key 기반 인증 구조 전환
- RDS 분리 및 운영 DB 고도화

배포 구조는 서비스 성장에 따라 점진적으로 확장한다.

---

## 3. 배포 성공 기준

배포는 단순 기동 성공이 아니라 다음 조건을 모두 만족해야 성공으로 간주한다:

1. `/actuator/health` 가 200이며 상태가 `UP`
2. 이벤트 저장 API가 정상 동작
3. 집계 조회 API가 정상 동작

이 성공 기준은 수동 배포와 자동 배포 모두 동일하게 적용한다.

---

## 4. 품질 게이트 전략

배포 이전에는 반드시 CI를 통과해야 한다.

구성:

- H2 기반 기본 테스트
- PostgreSQL 호환성 테스트
- 핵심 시나리오 통합 테스트

원칙:

- CI 실패 시 배포는 진행하지 않는다.
- 배포는 항상 “검증된 코드”만 대상으로 수행한다.

---

## 5. 롤백 원칙

- 배포 실패 시 직전 정상 커밋으로 복귀한다.
- 복귀 후 health를 재확인한다.
- 데이터 손상을 유발할 수 있는 파괴적 스키마 변경은 v1 단계에서 지양한다.

롤백은 예외 상황이 아니라, 설계 단계에서부터 고려되어야 하는 기본 조건이다.

---

## 6. 설계 원칙 요약

- 배포는 단순 실행이 아니라 운영 설계의 일부이다.
- 자동화는 수동 과정을 충분히 이해한 이후 도입한다.
- 성공 기준은 환경에 따라 달라지지 않는다.
- 항상 복구 가능성을 전제로 설계한다.
