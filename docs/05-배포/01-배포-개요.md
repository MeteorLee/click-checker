# 배포 전략 개요 (1.6)

## 1. 목적
- Click-Checker의 현재 배포 구조와 운영 기준을 한 문서에서 빠르게 파악할 수 있게 한다.
- 수동 배포에서 자동 배포, RDS 전환까지 어떤 흐름으로 발전했는지 정리한다.
- 세부 절차 문서를 보기 전에 “지금 운영이 어떤 상태인지”를 먼저 이해하게 한다.

---

## 2. 현재 운영 기준

### 2.1 현재 배포 형태
- 인프라: 단일 EC2
- 외부 진입점: Nginx + `clickchecker.dev` + HTTPS
- 관리자 진입점: `grafana.clickchecker.dev` + HTTPS + Basic Auth
- 애플리케이션: Spring Boot
- 운영 DB: Amazon RDS for PostgreSQL
- 관측: Actuator, Prometheus, Grafana, Sentry
- 배포 방식: GitHub Actions 기반 자동 배포

### 2.2 현재 실행 구조
- 외부 사용자는 `https://clickchecker.dev`로만 app에 접근한다.
- Nginx가 `80/443`에서 요청을 받고 app (`127.0.0.1:8080`)으로 reverse proxy 한다.
- Grafana는 `https://grafana.clickchecker.dev` 경로에서 Basic Auth + Grafana 로그인 기준으로 접근한다.
- prod 앱은 EC2 내부 postgres가 아니라 RDS를 바라본다.
- `docker-compose.prod.yml`은 `DB_URL`, `DB_USERNAME`, `DB_PASSWORD`를 `.env`에서 읽는다.
- prod 배포 경로는 `app + prometheus + grafana` 중심으로 동작한다.
- app `8080`은 외부 직접 노출하지 않는다.
- Grafana `3000`도 외부 직접 노출하지 않는다.
- 로컬 postgres는 local 구성에서만 사용한다.

### 2.3 현재 브랜치 기준
- `develop`: 빠른 CI
- `main`: 강한 검증(`test + postgresTest`)
- `prod`: 배포 전용 브랜치

---

## 3. 배포 진화 흐름

### 3.1 1단계 – 수동 배포
- EC2 환경 직접 구성
- build → test → deploy → health → smoke 순서 수동 확인
- 실패 시 수동 롤백

의미:
- 배포 메커니즘을 직접 이해하고, 환경 차이와 실패 지점을 파악하는 단계

### 3.2 2단계 – 자동 배포
- GitHub Actions로 `prod` push 시 배포
- health + smoke 검증
- 실패 시 직전 정상 커밋 기준 자동 롤백

의미:
- 수동 절차를 코드화해서 배포 안정성을 높이는 단계

### 3.3 3단계 – 운영 고도화
- API Key 기반 인증 구조 전환
- Flyway 기반 스키마 통제
- 운영 DB의 RDS 분리
- prod 앱의 RDS 기준 배포 경로 정리

의미:
- “앱이 뜬다” 수준을 넘어서, 운영 DB / 인증 / 배포 기준을 고정하는 단계

### 3.4 4단계 – 웹 진입점 정리
- `clickchecker.dev` 도메인 연결
- Nginx reverse proxy 구성
- Let's Encrypt 기반 HTTPS 적용
- `/actuator/health` 공개, 내부 actuator 차단
- app `8080` 외부 직접 노출 제거
- `grafana.clickchecker.dev` + Basic Auth 관리자 진입점 추가

의미:
- 외부 접근 구조를 운영형 진입점 기준으로 정리하는 단계

### 3.5 다음 확장 방향
- 무중단 배포 구조 실험
- Grafana 접근 제어 강화 검토
- ALB 도입 여부 검토
- 운영 구조 추가 단순화

---

## 4. 배포 성공 기준
- `/actuator/health`가 `UP`
- 이벤트 저장 API가 정상 동작
- 집계 조회 API가 정상 동작
- `https://clickchecker.dev` 진입이 정상 동작
- prod 앱의 `DB_URL`이 운영 기준 DB(RDS)를 바라봄

원칙:
- 단순 기동 성공만으로는 배포 성공으로 보지 않는다.
- health, smoke, 운영 DB 연결 상태를 함께 본다.

---

## 5. 품질 게이트
- `develop`: `./gradlew test`
- `main`: `./gradlew test` + `./gradlew postgresTest`
- CI 실패 시 배포 진행 금지
- 배포 대상은 항상 “검증된 코드”여야 한다.

---

## 6. 롤백 원칙
- 배포 실패 시 직전 정상 커밋으로 코드 롤백
- 롤백 후 health 재확인
- 앱 롤백과 DB 복구는 분리해서 판단

원칙:
- 코드 롤백은 배포 단위로 처리한다.
- DB 문제는 백업/restore 또는 forward-fix로 처리한다.

## 8. 요약
- 현재 Click-Checker는 단일 EC2 기반 자동 배포 구조를 갖고 있다.
- 외부 진입점은 `clickchecker.dev + HTTPS + Nginx` 기준으로 정리됐다.
- Grafana도 `grafana.clickchecker.dev + HTTPS + Basic Auth` 기준의 관리자 진입점을 갖는다.
- 운영 DB는 이미 RDS로 분리됐고, prod 앱은 RDS를 기준으로 동작한다.
- 배포 성공 판단은 health + smoke + 운영 DB 연결 확인까지 포함한다.
- 이후 확장 작업은 현재 운영 기준을 유지한 상태에서 점진적으로 진행한다.
