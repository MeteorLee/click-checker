# 14. 웹 진입점 구성 초안

## 0. 목적
- 웹 진입점 구성 착수 전에 현재 외부 진입 구조와 목표 진입 구조를 먼저 고정한다.
- Nginx / HTTPS / 엔드포인트 보호 정책을 어떤 방향으로 가져갈지 정리한다.
- 실제 반영 결과와 시행착오는 별도 문서로 분리한다.

---

## 1. 설계 시작 시점의 현재 구조

### 1.1 외부에서 직접 접근하던 경로
- app: `8080`
- grafana: `3000`

### 1.2 당시 특성
- app 컨테이너가 `8080:8080`으로 직접 외부에 노출된다.
- Grafana도 `3000:3000`으로 직접 외부에 노출된다.
- prod 앱은 RDS를 바라본다.
- reverse proxy 없이 앱 포트에 직접 접근하는 구조였다.

### 1.3 당시 한계
- 서비스 진입점이 운영형 구조라고 보기 어려웠다.
- HTTP/HTTPS 정책이 분리되어 있지 않았다.
- 외부 공개 경로와 내부 관리 경로가 명확히 분리되지 않았다.

---

## 2. 목표 구조

### 2.1 목표 진입점
- 외부 사용자는 `80/443`으로만 접근한다.
- Nginx가 외부 진입점이 된다.
- Nginx가 내부 app로 reverse proxy 한다.
- 운영 DB는 계속 RDS를 사용한다.

### 2.2 목표 흐름
사용자  
→ Nginx (`80/443`)  
→ app (내부 포트)  
→ RDS

### 2.3 목표 상태
- app의 직접 외부 노출을 줄인다.
- HTTP 요청은 HTTPS로 리다이렉트한다.
- 외부 공개 엔드포인트와 내부 관리 엔드포인트를 구분한다.

---

## 3. 공개/비공개 원칙

### 3.1 외부 공개 대상
- 서비스 API 진입 경로
- `/actuator/health`

### 3.2 내부/제한 대상
- `/actuator/prometheus`
- 기타 내부 관리용 actuator 경로
- 직접 app 포트 접근

### 3.3 Grafana 정책
- 현재는 운영 편의를 위해 직접 노출 중
- 유지 여부는 별도 운영 정책으로 결정

---

## 4. 설계 원칙
- 웹 진입점은 Nginx를 기준으로 정리한다.
- health 확인은 `/actuator/health`를 기본 경로로 유지한다.
- 내부 actuator 경로는 외부 진입점에서 차단한다.
- HTTPS는 도메인 확보 후 Let's Encrypt 기준으로 적용한다.
- 실제 적용 결과와 트러블슈팅은 별도 문서로 분리한다.

---

## 5. 후속 정리 항목
- app `8080` 직접 외부 노출 축소 여부 결정
- Grafana 직접 노출 유지 여부 결정
- Nginx/HTTPS 반영 절차를 운영 문서에 반영
- 인증서 자동 갱신 점검 절차 정리
