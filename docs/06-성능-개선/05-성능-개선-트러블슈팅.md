# 05. 성능 개선 트러블슈팅

## 문서 목적

성능 실험을 준비하고 실행하는 과정에서 발생한 문제와 해결 과정을 누적 기록한다.  
이번 문서는 run 결과 자체보다, 실험 체계와 산출물 자동화 과정에서 부딪힌 문제를 다시 밟지 않도록 기준을 남기는 데 목적이 있다.

## 기록 원칙

- 성능 수치 자체보다 실험 체계, 산출물, 관측 연결 문제를 우선 기록한다.
- 단순한 오타보다는 재현성과 비교 가능성에 영향을 준 문제를 우선 기록한다.
- 해결 후에는 남긴 기준까지 같이 적는다.

## 항목 1. Grafana renderer 기반 PNG 캡처 추가

### 문제 상황

- 성능 run 산출물에 Grafana 그래프를 남기려면 수동 캡처가 필요했다.
- 수동 캡처는 run마다 반복하기 번거롭고, baseline과 재검증을 같은 방식으로 비교하기 어려웠다.
- 브라우저 자동화도 가능하지만, 이번 단계에서는 Grafana의 정식 render 경로를 먼저 쓰고 싶었다.

### 확인한 원인

- 기존 observability 스택에는 `renderer` 서비스가 없었다.
- Grafana도 remote renderer 설정 없이 기본 상태로만 떠 있었다.
- 따라서 대시보드 PNG를 run 산출물 디렉터리에 자동으로 남길 수 있는 경로가 없었다.

### 적용한 조치

- `docker-compose.yml`
  - `renderer` 서비스 추가
  - `GF_RENDERING_SERVER_URL`
  - `GF_RENDERING_CALLBACK_URL`
  - `GF_RENDERING_RENDERER_TOKEN`
  설정 추가
- `prometheus.local.yml`, `prometheus.prod.yml`
  - renderer metrics scrape job 추가
- `scripts/observability/*`
  - deploy / restart / validate 흐름에 renderer 포함
- `scripts/perf/common/capture-grafana-render.sh`
  - renderer API를 호출해 PNG를 저장하는 스크립트 추가
- `scripts/perf/w1/run-local.sh`
  - `capture.mode=scripted`면 run 종료 후 자동 캡처 실행

### 중간 이슈

- 처음 시도한 renderer 이미지 태그는 실제로 맞지 않았다.
- Grafana 11.2.2와 맞는 `grafana/grafana-image-renderer:3.12.9`로 조정했다.

### 검증 결과

- renderer 컨테이너 기동 확인
- renderer metrics endpoint 응답 확인
- Grafana 로그에서 remote renderer version 인식 확인
- 대시보드 전체 PNG와 핵심 panel PNG 생성 확인
- W1 산출물 디렉터리에 아래 파일이 자동 저장되는 것 확인
  - `dashboard-overview.png`
  - `ingest-rps.png`
  - `ingest-latency.png`
  - `http-error-ratio.png`
  - `db-pool.png`
  - `db-connections.png`
  - `jvm-heap-by-app.png`

### 남긴 기준

- 성능 run 기본 캡처 방식은 `capture.mode=scripted`로 본다.
- 수동 캡처는 예외 상황이나 보조 확인 용도로만 사용한다.
- 캡처 결과와 상태는 `meta.json.capture.*`로 관리한다.
- 실행 산출물은 `artifacts/perf/<scenario>/<runId>/`에 모은다.

## 항목 2. `M2` first-run 편차와 warm-up / rerun 해석 기준

### 문제 상황

- local `M2 s2`는 같은 코드 상태에서도 app 재기동 직후 첫 run과 같은 인스턴스의 rerun 결과 차이가 컸다.
- 어떤 변경은 첫 run만 보면 실패처럼 보였지만, 같은 앱 상태에서 rerun하면 threshold를 통과하기도 했다.
- 반대로 특정 실험은 첫 run과 rerun이 모두 나빠 실제 회귀로 봐야 했다.

### 확인한 원인

- `M2`는 write/read mixed 경쟁이 강해서 cold app, cold cache, JIT 미가열 상태의 영향을 크게 받는다.
- overview `5분` 캐시, resolver metadata cache, rollup read 경로는 모두 first-run과 steady-state의 차이를 더 키울 수 있다.
- 따라서 app restart 직후 첫 run 하나만으로는 변경의 효과를 안정적으로 판단하기 어려웠다.

### 적용한 조치

- `M2` 검증 기본 warm-up을 `2m`까지 늘려 재검증했다.
- first-run 결과가 크게 흔들리면 같은 앱 인스턴스 상태에서 rerun 1회를 추가했다.
- 문서 해석도
  - first-run
  - steady-state rerun
  을 분리해서 기록하도록 정리했다.

### 검증 결과

- overview 응답 `5분` 캐시 적용 뒤
  - `m2-20260323-135843-s2-after-overview-cache-warm2m`
  - throughput `193.94 req/s`, read `p95 1.24s`, write `p95 1.11s`, dropped `29`
  - 로 매우 안정적으로 threshold를 통과했다.
- aggregate `routes/event-types` rollup 제거 뒤 baseline 복구 확인에서도
  - 첫 두 run은 fail이었지만
  - `m2-20260324-015022-s2-after-aggregate-rollup-revert-check-rerun2`
  - throughput `193.09 req/s`, read `p95 1.09s`, write `p95 1.00s`, dropped `120`
  - 으로 baseline이 다시 복구됐다.

### 남긴 기준

- local `M2`는 app restart 직후 first-run 하나만으로 채택/철회를 판단하지 않는다.
- `warm-up 2m` 또는 sacrificial warm run 뒤의 steady-state rerun을 함께 본다.
- first-run과 rerun이 모두 나쁘면 회귀 가능성이 높고,
- first-run만 나쁘고 rerun이 복구되면 noisy run 가능성을 먼저 본다.
