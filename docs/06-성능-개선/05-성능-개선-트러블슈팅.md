# 05. 성능 개선 트러블슈팅

## 문서 목적

성능 실험을 준비하고 실행하는 과정에서 발생한 문제와 해결 과정을 누적 기록한다.  
이번 문서는 run 결과 자체보다, 실험 체계와 산출물 자동화 과정에서 부딪힌 문제를 다시 밟지 않도록 기준을 남기는 데 목적이 있다.

## 기록 원칙

- 성능 수치 자체보다 실험 체계, 산출물, 관측 연결 문제를 우선 기록한다.
- 단순한 오타보다는 재현성과 비교 가능성에 영향을 준 문제를 우선 기록한다.
- 해결 후에는 남긴 기준까지 같이 적는다.

## 항목 1. Grafana renderer 기반 PNG 캡처 추가

### 문제 상황

- 성능 run 산출물에 Grafana 그래프를 남기려면 수동 캡처가 필요했다.
- 수동 캡처는 run마다 반복하기 번거롭고, baseline과 재검증을 같은 방식으로 비교하기 어려웠다.
- 브라우저 자동화도 가능하지만, 이번 단계에서는 Grafana의 정식 render 경로를 먼저 쓰고 싶었다.

### 확인한 원인

- 기존 observability 스택에는 `renderer` 서비스가 없었다.
- Grafana도 remote renderer 설정 없이 기본 상태로만 떠 있었다.
- 따라서 대시보드 PNG를 run 산출물 디렉터리에 자동으로 남길 수 있는 경로가 없었다.

### 적용한 조치

- `docker-compose.yml`
  - `renderer` 서비스 추가
  - `GF_RENDERING_SERVER_URL`
  - `GF_RENDERING_CALLBACK_URL`
  - `GF_RENDERING_RENDERER_TOKEN`
  설정 추가
- `prometheus.local.yml`, `prometheus.prod.yml`
  - renderer metrics scrape job 추가
- `scripts/observability/*`
  - deploy / restart / validate 흐름에 renderer 포함
- `scripts/perf/common/capture-grafana-render.sh`
  - renderer API를 호출해 PNG를 저장하는 스크립트 추가
- `scripts/perf/w1/run-local.sh`
  - `capture.mode=scripted`면 run 종료 후 자동 캡처 실행

### 중간 이슈

- 처음 시도한 renderer 이미지 태그는 실제로 맞지 않았다.
- Grafana 11.2.2와 맞는 `grafana/grafana-image-renderer:3.12.9`로 조정했다.

### 검증 결과

- renderer 컨테이너 기동 확인
- renderer metrics endpoint 응답 확인
- Grafana 로그에서 remote renderer version 인식 확인
- 대시보드 전체 PNG와 핵심 panel PNG 생성 확인
- W1 산출물 디렉터리에 아래 파일이 자동 저장되는 것 확인
  - `dashboard-overview.png`
  - `ingest-rps.png`
  - `ingest-latency.png`
  - `db-pool.png`
  - `db-connections.png`

### 남긴 기준

- 성능 run 기본 캡처 방식은 `capture.mode=scripted`로 본다.
- 수동 캡처는 예외 상황이나 보조 확인 용도로만 사용한다.
- 캡처 결과와 상태는 `meta.json.capture.*`로 관리한다.
- 실행 산출물은 `artifacts/perf/<scenario>/<runId>/`에 모은다.
