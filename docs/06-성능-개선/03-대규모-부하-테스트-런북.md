# 03. 대규모 부하 테스트 런북

## 문서 목적

현재 Click-Checker 구조에서 `write-heavy`, `read-heavy`, `mixed` 부하를 실제로 재현하고, 병목을 `그래프 + 수치 + 로그` 기준으로 기록하는 실행 절차를 정리한다.  
이번 문서는 성능 개선 아이디어를 나열하는 데 목적이 있지 않고, 동일 조건의 실험을 반복해 전/후 비교가 가능하도록 기준을 고정하는 데 목적이 있다.

## 현재 전제

- 운영/관측 기준선은 이미 정리된 상태다.
  - Grafana
  - Loki
  - Prometheus
  - Alertmanager
  - Sentry
- `deploy smoke`, `k6 smoke read/write`, local / prod 검증 흐름이 이미 동작한다.
- 이번 단계는 새 관측 도구를 붙이는 것이 아니라, 현재 구조에서 실제 병목을 찾는 실험 단계다.
- 우선순위는 아래와 같다.
  - 1차: 대규모 테스트로 병목 확인
  - 2차: 인덱스 / 쿼리 / 쓰기 경로 같은 값싼 개선
  - 3차: 같은 조건 재테스트
  - 4차: 그래도 필요하면 Redis / rollup 검토

## 이번 런북의 범위

- 부하 테스트 실행 조건 고정
- `W1 / R1 / R2 / R3 / M1` 시나리오 정의
- 부하 테스트 전 체크리스트 정리
- 실행 시 기록해야 할 항목 정리
- 성공 / 실패 / 추가 분석 필요 기준 정리

## 목표

- 동일 조건으로 반복 가능한 부하 테스트 기준을 만든다.
- 현재 구조의 병목 지점을 `p95 / p99 / 에러율 / DB / Hikari` 기준으로 설명 가능하게 만든다.
- 이후 인덱스/쿼리/쓰기 경로 개선 전후를 같은 실험으로 비교할 수 있게 한다.

## 사전 체크리스트

- 대상 커밋 해시 또는 이미지 태그를 기록한다.
- 대상 환경(local / prod direct / prod public)을 기록한다.
- 실행 위치와 요청 대상을 분리해서 기록한다.
- Grafana 캡처 소스가 local인지 prod인지 기록한다.
- 테스트 조직과 테스트 식별 규칙을 고정한다.
  - path prefix
  - eventType
  - runId
- observability 기준이 정상인지 확인한다.
  - Grafana 접근 가능
  - Prometheus scrape 정상
  - Loki 로그 조회 가능
  - 필요 시 Sentry 접근 가능
- 테스트용 API key 준비 상태를 확인한다.
- write / read 대상 API의 현재 계약이 맞는지 확인한다.
- Grafana에서 캡처할 패널과 시간창을 미리 정한다.
- `capture.mode=scripted`인 시나리오는 run 종료 후 renderer 기반 PNG 캡처가 자동 생성되는지 확인한다.

## 환경별 주의사항

### `local`

- 네트워크 변수와 외부 ingress 영향이 작다.
- 단일 인스턴스 기준 앱/DB 병목 확인에 적합하다.
- 스크립트, 산출물, Grafana 반응을 먼저 검증하는 용도로 사용한다.

### `prod direct`

- 실제 EC2 / RDS 자원과 네트워크 RTT가 포함된다.
- 공개 ingress 영향을 줄이고 앱/DB 중심으로 병목을 확인하기 좋다.
- local 결과와 실제 운영 자원 차이를 비교할 때 사용한다.

### `prod public`

- Nginx / ingress / 외부 네트워크 영향까지 포함된다.
- 실제 공개 경로 체인 전체를 보는 대신, 앱 자체 병목 해석은 더 어려워질 수 있다.
- 대규모 baseline보다는 최종 검증 성격으로 다루는 편이 낫다.

## 실행 위치와 캡처 소스 주의사항

- 실행 위치, 요청 대상, Grafana 캡처 소스, 산출물 저장 위치는 서로 다를 수 있다.
- 특히 아래 조합은 자연스럽지만, 명시하지 않으면 나중에 헷갈리기 쉽다.
  - 실행 위치: local
  - 요청 대상: prod public
  - Grafana 캡처 소스: prod Grafana
  - 산출물 저장 위치: local workspace

원칙:

- 부하를 local에서 실행해도 Grafana 캡처는 prod를 대상으로 할 수 있다.
- `scripted` 캡처를 쓸 때는 `GRAFANA_BASE_URL`이 어느 Grafana를 가리키는지 실행 전에 확인한다.
- prod public 실험에서는 네트워크/ingress 지연이 섞이므로, 앱/DB 순수 병목 해석과 분리해서 본다.

## 부하 조건 고정

### 초기 실험안(v1)

- warm-up: `2m`
- main run: `10m`
- cooldown: `1m`
- `W1`
  - `50 / 100 / 200 RPS`를 각각 별도 run으로 실행
- `R1 / R2 / R3`
  - `10 / 30 / 50 RPS`를 각각 별도 run으로 실행
- `M1`
  - write/read `10/10`
  - write/read `20/20`
  - write/read `30/30`

주의:

- 위 숫자는 첫 실행용 초안이다.
- baseline 결과에 따라 숫자는 조정할 수 있지만, 같은 회차 안에서는 고정한다.
- 전/후 비교 시에는 반드시 같은 RPS / duration / warm-up을 유지한다.
- baseline 단계에서는 `W1`, `R1`, `R2`, `R3` 모두 고정 RPS run을 분리 실행하는 것을 기본으로 본다.

## 실행 순서

### 1. 기준선 확인

- 현재 `20-대규모-부하-테스트-및-성능-개선-계획.md` 기준을 다시 확인한다.
- 이번 실행이 baseline인지, 개선 후 재검증인지 먼저 구분한다.
- 테스트 대상 커밋 해시와 환경을 기록한다.

### 2. 테스트 데이터 준비

- 테스트용 조직을 준비한다.
- API key를 준비한다.
- write 시나리오는 운영 데이터와 구분 가능한 식별 규칙을 사용한다.
  - 고정 path prefix
  - 고정 eventType 또는 loadtest 계열 eventType
  - `runId`
- W1은 별도 준비 스크립트로 org / api key / seed / meta 파일을 만든다.
- 구체적인 준비 흐름은 [11-W1-준비-및-실행-구조.md](11-W1-준비-및-실행-구조.md)를 따른다.

### 3. observability 확인

- Grafana 패널과 시간창을 미리 열어둔다.
- scripted capture를 쓰는 시나리오는 대시보드 수동 캡처 대신 산출물 디렉터리의 PNG 생성 여부를 확인한다.
- 캡처 대상 최소 패널:
  - latency
  - error rate
  - DB active connections
  - Hikari pool usage
- 필요 시 Logs / Sentry도 같이 확인 준비한다.

### 4. 시나리오 실행

기본 원칙:

```text
warm-up
-> main run
-> 종료
-> k6 summary 저장
-> Grafana 캡처 또는 scripted capture 산출물 확인
-> 짧은 해석 메모 작성
```

권장 순서:

```text
W1
-> R1
-> R2
-> R3
-> M1
```

주의:

- 원인 분리가 목적이므로 처음부터 mixed를 먼저 돌리지 않는다.
- write / read 단독 시나리오를 먼저 보고 mixed로 간다.

## 중단 조건

- 애플리케이션 프로세스 재시작 또는 OOM이 발생하면 현재 run을 중단하고 무효 처리 후보로 본다.
- DB connection pool 고갈이 일정 시간 이상 지속되면 현재 run을 중단하고 원인 확인으로 전환한다.
- 5xx 비율이 임계치를 지속 초과하면 현재 run을 중단한다.
- Grafana / Prometheus 수집 이상으로 관측이 불가능하면 현재 run을 중단하고 재실행 여부를 판단한다.
- 잘못된 테스트 파라미터나 잘못된 대상 환경이 확인되면 즉시 중단한다.

### 5. 결과 기록

반드시 남길 항목:

- 실행 시각
- 시나리오 이름
- 대상 API
- 대상 환경
- 대상 커밋 해시 / 이미지 태그
- RPS / duration / warm-up
- `http_req_failed`
- `p95 / p99`
- DB / Hikari 주요 수치
- Grafana 스크린샷
- 로그/Sentry 이상 유무
- 다음 액션 메모
- 실행 산출물 저장 방식은 [02-성능-실험-산출물-및-기록-규칙.md](02-성능-실험-산출물-및-기록-규칙.md)를 따른다.

## 관측 체크리스트

- `k6` summary 저장
- Grafana latency 패널 캡처
- Grafana error rate 패널 캡처
- DB active connections 패널 캡처
- Hikari pool usage 패널 캡처
- 애플리케이션 error log 확인
- 필요 시 Sentry issue 발생 여부 확인
- 테스트 커밋 해시 기록
- 실행 위치 / 요청 대상 / Grafana 캡처 소스 기록
- 대상 조직 / path prefix / eventType / runId 기록

## 병목 판정 가이드

- `p95/p99` 급등 + Hikari active 고정 상승
  - DB / connection 대기 우선 의심
- 5xx 증가 + timeout 증가
  - 애플리케이션 처리 한계 또는 pool 고갈 우선 의심
- write만 급격히 악화, read는 비교적 안정
  - insert / write path 문제 우선 의심
- mixed에서만 급격히 악화
  - shared resource 경쟁 또는 connection 경합 우선 의심
- DB CPU 상승 + 긴 쿼리 증가
  - 인덱스 / 실행 계획 문제 우선 검토
- DB는 안정적인데 app latency만 상승
  - application layer overhead 또는 flush/save 경로 검토

## 성공 기준

- 시나리오별로 결과가 `수치 + 그래프 + 메모` 형태로 남는다.
- `W1 / R1 / R2 / R3 / M1` 중 최소 1개 이상에서 병목 가설을 설명할 수 있다.
- 이후 개선 전/후 비교에 동일 조건을 다시 사용할 수 있다.
- `summary.json`, `console.log`, `meta.json`이 모두 남는다.
- 최소 Grafana 캡처 패널이 모두 확보된다.
- `04-대규모-부하-테스트-기록.md`에 run 요약이 반영된다.
- baseline인지 재검증인지 문서와 산출물에서 구분된다.

## 실패 기준

- RPS / duration / warm-up이 실행마다 흔들려 비교가 불가능하다.
- 테스트 데이터 식별 규칙이 일관되지 않아 결과 해석이 어렵다.
- Grafana / Loki / k6 결과가 서로 연결되지 않는다.
- baseline인지 개선 후 재검증인지 구분되지 않는다.
