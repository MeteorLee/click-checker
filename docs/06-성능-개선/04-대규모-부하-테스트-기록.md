# 04. 대규모 부하 테스트 기록

## 문서 목적

`03-대규모-부하-테스트-런북.md` 기준으로 수행한 부하 테스트 결과를 회차별로 누적 기록한다.  
이번 문서의 목적은 실행 결과를 단순 나열하는 것이 아니라, baseline과 개선 후 재테스트를 같은 형식으로 비교할 수 있게 증거를 축적하는 데 있다.

## 기록 원칙

- 회차마다 같은 항목을 같은 순서로 기록한다.
- 결과는 반드시 `수치 + 그래프 + 짧은 해석` 형태로 남긴다.
- 실행 시점의 커밋 해시 또는 이미지 태그를 남긴다.
- 테스트 데이터 식별 규칙(path prefix, eventType, runId)을 남긴다.
- baseline인지, 개선 후 재검증인지 먼저 명시한다.
- Grafana, Loki, Sentry에서 확인한 특이사항은 별도 메모로 남긴다.

## Round 1~N 요약

- 초기 baseline과 주요 개선 run이 3개 이상 쌓이면 이 섹션을 먼저 갱신한다.
- 오래된 Round의 상세는 유지하되, 상단에는 요약만 배치한다.
- `Round 1`: 첫 공식 `r50` baseline. `50 RPS`는 통과했지만 `dropped_iterations=24`, `vus max=114`가 보여 higher-rate 확인 필요 신호가 있었다.
- `Round 2`: 첫 `r100` baseline. `p95 6.39s`, `dropped_iterations=468`, `vus max=500`으로 local 기준 한계 구간이 드러났다.
- `Round 3`: `apiKeyLastUsedAt` update 완화와 org 재조회 제거 후 `r50` 재검증. `p95 111ms`, `dropped_iterations=0`, `vus max=29`로 같은 `50 RPS`를 훨씬 가볍게 처리했다.
- `Round 4`: 같은 변경 뒤 `r100` 재검증. 이전 failure 구간이 `p95 471ms`, `dropped_iterations=21`, `vus max=208` 수준의 통과 구간으로 이동했다.
- `Round 5`: `R1` 첫 `r10` dryrun. `p95 14.98s`, `dropped_iterations=88`로 overview read가 예상보다 훨씬 무겁다는 신호가 나왔다.
- `Round 6`: `events (organization_id, occurred_at)` 인덱스 적용 후 `R1 r10` 재검증. `p95 130ms`, `dropped_iterations=0`으로 첫 read baseline이 안정 구간으로 이동했다.
- `Round 7`: 같은 인덱스 상태에서 `R1 r30` 실행. `p95 6.52s`, `dropped_iterations=66`으로 여전히 한계 구간이었다.
- `Round 8`: overview 내부 raw path/raw event type 집계 재사용 후 `R1 r30` 재검증. `p95 5.09s`, `dropped_iterations=40`으로 일부 개선됐지만 아직 threshold는 넘지 못했다.
- `Round 9`: `R2 r10` 초기 baseline. 같은 read dataset을 재사용하는 `routes` 집계 read는 `p95 37.55ms`로 매우 가볍게 시작했고, 집계형 read가 요약형 read보다 훨씬 가볍다는 점이 드러났다.
- `Round 10`: `R2 r30` baseline. `p95 76.23ms`, `dropped_iterations=0`으로 여전히 안정 구간이었고, route group-by / topN은 local 기준 아직 충분한 여유가 있었다.
- `Round 11`: `R2 r50` 한계 구간 확인. `p95 5.98s`, `dropped_iterations=119`, `vus max=181`로 threshold fail이 발생해, `routes` read의 local 기준 새 한계 구간이 `30과 50 사이`로 좁혀졌다.
- `Round 12`: `R3 r10` 유효 baseline. 첫 `bucket` 계약 mismatch 수정 뒤 `p95 67.17ms`, `dropped_iterations=0`으로 time-buckets read의 첫 안정 구간이 확보됐다.
- `Round 13`: `R3 r30` 한계 구간 확인. `p95 4.14s`, `dropped_iterations=147`, `vus max=178`로 threshold fail이 발생해, time-bucket read의 local 기준 한계 구간이 `10과 30 사이`로 좁혀졌다.
- `Round 14`: `M1 w10/r10` mixed baseline. 같은 snapshot에서 write/read를 50:50으로 동시에 붙였을 때 read `p95 941ms`, write `p95 468ms`, `dropped_iterations=0`으로 첫 mixed 안정 구간이 확보됐다.
- `Round 15`: `M1 w20/r20` 한계 구간 확인. read `p95 3.78s`, write `p95 3.79s`, `dropped_iterations=77`, `vus max=153`으로 mixed 한계 구간이 `10/10과 20/20 사이`로 좁혀졌다.
- `Round 16`: `W1 r100` prod-direct 검증. EC2 내부 direct port 기준 `p95 22.19ms`, `p99 138.67ms`, 실패율 `0%`로 local에서 확보한 write 안정 구간이 실제 인프라에서도 그대로 재현됐다.
- `Round 17`: `R1 r10` prod-direct 검증. EC2 / RDS 조건에서도 overview read `p95 96.39ms`, `p99 138.28ms`, 실패율 `0%`로 local 안정 구간이 그대로 유지됐다.
- `Round 18`: `M1 w10/r10` prod-direct 검증. read `p95 111.53ms`, write `p95 44.78ms`, 실패율 `0%`로 mixed 첫 안정 구간도 실제 인프라에서 재현됐다.
