# 04. 대규모 부하 테스트 기록

## 문서 목적

`03-대규모-부하-테스트-런북.md` 기준으로 수행한 부하 테스트 결과를 회차별로 누적 기록한다.  
이번 문서의 목적은 실행 결과를 단순 나열하는 것이 아니라, baseline과 개선 후 재테스트를 같은 형식으로 비교할 수 있게 증거를 축적하는 데 있다.

## 기록 원칙

- 회차마다 같은 항목을 같은 순서로 기록한다.
- 결과는 반드시 `수치 + 그래프 + 짧은 해석` 형태로 남긴다.
- 실행 시점의 커밋 해시 또는 이미지 태그를 남긴다.
- 테스트 데이터 식별 규칙(path prefix, eventType, runId)을 남긴다.
- baseline인지, 개선 후 재검증인지 먼저 명시한다.
- Grafana, Loki, Sentry에서 확인한 특이사항은 별도 메모로 남긴다.

## Round 1~N 요약

- 초기 baseline과 주요 개선 run이 3개 이상 쌓이면 이 섹션을 먼저 갱신한다.
- 오래된 Round의 상세는 유지하되, 상단에는 요약만 배치한다.
- `Round 1`: 첫 공식 `r50` baseline. `50 RPS`는 통과했지만 `dropped_iterations=24`, `vus max=114`가 보여 higher-rate 확인 필요 신호가 있었다.
- `Round 2`: 첫 `r100` baseline. `p95 6.39s`, `dropped_iterations=468`, `vus max=500`으로 local 기준 한계 구간이 드러났다.
- `Round 3`: `apiKeyLastUsedAt` update 완화와 org 재조회 제거 후 `r50` 재검증. `p95 111ms`, `dropped_iterations=0`, `vus max=29`로 같은 `50 RPS`를 훨씬 가볍게 처리했다.
- `Round 4`: 같은 변경 뒤 `r100` 재검증. 이전 failure 구간이 `p95 471ms`, `dropped_iterations=21`, `vus max=208` 수준의 통과 구간으로 이동했다.
- `Round 5`: `R1` 첫 `r10` dryrun. `p95 14.98s`, `dropped_iterations=88`로 overview read가 예상보다 훨씬 무겁다는 신호가 나왔다.
- `Round 6`: `events (organization_id, occurred_at)` 인덱스 적용 후 `R1 r10` 재검증. `p95 130ms`, `dropped_iterations=0`으로 첫 read baseline이 안정 구간으로 이동했다.
- `Round 7`: 같은 인덱스 상태에서 `R1 r30` 실행. `p95 6.52s`, `dropped_iterations=66`으로 여전히 한계 구간이었다.
- `Round 8`: overview 내부 raw path/raw event type 집계 재사용 후 `R1 r30` 재검증. `p95 5.09s`, `dropped_iterations=40`으로 일부 개선됐지만 아직 threshold는 넘지 못했다.

### 1차 해석

- 병목 의심 지점: path/event type 집계 중복은 실제 비용이었지만, 이를 줄여도 `30 RPS` 전체를 안정 구간으로 옮기기엔 아직 부족하다.
- 근거 지표: `p95 6.52s -> 5.09s`, `dropped_iterations 66 -> 40`, `vus max 126 -> 96`으로 개선됐지만 여전히 `threshold_fail`이다.
