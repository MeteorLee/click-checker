# 04. 대규모 부하 테스트 기록

## 문서 목적

`03-대규모-부하-테스트-런북.md` 기준으로 수행한 부하 테스트 결과를 회차별로 누적 기록한다.  
이번 문서의 목적은 실행 결과를 단순 나열하는 것이 아니라, baseline과 개선 후 재테스트를 같은 형식으로 비교할 수 있게 증거를 축적하는 데 있다.

## 기록 원칙

- 회차마다 같은 항목을 같은 순서로 기록한다.
- 결과는 반드시 `수치 + 그래프 + 짧은 해석` 형태로 남긴다.
- 실행 시점의 커밋 해시 또는 이미지 태그를 남긴다.
- 테스트 데이터 식별 규칙(path prefix, eventType, runId)을 남긴다.
- baseline인지, 개선 후 재검증인지 먼저 명시한다.
- Grafana, Loki, Sentry에서 확인한 특이사항은 별도 메모로 남긴다.

## 현재 기준 baseline

- 기준 baseline run: `w1-20260319-132007-r50`
- 시나리오: `W1`
- 커밋 / 이미지: `d7bad34` / local docker build
- 환경: `local`
- 기준 이유: 첫 정식 `W1` 50 RPS baseline이며, 산출물 3종과 renderer PNG 캡처가 모두 남은 최초 공식 run이다.

원칙:

- baseline이 바뀌면 이 섹션부터 먼저 갱신한다.
- 이후 run은 가능하면 `compareTo` 기준으로 이 baseline과 연결한다.

## Round 1~N 요약

- 초기 baseline과 주요 개선 run이 3개 이상 쌓이면 이 섹션을 먼저 갱신한다.
- 오래된 Round의 상세는 유지하되, 상단에는 요약만 배치한다.

## Round 1 - Baseline

### 실행 정보

- 실행 일시: `2026-03-19 13:22:21 ~ 13:35:49 (+09:00)`
- 대상 환경: `local`
- 대상 커밋 해시 / 이미지 태그: `d7bad34` / local docker build
- 회차 성격: baseline
- 시나리오: `W1`

### 이전 Round 대비 변경점

- 첫 공식 baseline run
- 드라이런 대비 `warm-up 2m + main 10m + cooldown 1m` 정식 조건 적용
- `capture.mode=scripted` 기준 renderer PNG 자동 캡처 적용

### 부하 조건

- warm-up: `2m`
- main run: `10m`
- cooldown: `1m`
- RPS / 단계 상승: `50 RPS` 고정 run

### 테스트 데이터 식별 규칙

- organization: `orgId=10`, `perf-w1-r50-20260319132007`
- path prefix: `/loadtest/w1`
- eventType: `loadtest_write`
- runId: `w1-20260319-132007-r50`

### k6 결과

- `http_req_failed`: `0.00%`
- `p95`: `696.45ms`
- `p99`: `1.77s` (`console.log` threshold output 기준)
- 평균 RPS: `48.48 req/s`
- 기타 요약: `dropped_iterations=24`, `iterations=29977`, `vus max=114`, `max latency=4.96s`
- 산출물 디렉터리: `artifacts/perf/w1/w1-20260319-132007-r50/`
- `summary.json`: 생성됨
- `console.log`: 생성됨
- `meta.json`: 생성됨

### Grafana 확인

- latency: renderer PNG 생성 완료, `p95/p99` 추세 확인 가능
- error rate: `0%`로 기록됨
- DB active connections: renderer PNG 생성 완료
- Hikari pool usage: renderer PNG 생성 완료
- 캡처 시간 범위: `start-2m ~ end+2m`
- 추가 캡처 메모: `dashboard-overview.png`, `ingest-rps.png`, `ingest-latency.png`, `db-connections.png`, `db-pool.png` 자동 생성

### 로그 / Sentry 특이사항

- app log: 실행 산출물 기준 별도 오류 없음
- nginx log: local baseline이라 해당 없음
- Sentry: 확인 대상 아님

### 1차 해석

- 병목 의심 지점: `50 RPS`에서는 write path가 아직 포화 전 단계로 보이지만, 후반부 VU 급증 구간이 있어 지연 스파이크는 존재한다.
- 근거 지표: 실패율 `0%`, `p95 696ms`, `p99 1.77s`로 threshold 이내였고, `dropped_iterations=24`, `vus max=114`가 발생했다.
- 반례 / 불확실성: local single-tenant 기준이라 prod direct/public, 실제 RDS 지연, multi-tenant 영향은 반영되지 않았다.

### 다음 액션

- P0: 같은 조건으로 `r100` baseline 실행
- P1: `r200` baseline 실행 후 `50/100/200` 전후 비교
- P2: higher rate에서 DB/Hikari PNG와 dropped iterations 상관관계 확인

### 실행 신뢰도

- 신뢰도:
  - `high`
- 이유: 정식 조건 run이 끝까지 성공했고, `summary.json`, `console.log`, `meta.json`, renderer PNG가 모두 생성됐다.
