# 02. 성능 실험 산출물 및 기록 규칙

## 문서 목적

대규모 부하 테스트 결과를 실행마다 같은 형식으로 남기기 위한 산출물 저장 규칙을 정리한다.  
이번 문서의 목적은 "실험을 많이 했는데 비교가 안 되는 상태"를 막고, baseline과 개선 후 결과를 같은 기준으로 재검토 가능하게 만드는 데 있다.

## 기본 원칙

- 실행 산출물은 문서와 분리 저장한다.
- 같은 run의 산출물은 한 폴더에 모은다.
- summary json만이 아니라 실행 로그와 메타데이터도 같이 남긴다.
- 실패한 실행도 버리지 않는다.
- Grafana 캡처는 패널과 시간 범위를 고정한다.

## 산출물 저장 위치

### 기본 경로

- `artifacts/perf/<scenario>/<runId>/`

### W1 기준 예시

- `artifacts/perf/w1/w1-20260319-143000-r50/`

원칙:

- `scenario`와 `runId`가 디렉터리만 봐도 드러나야 한다.
- 하나의 run에 대한 증거가 한 디렉터리에 모여야 한다.

## 저장 파일 기준

### 필수

- `summary.json`
  - `k6` 요약 결과
- `console.log`
  - 실행 중 콘솔 출력
- `meta.json`
  - 실행 메타데이터
- `dashboard-overview.png`
  - overview 대시보드 전체 렌더링 결과

### 권장

- `command.txt`
  - 실제 실행 명령
- Grafana 스크린샷
  - `ingest-rps.png`
  - `ingest-latency.png`
  - `db-connections.png`
  - `db-pool.png`

## `meta.json` 기준

최소 포함 항목:

- `scenario`
- `env`
- `runId`
- `status`
- `isBaseline`
- `compareTo`
- `commit`
- `imageTag`
- `load`
- `dataset`
- `auth` (필요한 경우)
- `capture`

권장 값 예시:

- `status`
  - `success`
  - `threshold_fail`
  - `aborted`
  - `invalid`
- `isBaseline`
  - `true | false`
- `compareTo`
  - baseline이면 `null`
  - 개선 run이면 비교 대상 `runId`
- `env`
  - `name`
  - `executionLocation`
  - `prepareBaseUrl` (필요한 경우)
  - `baseUrl`
  - `grafanaBaseUrl`
  - `appCount`
  - `dbInstance`
- `load`
  - `rate`
  - `warmup`
  - `duration`
  - `cooldown`
- `dataset`
  - `orgId`
  - `orgName`
  - `pathPrefix`
  - `eventType`
  - `existingUserPoolSize`
  - `existingUserRatio`
  - `pathCount`
- `auth`
  - `apiKey`
  - `apiKeyPrefix`
- `capture`
  - `mode`
  - `timeRange`
  - `status`
  - `fromMs`
  - `toMs`
  - `generatedAt`

원칙:

- 나중에 summary만 봐도 설명이 안 되는 조건은 `meta.json`에 넣는다.
- 적어도 "어떤 환경에서 어떤 조건으로 돌렸는지"는 `meta.json`만 봐도 알 수 있어야 한다.
- 특히 "어디서 실행했는지"와 "어느 Grafana를 보고 캡처했는지"는 분리해서 남긴다.
- `meta.json`에는 실행 조건과 상태만 넣는다.
- `p95`, `p99`, 에러율 같은 결과값은 `meta.json`에 넣지 않는다.
- 결과값은 `summary.json`, `console.log`, 기록 문서에서 관리한다.
- 실행 스크립트가 실제 인증 정보를 읽어야 하는 경우, `meta.json`에 `auth` 섹션을 둘 수 있다.
- plain API key가 `meta.json`에 들어가는 경우 파일 권한은 `600`으로 제한한다.

## 실행 로그 보존 기준

- stdout/stderr는 `console.log`로 남긴다.
- threshold 실패나 경고 메시지도 그대로 남긴다.
- 실패 실행도 삭제하지 않는다.

이유:

- summary에는 실행 중 경고나 threshold 맥락이 부족할 수 있다.
- 실패 run도 병목 해석의 증거가 된다.

## baseline / 비교 run 기준

- baseline run은 `meta.json`에 `isBaseline=true`로 명시한다.
- 개선 후 재검증 run은 `compareTo=<baseline-runId>`를 넣는다.

원칙:

- baseline을 파일 시스템 alias나 symlink로 관리하지 않는다.
- baseline / 비교 관계는 `meta.json` 필드로 명시한다.

## Grafana 캡처 기준

### W1 최소 고정 패널

- `Ingest RPS by App`
- `Ingest Latency p95 / p99`
- `DB Active Connections`
- `DB Pool Usage`

### 파일명 규칙

- `ingest-rps.png`
- `ingest-latency.png`
- `db-connections.png`
- `db-pool.png`

### 시간 범위 기준

- 실행 시작 `2분 전`부터 실행 종료 `2분 후`까지를 기본 범위로 본다.

### 캡처 방식 기준

- `capture.mode`
  - `manual`
  - `scripted`

원칙:

- 수동 캡처인지 자동 스크립트 캡처인지 `meta.json`에 남긴다.
- 캡처 방식이 다르면 그래프 비교 해석도 달라질 수 있다.
- `scripted`인 경우 Grafana renderer를 통해 `dashboard-overview.png`와 핵심 panel PNG를 생성한다.
- local에서 prod direct / prod public 부하를 실행하는 경우, 캡처 대상 Grafana가 local인지 prod인지 별도로 명시한다.
- `scripted` 캡처는 필요하면 `GRAFANA_BASE_URL`을 prod Grafana나 SSH 터널 기준 주소로 바꿔 실행한다.

원칙:

- 패널뿐 아니라 시간 범위도 고정한다.
- 그래야 baseline과 개선 후 그래프를 같은 조건으로 비교할 수 있다.

## 실패 실행 보존 규칙

- threshold 실패
- 잘못된 파라미터
- 중간 중단
- 예상보다 낮은 처리량

위 경우도 실행 산출물을 그대로 남긴다.

이유:

- 실패 실행도 원인 분석 증거다.
- baseline 품질과 실험 설계 문제를 설명하는 근거가 된다.

## 한 줄 해석 기준

각 run의 요약은 아래 세 줄 구조를 권장한다.

- 병목 의심 지점:
- 근거 지표:
- 다음 액션:

예:

- 병목 의심 지점: DB connection pool 포화
- 근거 지표: `p99` 급증, DB Active Connections 고점 유지, Pool Usage 상한 근접
- 다음 액션: connection pool 및 write 경로 lookup 횟수 확인
