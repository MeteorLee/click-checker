# 00. 시나리오 문서 템플릿

## 문서 목적

성능 개선 단계에서 새로운 시나리오 문서를 추가할 때 빠뜨리면 안 되는 섹션을 고정한다.  
이번 템플릿의 목적은 `W1`에서 사용한 단순화나 가정이 이후 시나리오(`R1`, `R2`, `R3`, `M1` 등)에 무의식적으로 복붙되는 것을 막고, 각 시나리오가 무엇을 재사용하고 무엇을 바꾸는지 명확히 드러내는 데 있다.

## 사용 원칙

- 신규 시나리오 문서는 이 템플릿을 기준으로 작성한다.
- 섹션 순서를 바꿀 수는 있지만, 필수 섹션은 생략하지 않는다.
- `W1`에서 쓴 가정을 그대로 가져올 때도 "재사용"으로 명시한다.
- `W1`에서 쓰지 않은 변수가 새로 중요해지면 반드시 별도 섹션에서 설명한다.

## 필수 섹션

### 1. 문서 목적

- 이번 시나리오가 무엇을 검증하려는지
- 어떤 병목 가설을 보기 위한 것인지

### 2. 적용 범위

- 대상 시나리오 이름
- 대상 API
- 이번 문서가 다루는 범위

### 3. 공통 기준에서 재사용하는 항목

- [03-대규모-부하-테스트-런북.md](03-대규모-부하-테스트-런북.md) 또는 [02-성능-실험-산출물-및-기록-규칙.md](02-성능-실험-산출물-및-기록-규칙.md)에서 그대로 재사용하는 기준을 적는다.

예:

- 실행 환경 단계 구분
- 산출물 저장 규칙
- 기록 문서 반영 규칙
- Grafana 캡처 시간 범위

### 4. 이전 시나리오와 다른 점

- 직전 시나리오와 비교해 무엇이 달라졌는지 명시한다.

예:

- `W1`은 single-tenant write 기준이지만 `M1`은 write/read 동시 경합을 본다.
- `R1`은 데이터셋 범위와 조회 기간이 주요 변수다.

### 5. 이번 시나리오에서 새로 중요해지는 변수

- 이번 시나리오에서만 중요하거나, 이전보다 더 중요해지는 축을 적는다.

예:

- 조회 기간
- bucket 수
- topN
- read/write 비율
- tenant 혼합

### 6. 환경 고정값 / 변경값

- 이번 실험에서 결과 비교를 위해 고정한 값과, 의도적으로 바꾼 값을 분리해서 적는다.

예:

- 고정값:
  - app version / commit hash
  - DB instance class
  - Hikari 설정
  - JVM 옵션
  - 데이터 규모
- 변경값:
  - RPS
  - tenant 수
  - 조회 기간
  - topN
  - read/write 비율

원칙:

- 이 섹션만 봐도 "무엇이 같고 무엇이 달라졌는지"가 보여야 한다.
- 결과 차이를 해석할 때 실험 변수를 빠르게 복구할 수 있어야 한다.

### 7. 이번 시나리오에서 일부러 단순화한 항목

- 해석 가능성을 위해 이번에도 일부러 줄인 부분을 적는다.

예:

- eventType 종류를 1개로 제한
- tenant 수를 제한
- payload 크기를 고정

### 8. 이전 시나리오의 어떤 가정을 가져오지 않는지

- `W1`의 전용 가정 중 이번 시나리오에서는 그대로 쓰지 않는 항목을 명시한다.

예:

- single-tenant 고정
- path 6개 고정
- 기존 사용자 80 / 신규 20 혼합

원칙:

- 이 섹션은 "W1 복붙 방지" 용도다.
- 그대로 가져오지 않는 가정은 반드시 적는다.

### 9. 해석 시 주의사항

- 이번 결과를 어디까지 일반화할 수 있고, 어디부터는 조심해서 읽어야 하는지 적는다.

예:

- 이번 결과는 single-tenant 조건에서만 해석한다.
- 캐시 미도입 상태의 기준선이다.
- 응답 크기 차이는 이번 시나리오에서 통제하지 않았다.
- 데이터셋 규모 변화에 대한 일반화는 제한적이다.

원칙:

- 숫자만 남기지 말고, 그 숫자를 어디까지 믿어야 하는지 같이 남긴다.
- 이후 종합 문서에서 과한 일반화를 막는 안전장치로 사용한다.

### 10. 실행 조건

- executor
- RPS 또는 arrival rate
- duration
- warm-up
- VU 기준

### 11. 입력 데이터 규칙

- path
- eventType
- user mix
- dataset 조건
- seed 방식

read-heavy 시나리오에서는 필요하면 아래도 같이 적는다.

- 조회 구간 길이
- 응답 row 수 예상 범위
- cardinality 축(path 수, eventType 수, user 수)
- hot tenant / cold tenant 분포
- cache warm 여부
- seed dataset 생성 시점과 볼륨

### 12. threshold 기준

- `http_req_failed`
- `p95 / p99`
- 시나리오별 실패 감지선

### 13. 중단 조건

- 어떤 경우에 현재 run을 실패로 간주하고 즉시 중단할지 적는다.

예:

- 5xx 비율이 일정 시간 이상 임계치 초과
- DB connection pool 고갈 지속
- 앱 재시작 / OOM 발생
- Grafana / Prometheus 수집 이상

원칙:

- "유효한 실패"와 "실험 오류"를 구분할 수 있어야 한다.
- run 중단 기준이 없으면 결과 해석이 과도하게 흔들릴 수 있다.

### 14. 산출물 및 기록 규칙

- 어떤 파일이 생성되는지
- 어떤 패널을 캡처하는지
- 기록 문서 어디에 반영하는지

### 15. 실험 후 판단 기준

- 이번 결과를 보고 다음에 어떤 방향으로 갈지 적는다.

예:

- `p95`는 통과하지만 DB pool usage가 지속적으로 높으면 DB / Hikari 튜닝으로 간다.
- 앱 CPU가 먼저 포화되면 batch insert, async 처리, app scaling 후보를 검토한다.
- read 쿼리만 병목이면 인덱스 / 집계 구조 / cache 후보를 분리 검토한다.

원칙:

- 문서를 단순 실행 설명서가 아니라, 다음 액션을 고르는 의사결정 문서로 사용한다.
- 종합 문서에서 판단 근거를 재사용할 수 있어야 한다.

### 16. 다음 문서

- 준비/실행 구조
- 산출물 규칙
- 기록 문서

## 권장 체크 문장

새 시나리오 문서를 쓸 때 아래 질문에 답이 바로 보여야 한다.

- 이 시나리오는 무엇을 보기 위한 것인가?
- 공통 기준 중 무엇을 그대로 쓰는가?
- 직전 시나리오와 무엇이 다른가?
- 이번 시나리오에서만 중요해지는 변수는 무엇인가?
- 이번 실험에서 고정한 값과 바꾼 값은 무엇인가?
- 이 결과를 어디까지 믿어야 하는가?
- 언제 테스트를 중단해야 하는가?
- 이 결과를 보고 다음에 어떤 결정을 내려야 하는가?
- `W1`의 어떤 단순화를 그대로 가져오지 않는가?
