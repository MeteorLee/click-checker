# 10. W1 쓰기 부하 시나리오 명세

## 문서 목적

`W1`는 `POST /api/events` 쓰기 경로의 첫 baseline을 확보하기 위한 시나리오다.  
이번 문서는 `W1`에서 어떤 환경, 어떤 RPS, 어떤 입력 데이터 규칙으로 write-heavy 부하를 걸지 고정하고, 이후 인덱스/쿼리/쓰기 경로 개선 전후를 같은 조건으로 비교할 수 있게 만드는 데 목적이 있다.

## 적용 범위

- 대상 시나리오:
  - `W1`
- 대상 API:
  - `POST /api/events`
- 이번 문서가 다루는 범위:
  - baseline용 write-heavy 입력 조건
  - 환경 기준
  - 요청 헤더 / payload / 사용자 / path 규칙
  - threshold / 중단 조건
  - 실험 후 판단 기준

## 공통 기준에서 재사용하는 항목

- [02-성능-실험-산출물-및-기록-규칙.md](02-성능-실험-산출물-및-기록-규칙.md)의 산출물 저장 규칙을 그대로 따른다.
- baseline, retest, dry run 구분은 [04-대규모-부하-테스트-기록.md](04-대규모-부하-테스트-기록.md)에 같은 형식으로 기록한다.

## 이전 시나리오와 다른 점

- 해당 없음.
- `W1`는 성능 단계의 첫 공식 시나리오이므로, 직전 시나리오 비교보다 "첫 baseline을 만들기 위한 단순화"에 더 초점을 둔다.

## 이번 시나리오에서 새로 중요해지는 변수

- 고정 RPS별 쓰기 처리량 (`50 / 100 / 200`)
- write 경로 지연 (`p95 / p99`)
- DB active connections
- Hikari pool usage
- 기존 사용자 lookup과 신규 사용자 생성이 섞일 때의 비용
- 같은 `POST /api/events`라도 path cardinality가 insert 비용에 미치는 영향

## 환경 고정값 / 변경값

### 고정값

- 시나리오:
  - `W1`
- 대상 API:
  - `POST /api/events`
- executor:
  - `constant-arrival-rate`
- warm-up:
  - `2m`
- main run:
  - `10m`
- cooldown:
  - `1m`
- 조직 수:
  - run당 `1개 organization`
- path count:
  - `6개`
- eventType:
  - `loadtest_write`
- 기존 사용자 풀:
  - `1000명`
- 사용자 비율:
  - 기존 `80%`, 신규 `20%`
- 산출물 저장 규칙:
  - `artifacts/perf/w1/<runId>/`

### 변경값

- 환경:
  - `local` -> 이후 `prod direct`
- run별 rate:
  - `50`
  - `100`
  - `200`
- `runId`
- `organization`
- `API key`

원칙:

- baseline과 retest를 비교할 때 변경값은 최소화한다.
- 결과 차이 해석에 직접 쓰이는 값만 변경값으로 둔다.

## 이번 시나리오에서 일부러 단순화한 항목

- 첫 baseline은 single-tenant 기준으로만 본다.
- eventType을 `loadtest_write` 하나로 고정한다.
- path를 6개 고정 세트로 제한한다.
- payload 크기와 필드 종류를 최소 메타데이터 수준으로 제한한다.
- 복수 executor나 mixed load는 다루지 않는다.

## 이전 시나리오의 어떤 가정을 가져오지 않는지

- 해당 없음.
- 다만 이후 `R1`, `R2`, `R3`, `M1`에서는 아래 W1 전용 가정을 그대로 가져오지 않는다.
  - single-tenant 고정
  - path 6개 고정
  - 기존 사용자 80 / 신규 20 비율
  - eventType 1개 고정

## 해석 시 주의사항

- 이번 결과는 single-tenant write 기준에서만 해석한다.
- 캐시 미도입 상태의 baseline이다.
- nginx / public ingress 영향을 제외한 write path 해석은 `local` 또는 `prod direct`에서만 가능하다.
- path 수, tenant 수, payload 다양성이 늘어나면 결과를 그대로 일반화할 수 없다.
- 첫 baseline은 "실서비스 완벽 재현"보다 "병목 원인 분리"를 우선한다.

## 실행 조건

### 환경 단계

#### 1차

- 환경:
  - `local`
- 목적:
  - 스크립트 / 산출물 / 관측 흐름 검증
  - 첫 baseline 확보

#### 2차

- 환경:
  - `prod direct`
- 목적:
  - 실제 EC2 / RDS 조건에서 local 결과 재현 여부 확인

### executor

- `constant-arrival-rate`

이유:

- 전/후 비교에서 같은 RPS 조건을 재사용하기 쉽다.
- 응답 지연으로 실제 처리량이 흔들리는 상황을 더 명확히 드러낸다.

### run 단위

- `50 RPS`
- `100 RPS`
- `200 RPS`

원칙:

- `50 -> 100 -> 200`을 한 run에 섞지 않는다.
- 각 run은 별도 조직, 별도 runId, 별도 산출물로 남긴다.

### 시간 조건

- warm-up:
  - `2m`
- main run:
  - `10m`
- cooldown:
  - `1m`

### VU 기준

#### `50 RPS`

- `preAllocatedVUs = 100`
- `maxVUs = 300`

#### `100 RPS`

- `preAllocatedVUs = 200`
- `maxVUs = 500`

#### `200 RPS`

- `preAllocatedVUs = 400`
- `maxVUs = 800`

원칙:

- write-heavy는 지연이 커질 수 있으므로 초반에는 여유 있게 잡는다.
- baseline과 재검증은 같은 VU 기준을 유지한다.

## 입력 데이터 규칙

### 조직 기준

- 첫 `W1` baseline은 run당 `1개 organization` 기준으로 본다.
- 각 run은 fresh org를 사용한다.

예:

- `perf-w1-r50`
- `perf-w1-r100`
- `perf-w1-r200`

### 요청 헤더 기준

- `X-API-Key`
- `Content-Type: application/json`
- `X-Request-Id`

`X-Request-Id` 원칙:

- `runId`와 iteration이 드러나는 규칙 값을 사용한다.
- app log / Loki 추적 가능성을 위해 넣는다.

### 이벤트 식별 규칙

- `eventType = loadtest_write`
- `pathPrefix = /loadtest/w1`
- `runId = 실행별 고유 값`

원칙:

- 모든 테스트 데이터는 운영 데이터와 구분 가능해야 한다.
- `eventType`, `path`, `payload.runId` 기준으로 추적 가능해야 한다.

### path 규칙

고정 세트 6개를 순환한다.

- `/loadtest/w1/page/1`
- `/loadtest/w1/page/2`
- `/loadtest/w1/post/1`
- `/loadtest/w1/post/2`
- `/loadtest/w1/product/10`
- `/loadtest/w1/product/11`

원칙:

- 단일 path보다 약간 현실적이어야 한다.
- 완전 랜덤보다 결과 해석과 재현성이 좋아야 한다.
- cardinality를 과도하게 키우지 않는다.

### 사용자 혼합 규칙

- 기존 사용자:
  - `80%`
- 신규 사용자:
  - `20%`

#### 기존 사용자 풀

- `existingUserPoolSize = 1000`
- 규칙:
  - `existing-0`
  - `existing-1`
  - ...
  - `existing-999`

#### 신규 사용자 규칙

- `lt-${runId}-new-${iteration}`

원칙:

- 완전 랜덤 문자열보다 run별 추적과 재현성이 좋아야 한다.

### payload 기준

- 첫 baseline은 payload 구조를 단순하게 유지한다.
- payload에는 최소 메타데이터만 넣는다.

초기 메타데이터:

- `source = k6-w1`
- `scenario = W1`
- `runId`
- `vu`
- `iter`

## threshold 기준

- `http_req_failed < 1%`
- `p95 < 5000ms`
- `p99 < 8000ms`

주의:

- 이번 threshold는 SLA가 아니라 baseline용 실패 감지선이다.
- 핵심 목적은 전/후 비교이며, threshold는 명백한 이상 상태를 빨리 감지하는 용도다.

## 중단 조건

- 5xx 비율이 임계치를 지속 초과하면 현재 run을 중단한다.
- 앱 재시작, OOM, readiness 실패가 발생하면 현재 run을 무효 처리한다.
- Prometheus / Grafana 수집 이상으로 관측이 불가능하면 현재 run을 중단하고 재실행 판정한다.
- 잘못된 테스트 파라미터 투입이 확인되면 즉시 중단한다.

## 실험 후 판단 기준

- `p95 / p99` 급등과 함께 DB active connections, pool usage가 같이 상승하면 DB / connection 대기를 먼저 의심한다.
- 실패율은 낮지만 latency만 높으면 app layer overhead, lookup 비용, flush/save 구조를 먼저 본다.
- `50 RPS`는 안정적이고 `100` 또는 `200`에서 급격히 악화되면 그 구간을 1차 임계점 후보로 본다.
- baseline에서 이미 위험 신호가 명확하면 다음 단계는 인덱스 / EXPLAIN / write 경로 해부로 간다.
