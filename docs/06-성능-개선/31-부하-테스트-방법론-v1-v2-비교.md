# 31. 부하 테스트 방법론 v1 v2 비교

## 문서 목적

이번 문서는 `W1 / R1 / R2 / R3 / M1`로 진행한 v1과,  
이후 설계하는 `W2 / R4 / R5 / R6 / M2` 중심 v2가 무엇이 다른지 비교하기 위한 문서다.

핵심 목적은 아래 두 가지다.

- v1이 실패한 것이 아니라 **다른 질문에 답한 시나리오**였음을 분명히 남긴다.
- v2가 왜 필요한지, 그리고 어떤 현실성을 새로 가져오는지 설명한다.

## 한 줄 비교

- v1:
  - 병목 탐색형
- v2:
  - 현실형 사용 패턴 검증형

## 비교 표

| 항목 | v1 | v2 |
| --- | --- | --- |
| 핵심 질문 | 어디가 먼저 느린가 | 실제 서비스형 패턴에서도 어떤 구조가 먼저 흔들리는가 |
| tenant 모델 | single-tenant 중심 | `6개 org / 50-30-20` |
| read 방식 | 개별 API 단독 반복 | 계층형 read mix |
| mixed 방식 | `M1 50:50` 경쟁 탐지용 | `M2 95:5` realistic mixed |
| read 범위 | `overview`, `routes`, `time-buckets`를 따로 봄 | 집계(`R4`)와 분석(`R5`)을 현실형 mix로 봄 |
| 분석 API 취급 | 별도 성능 시나리오 없음 | `R5`로 명시적으로 포함 |
| 목적 | baseline / limit / 저위험 개선 검증 | realistic baseline / 구조 개선 우선순위 검증 |

## v1이 맡았던 역할

v1은 아래 역할을 충분히 수행했다.

- 재현 가능한 baseline 구축
- write / read / mixed 병목 분리
- low-risk 개선 전후 비교
- `local -> prod direct -> prod public` 확인

즉 v1은 "충분히 단순했기 때문에" 해석이 쉬웠다.

## v2가 새로 가져오는 것

v2는 아래 현실성을 새로 가져온다.

### 1. multi-tenant

- 한 조직만 보는 것이 아니라 여러 조직이 동시에 섞이는 조건

### 2. read mix

- `overview`만 반복하지 않고 집계 / 분석 화면이 섞이는 조건

### 3. realistic mixed

- write-heavy 성격에 더 가까운 `95:5`

### 4. 방법론 보강

- saturation 판단
- bottleneck 분류
- 개선 비용 대비 효과 기록

## v1을 버리지 않는 이유

v2를 만든다고 해서 v1을 폐기하지 않는다.

이유:

- v1은 여전히 원인 분리와 저위험 개선 검증에 좋다.
- v2는 현실형 검증에는 좋지만, 해석 난도가 더 높다.

즉 앞으로는 아래처럼 역할을 나눈다.

- v1:
  - 진단 / baseline / 저위험 개선 검증
- v2:
  - 현실형 baseline / realistic mixed / 구조 개선 우선순위 판단

## v2 시나리오 관계

- `W2`
  - multi-tenant write
- `R4`
  - 집계 read mix
- `R5`
  - 분석 read mix
- `R6`
  - `R4 + R5` 공용 read profile
- `M2`
  - `W2 + R6 v2` realistic mixed

## 이번 문서의 결론

v1과 v2의 관계는 아래 한 줄로 정리할 수 있다.

> v1은 병목을 찾는 체계이고, v2는 현실형 트래픽 조건에서 구조 개선 우선순위를 다시 판단하기 위한 체계다.
