# 14. R1 개요 조회 시나리오 명세

## 문서 목적

`R1`은 `GET /api/v1/events/analytics/aggregates/overview` read path의 첫 baseline을 확보하기 위한 시나리오다.  
이번 문서는 `overview` 조회를 어떤 데이터셋, 어떤 기간, 어떤 RPS 조건으로 반복 실행할지 고정하고, 이후 `R2`, `R3`, `M1`과 비교 가능한 첫 read 기준선을 만드는 데 목적이 있다.

## 적용 범위

- 대상 시나리오:
  - `R1`
- 대상 API:
  - `GET /api/v1/events/analytics/aggregates/overview`
- 이번 문서가 다루는 범위:
  - baseline용 read-heavy 조회 조건
  - read 전용 데이터셋 가정
  - query 범위와 필터 규칙
  - threshold / 중단 조건
  - 실험 후 판단 기준

## 이전 시나리오와 다른 점

- `W1`은 run마다 fresh org를 만들었지만, `R1`은 **고정 read 데이터셋 org 하나를 재사용**한다.
- `W1`은 입력 payload와 사용자 혼합이 핵심 변수였지만, `R1`은 **데이터셋 크기와 조회 범위**가 핵심 변수다.
- `W1`은 write path의 처리 비용을 봤고, `R1`은 **같은 overview 결과를 반복 조회할 때의 응답 지연과 DB read 비용**을 본다.

## 이번 시나리오에서 새로 중요해지는 변수

- read 전용 데이터셋 크기
  - total events
  - identified users
  - raw path 수
  - raw event type 수
- 조회 구간 길이
- `overview` 내부의 distinct user 계산 비용
- `topRoutes`, `topEventTypes` 후처리 비용
- 직전 동일 길이 구간 비교 비용
- route match coverage / event type mapping coverage 해석 가능성

## 환경 고정값 / 변경값

### 고정값

- 시나리오:
  - `R1`
- 대상 API:
  - `GET /api/v1/events/analytics/aggregates/overview`
- executor:
  - `constant-arrival-rate`
- warm-up:
  - `2m`
- main run:
  - `10m`
- cooldown:
  - `1m`
- 데이터셋 조직 수:
  - `1개 organization`
- 데이터셋 총 이벤트 수:
  - `50,000건`
- 식별 사용자 수:
  - `1,000명`
- raw path 수:
  - `30개`
- raw event type 수:
  - `5개`
- 데이터 분포 기간:
  - `최근 7일`
- 기본 조회 범위:
  - 고정된 `1일` 구간
- 기본 필터:
  - `externalUserId` 없음
  - `eventType` 없음
- route template / event type mapping 규칙:
  - 고정 세트 재사용
- 산출물 저장 규칙:
  - `artifacts/perf/r1/<runId>/`

### 변경값

- 환경:
  - `local` -> 이후 `prod direct`
- run별 rate:
  - `10`
  - `30`
  - `50`
- `runId`

원칙:

- baseline과 retest를 비교할 때 변경값은 최소화한다.
- 데이터셋 org와 query window는 같은 회차 안에서 바꾸지 않는다.

## 이번 시나리오에서 일부러 단순화한 항목

- 첫 `R1`은 single-tenant read 기준으로만 본다.
- 첫 `R1`은 `overview` 하나만 다룬다.
- 첫 `R1`은 `externalUserId`, `eventType` 필터를 쓰지 않는다.
- 첫 `R1`은 query window를 하나의 고정 `1일` 구간으로만 본다.
- route template와 event type mapping은 테스트 중 동적으로 바꾸지 않는다.

## 이전 시나리오의 어떤 가정을 가져오지 않는지

- `W1`의 fresh org per run 가정은 가져오지 않는다.
- `W1`의 path 6개 고정 입력 규칙은 가져오지 않는다.
- `W1`의 기존 사용자 `80%` / 신규 사용자 `20%` 혼합 규칙은 가져오지 않는다.
- `W1`의 `loadtest_write` 단일 eventType 입력 가정은 가져오지 않는다.

## 해석 시 주의사항

- `R1`의 응답 값은 거의 같은 결과를 반복해서 돌려줄 가능성이 높다. 이번 실험의 관심사는 **값 변화**가 아니라 **같은 값을 얼마나 빨리 안정적으로 돌려주는가**다.
- 이번 결과는 local, single-tenant, 고정 데이터셋 기준에서만 해석한다.
- 데이터셋 분포와 route / event type mapping 품질이 바뀌면 `overview` 비용도 달라질 수 있다.
- `R1`이 안정적이어도 `R2(paths)`나 `R3(time-buckets)`가 안정적이라는 뜻은 아니다.
- 첫 `R1`은 "현실 완벽 재현"보다 "첫 read baseline을 설명 가능하게 만드는 것"을 우선한다.

## 실행 조건

### 환경 단계

#### 1차

- 환경:
  - `local`
- 목적:
  - read 전용 데이터셋과 `overview` baseline 확보

#### 2차

- 환경:
  - `prod direct`
- 목적:
  - 실제 EC2 / RDS 조건에서 local 결과 재현 여부 확인

### executor

- `constant-arrival-rate`

이유:

- 같은 query를 같은 RPS로 반복하기 좋다.
- read 결과는 값보다 지연 비교가 중요하므로, 고정 arrival rate가 전/후 비교에 유리하다.

### run 단위

- `10 RPS`
- `30 RPS`
- `50 RPS`

원칙:

- 각 run은 별도 `runId`, 별도 산출물로 남긴다.
- 데이터셋 org와 query window는 그대로 재사용한다.

### 시간 조건

- warm-up:
  - `2m`
- main run:
  - `10m`
- cooldown:
  - `1m`

### VU 기준

#### `10 RPS`

- `preAllocatedVUs = 20`
- `maxVUs = 60`

#### `30 RPS`

- `preAllocatedVUs = 60`
- `maxVUs = 180`

#### `50 RPS`

- `preAllocatedVUs = 100`
- `maxVUs = 300`

원칙:

- baseline과 retest는 같은 VU 기준을 유지한다.
- read-heavy는 write-heavy보다 가볍더라도, `overview` 내부 후처리 비용을 고려해 초반에는 여유 있게 잡는다.

## 입력 데이터 규칙

### 조직 기준

- 첫 `R1` baseline은 **고정 read 데이터셋 org 1개**를 재사용한다.

예:

- `perf-r1-dataset`

원칙:

- `R1`, `R2`, `R3`는 가능하면 같은 read 데이터셋 org를 공유한다.
- read는 fresh org보다 **고정 데이터셋 재사용**이 우선이다.

### 데이터셋 기준

- total events:
  - `50,000`
- identified users:
  - `1,000`
- identified events:
  - `40,000`
- anonymous events:
  - `10,000`
- raw path:
  - `30개`
- raw event types:
  - `5개`
- 기간 분포:
  - 최근 `7일`

원칙:

- `overview`의 `identifiedEventRate`가 항상 `0` 또는 `100`으로 고정되지 않게 한다.
- 식별 / 익명 비율은 baseline과 retest에서 동일하게 유지한다.

### route / event type 정규화 기준

- raw path 30개 중 다수는 고정 route template 세트에 매칭되도록 구성한다.
- 일부 path는 의도적으로 `UNMATCHED_ROUTE`에 남긴다.
- raw event type 5개 중 다수는 고정 event type mapping 세트에 매핑되도록 구성한다.
- 일부 raw event type은 의도적으로 `UNMAPPED_EVENT_TYPE`에 남긴다.

원칙:

- `overview`의 `routeMatchCoverage`, `eventTypeMappingCoverage`가 항상 `0` 또는 `100`으로 고정되지 않게 한다.
- 다만 coverage 지표가 해석을 방해할 정도로 극단적으로 치우치지 않게 구성한다.

#### route template 구성

- route template 세트:
  - `24개`
- unmatched raw path 세트:
  - `6개`

대표 route template 예시:

- `/home`
- `/pricing`
- `/docs`
- `/signup`
- `/login`
- `/dashboard`
- `/projects/{id}`
- `/projects/{id}/overview`
- `/projects/{id}/events`
- `/projects/{id}/paths`
- `/projects/{id}/retention`
- `/teams/{id}`
- `/posts/{id}`
- `/products/{id}`
- `/checkout`

대표 unmatched path 예시:

- `/legacy/report/2024`
- `/debug/raw/1`
- `/temp/feature-x`
- `/unknown/path/a`
- `/misc/landing/1`
- `/preview/abc123`

원칙:

- route template 세트는 일반적인 웹 / 프로젝트 SaaS 경로로 구성한다.
- unmatched path는 운영형 dataset에서도 나올 수 있는 "정규화되지 않은 예외 경로" 정도로 본다.

#### raw event type / mapping 구성

- raw event type 세트:
  - `5개`
- canonical mapping 대상:
  - `4개`
- unmapped 유지:
  - `1개`

대표 event type / mapping 예시:

- `page_view -> view`
- `button_click -> click`
- `signup_submit -> signup`
- `purchase_complete -> purchase`
- `legacy_custom_event -> UNMAPPED_EVENT_TYPE`

원칙:

- `overview`의 `topEventTypes`와 `eventTypeMappingCoverage`가 동시에 의미 있게 계산되도록 한다.
- 첫 `R1`에서는 event type 세트를 더 늘리지 않고, coverage 해석이 가능한 최소 세트로 유지한다.

### 조회 조건

- endpoint:
  - `GET /api/v1/events/analytics/aggregates/overview`
- query range:
  - 고정된 `1일` 구간
- 필터:
  - `externalUserId` 없음
  - `eventType` 없음

원칙:

- query range는 상대 시각(`now-1d`)이 아니라, 데이터셋 metadata에 기록된 **고정 absolute range**를 사용한다.
- `R1` baseline과 retest는 같은 query window를 유지한다.

### 데이터셋 seed 기준

- read 데이터셋 seed는 deterministic해야 한다.
- 같은 `datasetVersion`이면 같은 org, 같은 분포, 같은 query window를 다시 재현할 수 있어야 한다.
- route template와 event type mapping도 같은 seed 세트로 맞춘다.
- 이벤트는 최근 `7일` 전체에 분포시키되, 마지막 `1일` query window 안에 상대적으로 더 많은 이벤트가 모이도록 구성한다.
- 식별 / 익명 비율, matched / unmatched 비율, mapped / unmapped 비율도 datasetVersion 기준으로 고정한다.

## threshold 기준

- `http_req_failed < 1%`
- `p95 < 3000ms`
- `p99 < 5000ms`

주의:

- 위 값은 첫 read baseline용 느슨한 실패 감지선이다.
- `overview`는 같은 결과를 반복 조회하므로, 절대 수치보다 전후 비교와 DB read 부담 변화도 함께 본다.

## 중단 조건

- 5xx 비율이 일정 시간 이상 임계치를 초과하면 중단한다.
- Grafana / Prometheus 수집 이상으로 관측이 불가능하면 중단하고 무효 처리한다.
- 앱 재시작, OOM, DB connection pool 고갈이 발생하면 중단한다.
- 잘못된 query range나 dataset org 오배치가 확인되면 해당 run은 무효 처리한다.

## 실험 후 판단 기준

- `10 RPS`부터 `p95/p99`가 높게 나오면 `overview` read path 자체가 현재 구조에서 이미 무거운 것으로 본다.
- `50 RPS`에서만 크게 악화되면 `R1`은 통과 구간이 남아 있다고 보고, 이후 `R2(paths)` / `R3(time-buckets)`로 확장한다.
- DB read 지표나 긴 쿼리가 먼저 튀면 `overview` 관련 query와 후처리 흐름을 분리해서 본다.
- `overview`가 안정적이어도 route / bucket 중심 비용은 아직 분리되지 않은 상태이므로, `R2`, `R3`로 그대로 이어간다.
