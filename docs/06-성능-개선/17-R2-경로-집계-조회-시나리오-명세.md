# 17. R2 경로 집계 조회 시나리오 명세

## 문서 목적

`R2`는 `GET /api/v1/events/analytics/aggregates/routes` read path의 첫 baseline을 확보하기 위한 시나리오다.  
이번 문서는 routeKey 기준 group by / sort / topN 비용을 어떤 데이터셋, 어떤 기간, 어떤 RPS 조건으로 반복 실행할지 고정하고, 이후 `R3`, `M1`과 비교 가능한 집계형 read 기준선을 만드는 데 목적이 있다.

## 적용 범위

- 대상 시나리오:
  - `R2`
- 대상 API:
  - `GET /api/v1/events/analytics/aggregates/routes`
- 보조 확인 API:
  - `GET /api/v1/events/analytics/aggregates/paths`
  - `GET /api/v1/events/analytics/aggregates/routes/unmatched-paths`
- 이번 문서가 다루는 범위:
  - baseline용 group-by read 조건
  - read 전용 데이터셋 재사용 기준
  - query 범위와 `top` 규칙
  - threshold / 중단 조건
  - 실험 후 판단 기준

## 이전 시나리오와 다른 점

- `R1`은 overview 요약 응답을 봤지만, `R2`는 **routeKey 기준 상세 집계**를 직접 본다.
- `R1`은 비교 구간과 coverage를 포함한 복합 요약 응답이 핵심이었지만, `R2`는 **group by / sort / topN**이 핵심 비용이다.
- `R2`는 raw `paths` API를 보조 확인 대상으로만 두고, **확장 중심 API는 `routes`**로 고정한다.

## 이번 시나리오에서 새로 중요해지는 변수

- raw path cardinality와 routeKey cardinality 차이
- route template 매칭률과 unmatched path 비율
- `top` 크기에 따른 정렬 / 상위 N 계산 비용
- routeKey 재집계 후 상위 route 분포
- `UNMATCHED_ROUTE` 비중
- `externalUserId`, `eventType` 필터 부재 상태에서의 전체 group-by 비용

## 환경 고정값 / 변경값

### 고정값

- 시나리오:
  - `R2`
- 대상 API:
  - `GET /api/v1/events/analytics/aggregates/routes`
- executor:
  - `constant-arrival-rate`
- warm-up:
  - `2m`
- main run:
  - `10m`
- cooldown:
  - `1m`
- 데이터셋 조직 수:
  - `1개 organization`
- 데이터셋:
  - `R1`과 같은 datasetVersion 재사용
- 기본 조회 범위:
  - 고정된 `7일` 구간
- 기본 필터:
  - `externalUserId` 없음
  - `eventType` 없음
- 기본 `top`:
  - `10`
- 산출물 저장 규칙:
  - `artifacts/perf/r2/<runId>/`

### 변경값

- 환경:
  - `local` -> 이후 `prod direct`
- run별 rate:
  - `10`
  - `30`
  - `50`
- `runId`

원칙:

- baseline과 retest를 비교할 때 변경값은 최소화한다.
- 데이터셋 org, datasetVersion, query window, `top` 값은 같은 회차 안에서 바꾸지 않는다.

## 이번 시나리오에서 일부러 단순화한 항목

- 첫 `R2`는 single-tenant read 기준으로만 본다.
- 첫 `R2`는 `routes` 하나를 주 대상 API로 본다.
- 첫 `R2`는 `externalUserId`, `eventType` 필터를 쓰지 않는다.
- 첫 `R2`는 `top=10`만 사용한다.
- 첫 `R2`는 `routes/unique-users`, `route-event-types`까지 확장하지 않는다.

## 이전 시나리오의 어떤 가정을 가져오지 않는지

- `R1`의 `overview` 비교 구간 해석을 그대로 가져오지 않는다.
- `R1`의 `topRoutes` 요약 응답을 `R2` 상세 집계와 같은 수준으로 보지 않는다.
- `W1`의 fresh org per run 가정은 가져오지 않는다.
- `W1`의 입력 payload / 사용자 혼합 규칙은 가져오지 않는다.

## 해석 시 주의사항

- `R2`는 `routes` 기준 집계이므로, raw path 수가 같아도 route template 품질에 따라 cardinality가 달라질 수 있다.
- `paths`는 호환용 API이므로, 첫 집계형 read baseline은 `paths`보다 `routes` 기준으로 읽는다.
- `UNMATCHED_ROUTE` 비율이 높으면 `routes` 집계 숫자 해석이 불안정할 수 있다.
- `R2`가 안정적이어도 `R3(time-buckets)`가 안정적이라는 뜻은 아니다.

## 실행 조건

### 환경 단계

#### 1차

- 환경:
  - `local`
- 목적:
  - routeKey 기준 group-by / topN read baseline 확보

#### 2차

- 환경:
  - `prod direct`
- 목적:
  - 실제 EC2 / RDS 조건에서 route 집계 read 비용 재현 여부 확인

### executor

- `constant-arrival-rate`

이유:

- 같은 기간, 같은 `top`, 같은 필터 조건으로 group-by read를 반복하기 좋다.
- `R2`는 응답 값 자체보다 정렬 / 집계 / 상위 N 계산 시의 지연 비교가 중요하다.

### run 단위

- `10 RPS`
- `30 RPS`
- `50 RPS`

원칙:

- 각 run은 별도 `runId`, 별도 산출물로 남긴다.
- 데이터셋 org, datasetVersion, query window, `top`은 그대로 재사용한다.

### 시간 조건

- warm-up:
  - `2m`
- main run:
  - `10m`
- cooldown:
  - `1m`

### VU 기준

#### `10 RPS`

- `preAllocatedVUs = 20`
- `maxVUs = 60`

#### `30 RPS`

- `preAllocatedVUs = 60`
- `maxVUs = 180`

#### `50 RPS`

- `preAllocatedVUs = 100`
- `maxVUs = 300`

원칙:

- baseline과 retest는 같은 VU 기준을 유지한다.
- `R2`는 `R1`보다 정렬 / group-by 비용이 더 직접적으로 드러날 수 있으므로, 첫 단계에서도 VU 상한을 넉넉하게 둔다.

## 입력 데이터 규칙

### 조직 기준

- 첫 `R2` baseline은 **`R1`과 같은 고정 read 데이터셋 org 1개**를 재사용한다.

예:

- `perf-r1-dataset`

원칙:

- `R2`, `R3`는 가능하면 같은 read 데이터셋 org와 datasetVersion을 공유한다.
- read는 fresh org보다 **고정 데이터셋 재사용**이 우선이다.

### 데이터셋 기준

- datasetVersion:
  - `R1`과 동일
- total events:
  - `50,000`
- identified users:
  - `1,000`
- raw path:
  - `30개`
- route template:
  - `24개`
- unmatched path:
  - `6개`
- raw event types:
  - `5개`

### query window 기준

- `R2` baseline query window는 데이터셋 metadata에 기록된 **고정 absolute 7일 구간**을 사용한다.
- 첫 `R2`는 `top=10`, `externalUserId` 없음, `eventType` 없음으로만 실행한다.

## threshold / 중단 조건

- `http_req_failed < 1%`
- `p95 < 1500ms`
- `p99 < 3000ms`
- `dropped_iterations = 0`를 우선 목표로 본다.

해석 기준:

- `10 RPS`는 baseline 적합 여부 확인 구간이다.
- `30 RPS`는 local 기준 한계 구간 진입 여부 확인 구간이다.
- `50 RPS`는 `10`, `30`이 안정적일 때만 본다.

## 기록 시 남길 핵심 항목

- datasetVersion
- queryWindowFrom / queryWindowTo
- `top`
- `UNMATCHED_ROUTE` 비율
- `http_req_failed`
- `p95 / p99`
- `dropped_iterations`
- `vus_max`
- DB read 관련 패널
- route 집계 응답 size / count 특이사항

## 이번 문서의 종료 조건

- `R2` baseline `10 / 30 / 50 RPS` 중 실행 가능한 구간의 결과를 기록한다.
- 첫 retest 또는 1차 개선 지점을 설명 가능하게 만든다.
- 이후 `18-R2-준비-및-실행-구조.md` 기준으로 prepare / run / artifacts 구조를 재현 가능하게 만든다.
