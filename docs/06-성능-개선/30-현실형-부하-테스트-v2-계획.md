# 30. 현실형 부하 테스트 v2 계획

## 문서 목적

`W1 / R1 / R2 / R3 / M1`로 진행한 1차 사이클은 병목 탐색과 저위험 개선 검증에 초점을 둔 실험 체계였다.  
이번 문서는 그 다음 단계로, 실제 서비스 사용 패턴에 더 가까운 **현실형 부하 테스트 v2**를 어떤 원칙과 어떤 시나리오 묶음으로 설계할지 고정하는 데 목적이 있다.

이번 문서는 아직 `prepare / run`의 구현 상세와 seed 구현 세부를 끝까지 고정하는 데 목적이 있지 않다.  
현재 시점에서 확정된 것은 **tenant 분포, read 계층화, mixed 비율, request preset의 default / heavy 구조, rate ladder, duration, success / saturation 기준, 공통 dataset / snapshot 설계, reset 전략**까지다.

## 왜 v2가 필요한가

v1은 아래 목적에는 충분히 효과가 있었다.

- 재현 가능한 baseline 확보
- write / read / mixed 병목 분리
- 인덱스와 집계 재사용 같은 저위험 개선 검증
- `local -> prod direct -> prod public` 검증 체계 확보

하지만 v1은 아래 점에서 의도적으로 단순했다.

- single-tenant 중심
- API 단일 반복 호출 중심
- `overview`, `routes`, `time-buckets`를 따로 봄
- `M1 50:50` 같은 경쟁 탐지용 mixed 비율 사용

즉 v1은 탐색형 실험으로는 좋았지만, **실제 서비스형 부하를 설명하기에는 단순화가 큰 상태**다.  
v2는 이 단순화를 줄이고, 제품 사용 흐름에 가까운 조건을 만들기 위한 단계다.

## v2에서 새로 고정하는 축

이번 단계에서 공통적으로 고정할 축은 아래 여덟 가지다.

1. tenant 분포
2. read mix
3. scenario 역할 분리
4. 성공 기준 분리
5. seed + dataset version 고정
6. saturation 판단 기준
7. bottleneck 분류 체계
8. 개선 비용 대비 효과 기록

즉 v2는 단순히 부하를 더 세게 주는 단계가 아니라, **실험 설계 자체를 해석 가능한 형태로 올리는 단계**다.

## 공통 tenant 분포

`W2 / R4 / R5 / R6 / M2`는 아래 tenant 모델을 공통으로 사용한다.

- 총 `6개 organization`
- hot tenant `1개`: 전체 요청의 `50%`
- warm tenant `2개`: 전체 요청의 `30%`
- cold tenant `3개`: 전체 요청의 `20%`

이 분포를 공통으로 유지하는 이유는 아래와 같다.

- 시나리오 간 비교가 쉬워진다.
- single-tenant 기준으로는 드러나지 않던 cache hit/miss 차이를 볼 수 있다.
- hot tenant와 cold tenant가 동시에 존재할 때의 DB access locality 차이를 볼 수 있다.

## 공통 dataset / snapshot

v2는 `W2 / R4 / R5 / R6 / M2`가 같은 dataset 세계를 바라보도록 공통 dataset / snapshot을 사용한다.

- 공통 dataset / snapshot 설계:
  - [37-v2-공통-dataset-및-snapshot-설계.md](37-v2-공통-dataset-및-snapshot-설계.md)

즉 v2에서는 시나리오별로 서로 다른 테스트용 데이터를 따로 만들지 않고, 공통 tenant 세계와 공통 시간 분포를 재사용한다.

## 공통 reset 전략

v2 prepare는 공통 dataset을 항상 다시 만드는 대신, **reset mode**를 통해 준비 비용과 재현성을 함께 관리한다.

- `full`
  - 공통 dataset logical snapshot replay
  - 가장 정확하지만 가장 느림
- `quick`
  - 공통 배경 dataset은 유지
  - 이전 run이 만든 `k6-v2` overlay event와 신규 user만 삭제
- `skip`
  - dataset이 이미 정상 상태라고 보고 reset을 생략
  - read-only 재실행에 사용

현재 기본값은 아래와 같다.

- `W2`
  - `quick`
- `R4`
  - `skip`
- `R5`
  - `skip`
- `R6`
  - `skip`
- `M2`
  - `quick`

즉 v2는 **공통 dataset 하나 + 시나리오별 reset 전략**으로 움직인다.

추가 원칙:

- reset mode는 시나리오 기본값만으로 결정하지 않는다.
- 직전 run이 `W2` 또는 `M2`처럼 write overlay를 남겼다면, 다음 `R4 / R5 / R6` prepare는 기본 `skip`이라도 자동으로 `quick`으로 승격될 수 있다.
- 즉 effective reset mode는 **시나리오 기본값 + 직전 dataset state**를 함께 보고 결정한다.

## v2 시나리오 묶음

v2에서 사용하는 시나리오는 아래 다섯 가지다.

### `W2`

- 목적:
  - multi-tenant realistic write
- 성격:
  - v1의 `W1`보다 현실성 높은 write baseline

### `R4`

- 목적:
  - 집계 read mix
- 성격:
  - `overview`, `routes`, `event-types`, `time-buckets`와 교차 집계를 계층형으로 섞는 집계 read 시나리오

### `R5`

- 목적:
  - 분석 read mix
- 성격:
  - `users/overview`, `funnels/report`, `retention/daily`, `retention/matrix`를 계층형으로 섞는 분석 read 시나리오

### `R6`

- 목적:
  - `R4 + R5`를 합친 공용 read profile 정의
- 성격:
  - standalone 진단용으로도 사용할 수 있지만, 우선은 `M2`가 재사용하는 공용 profile로 본다.

### `M2`

- 목적:
  - realistic mixed
- 성격:
  - `W2 realistic + R6 v2`를 섞는 최종 mixed 시나리오

## 공통 request preset 원칙

v2의 request preset은 모든 API를 하나의 기간으로 통일하지 않고, **default / heavy** 두 층으로 나눈다.

핵심 원칙은 아래와 같다.

- preset은 성능을 통과시키기 위해 낮추는 값이 아니라, 실제 화면 기본값과 무거운 분석값을 그대로 반영하는 값이어야 한다.
- 짧은 시계열은 `HOUR`, 긴 시계열은 `DAY`를 사용한다.
- `externalUserId`는 기본적으로 비워 둔다.
- `timezone`은 기본적으로 `UTC`를 사용한다.

현재 합의된 공통 시간 구간은 아래와 같다.

### default

- `R4` 집계 read:
  - `24시간`
- `users/overview`, `funnel`:
  - `7일`
- `retention/daily`, `retention/matrix`:
  - `30일`

### heavy

- `R4` 집계 read:
  - `7일`
- `users/overview`, `funnel`:
  - `30일`
- `retention/daily`, `retention/matrix`:
  - `90일`

### 시계열 bucket 원칙

- `24시간` 시계열:
  - `HOUR`
- `7일`, `30일`, `90일` 시계열:
  - `DAY`

## stage 와 profile version

- `stage 1 / 2 / 3`:
  - 부하 세기 ladder
  - baseline / stretch / upper-bound probe
- `v1 / v2 / v3`:
  - request mix 모양
  - 어떤 API 조합과 어떤 비율을 쓰는지

즉 `stage`와 `profile version`은 **서로 다른 축**이다.  
예를 들어 `R4 v1 + stage 3`처럼, 같은 profile에 더 높은 부하를 줄 수 있어야 한다.

## default / heavy 매핑 원칙

- `R4`
  - `v1`, `v2`는 `default`
  - `v3`는 `heavy`
- `R5`
  - `v1`, `v2`는 `default`
  - `v3`는 `heavy`
- `R6`
  - standalone read-only 해석에서는 `v1`, `v2`를 `default`, `v3`를 `heavy`로 본다.
- `M2`
  - read profile version은 `R6 v2`를 기본으로 유지한다.
  - request preset만 `default / heavy`로 나눈다.

즉 `M2`는 read profile version을 `R6 v2`로 두고, **`R6 v2 + default`와 `R6 v2 + heavy`**를 비교하는 방식으로 간다.

## 공통 rate ladder 원칙

v2는 모든 시나리오를 과도하게 많은 단계로 쪼개지 않고, **3단계 ladder**로 본다.

- stage 1:
  - baseline
- stage 2:
  - stretch
- stage 3:
  - upper-bound probe

현재 합의된 ladder는 아래와 같다.

- `W2`
  - `100 / 200 / 300`
- `R4`
  - `10 / 20 / 30`
- `R5`
  - `5 / 10 / 15`
- `R6`
  - `10 / 15 / 20`
- `M2`
  - total `100 / 200 / 300`
  - `95:5`를 그대로 적용하므로
    - `100 -> write 95 / read 5`
    - `200 -> write 190 / read 10`
    - `300 -> write 285 / read 15`

## 환경별 duration 원칙

### local

- stage 1, 2
  - `warm-up 30s`
  - `main 5m`
  - `cooldown 0`
- stage 3
  - `warm-up 30s`
  - `main 3m`
  - `cooldown 0`

### prod-direct

- stage 1, 2
  - `warm-up 15s`
  - `main 1m`
  - `cooldown 0`
- stage 3
  - `warm-up 15s`
  - `main 30s`
  - `cooldown 0`

### prod-public

- stage 1, 2
  - `warm-up 15s`
  - `main 30s`
  - `cooldown 0`
- stage 3
  - `warm-up 10s`
  - `main 30s`
  - `cooldown 0`

## success / saturation 기준

### stable success

- 모든 threshold 통과
- `error rate < 1%`
- `dropped_iterations = 0`
- achieved throughput `>= 목표의 95%`

### degraded but not saturated

- threshold는 통과
- 그러나 `dropped_iterations > 0`
- 또는 achieved throughput이 `목표의 90~95%`

### saturated

- threshold 하나 이상 실패
- 또는 achieved throughput `< 목표의 90%`
- 또는 `dropped_iterations`가 의미 있게 지속 발생

### 병목 분류 지표의 역할

- DB / App / Pool / I/O 지표는 saturation 판정의 단독 기준이 아니다.
- 이 지표들은 **왜 포화됐는지 분류하는 보조 근거**로 사용한다.

## v1과 달라지는 핵심 설계 원칙

### 1. API를 개별로 반복하지 않고 계층으로 묶는다

- `R4`, `R5`는 개별 API 나열이 아니라 `core / extended` 계층으로 정의한다.
- 모든 API를 균등 비율로 섞지 않는다.

### 2. realistic mixed는 탐지용 mixed와 분리한다

- `M1 50:50`은 경쟁 탐지용 mixed로 유지한다.
- `M2 95:5`는 realistic mixed로 새로 정의한다.

### 3. read-only와 mixed가 같은 read profile을 공유한다

- `R6`를 두는 이유는 read mix 정의가 `M2` 안에서 다시 중복되지 않게 하기 위해서다.

## 공통 threshold 원칙

threshold는 stage마다 바꾸지 않고, **시나리오 성격별로 고정**한다.

- `W2`
  - `p95 < 5000ms`
  - `p99 < 8000ms`
- `R4`
  - `p95 < 3000ms`
  - `p99 < 5000ms`
- `R5`
  - `p95 < 5000ms`
  - `p99 < 8000ms`
- `R6`
  - `p95 < 5000ms`
  - `p99 < 8000ms`
- `M2 write`
  - `p95 < 5000ms`
  - `p99 < 8000ms`
- `M2 read`
  - `p95 < 5000ms`
  - `p99 < 8000ms`

공통적으로 아래 기준을 함께 사용한다.

- `error rate < 1%`
- achieved throughput 판정은 success / saturation 기준과 함께 본다.

## 공통 VU 원칙

`preAllocatedVUs / maxVUs`는 서비스 성격 자체를 표현하는 값이라기보다,  
목표 rate를 k6가 안정적으로 따라가게 하기 위한 **부하 생성기 headroom**으로 본다.

원칙:

- `W2`는 기존 `W1`의 write VU 패턴을 그대로 확장한다.
- `R4`는 기존 집계 read VU 패턴을 그대로 확장한다.
- `R5`, `R6`는 요청당 계산 비용이 더 클 수 있으므로 `R4`보다 조금 더 넉넉하게 둔다.
- `M2`는 write/read를 분리해 각각 별도 VU를 둔다.
