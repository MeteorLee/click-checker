# 20. R3 시계열 조회 시나리오 명세

## 문서 목적

`R3`는 `GET /api/v1/events/analytics/aggregates/time-buckets` read path의 첫 baseline을 확보하기 위한 시나리오다.  
이번 문서는 `bucket` 수와 시간축 응답 조립 비용을 어떤 데이터셋, 어떤 기간, 어떤 RPS 조건으로 반복 실행할지 고정하고, 이후 `M1`과 비교 가능한 추세형 read 기준선을 만드는 데 목적이 있다.

## 적용 범위

- 대상 시나리오:
  - `R3`
- 대상 API:
  - `GET /api/v1/events/analytics/aggregates/time-buckets`
- 보조 확인 API:
  - `GET /api/v1/events/analytics/aggregates/route-time-buckets`
  - `GET /api/v1/events/analytics/aggregates/event-type-time-buckets`
- 이번 문서가 다루는 범위:
  - baseline용 trend read 조건
  - read 전용 데이터셋 재사용 기준
  - query 범위, `bucket`, `timezone` 규칙
  - threshold / 중단 조건
  - 실험 후 판단 기준

- `R3`도 `capture.mode=scripted`를 기본값으로 사용하고, 첫 baseline은 `R1`, `R2`와 같은 read 패널 세트를 기준으로 renderer 기반 PNG를 자동 생성하는 구조를 전제로 한다.

## 이전 시나리오와 다른 점

- `R1`은 overview 요약 응답을 봤고, `R2`는 routeKey 기준 group by / sort / topN 비용을 봤지만, `R3`는 **시간 bucket 조립과 빈 bucket 채우기 비용**을 직접 본다.
- `R2`는 `top=10`이 핵심 변수였지만, `R3`는 **`bucket`과 `timezone`**이 핵심 변수다.
- `R3`는 현재 dataset이 `7일`이므로, 첫 baseline은 `30일/day`가 아니라 **`7일/HOUR`** 기준으로 고정한다.

## 이번 시나리오에서 새로 중요해지는 변수

- `bucket` 개수 증가에 따른 응답 조립 비용
- `HOUR` bucket 기준 빈 bucket 채우기 비용
- `timezone`에 따른 bucket 경계 해석 비용
- 같은 query window 안에서 `eventType`, `externalUserId` 필터 없이 전체 추세를 읽을 때의 비용
- 전체 count 추세와 route/eventType 세부 추세 간 비용 차이

## 환경 고정값 / 변경값

### 고정값

- 시나리오:
  - `R3`
- 대상 API:
  - `GET /api/v1/events/analytics/aggregates/time-buckets`
- executor:
  - `constant-arrival-rate`
- warm-up:
  - `2m`
- main run:
  - `10m`
- cooldown:
  - `1m`
- 데이터셋 조직 수:
  - `1개 organization`
- 데이터셋:
  - `R1`과 같은 datasetVersion 재사용
- 기본 조회 범위:
  - 고정 absolute `7일` 구간
- 기본 필터:
  - `externalUserId` 없음
  - `eventType` 없음
- 기본 `timezone`:
  - `UTC`
- 기본 `bucket`:
  - `HOUR`
- 산출물 저장 규칙:
  - `artifacts/perf/r3/<runId>/`

### 변경값

- 환경:
  - `local` -> 이후 `prod direct`
- run별 rate:
  - `10`
  - `30`
  - `50`
- `runId`

원칙:

- baseline과 retest를 비교할 때 변경값은 최소화한다.
- 데이터셋 org, datasetVersion, query window, `bucket`, `timezone`은 같은 회차 안에서 바꾸지 않는다.

## 이번 시나리오에서 일부러 단순화한 항목

- 첫 `R3`는 single-tenant read 기준으로만 본다.
- 첫 `R3`는 전체 추세 `time-buckets`만 주 대상 API로 본다.
- 첫 `R3`는 `externalUserId`, `eventType` 필터를 쓰지 않는다.
- 첫 `R3`는 `bucket=HOUR`, `timezone=UTC`만 사용한다.
- 첫 `R3`는 `route-time-buckets`, `event-type-time-buckets`, `route-event-type-time-buckets`까지 확장하지 않는다.

## 이전 시나리오의 어떤 가정을 가져오지 않는지

- `R1`의 overview 비교 구간 해석을 그대로 가져오지 않는다.
- `R2`의 `top=10` group-by 해석을 그대로 가져오지 않는다.
- `W1`의 fresh org per run 가정은 가져오지 않는다.
- `W1`의 입력 payload / 사용자 혼합 규칙은 가져오지 않는다.

## 해석 시 주의사항

- `R3`는 같은 총 이벤트 수라도 `bucket` 수가 늘면 응답 조립 비용이 크게 달라질 수 있다.
- 현재 dataset은 `7일` 범위이므로, 첫 `R3` baseline은 `30일/day`가 아니라 `7일/HOUR` 기준으로 읽는다.
- `R3`가 안정적이어도 `route-time-buckets`, `event-type-time-buckets` 같은 확장 trend API가 안정적이라는 뜻은 아니다.
- `R3`는 전체 이벤트 추세만 보기 때문에 route/eventType 해석 비용은 상대적으로 덜 드러난다.

## 실행 조건

### 환경 단계

#### 1차

- 환경:
  - `local`
- 목적:
  - `HOUR` bucket 기준 전체 trend read baseline 확보

#### 2차

- 환경:
  - `prod direct`
- 목적:
  - 실제 EC2 / RDS 조건에서 time-bucket read 비용 재현 여부 확인

### executor

- `constant-arrival-rate`

이유:

- 같은 기간, 같은 `bucket`, 같은 `timezone` 조건으로 trend 응답을 반복하기 좋다.
- `R3`는 결과 값 자체보다 bucket 조립과 빈 bucket 채우기 시의 지연 비교가 중요하다.

### run 단위

- `10 RPS`
- `30 RPS`
- `50 RPS`

원칙:

- 각 run은 별도 `runId`, 별도 산출물로 남긴다.
- 데이터셋 org, datasetVersion, query window, `bucket`, `timezone`은 그대로 재사용한다.

### 시간 조건

- warm-up:
  - `2m`
- main run:
  - `10m`
- cooldown:
  - `1m`

### VU 기준

#### `10 RPS`

- `preAllocatedVUs = 20`
- `maxVUs = 60`

#### `30 RPS`

- `preAllocatedVUs = 60`
- `maxVUs = 180`

#### `50 RPS`

- `preAllocatedVUs = 100`
- `maxVUs = 300`

원칙:

- baseline과 retest는 같은 VU 기준을 유지한다.
- `R3`는 bucket 수에 따라 응답 조립 비용이 커질 수 있으므로, `R2`와 같은 초기 VU 기준을 유지한다.

## 입력 데이터 규칙

### 조직 기준

- 첫 `R3` baseline은 **`R1`, `R2`와 같은 고정 read 데이터셋 org 1개**를 재사용한다.

예:

- `perf-r1-dataset`

원칙:

- `R2`, `R3`는 가능하면 같은 read 데이터셋 org와 datasetVersion을 공유한다.
- read는 fresh org보다 **고정 데이터셋 재사용**이 우선이다.

### 데이터셋 기준

- datasetVersion:
  - `R1`과 동일
- total events:
  - `50,000`
- identified users:
  - `1,000`
- raw path:
  - `30개`
- route template:
  - `24개`
- unmatched path:
  - `6개`
- raw event types:
  - `5개`

### query window 기준

- `R3` baseline query window는 데이터셋 metadata에 기록된 **고정 absolute 7일 구간**을 사용한다.
- 첫 `R3`는 `bucket=HOUR`, `timezone=UTC`, `externalUserId` 없음, `eventType` 없음으로만 실행한다.

## threshold / 중단 조건

- `http_req_failed < 1%`
- `p95 < 1500ms`
- `p99 < 3000ms`
- `dropped_iterations = 0`를 우선 목표로 본다.

해석 기준:

- `10 RPS`는 baseline 적합 여부 확인 구간이다.
- `30 RPS`는 local 기준 한계 구간 진입 여부 확인 구간이다.
- `50 RPS`는 `10`, `30`이 안정적일 때만 본다.

## 기록 시 남길 핵심 항목

- datasetVersion
- queryWindowFrom / queryWindowTo
- `bucket`
- `timezone`
- bucket 개수
- `http_req_failed`
- `p95 / p99`
- `dropped_iterations`
- `vus_max`
- DB read 관련 패널
- 응답 item 수 / 빈 bucket 비율 특이사항

## 이번 문서의 종료 조건

- `R3` baseline `10 / 30 / 50 RPS` 중 실행 가능한 구간의 결과를 기록한다.
- 첫 retest 또는 1차 개선 지점을 설명 가능하게 만든다.
- 이후 `21-R3-준비-및-실행-구조.md` 기준으로 prepare / run / artifacts 구조를 재현 가능하게 만든다.
