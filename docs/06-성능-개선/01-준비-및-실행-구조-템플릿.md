# 01. 준비 및 실행 구조 템플릿

## 문서 목적

성능 시나리오별 준비 스크립트와 실행 스크립트 구조를 정리할 때 빠뜨리면 안 되는 항목을 고정한다.  
이번 템플릿의 목적은 `W1`에서 정의한 `prepare -> run -> meta -> artifacts` 구조가 이후 시나리오에서도 재사용 가능하게 유지되도록 하고, 시나리오별 차이점도 같이 드러내는 데 있다.

## 사용 원칙

- 준비/실행 구조 문서는 이 템플릿을 기준으로 작성한다.
- 시나리오 명세 문서와 분리해서, "어떻게 준비하고 어떻게 돌리는지"만 다룬다.
- 같은 시나리오에서도 local / prod direct 흐름이 다르면 그 차이를 별도 섹션에서 드러낸다.
- `W1`의 구조를 그대로 가져오지 않는 부분은 명시한다.
- `prepare -> run -> meta -> artifacts` 경계가 무너지지 않게 유지한다.

## 필수 섹션

### 1. 문서 목적

- 이번 구조 문서가 어떤 시나리오의 준비/실행 흐름을 다루는지
- 왜 준비와 실행을 분리하는지

### 2. 적용 범위

- 대상 시나리오
- 대상 환경
- 이번 문서가 다루는 준비/실행 범위

### 3. 공통 구조에서 재사용하는 항목

- 공통 산출물 규칙
- `meta.json` 사용 여부
- 결과 저장 경로 규칙
- 기록 문서 반영 규칙

### 4. 환경 의존성

- local / prod direct / prod public 중 어떤 환경을 전제로 하는지 적는다.
- 같은 구조라도 환경에 따라 결과가 달라질 수 있는 요소를 적는다.

예:

- base URL
- 인증 방식 차이
- 네트워크 RTT 영향
- DB 스펙 차이
- autoscaling 여부

원칙:

- "왜 local과 prod 숫자가 다른가"를 나중에 설명할 수 있어야 한다.
- 환경 차이를 적지 않으면 구조 문제와 환경 차이를 혼동하기 쉽다.

#### 실행 위치와 관측 소스 분리

- 아래 네 축은 가능한 한 분리해서 적는다.
  - 실행 위치
  - 요청 대상
  - Grafana 캡처 소스
  - 산출물 저장 위치

예:

- 실행 위치: local docker
- 요청 대상: prod public
- Grafana 캡처 소스: prod Grafana
- 산출물 저장 위치: local workspace

원칙:

- "로컬에서 실행했으니 로컬 Grafana를 본다"처럼 자동으로 가정하지 않는다.
- prod direct / prod public 실행에서는 Grafana 캡처 소스가 local이 아닌지 별도로 확인한다.
- 이 네 축이 섞이면 결과 해석이 쉬워 보이지만, 나중에 왜 수치가 달랐는지 설명하기 어려워진다.

### 5. 이전 구조 문서와 다른 점

- 직전 시나리오의 준비/실행 흐름과 비교해 달라진 점

예:

- `W1`은 org를 새로 만들지만 `R1`은 기존 baseline org를 재사용한다.
- `M1`은 write/read 두 실행기를 동시에 준비한다.

### 6. 실행 재현성 기준

- 동일한 `meta.json`으로 동일 조건 run을 다시 재현할 수 있는지 적는다.
- prepare 없이 run만 재실행 가능한지 적는다.
- seed 데이터가 deterministic한지 적는다.

예:

- 같은 `meta.json`이면 같은 org / path prefix / eventType / runId 규칙을 재사용한다.
- run 스크립트는 `meta.json`만 있으면 재실행 가능하다.
- seed는 같은 입력이면 같은 초기 상태를 만든다.

원칙:

- 이 섹션은 "한 번 돌리고 끝나는 실험"이 아니라 "다시 검증 가능한 실험"인지 확인하는 용도다.

### 7. 준비 스크립트 역할

- 예상 파일명
- 준비 스크립트가 만드는 것

예:

- 테스트 계정
- org / api key
- seed 데이터
- runId
- outDir
- `meta.json`

### 8. 실행 스크립트 역할

- 예상 파일명
- 실행 스크립트가 읽는 것
- 실행 스크립트가 저장하는 것

예:

- `meta.json`
- `summary.json`
- `console.log`

### 9. prepare / run 경계

- 무엇이 prepare 단계에 속하고, 무엇이 run 단계에 속하는지 명시한다.

기본 원칙:

- prepare:
  - 상태 생성
  - seed
  - org / api key 준비
  - `meta.json` 생성
- run:
  - 부하 실행
  - 결과 저장

주의:

- 가능하면 run 단계에서는 seed나 사전 상태 생성을 하지 않는다.
- prepare와 run이 섞이면 run 간 비교 가능성이 크게 떨어진다.

### 10. 준비 데이터 규칙

- seed 대상
- seed 방식
- seed 식별 규칙
- 준비 단계에서 남길 최소 메타데이터

### 11. run 단위 기준

- 한 번에 한 run인지
- 여러 rate/stage를 한 번에 돌리는지
- fresh org를 쓸지, 기존 org를 재사용할지

### 12. `meta.json` 기준

- 어떤 값을 넣는지
- 실행 스크립트가 무엇을 `meta.json`에서 읽는지
- 어떤 값은 실행 시점에 다시 계산하지 않는지

#### 권장 스키마 예시

```json
{
  "scenario": "W1",
  "runId": "w1-20260319-143000-r50",
  "timestamp": "2026-03-19T14:30:00+09:00",
  "status": "success",
  "isBaseline": true,
  "compareTo": null,
  "env": {
    "name": "local",
    "baseUrl": "http://app:8080",
    "appCount": 1,
    "dbInstance": "local-postgres"
  },
  "dataset": {
    "orgId": 1,
    "orgName": "perf-w1-r50",
    "pathPrefix": "/loadtest/w1",
    "eventType": "loadtest_write",
    "pathCount": 6,
    "existingUserPoolSize": 1000,
    "existingUserRatio": 80
  },
  "seed": {
    "eventType": "loadtest_seed",
    "pathPrefix": "/loadtest/w1/seed",
    "existingUserPoolSize": 1000
  },
  "load": {
    "rate": 50,
    "warmup": "2m",
    "duration": "10m",
    "cooldown": "1m"
  },
  "capture": {
    "mode": "manual",
    "timeRange": "start-2m ~ end+2m"
  }
}
```

원칙:

- 실제 필드는 시나리오에 맞게 확장할 수 있다.
- 하지만 최소한 "환경 / 데이터 / 부하 조건"은 구조적으로 분리해 넣는다.
- `meta.json`에는 실행 조건과 상태만 넣고, 결과값은 넣지 않는다.
- baseline 여부와 비교 대상 run은 명시 필드로 관리한다.

### 13. 실패 run 처리 기준

- 어떤 run을 유효한 실패로 볼지, 어떤 run을 무효로 보고 재실행할지 적는다.

예:

- 네트워크 오류로 중단된 경우 -> 무효
- 서버 재시작 / OOM 발생 -> 무효
- 일부 요청 실패지만 threshold 이내 -> 유효
- Grafana / Prometheus 수집 누락 -> 재실행

원칙:

- 실패 run도 삭제하지 않고 산출물은 남긴다.
- 다만 유효한 실패와 실험 오류를 구분해야 한다.

### 14. cleanup 기준

- 자동 cleanup 여부
- 초기 단계에서 보존을 우선할지
- 정리 스크립트를 별도로 둘지

### 15. 실행 순서

- 드라이런
- 공식 baseline
- 재검증

### 16. 이전 시나리오의 어떤 구조 가정을 그대로 가져오지 않는지

- `W1`에서만 썼던 준비/실행 가정 중 이번에는 그대로 쓰지 않는 항목

예:

- single-tenant fresh org
- seed를 앱 경유로만 수행
- run마다 완전 분리된 org 사용

## 권장 체크 문장

새 준비/실행 구조 문서를 쓸 때 아래 질문에 답이 보여야 한다.

- 준비 스크립트와 실행 스크립트의 역할이 분리돼 있는가?
- 무엇을 `meta.json`에 고정하고 무엇을 실행 시점에 읽는가?
- seed와 본 실험 데이터가 구분되는가?
- fresh org / 기존 org 재사용 여부가 명확한가?
- cleanup 정책이 명시돼 있는가?
- `W1` 구조를 그대로 가져오지 않는 부분이 드러나는가?

## 후속 시나리오 확장 항목

아래 항목은 `M1`처럼 복수 executor가 필요한 시나리오에서 추가로 고려한다.

### 동시 실행 구조

- write executor
- read executor
- 실행 시작 타이밍
- write/read 비율

원칙:

- 단일 시나리오 구조 문서에는 억지로 넣지 않는다.
- mixed 시나리오부터 별도 섹션으로 확장한다.
