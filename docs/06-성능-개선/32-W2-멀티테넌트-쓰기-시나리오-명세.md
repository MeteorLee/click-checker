# 32. W2 멀티테넌트 쓰기 시나리오 명세

## 문서 목적

`W2`는 `POST /api/events` write path를 단일 tenant가 아니라 여러 tenant 분포에서 보는 첫 realistic write 시나리오다.  
이번 문서는 `W1` 이후 write 시나리오를 현실형으로 확장하기 위해, 어떤 tenant 모델과 어떤 역할로 `W2`를 정의할지 먼저 고정하는 데 목적이 있다.

이번 문서는 아직 seed 구현 세부까지 고정하지 않는다.  
현재 문서가 고정하는 것은 `W2`의 역할, tenant 모델, payload 분포, rate ladder, duration, threshold, success / saturation 기준, 공통 dataset / snapshot 재사용 원칙까지다.

## 적용 범위

- 대상 시나리오:
  - `W2`
- 대상 API:
  - `POST /api/events`
- 이번 문서가 다루는 범위:
  - multi-tenant write 목적
  - 공통 tenant 분포
  - `W1`와의 차이
  - 아직 고정하지 않는 항목

## 이전 시나리오와 다른 점

- `W1`은 single-tenant write baseline이었다.
- `W2`는 multi-tenant write baseline이다.
- `W1`의 목적이 핫패스 병목 탐색이었다면, `W2`의 목적은 **tenant skew가 있는 realistic write**를 보는 것이다.

## 이번 시나리오에서 새로 중요해지는 변수

- hot tenant와 cold tenant의 write skew
- tenant 분포가 index locality와 contention에 주는 영향
- 동일 write API라도 조직이 섞일 때의 connection / insert 패턴 변화

## 공통 tenant 분포

`W2`는 아래 분포를 사용한다.

- 총 `6개 organization`
- hot tenant `1개`: `50%`
- warm tenant `2개`: `30%`
- cold tenant `3개`: `20%`

원칙:

- 이 tenant 모델은 `R4 / R5 / R6 / M2`와 공통으로 사용한다.
- `W2` 단독으로 다른 tenant 모델을 만들지 않는다.

### tenant별 skew

## stage 1 재보정 메모

- 첫 local v2 baseline에서 기존 `100 RPS` stage 1은 `p95 16.91s`, 실패율 `2.08%`, achieved throughput `~63 RPS`로 baseline이 아니라 limit 구간에 가까웠다.
- 같은 dataset / same warm state 기준 재실행에서
  - `60 RPS`는 `p95 187.82ms`, 실패율 `0%`, dropped `0`
  - `80 RPS`는 `p95 83.2ms`, 실패율 `0%`, dropped `0`
  로 안정 구간이 확인됐다.
- 따라서 현재 `W2`의 stage는 아래처럼 읽는다.
  - stage 1: `60`
  - stage 2: `80`
  - stage 3: `100`

## 후속 write 개선 메모

- `EventUser`의 `find -> save` 경로를 native upsert 기반 `findOrCreateUserId`로 변경한 뒤, 기존 실패 구간이던 `W2 100 RPS`를 다시 확인했다.
- 결과는 `p95 176.1ms`, `p99 789.83ms`, fail `0%`, achieved throughput `~97 RPS`였다.
- 즉 현재 코드 기준으로는 `100 RPS`도 write-only에선 성공 구간으로 이동했다.
- 같은 코드 상태에서 상단 probe를 더 본 결과:
  - `120 RPS`는 `p95 1.41s`, fail `0%`, achieved `~116 RPS`, dropped `105`
  - `140 RPS`는 `p95 2.12s`, fail `0%`, achieved `~136 RPS`, dropped `59`
  - `160 RPS`는 `p95 1.42s`, fail `0%`, achieved `~155 RPS`, dropped `22`
  - `200 RPS`는 `p95 989.51ms`, fail `0%`, achieved `~191 RPS`, dropped `660`
  - `250 RPS`는 `p95 7.61s`, `p99 8.81s`, achieved `~213 RPS`, dropped `4606`으로 threshold fail
  - `300 RPS`는 `p95 11.02s`, `p99 13.26s`, achieved `~199 RPS`, dropped `15789`로 threshold fail
  로 `200`까지는 통과하고 `250`부터는 실패했다.
- 즉 현재 local write-only 상한은 `200과 250 사이`로 보는 것이 가장 자연스럽다.

다만 이 문서의 stage를 즉시 다시 바꾸지는 않는다.

- `60 / 80 / 100`은 이미 확보한 baseline 비교 축이고,
- mixed 시나리오에서는 아직 `100` 근처가 limit 쪽 의미를 갖기 때문이다.

따라서 다음 write ladder 재정의는:

- 현재 확보된 `60 / 80 / 100` baseline 비교 축은 유지하되
- write-only upper band는 `120 / 140 / 160 / 200`
- failure band는 `250+`
로 별도 해석하는 편이 맞다.

## success / saturation 기준

- stable:
  - threshold pass
  - `error rate < 1%`
  - `dropped_iterations = 0`
  - achieved throughput `>= 95%`
- degraded:
  - threshold는 pass
  - 그러나 `dropped_iterations > 0`
  - 또는 achieved throughput `90~95%`
- saturated:
  - threshold fail
  - 또는 achieved throughput `< 90%`
  - 또는 `dropped_iterations` 지속 발생

### 아직 고정하지 않는 변경값

- seed 방식 상세
- existing/new와 eventType/path의 상관관계 강제 여부

## 이번 시나리오에서 일부러 단순화하지 않는 항목

- single-tenant 고정을 유지하지 않는다.
- 모든 요청을 같은 org로 보내지 않는다.

## 이전 시나리오의 어떤 가정을 가져오지 않는지

- `W1`의 single-tenant 가정은 가져오지 않는다.
- `W1`의 intentionally simple한 org 고정 구조는 그대로 재사용하지 않는다.

## 해석 시 주의사항

- `W2`는 `W1`보다 현실성이 높지만, 원인 분리는 더 어려워진다.
- 따라서 `W2` 결과는 `W1`과 나란히 봐야 한다.

## 이번 문서의 결론

> `W2`는 `W1`를 대체하는 것이 아니라, `6개 org / 50-30-20` 분포를 적용한 현실형 write baseline 시나리오로 추가된다.
