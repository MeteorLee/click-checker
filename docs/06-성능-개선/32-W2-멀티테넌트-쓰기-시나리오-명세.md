# 32. W2 멀티테넌트 쓰기 시나리오 명세

## 문서 목적

`W2`는 `POST /api/events` write path를 단일 tenant가 아니라 여러 tenant 분포에서 보는 첫 realistic write 시나리오다.  
이번 문서는 `W1` 이후 write 시나리오를 현실형으로 확장하기 위해, 어떤 tenant 모델과 어떤 역할로 `W2`를 정의할지 먼저 고정하는 데 목적이 있다.

이번 문서는 아직 seed 구현 세부까지 고정하지 않는다.  
현재 문서가 고정하는 것은 `W2`의 역할, tenant 모델, payload 분포, rate ladder, duration, threshold, success / saturation 기준, 공통 dataset / snapshot 재사용 원칙까지다.

## 적용 범위

- 대상 시나리오:
  - `W2`
- 대상 API:
  - `POST /api/events`
- 이번 문서가 다루는 범위:
  - multi-tenant write 목적
  - 공통 tenant 분포
  - `W1`와의 차이
  - 아직 고정하지 않는 항목

## 이전 시나리오와 다른 점

- `W1`은 single-tenant write baseline이었다.
- `W2`는 multi-tenant write baseline이다.
- `W1`의 목적이 핫패스 병목 탐색이었다면, `W2`의 목적은 **tenant skew가 있는 realistic write**를 보는 것이다.

## 이번 시나리오에서 새로 중요해지는 변수

- hot tenant와 cold tenant의 write skew
- tenant 분포가 index locality와 contention에 주는 영향
- 동일 write API라도 조직이 섞일 때의 connection / insert 패턴 변화

## 공통 tenant 분포

`W2`는 아래 분포를 사용한다.

- 총 `6개 organization`
- hot tenant `1개`: `50%`
- warm tenant `2개`: `30%`
- cold tenant `3개`: `20%`

원칙:

- 이 tenant 모델은 `R4 / R5 / R6 / M2`와 공통으로 사용한다.
- `W2` 단독으로 다른 tenant 모델을 만들지 않는다.

## 환경 고정값 / 변경값

### 현재 고정값

- 대상 API:
  - `POST /api/events`
- tenant 수:
  - `6개`
- tenant 분포:
  - `50-30-20`
- 시나리오 역할:
  - realistic write baseline
- rate ladder:
  - `100 / 200 / 300`
- threshold:
  - `p95 < 5000ms`
  - `p99 < 8000ms`
- 기본 reset mode:
  - `quick`

## payload 분포

### 사용자 구성

- 전체 이벤트 기준:
  - `identified 70%`
  - `anonymous 30%`
- identified 내부:
  - `existing 80%`
  - `new 20%`

즉 전체 이벤트 기준으로는 아래처럼 해석한다.

- existing identified:
  - `56%`
- new identified:
  - `14%`
- anonymous:
  - `30%`

### canonical event type 분포

- `view 50%`
- `click 30%`
- `signup 12%`
- `purchase 8%`

### path 그룹 분포

- browse:
  - `60%`
  - 예:
    - `home`
    - `docs`
    - `blog`
    - `posts`
    - `products`
- product / analysis:
  - `25%`
  - 예:
    - `dashboard`
    - `projects`
    - `teams`
- conversion:
  - `15%`
  - 예:
    - `pricing`
    - `signup`
    - `checkout`

### tenant별 skew

- tenant 분포는 `6개 org / 50-30-20`
- tenant 간 차이는 우선 **volume skew만** 준다.
- payload 분포는 tenant 공통으로 시작한다.

즉 첫 `W2`에서는:

- hot tenant가 더 많은 요청을 받지만
- identified / anonymous, eventType, path 분포 자체는 모든 tenant에서 동일하게 둔다.

## 환경별 duration

- local
  - stage 1, 2:
    - `30s / 5m / 0`
  - stage 3:
    - `30s / 3m / 0`
- prod-direct
  - stage 1, 2:
    - `15s / 1m / 0`
  - stage 3:
    - `15s / 30s / 0`
- prod-public
  - stage 1, 2:
    - `15s / 30s / 0`
  - stage 3:
    - `10s / 30s / 0`

## success / saturation 기준

- stable:
  - threshold pass
  - `error rate < 1%`
  - `dropped_iterations = 0`
  - achieved throughput `>= 95%`
- degraded:
  - threshold는 pass
  - 그러나 `dropped_iterations > 0`
  - 또는 achieved throughput `90~95%`
- saturated:
  - threshold fail
  - 또는 achieved throughput `< 90%`
  - 또는 `dropped_iterations` 지속 발생

### 아직 고정하지 않는 변경값

- seed 방식 상세
- existing/new와 eventType/path의 상관관계 강제 여부

## 이번 시나리오에서 일부러 단순화하지 않는 항목

- single-tenant 고정을 유지하지 않는다.
- 모든 요청을 같은 org로 보내지 않는다.

## 이전 시나리오의 어떤 가정을 가져오지 않는지

- `W1`의 single-tenant 가정은 가져오지 않는다.
- `W1`의 intentionally simple한 org 고정 구조는 그대로 재사용하지 않는다.

## 해석 시 주의사항

- `W2`는 `W1`보다 현실성이 높지만, 원인 분리는 더 어려워진다.
- 따라서 `W2` 결과는 `W1`과 나란히 봐야 한다.

## 다음에 정할 항목

## 이번 문서의 결론

> `W2`는 `W1`를 대체하는 것이 아니라, `6개 org / 50-30-20` 분포를 적용한 현실형 write baseline 시나리오로 추가된다.
