# 07. 성능 개선 종합

## 문서 목적

`W1`, `R1`, `R2`, `R3`, `M1`까지의 1차 사이클 결과와  
`prod-direct` 검증 결과를 한 문서로 묶어, 현재 병목 지도와 다음 개선 우선순위를 정리한다.

이번 문서는 개별 run의 상세 수치를 다시 나열하는 데 목적이 있지 않고,
각 시나리오에서 확보한 **안정 구간 / 한계 구간 / 적용 조치 / 남은 구조 비용**을 한 번에 비교하는 데 목적이 있다.

## 1. 이번 단계에서 실제로 확보한 것

이번 단계에서 확보한 것은 크게 다섯 가지다.

- 재현 가능한 성능 실험 체계
  - 시나리오 문서
  - `prepare / run / meta / artifacts` 구조
  - scripted Grafana capture
  - `showcase` 선별본
- write 첫 기준선과 1차 개선 효과
- read 3개 축의 기준선과 한계 구간
  - 요약형 read
  - 집계형 read
  - 추세형 read
- write/read mixed 기준선과 첫 경쟁 구간
- prod-direct stable baseline 재현과 limit 일부 확인

즉 지금 단계는 "몇 개 run을 돌려봤다"가 아니라,
**성능 실험 체계 + 첫 병목 지도**를 같이 확보한 단계로 보는 것이 맞다.

## 2. 시나리오별 결과 요약

### 2.1 write

- `W1`
  - 안정 구간: `50 RPS`, 개선 후 `100 RPS`
  - 한계 구간: `200 RPS`
  - 핵심 조치:
    - `apiKeyLastUsedAt` 반복 update 완화
    - organization 재조회 제거

### 2.2 read

- `R1 overview`
  - 안정 구간: `10 RPS`
  - 한계 구간: `30 RPS`
  - 핵심 조치:
    - `events (organization_id, occurred_at)` 인덱스 추가
    - overview 내부 raw path / raw event type 집계 재사용

- `R2 routes`
  - 안정 구간: `30 RPS`
  - 한계 구간: `50 RPS`
  - 별도 조치 없이도 가장 가벼운 read 축으로 확인

- `R3 time-buckets`
  - 안정 구간: `10 RPS`
  - 한계 구간: `30 RPS`
  - `routes`보다 무겁고 `overview`보다는 가벼운 중간 축으로 확인

### 2.3 mixed

- `M1 write + overview`
  - 안정 구간: `10/10`
  - 한계 구간: `20/20`
  - 핵심 해석:
    - mixed에서는 read만 느려지는 것이 아니라 write도 같이 느려진다
    - shared DB / connection / pool 경쟁이 실제로 드러난다

## 3. 현재 local 기준 병목 지도

현재까지의 결과를 한 줄로 압축하면 아래와 같다.

- write-only에서는 앞단 부가 비용을 줄이자 `100 RPS`까지는 안정 구간으로 이동했다.
- read-only에서는 `overview`가 가장 무겁고, `routes`가 가장 가볍고, `time-buckets`가 그 중간이다.
- mixed에서는 write와 read가 같이 느려지며, read-only / write-only보다 더 이른 구간에서 한계가 온다.

즉 현재 local 기준 비용 축은 아래 순서로 읽는 것이 가장 자연스럽다.

1. `R1 overview`
2. `M1 write + overview`
3. `R3 time-buckets`
4. `W1 write`
5. `R2 routes`

이건 "어느 기능이 중요하냐"가 아니라,
**지금 데이터셋과 조건에서 어느 축이 먼저 한계 신호를 보여줬는가** 기준이다.

## 4. 이번 단계에서 효과가 컸던 조치

이번 단계에서 실제로 효과가 컸던 조치는 아래 두 묶음이다.

### 4.1 write 앞단 비용 절감

- `apiKeyLastUsedAt` 반복 update 완화
- organization 재조회 제거

이 묶음은 `W1 r100`을 failure 구간에서 통과 구간으로 이동시켰다.

### 4.2 read 공통 범위 읽기 + overview 재사용 구조 정리

- `events (organization_id, occurred_at)` 인덱스 추가
- overview 내부 raw path / raw event type 집계 재사용

이 묶음은 `R1 r10`을 failure 구간에서 안정 구간으로 이동시켰고,
`R1 r30`도 일부 개선시켰다.

즉 이번 단계에서 먹힌 조치는 공통적으로,
**구조 전체를 뒤엎기보다 저위험이고 공통 영향이 큰 비용부터 줄인 것**이었다.

## 5. 아직 남아 있는 구조적 비용

지금까지 일부러 보류한 구조적 비용은 아래와 같다.

### 5.1 write 본체 비용

- `event_user` 조회/생성
- `event` 저장

이건 funnel, retention, users overview 같은 사용자 단위 분석 요구와 연결되어 있으므로,
지금 단계에서 성급히 단순화하지 않았다.

### 5.2 overview 조합 비용

- 여러 count / distinct / group-by를 한 요청 안에서 조합하는 구조

첫 인덱스와 재사용 리팩터링으로 큰 폭 개선은 있었지만,
`R1 30 RPS`를 통과 구간으로 밀어낼 정도까지는 아니었다.

### 5.3 mixed shared resource 경쟁

- 같은 org / 같은 DB 자원 위에서 write와 read가 같이 들어올 때의 지연 상승

이건 write-only나 read-only 최적화만으로는 충분히 설명되지 않는다.

## 6. 이번 단계에서 하지 않은 것

이번 1차 단계에서는 아래 항목을 일부러 바로 하지 않았다.

- 추가 인덱스 연속 투입
- `event_user` 구조 단순화
- route/time-bucket 전용 캐시
- raw / 후처리 분리
- batch / rollup / Redis
- `M1 50/50` 추가 실패 구간 확인
- public failure-side 구간 확인

이유는 지금 단계의 목표가
**큰 구조 변경**보다
**저위험 조치로 병목 지도를 먼저 분리하는 것**에 있었기 때문이다.

## 7. prod direct 확인 결과

`26`에서 시작한 prod-direct 검증 결과는 아래처럼 정리된다.

- `W1 r100`
- `R1 r10`
- `R1 r30`
- `R1 r50` threshold fail
- `M1 10/10`
- `M1 20/20`
- `M1 30/30`
- `M1 40/40` threshold fail

즉 local에서 먼저 확보한 stable baseline은 EC2 / RDS direct 조건에서도 그대로 재현됐고,
`R1 overview`는 prod-direct에서 한계 구간이 `30과 50 사이`로 확인됐다.
또한 `M1 mixed`는 prod-direct에서 `30/30`까지는 안정 구간이고 `40/40`에서는 한계 구간으로 들어갔다.

즉 local 병목 지도는 방향성은 맞았지만, 실제 인프라 여유는 더 컸다.

상세는 [27-Prod-Direct-1차-종합.md](27-Prod-Direct-1차-종합.md)를 기준으로 본다.

## 8. prod public 확인 결과

`28`에서 시작한 prod-public 검증 결과는 아래처럼 정리된다.

- `W1 r100`
- `R1 r30`
- `M1 30/30`

즉 `nginx + TLS + public route`가 포함된 공개 진입점 기준에서도
기존 stable baseline은 그대로 유지됐다.

`R1`은 direct보다 지연이 올라갔지만 여전히 안정 구간이었고,
`M1`도 public 경로에서 안정 구간을 유지했다.

상세는 [29-Prod-Public-1차-종합.md](29-Prod-Public-1차-종합.md)를 기준으로 본다.

## 9. 다음 우선순위

다음 단계 우선순위는 아래 순서가 가장 자연스럽다.

### 9.1 local + prod direct + prod public 결과 종합 기준 정리

- 지금까지의 `W1/R1/R2/R3/M1`를 기준으로
  - 어떤 비용이 저위험 조치로 줄어들었는지
  - 어떤 비용이 구조적 trade-off인지
를 더 분명히 정리

### 9.2 필요 시 public failure-side 또는 추가 mixed 확장 검증

- `M1 50/50`
- `R1 r50`
- 필요 시 `R2`, `R3`

### 9.3 그 뒤 구조 개선 우선순위 재판단

후보:

- write 본체 구조
  - `event_user` 정규화 비용
- overview read 구조
  - 더 큰 조합 비용 절감
- mixed 경쟁 완화
  - connection / pool / query 비용 조정

즉 다음 질문은 "`local`에서 본 stable baseline이 맞는가"가 아니라,
**"limit 구간까지 다시 볼 것인가, 아니면 public 경유 최종 확인으로 넘어갈 것인가"**에 더 가깝다.

## 10. v2 현실형 baseline 착수 상태

`W2 / R4 / R5 / R6 / M2`를 공통 dataset 위에서 돌리는 v2 realistic load test는 이제 **설계 단계에서 baseline 확보 단계**로 넘어갔다.

- `W2`
  - 기존 `100 RPS` stage 1은 baseline이 아니라 limit probe에 가까웠다
  - local 재보정 결과 `60`, `80`은 안정적으로 통과했다
  - 따라서 현재 write ladder는 `60 / 80 / 100`으로 보는 것이 맞다
- `R4 v1 stage1`
  - stable
- `R5 v1 stage1`
  - stable
- `R6 stage1`
  - stable
- `M2 stage1`
  - threshold는 통과했지만 dropped가 있어 약간 빡빡하다

즉 v2 1차 local baseline에서 먼저 드러난 것은,
**현실형 read는 이미 충분히 안정적이고, 현실형 write 시작선을 다시 잡는 일이 우선**이라는 점이다.

상세는 [38-v2-local-baseline-1차-종합.md](38-v2-local-baseline-1차-종합.md)를 기준으로 본다.
