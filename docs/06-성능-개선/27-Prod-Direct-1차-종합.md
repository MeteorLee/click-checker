# 27. Prod Direct 1차 종합

## 문서 목적

`26-Prod-Direct-검증-계획.md`에서 정의한 최소 검증 세트  
`W1 r100 -> R1 r10 -> M1 10/10`의 실행 결과를 한 문서로 정리한다.

이번 문서는 prod direct 결과를 세부 run 단위로 다시 길게 나열하기보다,
**local에서 확보한 안정 구간이 실제 EC2 / RDS 조건에서도 재현됐는지**를 빠르게 판단하는 데 목적이 있다.

## 1. 이번에 실제로 확인한 것

이번 prod direct 1차 검증에서 실제로 확인한 항목은 아래 세 개다.

- `W1 r100`
- `R1 r10`
- `M1 10/10`

즉 이번 단계는 read/write/mixed 전체를 넓게 다시 보는 것이 아니라,
**각 축의 대표 안정 구간이 실제 인프라에서도 그대로 유지되는지**를 최소 비용으로 확인한 단계다.

## 2. 핵심 결과

세 run 모두 성공했다.

- `W1 r100`
  - `p95 22.19ms`
  - `p99 138.67ms`
  - `http_req_failed 0%`
- `R1 r10`
  - `p95 96.39ms`
  - `p99 138.28ms`
  - `http_req_failed 0%`
- `M1 10/10`
  - read `p95 111.53ms`, `p99 190.20ms`
  - write `p95 44.78ms`, `p99 68.94ms`
  - `http_req_failed 0%`

즉 local에서 먼저 확보한 "첫 안정 구간"들은  
EC2 / RDS direct 조건에서도 그대로 재현됐다.

## 3. 이번에 확인한 의미

이번 단계에서 가장 중요한 의미는 아래 세 가지다.

### 3.1 local 결과가 방향성 면에서 유효했다

local에서 먼저 잡은 안정 구간이 실제 인프라에서도 그대로 성공했다.

이건 local 수치를 prod와 그대로 동일시해도 된다는 뜻은 아니지만,
적어도 **local에서 본 병목 지도와 개선 방향이 완전히 엇나가진 않았다**는 뜻에 가깝다.

### 3.2 첫 저위험 개선이 실제 인프라에서도 무의미하지 않았다

이번 prod direct에서 다시 확인한 대상은 모두 local 1차 사이클에서 안정 구간으로 정리된 run들이다.

즉 아래 조치들이 local 전용 착시가 아니었음을 간접적으로 확인했다.

- write 앞단 비용 절감
- `events (organization_id, occurred_at)` 인덱스
- overview 집계 재사용 구조
- mixed dataset / snapshot restore 구조

### 3.3 이번 단계는 "안정 구간 재현"까지만 확인했다

이번 prod direct 1차는 일부러 아래는 하지 않았다.

- `R1 30 RPS`
- `M1 20/20`
- `R2`, `R3`
- public ingress 경유 검증

즉 지금 문서의 결론은 "prod에서도 다 빠르다"가 아니라,
**대표 안정 구간은 prod direct에서도 다시 성립했다** 정도로 해석하는 것이 맞다.

## 4. 이번 단계에서 남겨둔 것

이번 단계에서 일부러 남겨둔 항목은 아래와 같다.

### 4.1 prod direct 한계 구간 재현

아직 확인하지 않은 것:

- `R1 30 RPS`
- `M1 20/20`

즉 local에서 보였던 한계 구간이 prod direct에서도 같은 위치에 오는지는 아직 확인하지 않았다.

### 4.2 automated Grafana capture

prod Grafana에서는 renderer가 일부 패널에서 `60s` timeout을 내고 있다.

따라서 이번 단계에서는:

- `capture.mode=manual`
- 수동 패널 확인만 수행
- showcase용 PNG 자동 생성은 보류

로 정리했다.

즉 이번 prod direct 1차는 **수치와 메타 중심 기록**으로 먼저 닫는다.

## 5. 지금 기준 판단

지금 기준 판단은 아래처럼 정리하는 것이 가장 자연스럽다.

- local에서 확보한 stable baseline은 실제 인프라에서도 유효했다.
- 따라서 local 단계에서 만든 병목 지도는 다음 단계 판단의 기준으로 계속 써도 된다.
- 다만 prod direct 1차는 안정 구간 재현까지만 확인했으므로, limit 구간까지 그대로 일반화하진 않는다.

즉 현재 단계는 "local 결과를 실제 인프라에서 1차 검증 완료"로 보는 것이 맞다.

## 6. 다음 단계

다음 단계 후보는 두 갈래다.

### 6.1 prod direct 확장 검증

필요하면 아래를 추가 확인한다.

- `R1 30 RPS`
- `M1 20/20`
- 필요 시 `R2`, `R3`

이건 "안정 구간"이 아니라 **한계 구간 위치까지 실제 인프라에서 다시 보고 싶을 때** 가는 단계다.

### 6.2 public 경유 최종 확인

또 다른 다음 단계는 public ingress를 포함한 최종 확인이다.

이 단계에서는:

- nginx / public route
- TLS
- 외부 네트워크 경로

까지 포함해 해석해야 한다.

즉 지금 문서 다음 질문은  
"local 안정 구간이 prod direct에서 재현되는가"에서  
"한계 구간까지 다시 볼 것인가, 아니면 public 경로 최종 확인으로 갈 것인가"로 옮겨간다.
