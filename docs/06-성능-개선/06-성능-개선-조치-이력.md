# 06. 성능 개선 조치 이력

## 문서 목적

성능 개선 단계에서 실제로 적용한 변경을 `개선 단위`로 누적 기록한다.  
이번 문서는 run 자체를 기록하는 문서가 아니라, "무엇을 왜 바꿨고, 실제로 어떤 효과가 있었는가"를 변경 중심으로 정리하는 데 목적이 있다.

## 기록 원칙

- 개선은 `run`이 아니라 `변경 묶음` 단위로 기록한다.
- 상세 수치는 [04-대규모-부하-테스트-기록.md](04-대규모-부하-테스트-기록.md)를 기준으로 하고, 이번 문서에는 비교에 필요한 핵심 수치만 요약한다.
- 긴 흐름 설명은 [12-W1-기준선-실행-해설.md](12-W1-기준선-실행-해설.md)로 연결한다.
- 결과가 기대보다 작거나 애매해도 기록에서 빼지 않는다.
- 한 번 적용한 개선은 `완료`, `부분 완료`, `보류` 중 하나의 상태로 남긴다.

## 개선 1. write 핫패스 앞단 비용 1차 절감

### 상태

- `완료`

### 적용 배경

첫 `W1 r100`에서 아래 신호가 동시에 나타났다.

- `p95 = 6.39s`
- `p99 = 8.12s`
- `dropped_iterations = 468`
- `vus max = 500`
- `Insufficient VUs` 경고

코드를 읽어보면 `/api/events` 핫패스 앞단에서 아래 비용이 반복되고 있었다.

- API key 인증 필터가 매 요청마다 같은 org row의 `apiKeyLastUsedAt`를 바로 update
- 서비스 내부에서 이미 확인한 org를 다시 조회

이 둘은 user 정규화나 event 저장 자체보다 기능 영향이 작은 비용이었고, 먼저 줄여볼 가치가 높다고 판단했다.

### 왜 이 개선을 먼저 골랐는가

첫 `r100` 결과를 본 뒤 병목 후보는 크게 세 갈래였다.

- `apiKeyLastUsedAt` update
- 서비스 내부 org 재조회
- user lookup/create

이 중 앞의 두 개를 먼저 고른 이유는 다음과 같다.

- 요청마다 항상 붙는 비용이다.
- 같은 org row를 반복적으로 건드리거나, 이미 알고 있는 org를 다시 읽는 성격이라 비교적 낭비가 명확하다.
- user lookup/create보다 도메인 의미를 덜 흔든다.

즉 이번 단계는 구조를 갈아엎는 실험이 아니라, **핫패스 앞단에서 기능 영향이 가장 작은 반복 비용을 먼저 걷어내는 단계**였다.

### 변경 내용

- `apiKeyLastUsedAt`는 최근 1분 안에 이미 기록된 상태면 다시 저장하지 않도록 조정
- 서비스 내부 org 재조회를 제거하고 JPA reference 사용으로 대체

대상 코드:

- [ApiKeyAuthFilter.java](../../src/main/java/com/clickchecker/web/filter/ApiKeyAuthFilter.java)
- [EventCommandService.java](../../src/main/java/com/clickchecker/event/service/EventCommandService.java)

### 변경 전 흐름 요약

변경 전 `/api/events` write path 앞단은 아래 순서로 움직였다.

1. API key로 org 조회
2. 같은 org row의 `apiKeyLastUsedAt` update
3. 서비스 안에서 org 재조회
4. 이후 user lookup/create와 event 저장 진행

즉 이벤트를 저장하기도 전에, 앞단에서 org row 읽기/쓰기 비용이 반복되고 있었다.

### 변경 후 흐름 요약

변경 후에는 아래처럼 정리됐다.

1. API key로 org 조회
2. `apiKeyLastUsedAt`는 최근 1분 안에 이미 기록돼 있으면 update 생략
3. 서비스 안에서는 org 재조회 대신 JPA reference 사용
4. 이후 user lookup/create와 event 저장 진행

즉 구조 전체는 유지하되, **핫패스 앞단의 불필요한 write/select 한 쌍을 줄인 상태**로 바뀌었다.

### 기대 효과

- 같은 org row에 대한 과도한 update 감소
- 인증 이후 hot path의 불필요한 select 제거
- 동일 rate에서 더 적은 VU와 더 낮은 지연으로 write 요청 처리

- `r100`
  - 기준: `w1-20260319-135509-r100`
  - 재검증: `w1-20260319-150314-r100-retest`

### 검증 방식

이번 개선은 아래 방식으로 검증했다.

- 동일한 `W1` 시나리오 유지
- 동일한 local single-tenant 조건 유지
- fresh org / fresh API key / 기존 사용자 1000명 seed 유지
- 바뀐 것은 코드와 runId뿐이고, rate는 기존 baseline과 같은 `50`, `100`으로 재검증

즉 결과 해석은 "다른 시나리오를 돌렸다"가 아니라, **같은 실험에서 코드 변경만 들어갔을 때 무엇이 달라졌는지**를 보는 방식이다.

### 결과 요약

#### `r50`

- `p95`: `696.45ms -> 111.12ms`
- `p99`: `1.77s -> 344.7ms`
- `dropped_iterations`: `24 -> 0`
- `vus max`: `114 -> 29`

#### `r100`

- `p95`: `6.39s -> 471.2ms`
- `p99`: `8.12s -> 1.44s`
- `dropped_iterations`: `468 -> 21`
- `vus max`: `500 -> 208`
- 상태: `threshold_fail -> success`

### 해석

- 첫 `r100` failure의 핵심 원인이 event insert 자체만은 아니었다.
- hot path 앞단의 작은 반복 비용 두 개만 줄여도 higher-rate 구간이 크게 안정됐다.
- 즉 현재 구조는 전체 재설계가 필요한 상태라기보다, 핫패스 비용을 순서대로 덜어내며 버티는 구간을 넓힐 수 있는 상태에 가깝다.

### 이번 개선으로 확인된 것

- 구조 전체가 잘못된 것은 아니었다.
- `apiKeyLastUsedAt` update와 org 재조회는 실제로 유의미한 hot path 비용이었다.
- `100 RPS`가 "한계 구간"처럼 보였던 이유 중 상당 부분이 event 저장 본체가 아니라 앞단 부가 비용에 있었다.
- 따라서 다음 단계도 같은 방식으로, 기능 의미를 크게 흔들지 않는 비용부터 순서대로 줄여보는 접근이 타당하다.

### 남은 리스크

- user lookup/create 비용은 그대로 남아 있다.
- local single-tenant 기준 결과라 prod direct, multi-tenant, 더 높은 rate에서는 다른 병목이 다시 나타날 수 있다.
- `r200` 이상에서는 여전히 event insert / user 정규화 / 인덱스 비용이 다시 주요 병목이 될 수 있다.

### `r200` 비교 probe로 확인한 것

앞단 비용을 줄인 뒤 `r200`는 짧은 probe로만 확인했다.

- 기본 `80/20`:
  - `runId=w1-20260319-155609-r200-probe`
  - `threshold_fail`
  - `p95=5.8s`
  - `dropped_iterations=2241`
- 기존 사용자 `100/0`:
  - `runId=w1-20260319-161200-r200-existing100`
  - `success`
  - `p95=4.66s`
  - `dropped_iterations=621`
- 기존/신규 `50/50`:
  - `runId=w1-20260319-161709-r200-existing50`
  - `threshold_fail`
  - `p95=6.35s`
  - `dropped_iterations=2655`

이 비교로 아래 두 가지가 분명해졌다.

- 신규 user 생성은 분명히 비용을 더 키운다.
- 하지만 신규 생성만 없애도 `200 RPS`가 완전히 해결되지는 않는다.

즉 현재 남은 문제는 느린 `users` 조회 쿼리라기보다, `event_user` 조회/생성과 `event` 저장이 함께 묶인 write 본체 경로에 더 가깝다.

### W1 1차 종료 판단

이번 단계에서는 write 경로의 앞단 비용을 먼저 줄이는 데 집중했다.

- `apiKeyLastUsedAt` 매요청 저장 완화
- organization 재조회 제거

이후 `r50`과 `r100`은 모두 뚜렷하게 개선됐고, 첫 `r100` failure 구간은 통과 구간으로 이동했다.

반면 `r200` probe와 사용자 비율 비교 결과, 현재 남은 비용은 불필요한 부가 작업보다 `event_user` 조회/생성 및 `event` 저장이 함께 묶인 write 본체 경로에 더 가깝다는 점이 확인됐다.

이 구조는 funnel, retention, users overview 같은 사용자 단위 분석 요구와 연결되어 있으므로, 지금 단계에서 성급하게 단순화하지 않는다.
write 경로는 여기서 1차 정리로 마무리하고, 다음에는 read(`R1/R2/R3`)와 mixed(`M1`) 부하를 확인한 뒤 전체 병목을 다시 비교해 구조 개선 우선순위를 정한다.
