# 24. M1 준비 및 실행 구조

## 문서 목적

`M1` mixed baseline을 반복 가능하게 실행하기 위한 준비와 실행 구조를 고정한다.  
이번 문서는 `prepare -> restore -> run -> meta -> artifacts` 경계를 유지하면서, mixed 전용 dataset snapshot을 같은 시작 상태로 되돌려 write/read 경합을 비교 가능하게 만드는 데 목적이 있다.

## 적용 범위

- 대상 시나리오:
  - `M1`
- 대상 환경:
  - `local`
  - 이후 `prod direct`
- 이번 문서가 다루는 범위:
  - mixed 전용 dataset/snapshot 구조
  - `seed`, `prepare`, `restore`, `run`의 역할
  - `meta.json` 구조
  - run 단위
  - cleanup / 실패 run 처리 기준

## 환경 의존성

### local

- `docker compose` 네트워크 안에서 app 컨테이너를 직접 호출한다.
- mixed 전용 dataset org와 snapshot을 로컬 DB 기준으로 복원한다.
- 첫 mixed baseline과 빠른 retest, snapshot restore 검증에 적합하다.

### prod direct

- 실제 EC2 / RDS 자원을 사용한다.
- public ingress를 거치지 않고 앱 포트를 직접 본다.
- local에서 본 write/read 경쟁이 실제 인프라에서도 비슷하게 나타나는지 확인할 때 사용한다.

원칙:

- 같은 구조라도 환경이 바뀌면 숫자 의미가 달라질 수 있다.
- local과 prod direct 결과는 그대로 등치하지 않고, 자원 차이를 같이 해석한다.

## 이전 구조 문서와 다른 점

- `W1`은 write-only라 fresh org per run 구조가 가능했지만, `M1`은 **같은 시작 상태를 맞춘 mixed snapshot restore**가 핵심이다.
- `R1`은 read-only라 dataset 재사용만 하면 됐지만, `M1`은 **run 전에 write 흔적까지 지워진 동일 상태**가 필요하다.
- `M1`은 `prepare -> restore -> run` 경계가 특히 중요하다.

## 실행 재현성 기준

- 같은 `snapshotVersion`, 같은 `queryPreset`, 같은 `meta.json`이면 같은 조건으로 run을 다시 재현할 수 있어야 한다.
- run 스크립트는 `meta.json`만 있으면 재실행 가능해야 한다.
- snapshot restore는 명시적으로 호출되며, run 시작 시점의 org 상태를 동일하게 맞춰야 한다.

## seed 스크립트 역할

예상 파일:

- `scripts/perf/m1/seed-dataset-local.sh`

역할:

- mixed 전용 org 생성
- API key 발급
- route template / event type mapping 생성
- read 가능한 base dataset 생성
- snapshotVersion 생성

원칙:

- seed 스크립트는 baseline 시작 전에만 수행한다.
- run마다 seed를 다시 만들지 않는다.

## restore 스크립트 역할

예상 파일:

- `scripts/perf/m1/restore-snapshot-local.sh`

역할:

- 지정한 snapshotVersion 기준으로 DB 상태 복원
- write 흔적 제거
- read query window 기준 데이터 상태 복원

원칙:

- 각 mixed run 시작 전 restore를 수행한다.
- `10/10`, `20/20`, `30/30`은 모두 같은 snapshot 상태에서 시작한다.

## prepare 스크립트 역할

예상 파일:

- `scripts/perf/m1/prepare-local.sh`

역할:

- mixed 전용 org / API key 상태 확인
- snapshotVersion 존재 여부 확인
- read query window 확정
- write 식별 규칙 확정
- restore 스크립트 호출
- `runId`와 `outDir` 확정
- `meta.json` 생성

원칙:

- prepare는 상태 확인과 snapshot restore, run 메타 확정까지 담당한다.
- 실제 부하 실행은 하지 않는다.

## 실행 스크립트 역할

예상 파일:

- `scripts/perf/m1/run-local.sh`

역할:

- `meta.json` 읽기
- write/read mixed `k6` 시나리오 실행
- `summary.json`, `console.log` 저장
- 필요 시 `meta.json.status` 업데이트
- scripted capture 실행

원칙:

- 실행 스크립트는 부하 실행과 산출물 저장만 담당한다.
- org 생성, snapshot 생성, restore 같은 상태 변경은 prepare/restore 단계에 속한다.

## prepare / restore / run 경계

### seed

- mixed 전용 org와 dataset snapshot 생성
- baseline 시작 전에만 수행

### restore

- snapshotVersion 기준 DB 상태 복원
- run 시작 상태 고정

### prepare

- org / API key / snapshot 상태 확인
- read query window와 write 식별 규칙 확정
- restore 호출
- `runId` 생성
- 산출물 디렉터리 생성
- `meta.json` 생성

### run

- mixed `k6` 시나리오 실행
- `summary.json` 저장
- `console.log` 저장
- mixed 패널 기준 PNG 생성

원칙:

- snapshot restore 이후에만 run을 시작한다.
- prepare와 run이 섞이면 mixed 전/후 비교 가능성이 크게 떨어진다.

## 준비 데이터 규칙

### org / API key 기준

- `M1`은 mixed 전용 org와 API key를 사용한다.
- 같은 snapshotVersion에서는 org와 API key를 바꾸지 않는다.

### dataset / snapshot 기준

- datasetVersion:
  - `m1-v1`
- snapshotVersion:
  - baseline 시작 상태를 식별하는 별도 값
- base dataset:
  - read 가능한 historical 데이터셋
- read query window:
  - 고정 absolute `1일`

### write 식별 기준

- path prefix:
  - `/loadtest/m1`
- eventType:
  - `loadtest_mixed`
- runId:
  - `m1-...`

### read query preset 기준

- API:
  - `overview`
- query window:
  - snapshot metadata에 기록된 고정 absolute `1일`
- 필터:
  - `externalUserId` 없음
  - `eventType` 없음

원칙:

- run 동안 생성되는 write 이벤트는 read query window 밖에 두는 것을 기본으로 한다.
- 첫 `M1`은 결과 값 자체보다 shared resource 경쟁 관찰이 목적이다.

## run 단위 기준

- `m1-w10-r10`
- `m1-w20-r20`
- `m1-w30-r30`

원칙:

- 각 run은 별도 `runId`, 별도 산출물로 남긴다.
- 매 run 시작 전에 같은 snapshotVersion으로 restore한다.
- write/read 비율은 `50:50`을 유지한다.

## meta.json 최소 구조

예시:

```json
{
  "scenario": "m1",
  "runId": "m1-20260320-000000-w10-r10",
  "target": {
    "environment": "local",
    "writeApi": "POST /api/events",
    "readApi": "GET /api/v1/events/analytics/aggregates/overview"
  },
  "dataset": {
    "datasetVersion": "m1-v1",
    "snapshotVersion": "m1-v1-snap1",
    "orgId": 21,
    "queryPreset": "m1-overview-1d-fixed",
    "queryWindowFrom": "2026-03-18T09:00:00Z",
    "queryWindowTo": "2026-03-19T09:00:00Z"
  },
  "write": {
    "pathPrefix": "/loadtest/m1",
    "eventType": "loadtest_mixed",
    "rate": 10
  },
  "read": {
    "rate": 10
  },
  "run": {
    "warmup": "2m",
    "duration": "10m",
    "cooldown": "1m"
  },
  "capture": {
    "mode": "scripted",
    "profile": "m1"
  }
}
```

## 실패 run / cleanup 기준

- prepare 단계에서 `meta.json`도 못 만든 빈 디렉터리는 보존 대상이 아니다.
- `meta.json`이 생성된 뒤 실패한 run은 실패 산출물로 남긴다.
- snapshot restore 실패 run은 원인 확인 전 다음 run으로 넘어가지 않는다.

## 재실행 규칙

- `M1` 재실행은 같은 `meta.json`과 같은 snapshotVersion으로 다시 실행 가능해야 한다.
- 같은 snapshotVersion, 같은 query preset, 같은 pair rate 조건이면 retest 비교가 가능해야 한다.
- snapshotVersion이 바뀌면 `M1` baseline도 새 회차로 다시 시작한다.

## 이번 문서의 종료 조건

- `M1` 준비 / 복원 / 실행 경계가 `seed -> restore -> prepare -> run -> artifacts` 기준으로 고정된다.
- mixed 전용 dataset과 snapshot reset 구조가 명확해진다.
- 이후 `scripts/perf/m1/*`, `k6/m1/*` 구현 기준으로 바로 옮길 수 있을 정도로 준비 정보가 정리된다.
