# 15. R1 준비 및 실행 구조

## 문서 목적

`R1` read-heavy overview baseline을 반복 가능하게 실행하기 위한 준비와 실행 구조를 고정한다.  
이번 문서는 `prepare -> run -> meta -> artifacts` 경계를 유지하면서도, `W1`와 달리 **고정 read 데이터셋 org를 재사용**하는 구조를 명확히 정리하는 데 목적이 있다.

## 적용 범위

- 대상 시나리오:
  - `R1`
- 대상 환경:
  - `local`
  - 이후 `prod direct`
- 이번 문서가 다루는 범위:
  - read 전용 데이터셋 준비 방식
  - `prepare`와 `run`의 역할
  - `meta.json` 구조
  - run 단위
  - cleanup / 실패 run 처리 기준

## 공통 구조에서 재사용하는 항목

- [02-성능-실험-산출물-및-기록-규칙.md](02-성능-실험-산출물-및-기록-규칙.md)의 산출물 저장 규칙을 그대로 따른다.
- [03-대규모-부하-테스트-런북.md](03-대규모-부하-테스트-런북.md)의 dry run -> baseline -> retest 흐름을 그대로 따른다.
- run 결과 요약은 [04-대규모-부하-테스트-기록.md](04-대규모-부하-테스트-기록.md)에 반영한다.
- baseline 여부와 비교 대상 run 여부는 `meta.json` 필드(`isBaseline`, `compareTo`)로 관리한다.

## 환경 의존성

### local

- `docker compose` 네트워크 안에서 app 컨테이너를 직접 호출한다.
- read 전용 데이터셋 org를 로컬 DB에 만들어 재사용한다.
- 첫 baseline, dataset seed 검증, 빠른 반복에 적합하다.

### prod direct

- 실제 EC2 / RDS 자원을 사용한다.
- public ingress를 거치지 않고 앱 포트를 직접 본다.
- local에서 본 `overview` read 비용이 실제 인프라에서도 비슷하게 드러나는지 확인할 때 사용한다.

원칙:

- 같은 구조라도 환경이 바뀌면 숫자 의미가 달라질 수 있다.
- local 결과와 prod direct 결과는 그대로 등치하지 않고, 환경 차이를 같이 해석한다.

## 이전 구조 문서와 다른 점

- `W1`은 run마다 fresh org를 만들었지만, `R1`은 **고정 데이터셋 org를 재사용**한다.
- `W1`은 run마다 입력 데이터를 새로 만들었지만, `R1`은 **같은 데이터셋을 여러 번 읽는 구조**다.
- `W1`은 사용자 seed를 run 준비의 일부로 다뤘지만, `R1`은 **dataset seed와 run 준비를 분리**해 본다.

## 실행 재현성 기준

- 같은 `datasetVersion`과 같은 `meta.json`이면 같은 query window와 같은 데이터셋 조건으로 run을 다시 재현할 수 있어야 한다.
- run 스크립트는 `meta.json`만 있으면 재실행 가능해야 한다.
- dataset seed는 deterministic해야 한다.
- dataset 재생성은 명시적 reset 없이는 자동으로 일어나지 않아야 한다.

## 준비 스크립트 역할

예상 파일:

- `scripts/perf/r1/prepare-local.sh`

역할:

- read 전용 org 확인 또는 생성
- API key 확인 또는 발급
- route template / event type mapping seed 확인 또는 생성
- read 데이터셋 seed 확인 또는 생성
- 고정 query window 확인
- `runId`와 `outDir` 확정
- `meta.json` 생성

원칙:

- 준비 스크립트는 **상태 생성과 run 메타 확정**만 담당한다.
- 실행 결과(`summary.json`, latency, 실패율)는 준비 스크립트가 만들지 않는다.

## 실행 스크립트 역할

예상 파일:

- `scripts/perf/r1/run-local.sh`

역할:

- `meta.json` 읽기
- 해당 rate 기준 `k6` 실행
- `summary.json`, `console.log` 저장
- 필요 시 `meta.json.status` 업데이트
- `capture.mode=scripted`, `capture.profile=r1` 기준으로 Grafana renderer 캡처 실행

원칙:

- 실행 스크립트는 부하 실행과 산출물 저장만 담당한다.
- dataset 재생성, org 생성, mapping 변경 같은 상태 생성은 하지 않는다.
- scripted capture는 run 산출물 생성의 일부로 본다.

## prepare / run 경계

### dataset seed

예상 파일:

- `scripts/perf/r1/seed-dataset-local.sh`

역할:

- route template / event type mapping 생성
- `users`, `events` 대량 seed
- query window metadata 생성
- dataset metadata 저장

원칙:

- dataset seed는 run과 분리된 독립 단계로 본다.
- `R1`, `R2`, `R3`는 같은 datasetVersion을 재사용할 수 있어야 한다.

### prepare

- read 전용 org 확인 또는 생성
- API key 준비
- route template / event type mapping seed 확인
- dataset seed 확인 또는 생성
- 고정 query window metadata 확정
- `runId` 생성
- 산출물 디렉터리 생성
- `meta.json` 생성

### run

- `k6` 시나리오 실행
- `summary.json` 저장
- `console.log` 저장
- Grafana 캡처 시간 범위 기록
- `capture.mode=scripted`, `capture.profile=r1`이면 renderer API로 `analytics-rps.png`, `analytics-latency.png` 등 read 패널 PNG 생성

원칙:

- dataset seed는 prepare 단계에만 속한다.
- run 단계에서 데이터셋을 바꾸지 않는다.
- prepare와 run이 섞이면 read run 간 비교 가능성이 크게 떨어진다.

## 준비 데이터 규칙

### org / API key 준비

공식 흐름:

```text
signup
-> admin org create
-> api key 발급
```

원칙:

- 현재 제품의 공식 진입 흐름을 그대로 사용한다.
- `R1`, `R2`, `R3`는 가능하면 같은 read 데이터셋 org와 API key를 재사용한다.

### read 데이터셋 기준

- org name:
  - `perf-r1-dataset`
- total events:
  - `50,000`
- identified users:
  - `1,000`
- raw path:
  - `30개`
- raw event types:
  - `5개`
- 기간 분포:
  - 최근 `7일`

### route / event type 정규화 seed 기준

- route template는 고정 세트를 먼저 만든다.
- raw path 일부는 의도적으로 `UNMATCHED_ROUTE`에 남긴다.
- event type mapping도 고정 세트를 먼저 만든다.
- raw event type 일부는 의도적으로 `UNMAPPED_EVENT_TYPE`에 남긴다.

원칙:

- `overview`의 coverage 지표가 의미 있게 계산되도록 seed한다.
- run 중 mapping이나 template를 바꾸지 않는다.

#### seed 순서

```text
read org 준비
-> API key 준비
-> route template 생성
-> event type mapping 생성
-> dataset seed
-> query window metadata 확정
```

원칙:

- route template와 event type mapping은 이벤트 적재 전에 먼저 만든다.
- 같은 datasetVersion에서는 seed 순서도 고정한다.

### query window 기준

- baseline query window는 데이터셋 metadata에 기록된 **고정 absolute 1일 구간**을 사용한다.
- `R1` run은 모두 같은 `from`, `to`를 사용한다.

예:

- dataset range:
  - 최근 `7일`
- query window:
  - 마지막 `1일`

### 준비 단계 메타데이터

준비 스크립트는 최소 아래를 남긴다.

- datasetVersion
- dataset seed 시작 / 종료 시각
- orgId
- queryWindowFrom
- queryWindowTo
- totalEvents
- identifiedUsers
- rawPathCount
- rawEventTypeCount
- identifiedEvents
- anonymousEvents
- matchedPathCount
- unmatchedPathCount
- mappedEventTypeCount
- unmappedEventTypeCount

### dataset seed 분포 원칙

- 이벤트는 최근 `7일` 전체에 퍼뜨린다.
- 마지막 `1일` query window에 상대적으로 더 많은 이벤트를 둔다.
- 식별 이벤트 `40,000`, 익명 이벤트 `10,000` 비율을 유지한다.
- raw path 30개 중 `24개`는 route template에 매칭되고, `6개`는 unmatched로 남긴다.
- raw event type 5개 중 `4개`는 canonical mapping 대상으로 두고, `1개`는 unmapped로 남긴다.
- 사용자 1,000명은 완전히 균등하게 분포시키지 않고, 일부 hot user와 일부 cold user가 섞인 형태로 seed한다.

원칙:

- read baseline의 목적은 현실 전체를 재현하는 것이 아니라, `overview`가 너무 단순하지 않으면서도 반복 비교 가능한 첫 기준선을 만드는 것이다.
- dataset seed 규칙이 바뀌면 baseline과 retest를 직접 비교하지 않는다.

## run 단위 기준

- `r1-10`
- `r1-30`
- `r1-50`

원칙:

- 준비 스크립트는 한 번에 한 run만 준비한다.
- 실행 스크립트도 한 번에 한 run만 실행한다.
- 각 run은 별도 `runId`와 별도 산출물을 갖되, 데이터셋 org는 재사용한다.

## `meta.json` 기준

위치:

- `artifacts/perf/r1/<runId>/meta.json`

원칙:

- 실행 스크립트는 필요한 값을 `meta.json`에서만 읽는다.
- 실행 스크립트가 dataset org를 다시 만들거나 query window를 다시 계산하지 않는다.
- `meta.json`에는 실행 조건과 상태만 넣고 결과값은 넣지 않는다.

### 권장 스키마

```json
{
  "scenario": "R1",
  "runId": "r1-20260319-180000-r30",
  "timestamp": "2026-03-19T18:00:00+09:00",
  "status": "prepared",
  "isBaseline": true,
  "compareTo": null,
  "env": {
    "name": "local",
    "prepareBaseUrl": "http://localhost:8080",
    "baseUrl": "http://app:8080",
    "appCount": 1,
    "dbInstance": "local-postgres"
  },
  "auth": {
    "apiKey": "plain-api-key",
    "apiKeyPrefix": "ck_live"
  },
  "dataset": {
    "orgId": 1,
    "orgName": "perf-r1-dataset",
    "datasetVersion": "r1-v1",
    "totalEvents": 50000,
    "identifiedUsers": 1000,
    "rawPathCount": 30,
    "rawEventTypeCount": 5,
    "rangeFrom": "2026-03-01T00:00:00Z",
    "rangeTo": "2026-03-08T00:00:00Z",
    "queryWindowFrom": "2026-03-07T00:00:00Z",
    "queryWindowTo": "2026-03-08T00:00:00Z"
  },
  "request": {
    "endpoint": "/api/v1/events/analytics/aggregates/overview",
    "from": "2026-03-07T00:00:00Z",
    "to": "2026-03-08T00:00:00Z",
    "externalUserId": null,
    "eventType": null
  },
  "load": {
    "rate": 30,
    "warmup": "2m",
    "duration": "10m",
    "cooldown": "1m"
  },
  "capture": {
    "mode": "scripted",
    "timeRange": "start-2m ~ end+2m"
  }
}
```

주의:

- `datasetVersion`과 `request.from/to`는 read 비교에서 특히 중요하다.
- 같은 시나리오라도 datasetVersion이 바뀌면 baseline과 직접 비교하지 않는다.

## 실패 run 처리 기준

- 잘못된 API key, 잘못된 query range, dataset org 누락으로 run이 실패하면 무효 처리한다.
- 앱 재시작, OOM, Grafana / Prometheus 수집 누락은 무효 처리하고 재실행한다.
- threshold를 넘긴 run은 유효한 실패로 본다.
- 4xx가 query parameter 오류로 발생하면 실험 오류로 간주한다.

## cleanup 기준

- dataset org와 dataset seed는 자동 삭제하지 않는다.
- run 산출물만 회차별로 별도 저장한다.
- dataset 재생성은 `RESET_DATASET=true` 같은 명시적 플래그가 있을 때만 수행한다.

원칙:

- read는 데이터셋 상태가 비교의 핵심이므로, 기본은 보존이다.
- `prepare` 단계에서 `meta.json`도 못 만든 빈 디렉터리는 보존 대상이 아니다.

## 실행 순서

### 1. 첫 준비

- read 데이터셋 org 생성
- API key 준비
- route template / event type mapping seed
- dataset seed
- 고정 query window 확정

### 2. 드라이런

- `10 RPS` 짧은 run으로 endpoint, query range, 산출물 구조 확인

### 3. 공식 baseline

- `10 RPS`
- `30 RPS`
- `50 RPS`

### 4. 재검증

- 코드 또는 인덱스 변경 후 같은 datasetVersion과 query window로 다시 실행

## 이전 시나리오의 어떤 구조 가정을 그대로 가져오지 않는지

- `W1`의 fresh org per run 구조는 가져오지 않는다.
- `W1`의 per-run 사용자 seed 구조는 가져오지 않는다.
- `W1`의 입력 path / eventType 규칙을 그대로 쓰지 않는다.
- `W1`의 "run이 곧 데이터 생성" 구조를 `R1`에 그대로 적용하지 않는다.
