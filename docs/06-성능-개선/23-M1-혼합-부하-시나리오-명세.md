# 23. M1 혼합 부하 시나리오 명세

## 문서 목적

`M1`은 write-heavy와 read-heavy가 같은 tenant와 같은 DB 자원을 함께 사용할 때 어떤 구간부터 경쟁이 발생하는지 보기 위한 첫 mixed 시나리오다.  
이번 문서는 `POST /api/events`와 `GET /api/v1/events/analytics/aggregates/overview`를 어떤 비율, 어떤 데이터셋, 어떤 run 조건으로 동시에 실행할지 고정하고, 이후 `local -> prod direct` 비교와 구조 개선 우선순위 판단의 기준선을 만드는 데 목적이 있다.

## 적용 범위

- 대상 시나리오:
  - `M1`
- 대상 API:
  - `POST /api/events`
  - `GET /api/v1/events/analytics/aggregates/overview`
- 이번 문서가 다루는 범위:
  - 첫 mixed baseline 조합
  - write/read 비율 규칙
  - mixed 전용 데이터셋과 snapshot reset 기준
  - query window / write 식별 규칙
  - threshold / 중단 조건

## 이전 시나리오와 다른 점

- `W1`은 write-only였고, `R1`은 overview read-only였지만, `M1`은 **같은 org / 같은 DB 안에서 write와 overview read를 동시에 붙인다.**
- `R1`은 read query window만 고정하면 됐지만, `M1`은 **run 시작 상태를 동일하게 맞추기 위한 snapshot reset**이 중요하다.
- `W1`는 fresh org per run 구조였지만, `M1`은 **mixed 전용 dataset org와 snapshot을 재사용**하는 쪽이 더 중요하다.

## 이번 시나리오에서 새로 중요해지는 변수

- write/read 비율
- 같은 org 안에서의 shared resource 경쟁
- write insert와 overview read가 동시에 들어올 때의 지연 상승
- fixed historical read window와 live write 데이터의 분리 방식
- run 전 snapshot reset 여부

## 환경 고정값 / 변경값

### 고정값

- 시나리오:
  - `M1`
- write API:
  - `POST /api/events`
- read API:
  - `GET /api/v1/events/analytics/aggregates/overview`
- executor:
  - write / read 각각 `constant-arrival-rate`
- warm-up:
  - `2m`
- main run:
  - `10m`
- cooldown:
  - `1m`
- 혼합 비율:
  - `50:50`
- read query window:
  - 고정 absolute `1일`
- read dataset:
  - `M1` 전용 dataset snapshot 재사용
- capture profile:
  - `m1`

### 변경값

- 환경:
  - `local` -> 이후 `prod direct`
- run별 pair rate:
  - write/read `10/10`
  - write/read `20/20`
  - write/read `30/30`
- `runId`

원칙:

- baseline과 retest를 비교할 때 write/read 비율과 read query window는 바꾸지 않는다.
- 같은 회차 안에서는 snapshot, org, datasetVersion을 바꾸지 않는다.

## 이번 시나리오에서 일부러 단순화한 항목

- 첫 `M1`은 `W1 + R1 overview` 조합만 본다.
- 첫 `M1`은 `50:50` 비율만 사용한다.
- 첫 `M1`은 single-tenant mixed 기준으로만 본다.
- 첫 `M1`은 `R2 routes`, `R3 time-buckets`를 같이 섞지 않는다.
- 첫 `M1`은 read query window를 고정 absolute historical range로 유지한다.

## 이전 시나리오의 어떤 가정을 가져오지 않는지

- `W1`의 fresh org per run 가정은 그대로 가져오지 않는다.
- `R1`의 read-only 안정 구간 해석을 그대로 mixed 결과에 대입하지 않는다.
- `R2`, `R3`의 집계형/추세형 비용 축은 첫 `M1`에 직접 포함하지 않는다.

## 해석 시 주의사항

- `M1`에서 read query는 fixed historical window를 사용하므로, run 중 새로 생성된 write 이벤트는 read 결과 자체에 직접 반영되지 않을 수 있다.
- 이건 첫 `M1`에서 **결과 값 변화보다 shared resource 경쟁**을 더 선명하게 보기 위한 의도적인 선택이다.
- 따라서 첫 `M1` 해석은 "write와 read가 같은 org/DB에서 동시에 들어올 때 지연이 얼마나 올라가는가"에 초점을 둔다.

## 실행 조건

### 환경 단계

#### 1차

- 환경:
  - `local`
- 목적:
  - mixed baseline과 첫 한계 구간 확보

#### 2차

- 환경:
  - `prod direct`
- 목적:
  - 실제 EC2 / RDS 조건에서 shared resource 경쟁 재현 여부 확인

### executor

- write / read 각각 `constant-arrival-rate`

이유:

- write와 read 비율을 명확하게 고정하기 좋다.
- 한쪽만 밀리는 것이 아니라 두 요청 축이 같이 들어가는 mixed 상태를 재현하기 좋다.

### run 단위

- write/read `10/10`
- write/read `20/20`
- write/read `30/30`

원칙:

- 각 run은 별도 `runId`, 별도 산출물로 남긴다.
- write/read 비율은 `50:50`을 유지한다.
- run 시작 전 snapshot을 같은 상태로 복원한다.

### 시간 조건

- warm-up:
  - `2m`
- main run:
  - `10m`
- cooldown:
  - `1m`

### VU 기준

#### `10/10`

- write `preAllocatedVUs = 20`, `maxVUs = 60`
- read `preAllocatedVUs = 20`, `maxVUs = 60`

#### `20/20`

- write `preAllocatedVUs = 40`, `maxVUs = 120`
- read `preAllocatedVUs = 40`, `maxVUs = 120`

#### `30/30`

- write `preAllocatedVUs = 60`, `maxVUs = 180`
- read `preAllocatedVUs = 60`, `maxVUs = 180`

원칙:

- baseline과 retest는 같은 VU 기준을 유지한다.
- 첫 `M1`은 `W1`, `R1`보다 보수적으로 시작하되, pair rate를 일정하게 올려간다.

## 입력 데이터 규칙

### 조직 기준

- 첫 `M1`은 **mixed 전용 org 1개**를 사용한다.
- run마다 빈 org를 새로 만들지 않고, snapshot reset 기준으로 같은 시작 상태를 재사용한다.

예:

- `perf-m1-dataset`

### 데이터셋 기준

- datasetVersion:
  - `m1-v1`
- base dataset:
  - `R1`과 유사한 read 가능 데이터셋
- total events:
  - `50,000`
- identified users:
  - `1,000`
- read query window:
  - absolute `1일`

### write 식별 규칙

- path prefix:
  - `/loadtest/m1`
- eventType:
  - `loadtest_mixed`
- runId:
  - `m1-...`

### read query 기준

- API:
  - `overview`
- query window:
  - snapshot metadata에 기록된 고정 absolute `1일`
- 필터:
  - `externalUserId` 없음
  - `eventType` 없음

## threshold / 중단 조건

- `http_req_failed < 1%`
- write / read 어느 한쪽이라도 `p95 >= 1500ms`면 한계 구간 신호로 본다.
- `p99 >= 3000ms`는 강한 경고 신호로 본다.
- `dropped_iterations > 0`는 shared resource 경쟁 신호로 우선 해석한다.

해석 기준:

- `10/10`은 first mixed baseline 적합 여부 확인 구간이다.
- `20/20`은 shared resource 경쟁이 시작되는지 보는 구간이다.
- `30/30`은 앞 구간이 안정적일 때만 본다.

## 기록 시 남길 핵심 항목

- datasetVersion / snapshotVersion
- orgId
- read query window
- write path prefix / eventType
- write rate / read rate
- write/read 각각의 p95 / p99
- dropped_iterations
- DB active connections
- Hikari pool usage
- mixed 패널 기준 latency 추세
