# 21. R3 준비 및 실행 구조

## 문서 목적

`R3` time-buckets read-heavy baseline을 반복 가능하게 실행하기 위한 준비와 실행 구조를 고정한다.  
이번 문서는 `prepare -> run -> meta -> artifacts` 경계를 유지하면서, `R1`과 같은 read 데이터셋을 재사용하되 **query window, bucket, timezone을 추세형 read에 맞게 고정하는 구조**를 명확히 정리하는 데 목적이 있다.

## 적용 범위

- 대상 시나리오:
  - `R3`
- 대상 환경:
  - `local`
  - 이후 `prod direct`
- 이번 문서가 다루는 범위:
  - `R1` 데이터셋 재사용 방식
  - `prepare`와 `run`의 역할
  - `meta.json` 구조
  - run 단위
  - cleanup / 실패 run 처리 기준

## 환경 의존성

### local

- `docker compose` 네트워크 안에서 app 컨테이너를 직접 호출한다.
- `R1`에서 만든 read 데이터셋 org와 datasetVersion을 그대로 재사용한다.
- 첫 baseline, 빠른 retest, bucket 조립 비용 확인에 적합하다.

### prod direct

- 실제 EC2 / RDS 자원을 사용한다.
- public ingress를 거치지 않고 앱 포트를 직접 본다.
- local에서 본 trend read 비용이 실제 인프라에서도 비슷하게 드러나는지 확인할 때 사용한다.

원칙:

- 같은 구조라도 환경이 바뀌면 숫자 의미가 달라질 수 있다.
- local 결과와 prod direct 결과는 그대로 등치하지 않고, 환경 차이를 같이 해석한다.

## 이전 구조 문서와 다른 점

- `R2`는 `routes` 기준 group-by / topN read였지만, `R3`는 **bucket 조립과 빈 bucket 채우기**가 핵심이다.
- `R2`는 `top=10`이 중요했지만, `R3`는 **`bucket=HOUR`, `timezone=UTC`**가 중요하다.
- `R3`는 현재 dataset 범위에 맞춰 **고정 `7일` query window**를 사용한다.

## 실행 재현성 기준

- 같은 `datasetVersion`, 같은 `queryPreset`, 같은 `meta.json`이면 같은 조건으로 run을 다시 재현할 수 있어야 한다.
- run 스크립트는 `meta.json`만 있으면 재실행 가능해야 한다.
- `R3`는 기본적으로 `R1` datasetVersion을 그대로 재사용한다.
- dataset 재생성은 명시적 reset 없이는 자동으로 일어나지 않아야 한다.

## 준비 스크립트 역할

예상 파일:

- `scripts/perf/r3/prepare-local.sh`

역할:

- read 전용 org / API key 재사용 상태 확인
- datasetVersion 존재 여부 확인
- `R3` 전용 query window, `bucket`, `timezone` 확정
- `runId`와 `outDir` 확정
- `meta.json` 생성

원칙:

- 준비 스크립트는 **상태 확인과 run 메타 확정**만 담당한다.
- `R3`에서 기본적으로 dataset seed를 다시 만들지 않는다.
- 실행 결과(`summary.json`, latency, 실패율)는 준비 스크립트가 만들지 않는다.

## 실행 스크립트 역할

예상 파일:

- `scripts/perf/r3/run-local.sh`

역할:

- `meta.json` 읽기
- 해당 rate 기준 `k6` 실행
- `summary.json`, `console.log` 저장
- 필요 시 `meta.json.status` 업데이트
- `capture.mode=scripted` 기준 Grafana renderer 캡처 실행

원칙:

- 실행 스크립트는 부하 실행과 산출물 저장만 담당한다.
- dataset seed, org 생성, route template 변경 같은 상태 생성은 하지 않는다.
- scripted capture는 run 산출물 생성의 일부로 본다.

## prepare / run 경계

### dataset 재사용

- `R3`는 `R1` datasetVersion과 org를 그대로 사용한다.
- `R3`에서 dataset seed를 다시 만드는 경우는 reset 또는 datasetVersion 변경 시뿐이다.

### prepare

- read 전용 org / API key 상태 확인
- datasetVersion 존재 확인
- `R3` 고정 query window 확정
- `bucket` 값 확정
- `timezone` 값 확정
- `runId` 생성
- 산출물 디렉터리 생성
- `meta.json` 생성

### run

- `k6` 시나리오 실행
- `summary.json` 저장
- `console.log` 저장
- Grafana 캡처 시간 범위 기록
- scripted capture 기준 read 패널 PNG 생성

원칙:

- dataset 상태와 query preset은 prepare 단계에만 속한다.
- run 단계에서 데이터셋, query window, `bucket`, `timezone`을 바꾸지 않는다.
- prepare와 run이 섞이면 trend read 전/후 비교 가능성이 크게 떨어진다.

## 준비 데이터 규칙

### org / API key 재사용

- `R3`는 `R1`에서 만든 read 전용 org와 API key를 그대로 사용한다.

원칙:

- 같은 datasetVersion에서는 org와 API key를 바꾸지 않는다.
- route template / event type mapping도 기본적으로 `R1` 상태를 유지한다.

### read 데이터셋 기준

- datasetVersion:
  - `R1`과 동일
- org name:
  - `perf-r1-dataset`
- total events:
  - `50,000`
- identified users:
  - `1,000`
- query window:
  - absolute `7일`

### query preset 기준

- baseline query window:
  - 데이터셋 metadata에 기록된 고정 absolute `7일` 구간
- baseline `bucket`:
  - `HOUR`
- baseline `timezone`:
  - `UTC`
- 필터:
  - `externalUserId` 없음
  - `eventType` 없음

원칙:

- `R3` run은 모두 같은 `from`, `to`, `bucket`, `timezone`을 사용한다.
- route/eventType time-buckets 보조 확인도 같은 query preset을 쓴다.

### 준비 단계 메타데이터

준비 스크립트는 최소 아래를 남긴다.

- datasetVersion
- orgId
- queryPreset
- queryWindowFrom
- queryWindowTo
- bucket
- timezone
- expectedBucketCount
- captureProfile

## run 단위 기준

- `r3-10`
- `r3-30`
- `r3-50`

원칙:

- 각 run은 별도 `runId`, 별도 산출물로 남긴다.
- datasetVersion, query preset, `bucket`, `timezone`은 그대로 재사용한다.

## meta.json 최소 구조

예시:

```json
{
  "scenario": "r3",
  "runId": "r3-20260319-000000-r10",
  "target": {
    "api": "GET /api/v1/events/analytics/aggregates/time-buckets",
    "environment": "local"
  },
  "dataset": {
    "datasetVersion": "r1-v1",
    "orgId": 17,
    "queryPreset": "r3-time-buckets-7d-hour-utc",
    "queryWindowFrom": "2026-03-12T09:00:00Z",
    "queryWindowTo": "2026-03-19T09:00:00Z",
    "bucket": "HOUR",
    "timezone": "UTC",
    "expectedBucketCount": 168
  },
  "run": {
    "rate": 10,
    "warmup": "2m",
    "duration": "10m",
    "cooldown": "1m"
  },
  "capture": {
    "mode": "scripted",
    "profile": "r1"
  }
}
```

## 실패 run / cleanup 기준

- prepare 단계에서 `meta.json`도 못 만든 빈 디렉터리는 보존 대상이 아니다.
- `meta.json`이 생성된 뒤 실패한 run은 실패 산출물로 남긴다.
- datasetVersion과 query preset을 바꾼 retest는 baseline과 직접 비교 대상으로 섞지 않는다.

## 재실행 규칙

- `R3` 재실행은 같은 `meta.json`으로 다시 실행 가능해야 한다.
- 같은 datasetVersion, 같은 query preset, 같은 rate 조건이면 retest 비교가 가능해야 한다.
- `R1` 데이터셋이 바뀌면 `R3` baseline도 새 회차로 다시 시작한다.

## 이번 문서의 종료 조건

- `R3` 준비 / 실행 경계가 `prepare -> run -> meta -> artifacts` 기준으로 고정된다.
- `R1` datasetVersion 재사용 구조가 명확해진다.
- 이후 `scripts/perf/r3/*`, `k6/r3/*` 구현 기준으로 바로 옮길 수 있을 정도로 준비 정보가 정리된다.
