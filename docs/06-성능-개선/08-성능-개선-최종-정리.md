# 08. 성능 개선 최종 정리

## 문서 목적

이번 문서는 성능 개선 사이클 전체를 마무리하며,

- 처음 어떤 수치에서 시작했는지
- 최종적으로 어떤 수치까지 개선됐는지
- 실제로 유지한 개선과 버린 실험이 무엇인지
- 지금 상태를 어떤 baseline으로 볼 것인지

를 한 번에 확인하기 위한 최종 결론 문서다.

상세 run 기록은 [04-대규모-부하-테스트-기록.md](04-대규모-부하-테스트-기록.md),
개선 단위 이력은 [06-성능-개선-조치-이력.md](06-성능-개선-조치-이력.md),
종합 해석은 [07-성능-개선-종합.md](07-성능-개선-종합.md),
v2 realistic baseline 흐름은 [38-v2-local-baseline-1차-종합.md](38-v2-local-baseline-1차-종합.md)를 기준으로 본다.

## 최종 결론

- 이번 사이클에서 **실제로 유지할 가치가 확인된 개선**은
  - `EventUser` 경로 최적화
  - payload `OID -> TEXT`
  - resolver metadata cache
  - overview summary 단일 쿼리
  - unique-user raw pair 경량화
  - overview `5분` 응답 캐시
  - hourly rollup 적재 인프라
  - time-series 전체 read rollup 전환
  이다.
- 반대로 **현재 기준으로 채택하지 않기로 한 실험**은
  - overview rollup read
  - aggregate `routes/event-types` rollup
  이다.
- 최종 keep 상태는 **`overview cache + time-series rollup`** baseline이다.
- local realistic mixed 판단은 **`M2 s2 + warm-up 2m + 필요 시 rerun`** 기준으로 읽는 것이 맞다.

## 처음 수치 vs 최종 수치

아래 표는 같은 시나리오 안에서, 처음 기준선과 최종 유지 상태를 가장 잘 대표하는 run만 골라 비교한 것이다.

## 최종 baseline

현재 baseline을 대표하는 run은 아래와 같다.

- `R4`
  - runId: `r4-20260323-201254-s2-after-full-timeseries-rollup`
  - `p95 26.10ms`
  - `p99 135.23ms`
  - fail `0%`
- `M2 s2`
  - runId: `m2-20260324-015022-s2-after-aggregate-rollup-revert-check-rerun2`
  - warm-up: `2m`
  - throughput `193.09 req/s`
  - read `p95 1.09s`
  - write `p95 1.00s`
  - dropped `120`

즉 지금 로컬 기준 final baseline은

- overview는 `5분` 응답 캐시
- time-series는 hourly rollup read
- aggregate와 overview의 추가 rollup read 실험은 제외

조합으로 정리된다.

## prod-public 추가 확인

배포 후 public 도메인에서 realistic mixed upper band를 짧게 다시 확인했다.

- 실행 방식:
  - `prepare`는 EC2에서 수행
  - `run`은 local에서 `clickchecker.dev`로 직접 수행
  - 따라서 결과에는 public ingress / TLS / 인터넷 RTT가 함께 포함된다
- 핵심 결과:

| 구간 | runId | 결과 | 핵심 수치 | 해석 |
| --- | --- | --- | --- | --- |
| `M2 s2` | `m2-20260324-050026-s2-prod-public-localrun-warm2m-3m` | `success` | throughput `193.55 req/s`, read `p95 297.14ms`, write `p95 38.16ms`, fail `0%`, dropped `0` | `2m` warm-up 기준 `200`은 public 경로에서도 안정 구간이다 |
| `M2 s3` | `m2-20260324-051516-s3-prod-public-localrun-probe` | `success` | throughput `289.45 req/s`, read `p95 1.02s`, write `p95 580.91ms`, fail `0%`, dropped `0` | `300`도 아직 안정 구간이다 |
| `M2 350 probe` | `m2-20260324-064109-s35-prod-public-localrun-probe-reset` | `threshold_fail` | throughput `152.22 req/s`, read fail `31.70%`, write fail `19.59%`, dropped `6605` | fresh quick reset 후에도 `350`은 실패 구간이다 |

즉 현재 `prod-public + local run` 기준 realistic mixed 안전 upper bound는 **`300`**으로 정리하는 것이 가장 자연스럽다.

## 유지한 개선

- write 핫패스 앞단 비용 절감
  - `apiKeyLastUsedAt` 반복 update 완화
  - organization 재조회 제거
- `EventUser` 경로 최적화
  - native upsert 기반 user id 확보
- payload 저장 방식 변경
  - `OID -> TEXT`
- quick reset 전략 재설계
  - event-only quick reset
- resolver metadata cache
  - route template
  - event type mapping
- overview summary 단일 쿼리
- unique-user raw pair 경량화
- overview `5분` 응답 캐시
- hourly rollup 적재 인프라
  - rollup table
  - watermark
  - refresh service
  - advisory-lock scheduler
- time-series 전체 read rollup 전환
  - `time-buckets`
  - `route-time-buckets`
  - `event-type-time-buckets`
  - `route-event-type-time-buckets`

## 비채택한 실험

- overview rollup read
  - partial hour 보정, raw tail, top regrouping이 겹치며 mixed에서 반복적으로 회귀했다
  - 결론: `overview cache keep / overview rollup read drop`
- aggregate `routes/event-types` rollup
  - `R4`처럼 단순한 축엔 좋아 보여도 `M2`에서는 요청당 fan-out이 커져 반복 rerun에서도 회귀했다
  - 결론: 현재 구조에서는 채택하지 않음

## 이번 사이클에서 배운 점

- local mixed는 first-run 편차가 커서 `warm-up 2m`와 rerun 해석 기준이 중요하다.
- time-series처럼 **최종 응답 모양과 rollup 축이 잘 맞는 경로**는 실제로 크게 좋아진다.
- 반대로 overview, aggregate처럼 최종 regrouping이 앱에 많이 남아 있는 경로는 같은 hourly rollup을 단순 확장해도 이득보다 fan-out 비용이 더 커질 수 있다.
- 작은 query tweak는 대부분 한 번씩 확인했고, 이제 남은 후보는 더 큰 구조 변경 쪽이다.

## 다음 사이클로 넘긴 과제

- `route_key`, `canonical_event_type` 비정규화 검토
- aggregate 전용 rollup 또는 최종 차원 기준 사전 집계 구조 검토
- current shared DB work를 더 직접적으로 줄이는 구조 변경 후보 비교

즉 다음 사이클은 작은 쿼리 튜닝의 연장이 아니라, **더 큰 저장/집계 구조 설계 단계**로 보는 편이 맞다.
