# 26. Prod Direct 검증 계획

## 문서 목적

local 기준으로 확보한 `W1`, `R1`, `R2`, `R3`, `M1` 1차 결과를  
실제 EC2 / RDS 조건에서 다시 확인하기 위한 최소 검증 계획을 정리한다.

이번 문서는 새 시나리오를 정의하는 문서가 아니라,
**왜 지금 prod direct를 보고, 무엇을 어떤 순서로 검증할지**를 한 번에 보게 만드는 실행 계획 문서다.

## 1. 왜 지금 prod direct인가

local에서는 아래 항목이 이미 정리됐다.

- write 기준선과 1차 개선 효과
- overview / routes / time-buckets read의 비용 차이
- write + overview mixed 경쟁 구간

즉 다음 단계의 질문은
"local에서 보인 패턴이 실제 인프라에서도 비슷한가"에 가깝다.

따라서 지금 prod direct는
**local 결과를 그대로 믿기 전에, EC2 / RDS 조건에서 같은 경향이 다시 나오는지 확인하는 단계**로 본다.

## 2. 이번 단계 최소 검증 세트

이번 prod direct 단계에서는 아래 세 개만 먼저 본다.

### 2.1 `W1 r100`

- 목적:
  - write 대표 안정 구간 확인
- 이유:
  - local에서 첫 개선 뒤 통과 구간으로 이동한 write 기준선이기 때문

### 2.2 `R1 r10`

- 목적:
  - 가장 무거운 read 대표 안정 구간 확인
- 이유:
  - overview는 read 축 중 가장 무거운 대표 read였기 때문

### 2.3 `M1 10/10`

- 목적:
  - mixed 대표 안정 구간 확인
- 이유:
  - local에서 shared resource 경쟁 전 단계의 첫 mixed baseline으로 확보됐기 때문

원칙:

- 첫 prod direct는 coverage를 넓히기보다 **대표 축 3개를 최소 비용으로 다시 확인하는 것**이 목적이다.
- `R2`, `R3`, `M1 20/20`은 이 세 개 결과를 보고 다음 단계에서 판단한다.

## 3. 실행 위치와 요청 대상

### 실행 위치

- 가능하면 **EC2 내부에서 실행**

이유:

- direct app port가 외부에 열려 있지 않을 수 있다.
- 외부 네트워크 변수를 줄일 수 있다.
- prod direct의 의미를 더 선명하게 유지할 수 있다.

### 요청 대상

- public URL이 아니라 **target color app port direct 호출**
- 예:
  - `127.0.0.1:8081`
  - `127.0.0.1:8082`

원칙:

- 현재 target color와 direct port를 먼저 확인한다.
- public ingress를 거치는 호출은 이번 단계 대상이 아니다.

## 4. 준비 체크리스트

실행 전 아래 항목을 먼저 확인한다.

- target color 확인
- direct 대상 포트 확인
- prod direct 요청 URL 확인
- Grafana 캡처 소스가 prod Grafana인지 확인
- perf용 org / API key 준비 방식 확인
- 산출물 저장 경로와 `runId` 규칙 확인

원칙:

- 실행 위치, 요청 대상, Grafana 캡처 소스가 서로 다를 수 있으므로 각각 분리해서 기록한다.
- prod direct는 public smoke와 같은 의미로 취급하지 않는다.

## 5. 실행 순서

이번 단계 실행 순서는 아래로 고정한다.

1. `W1 r100`
2. `R1 r10`
3. `M1 10/10`

이유:

- 먼저 write 단독 구간을 확인하고
- 그 다음 가장 무거운 read 단독 구간을 확인하고
- 마지막에 mixed 기준선을 확인하는 편이 해석이 쉽다.

즉 먼저 단독 축을 다시 확인한 뒤 mixed로 들어가는 순서가 자연스럽다.

## 6. 기록 원칙

- [04-대규모-부하-테스트-기록.md](04-대규모-부하-테스트-기록.md)에 `environment=prod direct`를 명시한다.
- 같은 시나리오의 local baseline과 연결해 비교한다.
- prod direct 결과를 local과 그대로 등치하지 않고, 자원 차이를 같이 적는다.
- `showcase`는 모든 run을 올리지 않고 대표 run만 선별한다.

## 7. 해석 시 주의사항

- prod direct는 local보다 EC2 / RDS 자원과 RTT가 포함된다.
- 따라서 숫자 자체보다 **패턴이 유지되는지**를 먼저 본다.
- public ingress가 빠진 상태이므로, public 성능과 혼동하지 않는다.
- local에서 가벼웠던 축이 prod direct에서 반드시 같은 비율로 가볍다고 단정하지 않는다.

## 9. 다음 단계

이번 prod direct 최소 검증 세트를 마친 뒤에는 아래 순서로 간다.

- local과 prod direct 패턴 비교
- 구조 개선 우선순위 재판단
- 필요하면 prod public 최종 검증

즉 이번 문서의 목적은
**local 결과 다음에 무엇을 어떤 순서로 확인할지 흐름을 고정하는 것**이다.

## 10. 실행 결과

이번 계획 문서 기준 최소 검증 세트는 실제로 모두 성공했다.

- `W1 r100`
- `R1 r10`
- `M1 10/10`

상세 결과와 해석은 [27-Prod-Direct-1차-종합.md](27-Prod-Direct-1차-종합.md)를 기준으로 본다.

## 결론

이번 단계에서 prod direct는 새 시나리오가 아니라,  
이미 확보한 local 결과를 실제 인프라에서 다시 확인하기 위한 최소 검증 단계로 본다.

따라서 이번 문서의 결론은 다음 한 줄로 정리할 수 있다.

> prod direct 첫 검증은 `W1 r100 -> R1 r10 -> M1 10/10` 순서의 최소 세트로 시작하는 것이 맞다.
