# 36. M2 현실형 혼합 부하 시나리오 명세

## 문서 목적

`M2`는 `W2` realistic write와 `R6 v2` combined read를 섞는 **현실형 mixed 시나리오**다.  
이번 문서는 `M1 50:50`과 다른 realistic mixed 기준으로 `M2`를 어떤 조합과 어떤 비율로 정의할지 고정하는 데 목적이 있다.

이번 문서는 아직 seed 상세와 prepare / run 구현까지 고정하지 않는다.  
현재 문서가 고정하는 것은 `M2`의 구성 조합, read/write 비율, read request preset 재사용 방식, rate ladder, duration, threshold, success / saturation 기준, 공통 dataset / snapshot 재사용 원칙까지다.

## 적용 범위

- 대상 시나리오:
  - `M2`
- 대상 구성:
  - `W2 realistic`
  - `R6 v2`
- 이번 문서가 다루는 범위:
  - realistic mixed 비율
  - `M1`과의 차이
  - tenant 분포와 read split

## 이전 시나리오와 다른 점

- `M1`은 `W1 + R1 overview`의 `50:50` mixed였다.
- `M2`는 `W2 + R6 v2`의 `95:5` realistic mixed다.

즉 `M1`이 shared resource 경쟁 탐지용 mixed였다면,
`M2`는 실제 서비스형 write-heavy mixed에 더 가깝다.

## 이번 시나리오에서 새로 중요해지는 변수

- write-heavy 비율에서 read가 어느 정도까지 영향을 주는지
- tenant 분포가 섞인 상태에서 write와 read가 함께 들어올 때의 경쟁
- 집계 read와 분석 read가 낮은 비율로 섞여 들어올 때의 mixed 비용

## 현재 고정값

- tenant 분포:
  - `6개 org / 50-30-20`
- write:
  - `W2 realistic`
- read:
  - `R6 v2`
- write:read:
  - `95:5`
- read 내부 분배:
  - `R4 70`
  - `R5 30`

## request preset 재사용 방식

- `M2`의 read는 `R6 v2`를 그대로 재사용한다.
- 따라서 `M2`는 read request preset을 별도로 다시 정의하지 않는다.
- `R6`가 가진 `default / heavy` 층을 그대로 따라간다.
- 기본 read profile version은 `R6 v2`다.
- preset은 `default / heavy`를 독립적으로 선택할 수 있다.
- 초기 기본값은 stage 1, 2에서 `default`, stage 3에서 `heavy`다.

### local first-run 해석 메모

- local 기본 duration은 계속 `30s / 5m / 0`으로 유지한다.
- 다만 app restart 직후 첫 `M2 s2`는 cold-start 편차가 크게 섞일 수 있다.
- overview 응답 캐시를 추가한 뒤 warm-up을 `2m`로 늘린 확인 run
  - `m2-20260323-135843-s2-after-overview-cache-warm2m`
  에서는 throughput `193.94 req/s`, read `p95 1.24s`, write `p95 1.11s`, dropped `29`로 매우 안정적으로 통과했다.

즉 local `M2` 해석은 아래처럼 나누는 편이 맞다.

- 일반 steady-state 비교:
  - 기본 `30s / 5m / 0`
- app restart 직후 cold-start 편차 제거 목적:
  - sacrificial warm run 1회
  - 또는 `2m` warm-up 확인 run

이 메모는 default ladder를 바꾸기 위한 것이 아니라, **cold-start 편차를 steady-state 성능과 혼동하지 않기 위한 해석 규칙**으로 둔다.

## success / saturation 기준

- stable:
  - threshold pass
  - `error rate < 1%`
  - `dropped_iterations = 0`
  - achieved throughput `>= 95%`
- degraded:
  - threshold는 pass
  - 그러나 `dropped_iterations > 0`
  - 또는 achieved throughput `90~95%`
- saturated:
  - threshold fail
  - 또는 achieved throughput `< 90%`
  - 또는 `dropped_iterations` 지속 발생

## threshold

- write
  - `p95 < 5000ms`
  - `p99 < 8000ms`
- read
  - `p95 < 5000ms`
  - `p99 < 8000ms`

## 기본 reset mode

- `quick`
- baseline / 문서용 기준 run / dataset 이상 의심 시에는 `full`을 사용한다.
- current quick reset은 `event-only`다.
  - `k6-v2` overlay event만 삭제한다.
  - 신규 user overlay는 누적 허용한다.
  - 따라서 반복 튜닝 run은 quick, strict 비교 run은 full restore가 기본 원칙이다.

## 최근 local 재검증 메모

- `payload OID -> TEXT` 전환 뒤 quick reset의 가장 큰 병목은 `users DELETE`였다.
- 실서비스 가치가 약한 FK 보조 인덱스는 추가하지 않고, quick reset을 event-only로 단순화했다.
- 그 결과 local `W2 quick prepare`는 다시 빠르게 통과했고, `M2 s2`도 prepare 단계는 정상적으로 끝났다.
- 그러나 `M2 s2` main run은 여전히 threshold fail이다.
  - runId:
    - `m2-20260322-185539-s2-after-event-only-quick-reset`
  - read:
    - `p95 10.46s`
    - `p99 24.36s`
  - write:
    - `p95 8.47s`
    - `p99 10.07s`
  - achieved throughput:
    - `166.35 req/s`
  - dropped iterations:
    - `8572`

즉 이번 변경은 `M2` 본체 병목 해결이 아니라 **prepare/reset 병목 해소**에 해당한다.

## 아직 고정하지 않는 항목

- seed 상세
- snapshot / dataset reset 구현 구조
- prepare / run 구조

## 이번 시나리오에서 일부러 단순화한 항목

- `M1`처럼 `50:50` 경쟁 탐지 비율을 그대로 가져오지 않는다.
- read 5% 안에서 `R4`, `R5`를 더 세분화한 추가 layer는 지금 만들지 않는다.

## 이전 시나리오의 어떤 가정을 가져오지 않는지

- `M1`의 single-tenant mixed 가정은 가져오지 않는다.
- `M1`의 `overview` 단일 read 가정은 가져오지 않는다.

## 해석 시 주의사항

- `M2`는 `M1`보다 현실성이 높지만, 어떤 read 조각이 정확히 영향을 줬는지는 더 분리하기 어렵다.
- 따라서 `M2`는 반드시 `W2`, `R4`, `R5` 결과와 함께 읽어야 한다.

## 다음에 정할 항목

- seed 상세
- snapshot reset 구현 방식
- prepare / run 구조

## 이번 문서의 결론

> `M2`는 `W2 realistic + R6 v2`를 `95:5`로 섞는 현실형 mixed 시나리오이며, read 안에서는 `R4 70 / R5 30` 분배를 기본으로 사용한다.
