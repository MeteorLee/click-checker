# 06. W1 준비 및 실행 구조

## 문서 목적

`W1` write-heavy baseline을 반복 가능하게 실행하기 위한 준비와 실행 구조를 고정한다.  
이번 문서는 `prepare -> run -> meta -> artifacts` 경계를 명확히 나누고, 각 run이 같은 시작 조건에서 재현 가능하도록 준비 스크립트와 실행 스크립트의 역할을 정리하는 데 목적이 있다.

## 적용 범위

- 대상 시나리오:
  - `W1`
- 대상 환경:
  - `local`
  - 이후 `prod direct`
- 이번 문서가 다루는 범위:
  - prepare 스크립트 역할
  - run 스크립트 역할
  - seed 방식
  - `meta.json` 구조
  - run 단위
  - cleanup / 실패 run 처리 기준

## 환경 의존성

### local

- `docker compose` 네트워크 안에서 app 컨테이너를 직접 호출한다.
- 네트워크 RTT와 외부 ingress 영향이 작다.
- 첫 baseline, prepare/run 구조 검증, 빠른 반복에 적합하다.

### prod direct

- 실제 EC2 / RDS 자원을 사용한다.
- public ingress를 거치지 않고 앱 포트를 직접 본다.
- app / DB 경로 기준 실제 비용을 확인하기 좋다.

원칙:

- 같은 구조라도 환경이 바뀌면 숫자 의미가 달라질 수 있다.
- local 결과와 prod direct 결과는 그대로 등치하지 않고, 환경 차이를 같이 해석한다.

## 이전 구조 문서와 다른 점

- 해당 없음.
- `W1`는 성능 단계의 첫 준비/실행 구조 문서이므로, 직전 구조 문서 비교보다 공통 템플릿을 실제 운영 규칙으로 구체화하는 데 집중한다.

## 실행 재현성 기준

- 같은 `meta.json`이 있으면 같은 run 조건을 다시 재현할 수 있어야 한다.
- run 스크립트는 `meta.json`만 있으면 재실행 가능해야 한다.
- seed 데이터 규칙은 deterministic해야 한다.
- `prepare`와 `run`을 분리해, state 생성과 부하 실행을 독립적으로 다시 수행할 수 있어야 한다.

## 준비 스크립트 역할

예상 파일:

- `scripts/perf/prepare-w1-local.sh`

역할:

- 테스트 계정 준비
- 새 org 생성
- API key 발급
- 기존 사용자 `1000명` seed
- `runId`와 `outDir` 확정
- `meta.json` 생성

원칙:

- 준비 스크립트는 시작 상태를 만드는 역할만 한다.
- 실행 결과(`summary.json`, latency, 실패율)는 준비 스크립트가 만들지 않는다.

## 실행 스크립트 역할

예상 파일:

- `scripts/perf/run-w1-local.sh`

역할:

- `meta.json` 읽기
- 해당 rate 기준 `k6` 실행
- `summary.json`, `console.log` 저장
- 필요 시 `meta.json.status` 업데이트
- `capture.mode=scripted`면 Grafana renderer 캡처 실행

원칙:

- 실행 스크립트는 부하 실행과 산출물 저장만 담당한다.
- org 생성, API key 발급, seed 같은 상태 생성은 하지 않는다.
- scripted capture는 run 산출물 생성의 일부로 본다.

## prepare / run 경계

### prepare

- 테스트 계정 준비
- org / API key 생성
- seed 이벤트 입력
- `runId` 생성
- 산출물 디렉터리 생성
- `meta.json` 생성

### run

- `k6` 시나리오 실행
- `summary.json` 저장
- `console.log` 저장
- Grafana 캡처 시간 범위 기록
- `capture.mode=scripted`면 renderer API로 PNG 생성

원칙:

- seed는 prepare 단계에만 속한다.
- run 단계에서 seed나 사전 상태 생성을 하지 않는다.
- prepare와 run이 섞이면 run 간 비교 가능성이 크게 떨어진다.

## 준비 데이터 규칙

### org / API key 준비

공식 흐름:

```text
signup
-> admin org create
-> api key 발급
```

원칙:

- 현재 제품의 공식 진입 흐름을 그대로 사용한다.
- 레거시 조직 생성 방식은 사용하지 않는다.

### seed 대상

- 기존 사용자 `1000명`
- 각 사용자당 seed 이벤트 `1건`

사용자 규칙:

- `existing-0`
- `existing-1`
- ...
- `existing-999`

### seed 이벤트 식별 규칙

- `eventType = loadtest_seed`
- `pathPrefix = /loadtest/w1/seed`

원칙:

- seed 이벤트와 본 실험 이벤트(`loadtest_write`)는 구분된다.
- 이후 DB / Loki / 기록 문서에서 seed와 본 실험을 분리할 수 있어야 한다.

### seed 처리 방식

- 완전 직렬 대신 가벼운 병렬 처리
- 목적:
  - 준비 시간 단축
  - 앱을 과도하게 압박하지 않는 수준 유지

### 준비 단계 메타데이터

준비 스크립트는 최소 아래를 남긴다.

- seed 시작 시각
- seed 종료 시각
- 성공 건수
- orgId
- existingUserPoolSize

## run 단위 기준

- `r50`
- `r100`
- `r200`

원칙:

- 준비 스크립트는 한 번에 한 run만 준비한다.
- 실행 스크립트도 한 번에 한 run만 실행한다.
- `50 -> 100 -> 200`을 한 번에 섞지 않는다.
- 각 run은 fresh org를 사용한다.

## `meta.json` 기준

위치:

- `artifacts/perf/w1/<runId>/meta.json`

원칙:

- 실행 스크립트는 필요한 값을 `meta.json`에서만 읽는다.
- 실행 스크립트가 org를 다시 만들거나 API key를 다시 찾지 않는다.
- `meta.json`에는 실행 조건과 상태만 넣고 결과값은 넣지 않는다.

### 권장 스키마

```json
{
  "scenario": "W1",
  "runId": "w1-20260319-143000-r50",
  "timestamp": "2026-03-19T14:30:00+09:00",
  "status": "prepared",
  "isBaseline": true,
  "compareTo": null,
  "env": {
    "name": "local",
    "prepareBaseUrl": "http://localhost:8080",
    "baseUrl": "http://app:8080",
    "appCount": 1,
    "dbInstance": "local-postgres"
  },
  "auth": {
    "apiKey": "plain-api-key",
    "apiKeyPrefix": "ck_live"
  },
  "dataset": {
    "orgId": 1,
    "orgName": "perf-w1-r50",
    "pathPrefix": "/loadtest/w1",
    "eventType": "loadtest_write",
    "pathCount": 6,
    "existingUserPoolSize": 1000,
    "existingUserRatio": 80
  },
  "seed": {
    "eventType": "loadtest_seed",
    "pathPrefix": "/loadtest/w1/seed",
    "existingUserPoolSize": 1000
  },
  "load": {
    "rate": 50,
    "warmup": "2m",
    "duration": "10m",
    "cooldown": "1m"
  },
  "capture": {
    "mode": "scripted",
    "timeRange": "start-2m ~ end+2m",
    "status": "success",
    "fromMs": 1773894480000,
    "toMs": 1773895320000,
    "generatedAt": "2026-03-19T14:42:00+09:00"
  }
}
```

주의:

- `W1` local 구조에서는 prepare 단계와 run 단계의 base URL이 다를 수 있다.
- run 스크립트가 실제 요청에 써야 하는 API key는 `meta.json.auth.apiKey`에서 읽는다.
- plain API key가 들어가는 `meta.json` 파일 권한은 `600`으로 제한한다.

## 실패 run 처리 기준

- 네트워크 오류로 중단된 경우:
  - 무효
- 서버 재시작 / OOM / readiness 실패:
  - 무효
- 일부 요청 실패지만 threshold 이내:
  - 유효
- Grafana / Prometheus 수집 누락:
  - 재실행 판정

원칙:

- 실패 run도 산출물은 삭제하지 않는다.
- 다만 유효한 실패와 실험 오류를 구분해 기록 문서에 남긴다.

## cleanup 기준

초기 단계 기준:

- 자동 cleanup 하지 않음

이유:

- 이상한 결과가 나온 run을 나중에 다시 확인할 수 있어야 한다.
- org / seed / event 데이터가 남아 있어야 DB 상태 분석이 쉽다.

원칙:

- 초기 W1 단계에서는 보존을 우선한다.
- 정리는 이후 별도 cleanup 스크립트로 분리한다.

## 실행 순서

### 1. 로컬 드라이런

- 짧은 duration
- 목적:
  - prepare / run 스크립트 정상 동작 확인
  - `meta.json`, `summary.json`, `console.log` 생성 확인
  - Grafana 패널 반응 확인

### 2. 공식 baseline

1. `prepare-w1-local.sh`로 `r50` 준비
2. `run-w1-local.sh`로 `50 RPS` 실행
3. 기록 문서 업데이트
4. `r100` 준비 / 실행
5. 기록 문서 업데이트
6. `r200` 준비 / 실행
7. 기록 문서 업데이트

### 3. prod direct 재현

- local baseline 구조가 안정된 뒤 같은 패턴으로 prod direct run을 준비한다.
- 이때도 `prepare -> run -> meta -> artifacts` 경계를 유지한다.

## 이전 시나리오의 어떤 구조 가정을 그대로 가져오지 않는지

- 해당 없음.
- 다만 이후 `R1`, `R2`, `R3`, `M1`에서는 아래 W1 전용 구조 가정을 그대로 가져오지 않는다.
  - fresh org를 run마다 무조건 새로 만드는 구조
  - seed를 앱 경유 write 이벤트로만 만드는 구조
  - single-tenant 전용 준비 구조
