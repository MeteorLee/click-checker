## 1. 경량 검증 단계에서 먼저 확인한 것

`W1` 전에 이미 운영/관측 정비 단계에서 경량 부하 검증을 한 번 거쳤다.

### 실제로 확인한 것

- local `deploy smoke` 성공
- 짧은 `k6` write burst 성공
  - 약 `315`건
  - error `0%`
  - `p95 ≈ 3.96s`
  - `p99 ≈ 5.26s`
- 짧은 `k6` read burst 성공
  - 약 `383`건
  - error `0%`
  - `p95 ≈ 188ms`
  - `p99 ≈ 454ms`

### 이 단계의 의미

- `deploy smoke`, `k6`, Grafana, Loki, Sentry가 현재 API 흐름과 맞는지 확인하는 단계였다.
- 즉 "실험 체계가 대체로 맞물린다"는 것을 먼저 확인한 것이지, 이 수치 자체를 baseline으로 삼은 것은 아니다.
- 특히 write가 read보다 먼저 무거워 보인다는 힌트는 여기서 이미 나왔다.

### 왜 baseline으로 쓰지 않았는가

- duration이 짧았다.
- `W1` 전용 조직/seed/meta/artifacts 체계가 아직 없었다.
- scripted capture와 공식 기록 규칙이 완전히 고정되기 전 단계였다.

### 이 단계에서 실제로 얻은 것

- `prepare-w1-local.sh`가 fresh org, API key, 기존 사용자 1000명 seed, `meta.json` 생성을 끝까지 수행하는지 확인했다.
- `run-w1-local.sh`가 `summary.json`, `console.log`, `command.txt`를 남기는지 확인했다.
- Grafana renderer 기반 PNG 자동 캡처가 실제로 생성되는지 확인했다.

### 왜 수치 해석은 조심해야 하는가

- 정식 baseline보다 훨씬 짧은 run이었다.
- scripted capture는 성공했지만, 이 단계의 숫자는 "절차 검증" 성격이 더 강하다.
- 따라서 드라이런 수치는 `W1` baseline의 공식 비교값이 아니라, 실험 체계가 실제로 작동하는지 보여주는 보조 증거로 봐야 한다.

### 결과

- 평균 처리량: `48.48 req/s`
- `dropped_iterations`: `24`

### 산출물

- `meta.json`
- `summary.json`
- `console.log`
- `dashboard-overview.png`
- `ingest-rps.png`
- `ingest-latency.png`
- `db-connections.png`
- `db-pool.png`

### 이 수치가 의미하는 것

- `50 RPS` 자체는 현재 local write path가 견디는 구간으로 보인다.
- threshold(`p95 < 5s`, `p99 < 8s`, `error rate < 1%`)도 모두 통과했다.
- 다만 `dropped_iterations=24`와 `vus max=114`는 "아직 완전히 여유롭다"기보다, higher rate에서 더 빨리 악화될 수 있다는 신호다.
- 즉 이번 결과는 "`50 RPS`는 통과"이면서도, "`100`과 `200`을 빨리 봐야 한다"는 결론으로 이어진다.

## 드라이런과 정식 baseline의 차이

| 항목 | 드라이런 | 정식 baseline |
| --- | --- | --- |
| 목적 | 절차 검증 | 공식 비교 기준 확보 |
| warm-up / main / cooldown | `15s / 30s / 0` | `2m / 10m / 1m` |
| 산출물 | 동일 | 동일 |
| 숫자 해석 비중 | 낮음 | 높음 |
| compare 기준 사용 여부 | 없음 | 있음 |

핵심은 이거다.

- 드라이런은 "스크립트와 산출물 구조가 끝까지 돈다"를 확인하는 단계
- 정식 baseline은 "이후 비교의 기준점"을 확보하는 단계

둘 다 필요하지만, 의미는 다르다.

## 지금까지의 해석

### 1. write path는 이미 read보다 더 무거운 쪽이다

이건 경량 검증 단계에서 이미 힌트가 나왔고, `W1 r50` baseline에서도 뒤집히지 않았다.

### 2. `50 RPS`는 통과했지만 여유 구간이라고 단정하긴 이르다

- 실패율은 `0%`
- `p95`, `p99`도 threshold 이내

여기까지만 보면 괜찮다.  
하지만 `dropped_iterations`와 후반 VU 급증은 higher rate에서 더 빨리 포화될 가능성을 보여준다.

### 3. 다음 비교점은 자연스럽게 `r100`, `r200`이다

지금 단계에서 바로 인덱스나 Redis로 가지 않는 이유는, 아직 `50 / 100 / 200` 중 어느 구간에서 임계점이 생기는지 공식 baseline이 다 닫히지 않았기 때문이다.

## 한 줄 정리

이전 경량 검증은 "관측과 현재 API 흐름이 맞다"를 확인한 단계였고, `W1` 드라이런은 "실험 체계가 실제로 작동한다"를 확인한 단계였다.  
이번 `w1-20260319-132007-r50`은 그 둘 위에서 확보한 **첫 공식 write baseline**이다.
