## 1. 왜 `W1`가 첫 공식 시나리오인가

`W1`는 가장 먼저 `POST /api/events` write path를 고정된 조건으로 보는 시나리오다.

이 시나리오를 첫 공식 시나리오로 둔 이유는 다음과 같다.

- 경량 검증 단계에서 이미 write가 read보다 먼저 무거워 보이는 힌트가 있었다.
- 이벤트 수집 경로는 이후 funnel, retention, users overview의 입력 원천이므로 먼저 안정성을 확인할 가치가 있다.
- mixed나 read 시나리오보다 원인 분리가 쉽다.

즉 `W1`는 "가장 현실적인 사용자 흐름"을 재현하려는 시나리오가 아니라, **가장 먼저 write 병목을 설명 가능하게 만드는 기준선 시나리오**다.

## 2. 경량 검증 단계에서 먼저 확인한 것

`W1` 전에 이미 운영/관측 정비 단계에서 경량 부하 검증을 한 번 거쳤다.

### 실제로 확인한 것

- local `deploy smoke` 성공
- 짧은 `k6` write burst 성공
  - 약 `315`건
  - error `0%`
  - `p95 ≈ 3.96s`
  - `p99 ≈ 5.26s`
- 짧은 `k6` read burst 성공
  - 약 `383`건
  - error `0%`
  - `p95 ≈ 188ms`
  - `p99 ≈ 454ms`

### 이 단계의 의미

- `deploy smoke`, `k6`, Grafana, Loki, Sentry가 현재 API 흐름과 맞는지 확인하는 단계였다.
- 즉 "실험 체계가 대체로 맞물린다"는 것을 먼저 확인한 것이지, 이 수치 자체를 baseline으로 삼은 것은 아니다.
- 특히 write가 read보다 먼저 무거워 보인다는 힌트는 여기서 이미 나왔다.

### 왜 baseline으로 쓰지 않았는가

- duration이 짧았다.
- `W1` 전용 조직/seed/meta/artifacts 체계가 아직 없었다.
- scripted capture와 공식 기록 규칙이 완전히 고정되기 전 단계였다.

### 이 단계에서 실제로 얻은 것

- `scripts/perf/w1/prepare-local.sh`가 fresh org, API key, 기존 사용자 1000명 seed, `meta.json` 생성을 끝까지 수행하는지 확인했다.
- `scripts/perf/w1/run-local.sh`가 `summary.json`, `console.log`, `command.txt`를 남기는지 확인했다.
- Grafana renderer 기반 PNG 자동 캡처가 실제로 생성되는지 확인했다.

### 왜 수치 해석은 조심해야 하는가

- 정식 baseline보다 훨씬 짧은 run이었다.
- scripted capture는 성공했지만, 이 단계의 숫자는 "절차 검증" 성격이 더 강하다.
- 따라서 드라이런 수치는 `W1` baseline의 공식 비교값이 아니라, 실험 체계가 실제로 작동하는지 보여주는 보조 증거로 봐야 한다.

## 4. W1에서 일부러 단순화한 것

`W1`는 첫 baseline 시나리오라서, 현실성보다 해석 가능성을 우선해 일부러 단순화한 항목이 있다.

### 단순화한 항목

- org는 한 run당 하나만 사용한다.
- eventType은 `loadtest_write` 하나만 사용한다.
- path는 고정 패턴 6개만 순환한다.
- 사용자 혼합은 `기존 80 / 신규 20`으로 고정한다.
- 첫 실행 환경은 `local`로 제한한다.

### 왜 이렇게 했는가

- 초기 병목 후보를 write path 안쪽에서 분리해서 보기 위해서
- "실서비스 완전 재현"보다 "왜 느린지 설명 가능"한 기준선을 먼저 만들기 위해서
- 이후 `R1`, `R2`, `R3`, `M1`에서 변수를 하나씩 늘릴 수 있게 하기 위해서

즉 `W1`는 intentionally simple한 시나리오다.

### 결과

- 실패율: `0.00%`

- 평균 처리량: `48.48 req/s`
- `dropped_iterations`: `24`

### 산출물

- `meta.json`
- `summary.json`
- `console.log`
- `dashboard-overview.png`
- `ingest-rps.png`
- `ingest-latency.png`
- `db-connections.png`
- `db-pool.png`

### 이 수치가 의미하는 것

- `50 RPS` 자체는 현재 local write path가 견디는 구간으로 보인다.
- threshold(`p95 < 5s`, `p99 < 8s`, `error rate < 1%`)도 모두 통과했다.
- 다만 `dropped_iterations=24`와 `vus max=114`는 "아직 완전히 여유롭다"기보다, higher rate에서 더 빨리 악화될 수 있다는 신호다.
- 즉 이번 결과는 "`50 RPS`는 통과"이면서도, "`100`과 `200`을 빨리 봐야 한다"는 결론으로 이어진다.

## 6. 드라이런과 정식 baseline의 차이

| 항목 | 드라이런 | 정식 baseline |
| --- | --- | --- |
| 목적 | 절차 검증 | 공식 비교 기준 확보 |
| warm-up / main / cooldown | `15s / 30s / 0` | `2m / 10m / 1m` |
| 산출물 | 동일 | 동일 |
| 숫자 해석 비중 | 낮음 | 높음 |
| compare 기준 사용 여부 | 없음 | 있음 |

핵심은 이거다.

- 드라이런은 "스크립트와 산출물 구조가 끝까지 돈다"를 확인하는 단계
- 정식 baseline은 "이후 비교의 기준점"을 확보하는 단계

둘 다 필요하지만, 의미는 다르다.

## 7. W1 결과를 읽을 때 주의할 점

### 1. W1는 local single-tenant 기준이다

이 결과는 local docker, single-tenant 조건에서 나온다.  
따라서 prod direct, prod public, multi-tenant 상황에 그대로 일반화하면 안 된다.

### 2. W1는 write 병목을 보기 위한 시나리오다

read-heavy 비용, mixed 경합, ingress/public 네트워크 비용은 별도 시나리오에서 더 잘 드러난다.

### 3. W1의 단순화는 다음 시나리오에 그대로 복붙하면 안 된다

`W1`의 path 수, eventType 수, org 수, 환경 가정은 `R1`, `R2`, `R3`, `M1`에 그대로 옮기면 안 된다.  
W1는 어디까지나 "첫 기준선"을 위한 단순화다.

## 8. W1에서 확인한 해석 포인트

### 1. write path는 이미 read보다 더 무거운 쪽이다

이건 경량 검증 단계에서 이미 힌트가 나왔고, `W1 r50` baseline에서도 뒤집히지 않았다.

### 2. `50 RPS`는 통과했지만 여유 구간이라고 단정하긴 이르다

- 실패율은 `0%`
- `p95`, `p99`도 threshold 이내

여기까지만 보면 괜찮다.  
하지만 `dropped_iterations`와 후반 VU 급증은 higher rate에서 더 빨리 포화될 가능성을 보여준다.

### 3. `100 RPS`가 처음엔 한계 구간처럼 보였지만, 앞단 비용 절감 뒤에는 통과 구간으로 이동했다

이건 구조 전체가 아니라, 요청 앞단의 작은 반복 비용이 의외로 컸다는 뜻이다.  
자세한 변경 내용과 전후 비교는 [06-성능-개선-조치-이력.md](06-성능-개선-조치-이력.md)에서 본다.

### 4. 다음 비교점은 자연스럽게 `r200` 또는 다음 병목 후보다

지금 단계에서 바로 인덱스나 Redis로 가지 않는 이유는, 아직 `50 / 100 / 200` 중 어느 구간에서 임계점이 생기는지 공식 baseline이 다 닫히지 않았기 때문이다.

### 5. `event_user` 비용은 즉시 제거 대상이 아니다

`W1`의 후반 비교에서는 `event_user` 조회/생성 비용이 write 경로의 다음 구조적 비용으로 드러났다.
하지만 이 항목은 funnel, retention, users overview 같은 사용자 단위 분석 요구와 직접 연결되어 있으므로, 이번 단계에서 바로 단순화 대상으로 보지 않는다.

즉 `W1`는 write 특성을 먼저 드러내는 역할까지만 맡고, `event_user` 구조는 read 및 mixed 결과까지 확인한 뒤 통합 개선 대상으로 다시 판단한다.

## 다음에 볼 것

- read(`R1/R2/R3`) baseline 시작
- mixed(`M1`) baseline으로 shared resource 경합 확인
- `event_user` 구조는 write 단독 결과만으로 확정하지 않고, read/mixed 이후 다시 판단
- `R1`, `R2`, `R3`, `M1` 문서에서 W1의 단순화를 그대로 복붙하지 않기

## 한 줄 정리

`W1`는 현실 전체를 재현하려는 시나리오가 아니라, `POST /api/events` write path의 첫 기준선을 **설명 가능하게 만들기 위한 intentionally simple한 시나리오**다.  
그래서 W1 결과는 write 병목과 앞단 비용을 보는 데는 유효하지만, read/mixed/public 결과로 그대로 일반화하면 안 된다.
