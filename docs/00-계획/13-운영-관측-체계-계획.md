# 13. 운영 관측 체계 계획 (v2.0)

## 목표
- 현재 운영 배포 구조(단일 EC2 + Blue/Green + Nginx + RDS)에 맞는 관측 체계를 한 번에 정리한다.
- `Prometheus + Grafana + Loki + Alertmanager + Sentry + k6`를 각각 역할이 분명한 구조로 연결한다.
- 이후 배포 검증, 장애 확인, 성능 비교에서 공통으로 참고할 수 있는 운영 관측 기준을 남긴다.

---

# 13.0 고정 원칙

## 1. 이번 단계는 “도구 추가”가 아니라 관측 구조 정리 단계다
- 이번 단계의 목적은 단순히 Loki, Alertmanager를 붙이는 데 있지 않다.
- `메트릭 -> 로그 -> 예외 -> 알림` 흐름을 현재 프로젝트 규모에 맞게 설명 가능한 구조로 정리하는 데 집중한다.

## 2. 각 도구의 역할을 분리한다
- Prometheus:
  - 메트릭 수집
  - alert rule 평가
- Grafana:
  - 메트릭 / 로그 시각화
  - 배포 시점 전후 비교 화면
- Loki:
  - 로그 저장 / 조회
  - `requestId`, `appColor` 기준 검색
- Alertmanager:
  - 알림 라우팅
  - 최소 알림 전달
- Sentry:
  - 예외 stacktrace / env / release / endpoint 확인
- k6:
  - 관측 검증용 입력 부하

## 2.1 요청 추적 기준은 `X-Request-Id`로 고정한다
- `nginx -> app -> 로그 -> Sentry` 흐름에서 같은 요청을 추적할 수 있어야 한다.
- 이번 단계에서는 `X-Request-Id` 전달이 운영 관측 체계의 기본 전제로 작동하는지 함께 확인한다.
- 목적은 메트릭 이상 징후를 실제 요청 단위 분석으로 이어지게 만드는 것이다.

## 3. 현재 운영 구조를 기준으로 설계한다
- 기준 구조는 다음으로 고정한다.
  - 단일 EC2
  - Nginx
  - `app-blue:8081`, `app-green:8082`
  - RDS
- 관측 체계도 이 운영 구조를 기준으로 설명 가능해야 한다.

## 4. 관측은 “판단 가능한 흐름”으로 본다
- Grafana에서 지표만 보는 것으로 끝내지 않는다.
- 기본 확인 흐름은 다음으로 고정한다.
  1. 메트릭 이상 징후 확인
  2. Loki에서 로그 확인
  3. 필요 시 Sentry에서 예외 원인 확인

## 5. 이번 단계는 운영형 관측 체계를 닫는 데 목적이 있다
- 로그 장기 보관 플랫폼까지 가지 않는다.
- tracing / ELK / Cloud 관측 스택까지 확장하지 않는다.
- 대신 현재 프로젝트에서 설명 가능한 관측 구조를 이번 단계에서 마감한다.

# 13.4 설계 기준

## 1) 메트릭 축
- HTTP 요청 계열
  - RPS
  - `p95`
  - `p99`
  - `4xx ratio`
  - `5xx ratio`
- DB 계열
  - HikariCP active connection
  - max 대비 사용률
  - 포화/대기 여부
- 기본 상태 계열
  - JVM 메모리
  - 프로세스/시스템 기본 상태
  - app up/down 상태

## 2) 로그 축
- 수집 대상:
  - app stdout 로그
  - nginx access log
  - nginx error log
- 수집 경로:
  - Promtail -> Loki
- label 기준:
  - `job=click-checker`
  - `instance=<host or container>`
  - `app=app-blue | app-green | nginx`
  - `level=INFO | WARN | ERROR` (가능한 경우)
- 조회 기준:
  - `requestId`
  - `appColor`
  - log level
  - 필요 시 path / logger / upstream

## 3) 예외 축
- Sentry는 계속 예외 전담 도구로 유지한다.
- 운영 해석 순서는 다음으로 고정한다.
  1. 메트릭 이상 징후 확인
  2. Loki 로그 확인
  3. Sentry 예외 이벤트 확인

## 4) 알림 축
- 최소 알림 3종:
  - `5xx ratio` 급증
  - `http.server.requests` `p95` 급등
  - DB pool active connection 과다 또는 포화 지속
- 추가 기본 알림:
  - app down (`up == 0` 또는 health DOWN)
- 초기 임계값 초안:
  - 5xx 비율: 5분 평균 `>= 2%`
  - p95 응답시간: 5분 평균 `>= 800ms`
  - DB pool active connection: max 대비 `>= 80%` 5분 지속
- 기본 down 기준 초안:
  - `up{job="click-checker-app"} == 0`
- 실제 전달 경로는 이번 단계에서 최소 1개 이상 연결한다.

## 5) 대시보드 축
- 유지 대상
  - RPS
  - `p95`
  - `5xx ratio`
- 추가 대상
  - `p99`
  - `4xx ratio`
  - DB pool active connection
  - JVM 메모리
- 필요 시 app color별 분리 관찰 기준을 둔다.

# 13.8 추가 고려 사항

## Blue/Green color 전환 관찰 패널
- 목적:
  - 배포 전환 시점에 active color가 실제로 어떻게 바뀌는지 더 직접적으로 확인한다.
  - 전환 직후 color별 요청 비중과 에러 여부를 비교할 수 있게 한다.

- 후보 방향:
  1. Nginx access log를 Loki로 수집한 뒤, upstream 또는 app color 기준으로 로그 기반 패널 구성
  2. 필요 시 이후 단계에서 app color를 메트릭 tag로 넣는 방식 검토

- 현재 판단:
  - 관측 체계 핵심 범위에 비해 구현/설계 비용이 한 단계 더 든다.
  - 이번 단계 필수 항목으로 넣기보다, `Loki + nginx 로그 + Alertmanager`까지 정리한 뒤 여유가 있으면 이어서 검토한다.
