# 05. 웹 진입점 구성 계획 (v1.0)

## 5단계 목표
- 메인 서비스 진입점을 `clickchecker.dev` 기준으로 정리한다.
- Nginx reverse proxy와 HTTPS를 적용해 운영형 웹 진입점을 만든다.
- 외부 공개 대상과 내부 관리 대상을 분리한다.

---

# 5.0 고정 원칙

## 1. 메인 서비스 진입점은 단일 도메인으로 고정한다
- 메인 앱은 `clickchecker.dev` 하나를 기준으로 운영한다.
- `www`는 필수가 아니며, 초기 단계에서는 사용하지 않아도 된다.

## 2. 외부 사용자는 Nginx를 통해서만 접근한다
- app `8080` 직접 접근은 운영 경로에서 제거한다.
- Nginx가 `80/443`에서 외부 요청을 받고 내부 app로 reverse proxy 한다.

## 3. HTTPS는 선택이 아니라 기본값으로 본다
- 도메인 연결 후 Let's Encrypt 기준으로 HTTPS를 적용한다.
- HTTP는 HTTPS 리다이렉트 경로로만 유지한다.

## 4. actuator는 최소 공개 원칙을 따른다
- `/actuator/health`만 외부 공개한다.
- `/actuator/prometheus` 포함 기타 `/actuator/**`는 외부 진입점에서 차단한다.

## 5. 관리 도구와 메인 서비스는 분리해서 판단한다
- 메인 앱 도메인과 Grafana/Prometheus 노출 정책은 같은 문제로 취급하지 않는다.
- Grafana는 별도 정책으로 판단하고, Prometheus는 기본 비공개로 둔다.

---

# 5.1 범위 정의

## 포함(in scope)
- 도메인 구매 및 DNS 연결
- EC2에 Nginx 설치
- `80 -> Nginx -> app:8080` reverse proxy 구성
- Let's Encrypt 인증서 발급
- `80 -> 443` 리다이렉트
- 루트(`/`) 최소 응답 추가
- `/actuator/health` 공개, 기타 actuator 차단
- app `8080` 외부 직접 노출 제거
- 관련 런북 / 트러블슈팅 / 종합 문서화

## 제외(out of scope)
- `grafana.clickchecker.dev` 구성
- `www.clickchecker.dev` 지원
- Prometheus 외부 공개
- ALB / CloudFront / CDN
- Blue/Green 또는 무중단 배포

# 5.2 완료 기준 (Done)
- `clickchecker.dev`가 EC2 퍼블릭 IP로 연결된다.
- `https://clickchecker.dev`가 정상 응답한다.
- `80` 요청은 `443`으로 리다이렉트된다.
- `/actuator/health`는 `200`, `/actuator/prometheus`는 차단된다.
- app `8080`은 외부에서 직접 접근하지 않는다.
- `certbot renew --dry-run`이 성공한다.

- `clickchecker.dev`가 EC2 퍼블릭 IP로 해석되는지 확인

## nginx

- reverse proxy가 `127.0.0.1:8080`을 바라보는지 확인

## HTTPS
