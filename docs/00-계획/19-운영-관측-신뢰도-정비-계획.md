# 19. 운영/관측 신뢰도 정비 계획 (v1.0)

## 목표
- 바로 큰 구조 변경으로 들어가기 전에, 현재 운영/관측 도구가 믿을 수 있는 기준선 역할을 하도록 정리한다.
- 부하 테스트, 배포 스모크, Grafana 대시보드, Alert rule, Sentry 추적 정보가 현재 제품 흐름과 실제 API 경로를 제대로 반영하도록 맞춘다.
- 즉 이번 단계는 "도구를 더 붙이는 단계"가 아니라, "이미 있는 운영/관측 도구를 실제 성능 개선의 기준선으로 쓸 수 있게 다잡는 단계"다.

---

# 19.0 고정 원칙

## 1. 이번 단계는 "운영/관측 신뢰도 정비"에 집중한다
- 이번 단계의 핵심은 새로운 인프라를 크게 추가하는 것이 아니다.
- 지금 이미 있는 `Prometheus + Grafana + Loki + Alertmanager + Sentry + k6`를 믿고 사용할 수 있게 정리한다.
- 즉 "관측 스택 확장"보다 "관측 스택 정합성 확보"가 우선이다.

## 2. 측정 기준이 먼저다
- 이후 인덱스 개선, write/read 경로 최적화, rollup/retention 전략은 모두 기준선이 있어야 비교할 수 있다.
- 따라서 k6 시나리오, Grafana 패널, Alert rule이 현재 API 경로와 제품 흐름을 제대로 반영해야 한다.
- 감으로 최적화하지 않고, 비교 가능한 수치와 스크린샷을 남길 수 있어야 한다.

## 3. 도구는 가능한 한 줄인다
- 부하 테스트 도구는 `k6`를 기준으로 고정한다.
- 이번 단계에서는 `nGrinder`, Kafka, Redis cache, Loki S3 확장 같은 추가 확장을 같이 끌고 오지 않는다.
- 먼저 현재 스택만으로 read/write 기준선을 재현 가능하게 만든다.

## 4. 배포 검증도 현재 공식 흐름을 따라야 한다
- deploy smoke가 옛 공개 경로를 기준으로 돌아가면 현재 제품 흐름과 어긋난다.
- 따라서 배포 검증은 지금 공식 흐름과 최대한 맞아야 한다.
  - `signup`
  - admin organization 생성
  - API key 수령
  - event ingest
  - analytics 조회

## 5. 운영 지표는 제품 트래픽 중심으로 본다
- health, root, 스캐너 404, 내부 운영 경로가 제품 트래픽 지표를 심하게 오염시키지 않도록 정리한다.
- RPS, p95/p99, 4xx/5xx 비율은 "실제 제품 API"에 더 가깝게 해석할 수 있어야 한다.
- 즉 이번 단계는 메트릭 자체를 더 많이 추가하는 것보다, 기존 메트릭을 더 정확히 읽게 만드는 데 우선순위를 둔다.

## 6. Grafana는 "보기 쉬운 운영 도구"가 되어야 한다
- 현재 Overview 대시보드는 최소 구성으로 시작했지만, 실제 운영에서 바로 읽기엔 다소 거칠 수 있다.
- app 로그와 nginx 로그를 구분하고, 운영자가 바로 봐야 할 패널을 더 선명하게 만든다.
- 이번 단계에서는 대시보드의 "가독성"과 "실전 해석 가능성"을 함께 정리한다.

## 7. Sentry는 예외 저장소가 아니라 추적 도구가 되어야 한다
- 예외를 저장하는 것만으로는 충분하지 않다.
- requestId, appColor, release 같은 운영 문맥이 있어야 "이 에러가 어느 요청/배포/색상에서 났는지" 설명 가능하다.
- 이번 단계에서는 Sentry를 로그/배포와 연결되는 예외 추적 도구로 다듬는다.

# 19.1 범위 정의

## 포함(in scope)
- k6 read/write 시나리오 정합성 보정
- 현재 analytics API 경로와 부하 테스트 대상 경로 일치화
- deploy smoke를 현재 공식 조직/API key 흐름에 맞추는 작업
- Grafana Overview 대시보드 패널/로그 쿼리 정리
- app 로그 / nginx 로그 분리 및 운영자 가독성 보강
- Alert rule의 지표 해석 기준 정리
- requestId / appColor / release 기준 Sentry 추적 정보 보강
- 운영 기준 문서와 실제 설정 사이의 불일치 정리

## 제외(out of scope)
- Loki 로그 저장소의 S3 전환
- Loki retention / compactor / 장기 보관 본격 설계
- Redis cache 도입
- Kafka 도입
- rollup / retention 배치 구현
- bulk ingest API 확장
- 관리자 콘솔 화면 구현

# 19.2 완료 기준 (Done)

- read-heavy / write-heavy k6 시나리오가 현재 공식 API 경로 기준으로 정상 동작한다.
- deploy smoke가 현재 공식 organization/API key 흐름을 기준으로 통과한다.
- Grafana Overview 대시보드에서 app 로그와 nginx 로그를 분리해 읽을 수 있다.
- 핵심 지표(RPS, p95/p99, 4xx/5xx, DB pool usage)가 현재 제품 트래픽 기준으로 더 일관되게 해석된다.
- Sentry에서 예외를 볼 때 requestId, appColor, release 중 최소 1개 이상 운영 추적 문맥을 함께 연결할 수 있다.
- 이후 성능 개선 전/후 비교에 사용할 기준선 문서와 스크린샷 포인트를 설명할 수 있다.

# 19.3 현재 상태

## 이미 있는 것
- Docker Compose 기반 app / postgres / prometheus / loki / promtail / alertmanager / grafana 구성
- `/actuator/health`, `/actuator/prometheus` 기반 메트릭 노출
- Grafana 기본 Overview 대시보드
- Alertmanager + Discord webhook
- requestId 필터와 appColor 로그 출력
- Sentry 최소 모드
- k6 read/write smoke 스크립트

## 현재 구조의 한계
- k6 read script가 현재 analytics 경로와 완전히 맞지 않는다.
- deploy smoke는 아직 예전 organization 생성 흐름에 기대고 있다.
- Grafana의 기본 로그 패널은 app 로그와 nginx 로그를 함께 보여줘 읽기 불편하다.
- 일부 메트릭은 health/root/스캐너 요청까지 포함해 제품 트래픽 해석을 흐릴 수 있다.
- Sentry는 예외 수집은 되지만, 운영 추적 문맥(requestId/appColor/release)이 충분히 연결되지 않았다.

즉 이번 단계는 "운영/관측 도구가 있다"에서 끝나지 않고, "지금부터 성능 개선 기준선으로 믿고 쓸 수 있다"로 올리는 단계다.

# 19.4 선행 확정 항목

## 부하 테스트 기준
- 부하 테스트 도구는 `k6`로 고정한다.
- 시나리오는 `read-heavy`, `write-heavy` 두 개를 기준으로 둔다.
- 비교 시 다음 조건을 고정한다.
  - 동일 RPS
  - 동일 duration
  - 동일 warm-up
  - 동일 데이터셋 조건
  - 동일 배포 버전(커밋 해시)

## 운영 대시보드 기준
- 기본 대시보드는 `Click Checker Overview`를 계속 사용한다.
- 다만 패널은 "보여주기"보다 "운영자가 실제로 읽기 좋은가" 기준으로 정리한다.
- 로그 패널은 최소한 app 로그와 nginx 로그를 분리한다.

## 운영 추적 기준
- request 추적 기준은 `X-Request-Id`
- Blue/Green 구분 기준은 `APP_COLOR`
- 예외 추적 기준은 `Sentry + requestId + release`

## 보안 원칙
- deploy smoke나 운영 스크립트가 plain API key를 불필요하게 오래 남기지 않도록 정리한다.
- 브라우저/운영 문서에서 API key와 JWT 경계를 계속 분리해 설명한다.

# 19.5 세부 작업 항목

## 1) k6 / smoke 정합성 정리
- read-heavy k6가 현재 analytics 경로를 정확히 치도록 수정
- write-heavy k6 입력 조건 정리
- deploy smoke를 현재 공식 흐름 기준으로 정리
- smoke 결과 파일에 plain API key를 과도하게 남기지 않는 방향 검토

## 2) Grafana 대시보드 정리
- Overview 패널 구성 재검토
- `Recent Logs`를 app 중심 로그로 조정
- nginx 로그는 별도 패널 또는 별도 쿼리로 분리
- 필요 시 운영자가 자주 보는 최소 패널 재배치

## 3) Alert / 메트릭 해석 기준 정리
- health/root/스캐너성 요청이 핵심 지표를 얼마나 오염시키는지 확인
- 필요 시 PromQL 필터 또는 대시보드 쿼리 기준 조정
- Alert rule도 같은 기준으로 재검토

## 4) Sentry 추적 문맥 보강
- requestId 연결 방식 점검
- appColor / release 태그 연결 가능성 검토
- 운영 문서에서 "로그 -> Sentry" 추적 흐름을 더 구체화

## 5) 문서 / 기준선 정리
- 이후 성능 개선 전/후 비교용 기준선 포인트 정의
- 어떤 대시보드 패널을 스크린샷 기준으로 쓸지 정리
- 운영/관측 문서와 실제 설정 차이가 있으면 현재 코드 기준으로 맞춘다
