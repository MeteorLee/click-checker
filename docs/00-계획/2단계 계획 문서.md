# 2단계 실행 계획표 v1.3

## 2단계 목표
- `X-API-Key`로 **Organization을 식별**한다.
- `/api/events/**`는 **키 없으면 401**.
- 이벤트 저장/조회는 **인증된 org 스코프**로 강제한다.
- 평문 키는 **생성/rotate에서만 1회 반환**하고 **DB/로그에 남기지 않는다**.

---

## 2.0 고정 결정(변경 금지)
1. **저장은 해시만**
- `apiKeyHash` = `HMAC-SHA256(pepper, plainKey)` 결과를 **hex(64 chars)** 로 저장(권장)
- pepper는 환경변수(예: `API_KEY_PEPPER`)로 주입

2. **조직당 단일 키**
- rotate 시 즉시 교체(이전 키는 곧바로 무효)

3. **prefix는 운영 편의용**
- `apiKeyPrefix` = plainKey 앞 8자
- 인증에는 사용하지 않음(표시/검색/로그용 “일부만” 허용)

## 2.1 데이터 모델 변경 (Organization)
### 변경 사항
- 필드 추가:
  - `apiKeyKid` (NOT NULL, UNIQUE)
  - `apiKeyHash` (NOT NULL, UNIQUE)
  - `apiKeyPrefix` (NOT NULL)
  - `apiKeyStatus` (NOT NULL, default `ACTIVE`)
  - `apiKeyCreatedAt` (NOT NULL)
  - `apiKeyRotatedAt` (NULL)
  - `apiKeyLastUsedAt` (NULL)
- enum:
  - `ApiKeyStatus { ACTIVE, DISABLED }`

### 완료 기준
- 애플리케이션 기동 시 스키마 반영(현재는 ddl-auto로 OK)
- unique 제약으로 중복 hash 저장 불가

## 2.2 키 발급/해시 유틸 (새 파일)
### 클래스: `ApiKeyIssuer`

#### 메서드
- `IssuedApiKey issue()`
- `String hash(String plainKey)` (내부/테스트용)

#### 규칙
- plainKey:
  - 키 형식: `ck_<env>_v1_<kid>_<secret>`
  - `env`: `live` 또는 `test`
  - `kid`: 랜덤 식별자(8~12 bytes 기반 인코딩)
  - `secret`: `SecureRandom`으로 생성한 바이트를 URL-safe Base64 인코딩(패딩 제거)
  - 길이: “충분히 길게” (예: 32 bytes 이상 원문) _정확 길이는 구현에서 고정_
- hash:
  - HMAC-SHA256(pepper, plainKey) → hex(64)
  - `API_KEY_PEPPER` 환경변수가 비어 있으면 애플리케이션 시작 단계에서 즉시 실패(fail-fast) 처리
- prefix:
  - plainKey의 앞 8자
- 비교:
  - 인증 시 hash 비교는 constant-time compare를 사용한다.

#### 반환 record
- `IssuedApiKey(String plainKey, String kid, String prefix, String hash)`

### 완료 기준
- 유닛 테스트:
  - issue()가 매번 다른 키를 생성
  - hash 결과가 64 hex 형식
- `API_KEY_PEPPER` 미설정 시 앱이 조기에 실패하는 동작 확인

## 2.3 Organization 생성 로직 연결
### 서비스: `OrganizationService` (또는 기존 커맨드 서비스)

#### 로직
- 조직 생성 시 `ApiKeyIssuer.issue()` 호출
- DB 저장: hash/prefix/status/timestamps
- 응답에는 plainKey 1회 반환

### DTO 분리 (핵심)
- `OrganizationCreateResponse`에만 `apiKey` 포함
- `OrganizationResponse`(조회/리스트)는 `apiKey` 필드 자체가 없음

### 완료 기준
- 생성 응답에서만 평문 키 확인 가능
- org 조회 API가 있다면 평문 키가 절대 내려가지 않음

## 2.5 API Key 인증 필터 도입
### 파일: `ApiKeyAuthFilter` (OncePerRequestFilter)

### 필터 순서(고정)
- `ApiKeyAuthFilter`는 요청 초반에 실행한다.
- 기존 `RequestIdFilter`와 함께 사용 시, 필터 순서는 설정 파일에서 명시적으로 고정한다.

#### 동작
- 보호 대상: `/api/events/**`
- 헤더: `X-API-Key`
- 주: 현재 보호 범위는 `/api/events/**`이므로 `/api/organizations/**`, `/actuator/**`는 기본적으로 필터 대상이 아니다.
- 보호 범위를 `/api/**`로 확장하는 시점에 `/api/organizations/**`, `/actuator/**`를 예외 경로로 승격한다.
- 추가 통과 규칙:
  - `OPTIONS` 요청(CORS preflight)은 인증 없이 통과
  - `ERROR` 디스패치(`/error`)는 인증 없이 통과
  - (선택) `ASYNC` 디스패치는 인증 재검사 없이 통과(중복 인증 방지)

#### 검증 흐름
1. 보호 경로 아니면 통과
2. 보호 경로면 헤더 확인
  - 없거나 blank → 401
3. hash 계산 → repository 조회 (status ACTIVE)
  - `kid` 파싱 -> `findByApiKeyKidAndApiKeyStatus(...)` 조회
  - row의 hash와 요청 hash를 constant-time compare로 검증
  - 실패 시 401
4. 성공 시 orgId(또는 Organization)를 request attribute에 저장
  - 예: `request.setAttribute("AUTH_ORG_ID", org.getId())`

#### 응답 규칙(간단 고정)
- 2단계 MVP에서는 필터 401 응답을 **바디 없이** 통일한다.
- 공통 `ErrorResponse` 연계는 후속 단계에서 적용한다.

### 완료 기준
- `/api/events/**`는 키 없으면 401

## 2.5.2 API Key 조회 추상화(최소)
### 목적
- 현재는 `Organization` 단일 키 모델을 쓰되, 추후 `ApiKey` 별도 테이블로 쉽게 전환할 수 있게 경계를 만든다.

### 구성
- 인터페이스: `OrganizationApiKeyLookup` (또는 `ApiKeyLookupPort`)
- 현재 구현체: `OrganizationRepository` 기반 조회
- 필터는 repository 직접 접근 대신 인터페이스를 사용

### 완료 기준
- 필터 코드가 DB 모델(`Organization` 컬럼 구조)에 직접 결합되지 않는다.

## 2.5.1 Organization 컨텍스트 Resolver 도입
### 파일
- `@CurrentOrganizationId` (커스텀 파라미터 애노테이션)
- `CurrentOrganizationIdResolver` (HandlerMethodArgumentResolver)
- `WebMvcConfigurer` 등록

### 동작
- `ApiKeyAuthFilter`가 저장한 request attribute(`AUTH_ORG_ID`)를 컨트롤러 파라미터로 주입한다.
- 컨트롤러는 `HttpServletRequest` 직접 접근 없이 `@CurrentOrganizationId Long orgId`를 사용한다.
- 2단계에서는 org 컨텍스트 전달 수단을 request attribute로만 사용한다. (`AUTH_ORG_ID` 키 고정)

### 완료 기준
- 이벤트 API 컨트롤러에서 조직 컨텍스트를 Resolver로 주입받아 사용한다.
- request attribute 조회 중복 코드가 컨트롤러에 남아있지 않다.

## 2.6 Organization 스코프 강제 (이벤트 API) — _Event 연결 변경 포함_
### 목표
- 이벤트 생성/조회에서 **Organization 결정권을 요청 body → 인증 컨텍스트(API Key)**로 완전히 이동한다.

### 변경 포인트(핵심)
- **Event의 `organization`(또는 `organization_id`)는 `EventCreateRequest.organizationId`가 아니라,**
- **`X-API-Key → Organization resolve 결과(authOrgId)`로 세팅된다.**

### 구현 규칙(2단계 초반: 마이그레이션 모드)
1. **Request DTO에 `organizationId`가 남아있다면**
- 서버는 `req.organizationId()`를 **사용하지 않는다(무시)**
- _이유: 테넌트 결정은 인증으로만 해야 함(바디 변조 방지)_

2. **필터에서 org 컨텍스트 주입**
- `X-API-Key` 검증 성공 시 `authOrgId`를 request attribute(또는 컨텍스트)에 저장
  - 예: `request.setAttribute("AUTH_ORG_ID", orgId)`

3. **컨트롤러에서 authOrgId 꺼내서 서비스로 전달**
- `@CurrentOrganizationId Long authOrgId`를 주입받아
- `eventService.create(authOrgId, req)` 형태로 호출

### 서비스 시그니처 고정
- `EventCommandService.create(Long authOrgId, EventCreateRequest req)`로 변경한다.

4. **서비스에서 Event를 “authOrgId 기준 Organization”에 연결**
- 기존: `Organization organization = findById(req.organizationId())`
- 변경: `Organization organization = findById(authOrgId)`
- Event 생성 시:
  - `Event.organization = organization`은 **authOrgId로 조회한 Organization**을 사용

### 마이그레이션 종료 조건(고정)
- 아래 조건이 모두 충족되면 `EventCreateRequest.organizationId`를 DTO에서 제거한다.
  1. 이벤트 저장/조회 API가 모두 `authOrgId`만 사용하도록 전환 완료
  2. 관련 통합 테스트(정상/무효키/조직 스코프)가 전부 통과
  3. CI(`test`, `postgresTest`)에서 회귀 없음 확인

### 완료 기준
- 동일한 request body라도:
  - `X-API-Key=A`면 `event.organization_id = orgA`
  - `X-API-Key=B`면 `event.organization_id = orgB`
- request body의 `organizationId` 값이 달라도 결과는 **항상 헤더(API Key) 기준**으로 결정된다.

## 2.7 보안 가드레일 (로그/응답)
### 최소 규칙
- 로그에 `X-API-Key` 원문을 찍지 않는다
- 로그 식별은 `kid` 또는 `prefix`만 사용한다
- 에러 메시지에 “전달된 키”를 포함하지 않는다
- (가능하면) request logging 필터가 있다면 헤더 마스킹 적용

### 완료 기준
- 샘플 요청 시 서버 로그에 평문 키 미노출 확인(수동 점검 OK)

---

## 2.8 테스트 고정 (MVP 통합 테스트 6개)
1. org 생성 시 apiKeyHash/prefix/status 저장
2. 생성 응답에만 평문 키 포함
3. events 보호 경로: 키 누락 401
4. events 보호 경로: 키 무효 401
5. events 보호 경로: 키 정상 2xx
6. org 경로: 키 없이 2xx

# “한 파일씩” 실제 실행 순서 (Codex용)
1. `Organization` + `ApiKeyStatus`
2. `OrganizationRepository` 메서드
3. `ApiKeyIssuer` (+ 단위 테스트)
4. `OrganizationService` 생성 연결
5. `OrganizationController` DTO 분리(1회 노출)
6. `ApiKeyAuthFilter` (+ 통합 테스트)
7. `@CurrentOrganizationId` + `CurrentOrganizationIdResolver` 등록
8. `Event API` org 스코프 강제(+ 통합 테스트)
9. 로그/응답 점검(필요시 마스킹 보강)
