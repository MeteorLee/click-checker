# 23. 프런트엔드 콘솔 계획 (v1.3)

## 목표
- Click-Checker에 브라우저에서 직접 사용할 수 있는 관리자 콘솔을 추가한다.
- JWT 기반 admin analytics / 운영 API를 실제 화면에 연결한다.
- 현재 범위는 `회원가입/로그인 -> organization 선택/생성 -> overview -> 상세 분석 -> 설정/운영` 흐름까지 포함한다.

---

## 23.0 고정 원칙

### 1. 프런트는 현재 저장소 안에서 함께 관리한다
- 별도 저장소로 분리하지 않는다.
- 코드 위치는 루트 아래 `frontend/` 디렉토리로 둔다.

### 2. 브라우저는 JWT만 사용한다
- 브라우저는 `Authorization: Bearer <accessToken>`만 사용한다.
- 브라우저는 `X-API-Key`를 직접 저장하거나 전송하지 않는다.

### 3. overview를 허브로 확장한다
- overview는 analytics 진입점이자 설정/운영 화면 진입점 역할을 맡는다.
- 상세 분석은 `routes / event-types / trends / users / activity / retention / funnels`까지 포함한다.
- 운영 화면은 `API key / route 규칙 / event type 규칙 / 멤버 관리`까지 포함한다.

### 4. organization 삭제 UI는 membership 해제로 처리한다
- UI 문구는 `삭제`로 둔다.
- 실제 동작은 현재 계정의 membership 연결 해제다.
- 마지막 OWNER 규칙은 별도 확인 문구로 제어한다.

### 5. 초대는 토큰 방식 대신 loginId 직접 입력으로 단순화한다
- 별도 초대 링크/수락 흐름은 도입하지 않는다.
- 기존 계정을 `loginId` 기준으로 조직 멤버에 바로 추가한다.

## 23.1 포함 범위

### 포함(in scope)
- `frontend/` 앱 초기화
- 회원가입 / 로그인 화면
- organization 선택 / 생성 화면
- organization 삭제(=membership 해제) 흐름
- overview 대시보드
- routes / event types / trends / users / activity / retention / funnels 화면
- API key 1회 표시 / metadata / rotate
- route 규칙 / event type 규칙 관리
- 멤버 관리
- 최소 layout / header / card UI

### 제외(out of scope)
- 이메일/토큰 기반 초대 수락
- organization 상태 기반 deactivate 모델
- 복잡한 전역 상태관리
- 모바일 우선 최적화
- 디자인 시스템 고도화

---

## 23.2 완료 기준

- 사용자는 로그인 후 organization을 선택하거나 새로 만들 수 있다.
- overview와 상세 분석 화면을 브라우저에서 이동하며 확인할 수 있다.
- route 규칙 / event type 규칙 / 멤버를 콘솔에서 관리할 수 있다.
- organization 생성/rotate 직후 API key를 1회 확인하고 복사할 수 있다.
- organization 삭제 UI는 membership 해제로 동작하며 마지막 OWNER 규칙을 따른다.
- 브라우저는 JWT만 사용하고, API key를 인증에 사용하지 않는다.

## 23.3 현재 상태

### 이미 있는 것
- JWT 로그인 / 회원가입 / me
- organization membership 기반 RBAC
- overview / routes / event types / trends / users / activity / retention / funnels
- API key 관리
- route 규칙 / event type 규칙 관리
- 멤버 관리
- organization 삭제(=membership 해제)

### 아직 없는 것
- 이메일/토큰 기반 초대 수락
- organization 상태 모델
- 문서/실행기록 최신화

즉 현재는 "핵심 관리자 콘솔은 열렸고, 남은 것은 초대/조직 상태 모델 같은 운영 고도화" 상태다.

## 23.5 설계 방향

### 제품 메시지
- 콘솔은 "백엔드 API를 감싼 데모 화면"이 아니라, Click-Checker의 실제 제품 흐름을 보여주는 관리자 콘솔이어야 한다.

### 화면 우선순위
- 1순위: 로그인 / organization 선택
- 2순위: overview 대시보드
- 3순위: 상세 분석 화면
- 4순위: 운영/설정 화면

### 디자인 방향
- 운영 콘솔형 분석 대시보드
- 밝은 배경, 카드 중심 구조
- 숫자와 상태가 잘 읽히는 UI

### 기술 방향
- Next.js
- Mantine
- 최소 상태관리
- API 호출 함수 분리
