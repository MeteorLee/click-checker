# 23. 프론트엔드 콘솔 계획 (v1.0)

## 목표
- Click-Checker에 브라우저에서 직접 사용할 수 있는 최소 관리자 콘솔을 추가한다.
- 22단계에서 만든 JWT 기반 admin overview API를 실제 화면에 연결한다.
- 1차는 `로그인 -> organization 선택 -> overview 대시보드` 흐름만 닫고, routes/event-types/trends 등 확장 화면은 후속 단계로 둔다.

---

# 23.0 고정 원칙

## 1. 이번 단계는 "데모 가능한 최소 콘솔"에 집중한다
- 목표는 완성형 프런트 제품이 아니라, 현재 백엔드 기능을 실제 사용자 흐름으로 보여줄 수 있는 최소 콘솔이다.
- 화면 수를 늘리기보다, 로그인부터 overview 조회까지의 핵심 경로를 먼저 닫는다.

## 2. 프런트는 현재 저장소 안에서 함께 관리한다
- 프런트는 별도 저장소로 분리하지 않는다.
- 현재 프로젝트는 백엔드, 문서, 배포, 프런트가 하나의 제품 흐름으로 강하게 연결되어 있다.
- 코드 위치는 루트 아래 `frontend/` 디렉토리로 둔다.

## 3. 브라우저는 JWT만 사용한다
- 콘솔 화면은 `Authorization: Bearer <accessToken>`만 사용한다.
- 브라우저는 `X-API-Key`를 직접 저장하거나 전송하지 않는다.
- organization scope는 path의 `organizationId`와 membership 검증으로 결정한다.

## 4. 1차는 overview 하나로 충분하다
- routes / event-types / trends까지 한 번에 열지 않는다.
- 1차 화면은 overview만으로도
  - 로그인
  - organization membership
  - admin analytics
  흐름을 충분히 보여줄 수 있다.

## 5. UI 기술은 안정성과 속도를 우선한다
- 1차 프런트는 Next.js 기반으로 간다.
- UI 시스템은 Mantine를 기준으로 한다.
- 목표는 디자인 실험보다, 빠르고 안정적으로 일관된 콘솔을 만드는 것이다.

## 6. 복잡한 상태관리는 1차 범위 밖이다
- 전역 상태관리 도구를 처음부터 크게 도입하지 않는다.
- access token, 현재 사용자, 선택 organization, overview 결과 중심의 최소 상태만 다룬다.

## 7. 화면보다 흐름을 먼저 완성한다
- 로그인 화면
- organization 선택 화면
- overview 대시보드 화면

이 세 화면의 연결이 1차 완료 기준이다.

# 23.1 범위 정의

## 포함(in scope)
- `frontend/` 앱 초기화
- 로그인 화면
- organization 선택 화면
- overview 대시보드 화면
- JWT 기반 auth 흐름 연결
- `/api/v1/admin/me` 연동
- `/api/v1/admin/organizations/{organizationId}/analytics/overview` 연동
- 최소 layout / header / card UI 구성

## 제외(out of scope)
- routes / event-types / trends 화면
- users / activity / funnels / retention 화면
- member 관리 / API key 관리 화면
- 차트 중심 시각화 고도화
- 복잡한 전역 상태관리
- 모바일 우선 최적화
- 디자인 시스템 고도화

# 23.2 완료 기준 (Done)

- 사용자는 로그인 화면에서 계정으로 로그인할 수 있다.
- 로그인 후 자신이 속한 organization 목록을 볼 수 있다.
- 사용자는 organization을 선택해 overview 대시보드로 이동할 수 있다.
- overview 대시보드는 JWT 기반 admin overview API 결과를 실제 화면에 표시한다.
- 브라우저는 JWT만 사용하고, API key를 직접 다루지 않는다.
- 현재 프로젝트의 핵심 데모 흐름을 브라우저에서 시연할 수 있다.

# 23.3 현재 상태

## 이미 있는 것
- JWT 로그인 / 회원가입 / refresh / me
- organization membership 기반 RBAC
- Spring Security 기반 인증 전달 구조
- JWT 기반 admin overview API
- overview 결과 계산 로직 및 통합 테스트

## 아직 없는 것
- 브라우저 콘솔 UI
- organization 선택 화면
- admin overview 화면
- 프런트에서 실제 제품 흐름을 시연할 수 있는 진입점

즉 현재는 "백엔드는 준비됐고, 사용자에게 보여줄 프런트 콘솔이 아직 없는 상태"다.

# 23.4 선행 확정 항목

## 화면 흐름
```text
/login
-> /organizations
-> /dashboard/[organizationId]
```

## 1차 API 사용 범위
- `POST /api/v1/admin/auth/login`
- `GET /api/v1/admin/me`
- `GET /api/v1/admin/organizations/{organizationId}/analytics/overview`

## overview 파라미터
- `from`
- `to`

설명:
- timezone 파라미터는 받지 않는다.
- overview 시간 해석은 백엔드 기준 `Asia/Seoul` 고정 정책을 따른다.

## 문서 역할 분리
- 이 문서는 전체 단계 계획이다.
- 프런트 세부 설계는 `docs/07-front/01-프론트엔드-계획.md`를 따른다.

# 23.5 설계 방향

## 1) 제품 메시지
- 이번 프런트는 "백엔드 API를 감싼 단순 데모 화면"이 아니라, Click-Checker의 실제 제품 흐름을 보여주는 관리자 콘솔이어야 한다.
- 즉 로그인, organization 선택, overview 확인까지가 한 제품 경험으로 이어져야 한다.

## 2) 화면 우선순위
- 1순위: 로그인
- 2순위: organization 선택
- 3순위: overview 대시보드

## 3) 디자인 방향
- 운영 콘솔형 분석 대시보드
- 밝은 배경, 카드 중심 구조
- 숫자와 상태가 잘 읽히는 UI
- 과한 장식보다 실사용 느낌을 우선

## 4) 기술 방향
- Next.js
- Mantine
- 최소 상태관리
- API 호출 함수 분리
- 페이지는 얇고 컴포넌트는 역할별 분리

# 23.7 검증 계획

## 기능 검증
- 로그인 성공 후 organization 선택으로 이동
- organization 선택 후 overview 화면 이동
- overview 데이터 정상 표시

## 보안 검증
- JWT 없으면 보호 화면 접근 불가
- API key는 브라우저에서 사용하지 않음

## 회귀 기준
- 프런트 구현 전후로 기존 백엔드 API 계약을 유지한다.
- overview 결과는 백엔드 테스트로 이미 검증된 값을 그대로 사용한다.
