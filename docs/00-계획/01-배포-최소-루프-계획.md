# 01. 배포 최소 루프 계획 (v1.0)

## 1단계 목표
- 로컬 검증에서 배포까지 이어지는 최소 배포 루프를 완성한다.
- CI 품질선을 고정하고, 수동/자동 배포 1회 완주 기준을 만든다.
- 배포 후 health/smoke로 성공을 판정한다.

---

## 1.0 고정 원칙
1. 배포 성공은 `기동 + health + smoke` 3단계로 판단한다.
2. CI는 빠른 테스트와 Postgres 호환 테스트를 분리한다.
3. 실패 배포는 원인 로그를 남기고 즉시 복구 경로를 실행한다.

---

## 1.1 컨테이너 기반 배포 준비
- 배포 대상 구성: `app/postgres/prometheus/grafana`
- 환경변수 주입 구조와 프로파일 분리(local/prod)

완료 기준:
- 배포 환경에서 `docker compose up -d --build`로 재기동 가능

---

## 1.2 CI 품질선 확립
- `test`(fast lane) + `postgresTest`(호환성) 분리
- main 진입 전 강한 검증 통과를 기준으로 고정

완료 기준:
- PR 기준 품질선이 문서/워크플로우에 명시됨
- 회귀 시 CI에서 즉시 실패

---

## 1.3 수동 배포 1회 완주
- EC2에서 pull/build/up/health/smoke 전체 절차 수행
- 실패 시 로그 수집/원인 파악 흐름 점검

완료 기준:
- 수동 배포 체크리스트 1회 완주 기록 보유

---

## 1.4 자동 배포 1회 완주
- GitHub Actions `prod` 배포 워크플로우 구성
- health/smoke 실패 시 실패 처리 + 복구 경로 실행

완료 기준:
- 실제 배포 1회 이상 성공
- 실패 케이스에서 복구/트러블슈팅 경로 검증

---

## 1.5 배포 후 스모크 기준
- 쓰기 스모크: Organization 생성 + Event 저장
- 읽기 스모크: aggregate 조회
- 성공 기준: 2개 모두 HTTP 성공 코드

완료 기준:
- 배포 후 자동 스모크 통과 로그 확인

---

## 1.6 운영 진입점 정의
- 외부 공개 API 범위와 내부 운영 엔드포인트 구분
- 추후 인증/API Key 확장 전 임시 정책 명확화

완료 기준:
- 운영 진입점 정책 문서화 완료
