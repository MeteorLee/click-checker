# 25. 프런트엔드 배포 계획

## 목표
- 현재 운영 중인 `CodeDeploy + ECR + backend blue/green + EC2 nginx` 구조 위에 프런트를 자연스럽게 올린다.
- backend의 blue/green 무중단 전환 구조는 유지하고, frontend는 별도 단일 컨테이너로 운영한다.
- 외부 사용자는 `clickchecker.dev` 하나로 접속하고, nginx가 same-origin 기준으로 frontend와 backend를 분기하도록 만든다.

---

# 25.0 고정 원칙

## 1. backend blue/green은 유지한다
- 현재 운영 강점은 backend의 blue/green 전환 구조다.
- `app-blue:8081`, `app-green:8082`, nginx upstream 전환, readiness 확인, old color drain 구조는 그대로 유지한다.
- 프런트 배포 때문에 backend 전환 구조를 단순화하거나 제거하지 않는다.

## 2. frontend는 single container로 둔다
- frontend까지 blue/green으로 확장하지 않는다.
- 이유:
  - 현재 프런트는 UI 서버 성격이 강하고, backend만큼 무중단 전환 복잡도를 감당할 이유가 약하다.
  - 운영 복잡도 증가를 최소화한다.
  - 현재 배포 목표는 "프런트를 운영 구조에 얹는 것"이지, frontend까지 완전한 blue/green을 설계하는 것이 아니다.
- 감수하는 trade-off:
  - frontend는 single container 재기동 시 짧은 공백이 생길 수 있다.
  - 현재 단계에서는 이 공백을 frontend blue/green 추가 복잡도보다 작은 비용으로 본다.

## 3. 외부 도메인은 계속 하나로 유지한다
- 외부 사용자는 계속 `https://clickchecker.dev` 하나로 접근한다.
- 프런트와 백엔드를 서브도메인으로 나누지 않는다.
- nginx가 path 기준으로 same-origin 라우팅을 담당한다.

## 4. nginx가 frontend / backend를 경로 기준으로 분기한다
- `/api/**`는 active backend로 보낸다.
- `/actuator/health`는 active backend로 보낸다.
- `/actuator/health/readiness`는 active backend로 보낸다.
- 필요 시 `/swagger-ui/**`, `/v3/api-docs/**`도 active backend로 보낸다.
- `/_next/**`, `/favicon.ico`, 일반 페이지 라우트(`/`)는 frontend로 보낸다.
- "나머지는 frontend"로 가더라도, backend로 가야 하는 경로를 먼저 명시적으로 빼는 방식으로 구현한다.

## 5. frontend 배포 실패가 backend active color 전환을 오염시키면 안 된다
- backend 전환 성공/실패 판단은 계속 backend readiness, nginx switch, public smoke 기준으로 유지한다.
- frontend 배포는 별도 책임으로 다룬다.
- frontend 재기동 실패가 backend active color rollback 로직과 섞이지 않도록 스크립트 책임을 분리한다.

## 6. 이미지와 배포 책임은 구조상 분리한다
- 처음에는 같은 GitHub Actions / CodeDeploy bundle 안에서 함께 다룰 수 있다.
- 하지만 구조상
  - `APP_IMAGE`
  - `FRONTEND_IMAGE`
  를 별도로 관리한다.
- 나중에 frontend만 독립 배포할 수 있는 여지를 남긴다.

# 25.1 현재 운영 구조 요약

## 이미 있는 것
- backend runtime image
  - `Dockerfile`
- local compose
  - `docker-compose.yml`
  - `docker-compose.local.yml`
- prod compose
  - `docker-compose.prod.yml`
- CodeDeploy hooks
  - `appspec.yml`
  - `scripts/codedeploy/after-install.sh`
  - `scripts/codedeploy/application-start.sh`
  - `scripts/codedeploy/validate-service.sh`
- prod deploy orchestrator
  - `scripts/deploy/deploy-prod-orchestrator.sh`
  - `scripts/deploy/blue-green-prod-lib.sh`
- EC2 nginx single entrypoint
  - `nginx/click-checker.conf`

## 현재 외부 진입 구조
- `https://clickchecker.dev`
  - nginx
  - active backend (`127.0.0.1:8081` or `127.0.0.1:8082`)
- `https://grafana.clickchecker.dev`
  - nginx
  - Grafana

## 현재 한계
- backend는 운영 배포 구조가 정리되어 있지만, frontend는 아직 운영 배포 대상으로 포함되지 않았다.
- 따라서 지금 frontend는 local `npm run dev/build` 기준까지만 닫혀 있고, 운영 same-origin 경로에는 올라가 있지 않다.

## 현재 구현 상태
- frontend 운영 Dockerfile을 추가했다.
- frontend는 `Next standalone + /healthz` 기준으로 빌드/이미지 생성이 가능하다.
- `docker-compose.prod.yml`에 `frontend` 서비스를 추가했다.
- repo nginx 설정은 frontend/backend path routing 구조로 정리했다.
- GitHub Actions / CodeDeploy / deploy script는 `FRONTEND_IMAGE`, frontend deploy, frontend recover 흐름까지 연결했다.
- 아직 실제 운영 서버에 첫 적용을 수행한 상태는 아니다.

# 25.3 기술 방향

## 1. frontend는 Next standalone로 빌드한다
- 현재 프런트는 Next.js 기반이라, 지금 단계에서는 프레임워크를 바꾸지 않는다.
- `output: "standalone"` 기준으로 운영 이미지를 만든다.
- multi-stage Docker build를 사용한다.

## 2. frontend runtime은 Node 프로세스 단일 컨테이너
- prod compose에 `frontend` 서비스를 추가한다.
- 외부 포트 전체 공개는 하지 않고, nginx가 접근할 수 있는 loopback 포트만 연다.
- 예:
  - `127.0.0.1:3001:3000`

## 3. 브라우저 API base URL은 same-origin 기준으로 단순화
- 가능하면 프런트는 절대 base URL보다 same-origin 상대 경로를 우선 사용한다.
- 즉 운영에서는 `clickchecker.dev/api/...`를 자연스럽게 타도록 한다.
- local에서는 개발 편의를 위해 현재 방식(`localhost:8080`)을 유지할 수 있게 env를 둔다.

## 4. frontend health check를 둔다
- 최소 응답 확인용 path가 필요하다.
- 경로는 `/healthz`로 고정한다.
- 이 경로는 auth 가드, 데이터 fetch, 리다이렉트 영향을 받지 않는 가벼운 응답으로 유지한다.
- 배포 스크립트에서는 frontend 컨테이너가 정상 응답하는지 먼저 확인한다.

# 25.5 완료 기준

- 운영에서 `https://clickchecker.dev`가 frontend를 응답한다.
- 같은 도메인에서 `/api/**`는 active backend로 정상 프록시된다.
- backend blue/green 전환은 기존처럼 유지된다.
- frontend는 prod compose에서 단일 컨테이너로 재기동 가능하다.

- 운영 문서에서

  - 왜 frontend는 single인지
  - nginx가 어떻게 분기하는지
  설명할 수 있다.

# 25.7 주요 리스크와 대응

## 1. nginx location 우선순위 오류
- 증상:
  - API 요청이 frontend로 가거나
  - frontend path가 backend로 가는 문제
- 대응:
  - backend 경로를 명시적으로 먼저 선언
  - 마지막에 frontend catch-all
  - `/api`, `/actuator/health`, `/actuator/health/readiness`, `/_next`, `/healthz` 동작을 별도 체크리스트로 검증

## 2. frontend 재기동 실패가 전체 배포처럼 보이는 문제
- 증상:
  - backend는 멀쩡한데 메인 사이트가 뜨지 않음
- 대응:
  - frontend health check 추가
  - frontend failure와 backend failure를 운영 로그/문서에서 분리

## 3. frontend와 backend 버전 결합도 증가
- 증상:
  - 한쪽만 바꿔도 둘 다 같이 배포해야 하는 압박
- 대응:
  - `APP_IMAGE`, `FRONTEND_IMAGE`를 구조상 분리
  - 초기에 한 파이프라인으로 가더라도 나중에 분리 가능한 구조 유지

## 4. Next runtime이 생각보다 무거울 수 있음
- 현재는 기존 Next 구조를 유지하는 편이 현실적이다.
- 현재 단계에서는 static export로 바로 전환하지 않는다.
- 우선은 Next standalone를 운영 기준으로 확정한다.
- 다만 이후 프런트가 완전히 CSR 대시보드로 굳으면
  - static export
  - 별도 정적 호스팅
  도 후속 선택지로 남겨둔다.

## 5. Next 캐시/ISR 고려는 후속 과제로 남긴다
- 현재 계획은 frontend single container 기준이라 캐시 일관성 문제가 크지 않다.
- 다만 이후 App Router 캐시, ISR, scale-out이 필요해지면
  - reverse proxy
  - 다중 인스턴스
  - 캐시 저장 위치
  를 함께 다시 검토한다.

# 25.8 문서화 시 설명 포인트

- backend는 요청 무중단 전환이 중요해서 blue/green을 유지한다.
- frontend는 stateless UI 서버이며, 배포 복잡도를 낮추기 위해 single container로 운영한다.
- nginx는 same-origin을 위해 frontend와 backend를 path 기준으로 라우팅한다.
- 사용자는 하나의 도메인만 보지만, 내부적으로는 frontend와 active backend가 분리되어 있다.
