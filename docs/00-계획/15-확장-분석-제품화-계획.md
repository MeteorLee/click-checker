# 15. 확장 분석 / 제품화 계획 (v1.2)

## 목표
- 14번 문서에서 정리한 핵심 집계 기반 위에, 사용자 단위 분석과 확장 분석 API를 올린다.
- funnel / retention / cohort를 최소 지원 버전부터 도입해 Click-Checker를 “기본 집계 시스템”이 아니라 “분석 기능을 가진 시스템”으로 확장한다.
- 이번 단계에서는 확장 분석의 계산 규칙과 API 계약을 먼저 고정하고, 성능 최적화는 후속 단계로 분리한다.

---

# 15.0 고정 원칙

## 1. 이번 단계는 핵심 집계 기반 위에 확장 분석을 올리는 단계다
- 14번 문서의 `route normalization`, `eventType mapping`, `overview / Top N / trends`가 먼저 안정돼 있다는 전제를 둔다.
- 이번 단계의 목적은 그 기반 위에 사용자 분석, funnel, retention, cohort를 올리는 것이다.

## 2. 계산 규칙을 먼저 고정한다
- funnel과 retention은 count 집계보다 해석 규칙이 훨씬 중요하다.
- 특히 아래 항목은 구현 전에 먼저 고정한다.
  - funnel anchor 규칙
  - step 순서 인정 규칙
  - conversion window 기본 정책
  - retention의 `first seen` 정의
  - cohort 관찰 범위

## 3. 이번 단계도 event table 직접 계산을 유지한다
- funnel / retention / cohort도 우선 `events` 원본 테이블 기준으로 계산한다.
- 이 단계에서는 정합성과 계약을 먼저 확인하고, 느려지는 시점에 별도 rollup / cache / batch 전략을 검토한다.

## 4. anonymous / login 병합은 아직 다루지 않는다
- 이번 단계의 사용자 기준은 여전히 `organization` 범위 내 `eventUser.id`를 기본으로 본다.
- anonymous 포함 유저 분석, alias 병합, cross-device 식별 문제는 후속 단계로 미룬다.

## 5. 제품화는 “응답 구조와 버저닝”까지 포함한다
- 이번 문서는 분석 기능만 적는 것이 아니라, 그 기능을 제품 API처럼 다듬는 단계까지 포함한다.
- 따라서 응답 메타 정리와 버저닝 정책도 이번 단계 범위에 둔다.

# 15.3 현재 상태

## 선행 전제
- `Route Template` 기반 path 정규화가 존재한다.
- `raw eventType -> canonical eventType` 매핑이 존재한다.
- `overview / Top N / trends` 같은 핵심 집계 기반이 먼저 정리돼 있다.
- unique user 기본 기준은 `organization` 범위 내 `eventUser.id`로 이미 고정돼 있다.
- `routeKey별 unique users`, `eventType별 unique users` 같은 기본 user 집계는 14단계에서 이미 제공된다.

## 현재 구조의 한계
- 현재 집계 API만으로는 “언제 어떤 route가 많이 쓰였는가”는 볼 수 있어도, 사용자의 순차 행동까지는 설명할 수 없다.
- 신규 / 재방문, 전환 흐름, 재방문율 같은 질문에는 답할 수 없다.
- 응답 구조도 집계 API마다 아직 제각각일 가능성이 높아, 확장 분석으로 갈수록 계약 흔들림이 커질 수 있다.

즉 이번 단계는 “핵심 집계 API가 있다” 상태에서 “확장 분석 API를 가진 시스템”으로 넘어가는 단계다.

# 15.4 선행 확정 항목

## 사용자 기준
- 기본 사용자 기준은 `organization` 범위 내 `eventUser.id`
- `eventUser`가 없는 이벤트는 funnel / retention / cohort 대상에서 제외한다.
- anonymous 포함 분석은 이번 단계에서 다루지 않는다.
- 따라서 15단계 사용자 분석은 “기본 unique user 집계”를 다시 만드는 것이 아니라, 식별 사용자 기준의 확장 분석을 추가하는 단계로 본다.

## funnel step 정의 단위
- 15단계 최소 funnel step은 `canonicalEventType` 기준으로만 지원한다.
- 즉 최소 버전의 step 조건은 `canonicalEventType = X` 형태로 고정한다.
- `canonicalEventType + optional routeKey` 조합 step은 후속 확장으로 둔다.

## funnel anchor 규칙
- 각 사용자는 step1의 최초 발생 시점을 anchor로 삼는다.
- anchor 이전에 발생한 step2~N 이벤트는 funnel 계산에 포함하지 않는다.
- 각 step은 anchor 이후 시간 순서로만 인정한다.
- 각 step은 직전 인정 step 시각 이상에서 조건을 만족하는 최초 이벤트 1건만 인정한다.
- 한 사용자가 같은 step 이벤트를 여러 번 발생시켜도 가장 이른 인정 1건만 사용한다.
- step2 없이 step3만 발생한 사용자는 step3 전환자로 인정하지 않는다.
- 같은 timestamp의 이벤트는 허용한다.
  - 즉 후속 step은 이전 step과 `같거나 이후` 시각이면 인정한다.
  - 동일 timestamp에서는 step 순서 위반으로 보지 않으며, 정확한 미세 순서 재구성은 시도하지 않는다.

## conversion window
- funnel 기본 conversion window는 `7일`로 고정하고 시작한다.
- custom window는 처음부터 넓게 열지 않고, 기본값이 실제 운영에서 부족하다고 확인될 때 후속 옵션으로 확장한다.

## retention의 first seen 정의
- `first seen`은 해당 organization 범위에서 사용자의 최초 이벤트 발생 시각으로 정의한다.
- 조회 기간은 cohort 관찰 범위를 제한할 뿐, `first seen` 자체를 다시 계산하지 않는다.
- 즉 `first seen`은 “조회 구간 내 첫 이벤트”가 아니라 “organization 내 전체 이벤트 기준 earliest occurredAt”이다.

## cohort 관찰 범위
- 최소 버전은 daily cohort 기준만 지원한다.
- Day 1 / 7 / 30만 제공한다.
- Day N retention은 cohort 기준일로부터 N일 후의 동일 일자 bucket에 활동이 있는 사용자의 비율로 계산한다.
- 즉 최소 버전 retention은 `exact-day retention`이며, `on or after` 방식은 이번 단계에서 지원하지 않는다.
- custom N-day range와 복수 cohort 기준은 후속 확장으로 둔다.

## timezone 기준
- cohort date와 retention day bucket은 요청 `timezone` 기준의 local date로 계산한다.
- `first seen` timestamp 자체는 원시 이벤트 시각 기준으로 보되, cohort 분류는 timezone 적용 후의 local date를 사용한다.
- funnel도 요청 timezone을 메타에 포함하지만, 최소 버전 funnel 판정의 핵심은 local date보다 step timestamp 비교에 둔다.

# 15.5 기능 설계

### 1) 사용자 분석
- 신규 / 재방문 사용자
- 식별 사용자 1인당 평균 이벤트 수

최소 버전 정의:
- `identifiedUsers`: 조회 구간 내 활동한 식별 사용자 수
- `newUsers`: 조회 구간 내 활동했고, `firstSeen`도 조회 구간 내인 사용자 수
- `returningUsers`: 조회 구간 내 활동했고, `firstSeen`은 조회 구간 이전인 사용자 수
- `avgEventsPerIdentifiedUser`: 조회 구간 내 식별 사용자 기준 평균 이벤트 수

현재 구현 상태:
- `GET /api/v1/events/analytics/users/overview` 구현됨
- 현재 응답은
  - `identifiedUsers`
  - `newUsers`
  - `returningUsers`
  - `avgEventsPerIdentifiedUser`
  를 포함한다

주의:
- `routeKey별 unique users`
- `eventType별 unique users`
는 14단계에서 이미 구현한 기본 user 집계로 간주하고, 15단계의 신규 범위에서는 제외한다.

### 2) funnel
- canonical eventType 기준 2~4 step 정의
- 최소 버전 step은 `canonicalEventType only`
- step1-anchor 기준 분모 고정
- 동일 user 기준 집계
- conversion window 기본값은 `7일`
- step별 최초 이벤트 인정 규칙 적용
- funnel report API
- 같은 timestamp는 허용하고, 후속 step은 이전 step과 같거나 이후 시각이면 인정

최소 지원 버전:
- step은 2~4개까지 지원
- 같은 유저 기준으로 계산
- 각 step은 직전 인정 step 시각 이상에서 조건을 만족하는 최초 이벤트 1건만 인정
- step1 진입자 기준으로 전환율을 계산

### 3) retention / cohort
- cohort 기준은 `first seen`
- daily cohort만 지원
- Day 1 / 7 / 30 retention 제공
- 최소 버전 retention은 `exact-day retention`
- retention report / cohort report API

### 4) 응답 구조 고도화
- 공통 응답 메타
  - from / to
  - timezone
  - appliedFilters
  - comparison
  - summary
- 분석 API DTO 표준화

권장 응답 틀:
- `meta`
- `summary`
- `data`

`appliedFilters` 최소 뼈대 예시:
```json
{
  "organizationId": 1,
  "from": "2026-03-01T00:00:00Z",
  "to": "2026-03-08T00:00:00Z",
  "timezone": "Asia/Seoul",
  "externalUserId": null,
  "canonicalEventTypes": ["SIGN_UP", "PURCHASE"],
  "routeKey": null
}
```

### 5) API 버저닝
- `/api/v1`
- 호환 / 비호환 변경 기준 고정
- breaking change는 v2로 분리

breaking change 예시:
- 필드 제거 / 이름 변경
- 기존 필드 의미 변경
- 응답 구조 중첩 변경
- 기본 계산 규칙 변경

non-breaking change 예시:
- nullable 신규 필드 추가
- 새로운 선택 필터 추가
- 새로운 endpoint 추가

# 15.7 체크포인트

## 사용자 기준
- `eventUser.id`가 없는 이벤트가 확장 분석 결과를 과도하게 왜곡하지 않는지
- unique user coverage를 같이 확인할 수 있는지

## funnel
- anchor 이전 이벤트가 제외되는지
- 같은 timestamp나 순서 경계가 문서 기준과 같이 해석되는지
- 최소 1개 funnel이 예시 응답으로 재현 가능한지

## retention / cohort
- `first seen`이 조회 기간과 혼동되지 않는지
- Day 1 / 7 / 30 결과가 같은 기준으로 계산되는지
- daily cohort가 응답 메타와 함께 해석 가능한지

## 계약 / 제품화
- 공통 응답 메타가 API마다 과하게 흔들리지 않는지
- `/api/v1` 정책이 breaking change 기준과 함께 설명 가능한지
- 대표 요청/응답 예시가 실제 구현과 일치하는지

# 15.8 후속 작업

- 실제 구현 결과를 실행기록 문서에 반영
- 필요 시 운영 문서에 확장 분석 API 확인 포인트 반영
- 성능 병목이 확인되면 별도 문서에서 rollup / cache / aggregate table 전략으로 이어감
