# 12. 종료 절차 개선 계획 (v1.0)

## 목표
- Blue/Green 전환 이후 old color 종료 절차를 현재보다 더 설명 가능한 구조로 정리한다.
- 기존 graceful shutdown, shutdown timeout, container stop grace period가 실제 종료 시점에서 어떤 역할을 하는지 다시 확인한다.
- 개선 후 같은 `write` 배포 검증 조건에서 실패가 재발하지 않는지 다시 확인한다.

---

# 12.0 고정 원칙

## 1. 이번 단계는 배포 구조 변경이 아니라 종료 절차 개선 단계다
- 현재 운영 배포 구조(`GitHub Actions -> ECR -> S3 -> CodeDeploy -> EC2 -> Blue/Green`) 자체를 바꾸지 않는다.
- 이미 검증한 배포 구조에서 old color 종료 절차를 더 명확하게 다듬는 데 집중한다.

## 2. 현재 보호 장치를 먼저 다시 읽는다
- graceful shutdown이 없는 상태에서 새로 도입하는 단계가 아니다.
- 현재도 다음 장치가 이미 들어가 있다.
  - `server.shutdown=graceful`
  - `spring.lifecycle.timeout-per-shutdown-phase`
  - `stop_grace_period`
  - `OLD_COLOR_DRAIN_SECONDS`
- 이번 단계는 이 장치들이 실제 종료 시점에서 어떻게 작동하는지 다시 정리하는 데 의미가 있다.

## 3. 고정 sleep을 바로 없애는 것이 목표는 아니다
- 지금 `OLD_COLOR_DRAIN_SECONDS`는 1차 완화책으로 효과를 보였다.
- 이번 단계의 목표는 sleep 자체를 지우는 것이 아니라, 신규 요청 차단 이후 old color 종료 절차를 graceful shutdown 중심으로 더 설명 가능하게 만드는 것이다.

## 4. 검증 조건은 이전 `write` 실전 검증과 맞춘다
- 개선 효과는 다른 조건이 아니라, 기존 `write` 배포 검증과 최대한 같은 조건에서 비교한다.
- 그래야 종료 절차 변경의 효과를 더 명확하게 볼 수 있다.

## 5. 변경은 작게 하고 재검증으로 확인한다
- 처음부터 큰 구조 변경으로 가지 않는다.
- 종료 절차에서 가장 영향이 큰 포인트 1개를 먼저 고르고, 최소 수정 후 재검증한다.

# 12.1 범위 정의

## 포함(in scope)
- 현재 Blue/Green 종료 순서 재확인
- `scripts/blue-green-prod-switch.sh` 종료 절차 검토
- 필요 시 `scripts/deploy-prod-blue-green.sh`와의 연결 방식 점검
- 종료 관련 운영 기준 정리
- 같은 `write` 배포 검증 조건으로 재검증

## 제외(out of scope)
- ALB / 멀티 EC2 / 고가용성 구조 변경
- Kubernetes / 서비스 메시 기반 drain 처리
- 대규모 부하 테스트 또는 한계 부하 측정
- 전체 배포 아키텍처 재설계

# 12.2 완료 기준 (Done)

- old color 종료 절차를 현재보다 더 명확한 기준과 순서로 설명할 수 있다.
- 같은 `write` 배포 검증 조건에서 `http_req_failed`가 재발하지 않거나 기존 실패율 대비 유의미하게 감소한다.
- 실패 구간에서 관찰됐던 Nginx upstream reset 패턴이 동일하게 재현되지 않는다.
- 개선 내용과 재검증 결과가 실행기록 또는 운영 문서에 남는다.

---

# 12.3 문제 정의

- 현재 종료 절차는 drain 대기 추가 이후 `write` 재검증까지는 통과했다.
- 다만 종료 기준이 여전히 고정 시간 기반(`OLD_COLOR_DRAIN_SECONDS`)에 크게 의존하고 있어, 왜 이 시간이 충분한지 구조적으로 설명하기는 어렵다.
- 또한 현재 graceful shutdown, shutdown timeout, container stop grace period가 이미 존재하지만, 이 보호 장치들이 old color 종료 시점에서 실제로 어떻게 작동하는지는 아직 충분히 정리되지 않았다.
- 다음 단계의 목표는 고정 sleep 자체를 없애는 것이 아니라, 신규 요청 차단 이후 old color 종료 절차를 graceful shutdown 중심으로 더 설명 가능하게 정리하고 stop 시점 제어를 더 분명히 만드는 것이다.

# 12.5 체크포인트

## 종료 순서
- nginx 전환 이후 old color에 신규 요청이 더 이상 들어가지 않는지
- old color stop이 너무 이른 시점에 들어가지 않는지
- 종료 순서가 문서와 스크립트에서 같은 흐름으로 설명 가능한지

## 보호 장치
- Spring graceful shutdown이 실제 종료 시점에서 어떤 역할을 하는지
- `timeout-per-shutdown-phase`와 `stop_grace_period`가 서로 충돌 없이 작동하는지
- `OLD_COLOR_DRAIN_SECONDS`가 현재 절차에서 어떤 의미를 가지는지

## 재검증
- 같은 `write` 배포 검증 조건에서 실패가 재발하는지
- `http_req_failed`가 다시 올라가는지
- 실패 시간대 Nginx upstream reset 패턴이 다시 나타나는지

## 기록
- 어떤 종료 포인트를 바꿨는지
- 왜 그 포인트를 골랐는지
- 재검증에서 무엇이 달라졌는지
