# 00. 운영 관측 기반 최소세트 계획 (v1.0)

## 0단계 목표
- 로컬에서도 운영과 유사한 실행/검증 루프를 재현한다.
- health/로그/메트릭/대시보드/스모크 부하를 하나로 연결한다.
- 이후 단계 성능/안정성 비교를 위한 기준선을 확보한다.

---

## 0.0 고정 원칙
1. 실행 성공 판정은 `health` 기준으로 고정한다.
2. 관측은 기본 차단 + 필요한 항목만 허용(allowlist)한다.
3. 민감정보는 로그/Sentry 모두 기본 차단한다.
4. 성능 평가는 단일 수치가 아니라 `p95/p99 + 에러율 + 커넥션`으로 본다.

---

## 0.1 Docker 기반 실행 골격
- `Dockerfile` + `docker-compose`로 `app + postgres` 실행
- healthcheck 기반 기동 성공 판정

완료 기준:
- `docker compose up` 1회로 앱/DB 기동
- 앱/DB 모두 `healthy` 확인

---

## 0.2 Actuator/Health 기준 확립
- `/actuator/health`를 성공 판정 기준으로 사용
- 프로파일(local/ci/prod)별 노출 정책 분리
- 기본은 안전값(불필요한 actuator 노출 금지)

완료 기준:
- DB 연결 이상 시 health가 `DOWN`으로 바뀌는 것 확인

---

## 0.3 로그 정책 + requestId
- `X-Request-Id` 수용/생성/응답 전파
- 로그 상관관계 키로 requestId 고정
- API Key/토큰/비밀번호/쿠키 원문 미기록

완료 기준:
- 요청 1건 단위로 requestId 추적 가능
- 에러 시 requestId로 로그/Sentry 연계 가능

---

## 0.4 메트릭/대시보드
- Micrometer + Prometheus 노출(`/actuator/prometheus`)
- Grafana 기본 패널 구성:
  - RPS
  - p95/p99
  - 5xx 비율

완료 기준:
- API 호출 시 지표가 즉시 변화
- 부하 입력 시 p95/p99/에러율 추이를 시각적으로 확인 가능

---

## 0.5 초경량 스모크 부하(k6)
- Read-heavy / Write-heavy 2개 시나리오 고정
- 짧은 고정 부하로 회귀 기준선 확보

완료 기준:
- 부하 중 지표 상승/변동 확인
- 부하 종료 후 정상 범위 복귀 확인

---

## 0.6 선택 항목: Sentry 최소 도입
- Unhandled exception 자동 수집
- `env`, `release`, `endpoint` 태그 부여
- 민감정보 스크러빙 정책 적용

완료 기준:
- 500 유도 시 Sentry 이벤트 수신 확인
- 민감정보 미노출 확인
