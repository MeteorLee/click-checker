# 20. 대규모 부하 테스트 및 성능 개선 계획 (v1.0)

## 목표
- 19단계에서 정리한 운영/관측 기준선을 바탕으로, 현재 구조가 실제로 어디서 먼저 병목이 되는지 대규모 부하 테스트로 확인한다.
- 병목을 `p95 / p99 / 에러율 / DB / Hikari / Grafana 스크린샷` 기준의 증거로 남기고, 가장 값싼 개선부터 적용한 뒤 같은 조건으로 다시 검증한다.
- 즉 이번 단계는 “기술을 많이 추가하는 단계”가 아니라, “부하 -> 병목 확인 -> 작은 개선 -> 재검증 -> 더 큰 구조 개선 판단”의 반복을 실제로 시작하는 단계다.

---

# 20.0 고정 원칙

## 1. 이번 단계는 "대규모 테스트 + 점진적 개선"에 집중한다
- 이번 단계의 핵심은 처음부터 Redis, rollup, Kafka를 다 넣는 것이 아니다.
- 먼저 현재 구조로 실제 부하를 걸어 병목을 확인하고, 가장 값싼 개선부터 적용한다.
- 즉 이번 단계의 기본 리듬은 아래와 같다.

```text
대규모 테스트
-> 병목 확인
-> 값싼 개선
-> 같은 조건 재테스트
-> 필요 시 더 큰 구조 개선
```

## 2. 부하 테스트는 "증거 수집"이 먼저다
- 부하 테스트의 목적은 "서비스를 괴롭히는 것"이 아니라, 어느 지점에서 어떤 수치가 먼저 나빠지는지 기록하는 것이다.
- 따라서 테스트 결과는 최소한 다음을 포함해야 한다.
  - `http_req_failed`
  - `p95 / p99`
  - `RPS`
  - `4xx / 5xx`
  - DB active connections
  - Hikari pool usage
  - 관련 Grafana 패널 스크린샷

## 3. 1차 개선은 가장 싼 카드부터 쓴다
- 첫 번째 개선 수단은 보통 다음 순서로 본다.
  - EXPLAIN / 쿼리 구조 확인
  - 인덱스 설계/조정
  - 불필요한 조회 제거
  - flush/save 경로 정리
  - batch insert
- Redis, rollup, bulk ingest는 1차 카드가 아니라 2차 이후 카드로 본다.

## 4. Redis는 목적을 분리해서 본다
- Redis는 하나의 기술이지만 역할은 두 가지로 분리해서 다룬다.
- 쓰기 보호:
  - org 단위 rate limit
  - backpressure
- 읽기 절감:
  - 집계 결과 TTL cache
- 즉 `Redis rate limit`과 `Redis cache`는 같은 단계에서 무조건 같이 넣지 않는다.

## 5. rollup / retention은 장기 해법이다
- raw table을 직접 읽는 현재 구조는 기준선과 초기 분석에는 유리하지만, 장기 운영에서는 비용이 커질 수 있다.
- 따라서 rollup / retention은 초기 병목 확인 이후 "구조적으로 계속 무거운 경우"에 들어가는 해법으로 본다.
- 이때 핵심은 다음 둘의 경계를 정하는 것이다.
  - 최신 구간: raw
  - 확정 구간: rollup

## 6. 비교 조건은 끝까지 고정한다
- 전/후 비교는 같은 시험이어야 의미가 있다.
- 따라서 아래는 반드시 고정한다.
  - 동일 RPS / duration / warm-up
  - 동일 데이터셋 조건
  - 동일 대상 API
  - 동일 대시보드 시간창
  - 동일 배포 버전(커밋 해시 기록)

## 7. Kafka는 이번 단계 기본 전제가 아니다
- 이번 단계는 "Kafka 없이 현재 구조를 어디까지 끌어올릴 수 있는가"를 먼저 본다.
- 즉 Kafka는 1차 해결 수단이 아니라, 현재 구조로 설명 가능한 개선을 모두 소진한 뒤 검토하는 확장 트랙으로 둔다.

# 20.2 완료 기준 (Done)

- `write-heavy`, `read-heavy`, `mixed` 대규모 테스트를 현재 구조 기준으로 최소 1회 이상 재현할 수 있다.
- 병목 지점이 `그래프 + 수치 + 로그` 기준으로 설명 가능하다.
- 인덱스/쿼리/쓰기 경로 1차 개선 후, 동일 조건 재테스트 결과를 전/후 비교할 수 있다.
- Redis가 필요한지, 필요하다면 `rate limit`인지 `cache`인지 목적을 분리해 판단할 수 있다.
- rollup / retention이 실제로 필요한 이유와 적용 구간(최신/raw vs 확정/rollup)을 설명할 수 있다.

# 20.3 현재 상태

## 이미 있는 것
- 운영/관측 기준선
  - `Grafana / Loki / Prometheus / Alertmanager / Sentry`
- `deploy smoke`
- `k6` smoke read/write
- local / prod 기준 검증 흐름
- 운영 관측 신뢰도 정비 완료

## 이미 관찰된 힌트
- 짧은 local burst 기준:
  - write: `p95≈3.96s`, `p99≈5.26s`
  - read: `p95≈188ms`, `p99≈454ms`
- 즉 현재 구조에서는 read보다 write 쪽이 먼저 무거울 가능성이 높다.
- 다만 현재 수치는 경량 burst 기준이므로, 대규모 테스트 전의 첫 힌트로만 본다.

즉 이번 단계는 “대충 write가 더 느려 보인다” 상태에서, “어떤 부하에서 정확히 어디가 먼저 병목이 되는가”를 증거로 남기는 단계다.

# 20.4 시나리오 전략

## 1) write-heavy
- 대상:
  - `POST /api/events`
- 목적:
  - 수집 경로의 쓰기 병목 확인
  - DB connection / Hikari / insert 비용 관찰
- 핵심 관찰:
  - `http_req_failed`
  - `p95 / p99`
  - insert 지연
  - DB active connections
  - Hikari saturation 여부

## 2) read-heavy
- 대상:
  - `GET /api/v1/events/analytics/aggregates/paths`
  - 필요 시 `overview`, `time-buckets`
- 목적:
  - 집계 조회 비용 확인
  - 범위, bucket, route/eventType 해석 비용 확인
- 핵심 관찰:
  - `p95 / p99`
  - 4xx / 5xx
  - DB read 부하
  - 긴 쿼리 여부

## 3) mixed
- 대상:
  - write + read 동시
- 목적:
  - "수집 중 조회"가 실제로 어떤 영향을 주는지 확인
  - write/read 경합 상황 관찰
- 핵심 관찰:
  - write p95/p99 변화
  - read p95/p99 변화
  - DB connection 경쟁
  - 5xx 상승 여부

## 초기 실험안(v1)
- warm-up: `2m`
- main run: `10m`
- cooldown: `1m`
- write-heavy:
  - `50 -> 100 -> 200 RPS` 단계 상승
- read-heavy:
  - `10 -> 30 -> 50 RPS` 단계 상승
- mixed:
  - write `100 RPS` + read `20 RPS`

주의:
- 위 숫자는 첫 실행용 초안이며, 실제 baseline 결과에 따라 조정할 수 있다.
- 중요한 것은 숫자 자체보다, 같은 시나리오를 전/후 비교에 반복 사용할 수 있게 고정하는 것이다.

# 20.6 개선 우선순위

## 1차: 값싼 개선

### 1. EXPLAIN / 쿼리 구조
- `EXPLAIN ANALYZE` 기준으로 어떤 쿼리가 먼저 비싼지 확인
- 특히 아래 축을 우선 본다.
  - `organization_id + occurred_at`
  - `routeKey`
  - `canonicalEventType`

### 2. 인덱스 설계/조정
- `organization_id + occurred_at` 복합 인덱스 점검
- routeKey 포함 인덱스 전략 검토
- eventType 포함 인덱스 전략 검토
- covering index 가능성 분석

### 3. 쓰기 경로 정리
- `/api/events` 요청 1건당 DB 조회/저장 수 확인
- event user lookup / create 경로 확인
- flush/save 구조 점검
- 필요 시 Hibernate/JDBC batch 적용 검토

### write 경로 병목 후보(우선 가설)
- organization 인증/조회 비용
- event user lookup 후 create 여부 판단 비용
- 중복되는 `select + insert` 패턴
- transaction 당 flush 발생 지점
- JPA dirty checking / persistence context overhead
- 인덱스 증가로 인한 insert 비용 상승
- DB connection pool 대기
- unique constraint 경합 또는 lock 가능성

### read-heavy 세부 시나리오 메모
- `R1 overview`
  - 최근 `1일`
  - 핵심 요약 응답 비용 확인
- `R2 paths`
  - 최근 `7일`, `top=10`
  - group by + sort + topN 비용 확인
- `R3 time-buckets`
  - 최근 `30일`, `day` bucket
  - bucket 수 증가와 응답 조립 비용 확인

주의:
- read가 느리다는 식의 뭉뚱그린 해석을 피하기 위해, overview / paths / time-buckets를 분리해서 본다.
- 이후 캐시나 인덱스 판단도 어떤 조회가 느린지 쪼개서 본 결과를 기준으로 한다.

## 2차: 구조 보조 수단

### Redis rate limit
- org 단위 쓰기 보호
- 과부하 시 429 / Retry-After 정책 검토
- 목적:
  - 서비스 보호
  - 백프레셔

### Bulk ingest
- 배열 수신 구조
- insert 최적화
- 부분 실패/전체 실패 정책 검토

### Redis cache
- 집계 API 결과 TTL cache
- 목적:
  - 반복 조회 부하 절감
- 적용 후보:
  - `overview`
  - `paths`
  - `time-buckets`

## 3차: 구조 변경

### rollup / retention
- 최신 구간은 raw
- 확정 구간은 rollup
- 오래된 raw는 retention
- Spring Batch 실행체 도입

### 재실행 안전 원칙
- `rollup_key` 기준 upsert
- cursor 저장
- late arrival 허용 범위
- 재집계 정책

## 1차 개선 목표(약한 목표값)
- write-heavy에서 baseline 대비 `p95 30% 이상 개선`이면 유의미한 개선으로 본다.
- 같은 부하에서 5xx 발생 구간이 뒤로 밀리면 개선으로 본다.
- 같은 부하에서 Hikari active/idle 균형이 더 안정되면 개선으로 본다.
- read-heavy는 평균보다 `p99 안정화`를 더 중요하게 본다.

## 관측 체크리스트
- `k6` summary 저장
- Grafana latency 패널 캡처
- Grafana error rate 패널 캡처
- DB active connections 패널 캡처
- Hikari pool usage 패널 캡처
- 애플리케이션 error log 확인
- 필요 시 Sentry issue 발생 여부 확인
- 테스트 커밋 해시 기록
- 대상 조직 / path prefix / eventType 기록

## 병목 판정 규칙(가이드)
- `p95/p99` 급등 + Hikari active 고정 상승
  - DB / connection 대기 우선 의심
- 5xx 증가 + timeout 증가
  - 애플리케이션 처리 한계 또는 pool 고갈 우선 의심
- write만 급격히 악화, read는 비교적 안정
  - insert / write path 문제 우선 의심
- mixed에서만 급격히 악화
  - shared resource 경쟁 또는 connection 경합 우선 의심
- DB CPU 상승 + 긴 쿼리 증가
  - 인덱스 / 실행 계획 문제 우선 검토
- DB는 안정적인데 app latency만 상승
  - application layer overhead 또는 flush/save 경로 검토

# 20.7 단계별 실행 순서

## 1. 대규모 테스트 설계 문서 고정
- 시나리오
- RPS
- duration
- warm-up
- 스크린샷 기준

## 2. 1차 대규모 테스트
- 현재 구조 그대로 실행
- 병목 지점 식별

## 3. 인덱스 / 쿼리 / 쓰기 경로 1차 개선
- EXPLAIN
- 인덱스
- batch / lookup / save 경로 정리

## 4. 2차 대규모 테스트
- 동일 조건 재실행
- before / after 비교

## 5. Redis / bulk ingest 판단
- 그래도 write/read가 무거우면 2차 카드 검토

## 6. rollup / retention 설계
- 구조적으로 raw 직접 집계 비용이 큰지 판단
- 필요 시 배치/rollup 트랙으로 이동

즉 이번 단계는 "부하 테스트를 한 번 해본다"가 아니라, "부하 테스트와 개선을 반복 가능한 운영 루프로 만든다"는 데 목적이 있다.
