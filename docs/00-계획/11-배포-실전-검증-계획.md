# 11. 배포 실전 검증 계획 (v1.0)

## 목표
- 현재 운영 배포 구조(CodeDeploy + Blue/Green + readiness + graceful shutdown)가 실제 요청 지속 상태에서도 안정적으로 동작하는지 검증한다.
- 배포 전환 중 `5xx`, timeout, 응답 지연 급증이 없는지 최소 관측 기준으로 확인한다.
- 이후 운영 문서와 실행기록에서 설명 가능한 “실전 검증 증거”를 남긴다.

---

# 11.0 고정 원칙

## 1. 이번 단계는 기능 추가가 아니라 검증 단계다
- 새로운 기능/API 스펙 변경은 하지 않는다.
- 이미 반영된 배포/전환 구조가 실제로 안전한지 확인하는 데 집중한다.

## 2. 실사용자 트래픽이 없어도 “지속 요청 상태”를 재현하면 충분하다
- 현재는 실제 사용자 트래픽을 전제로 하지 않는다.
- 대신 `k6` 등으로 일정한 지속 요청을 만들어 배포 시나리오를 재현한다.

## 3. 배포 성공 판단은 health만으로 끝내지 않는다
- `health`, `readiness` 확인은 기본이다.
- 전환 중 `5xx`, timeout, `p95/p99` 변화를 함께 본다.

## 4. 관측은 최소 지표에 집중한다
- 모든 메트릭을 다 보지 않는다.
- 이번 단계의 핵심 지표는 다음으로 고정한다.
  - RPS
  - `p95`
  - `p99`
  - `5xx ratio`
  - 필요 시 배포 시각 전후 health / readiness

## 5. 결과는 숫자와 문서로 남긴다
- “문제 없었다” 수준으로 끝내지 않는다.
- 실행 조건, 배포 시각, 지표 결과, 이상 여부를 문서에 남긴다.

## 6. 읽기와 쓰기 검증은 분리해서 시작한다
- 처음부터 read/write 혼합 시나리오로 시작하지 않는다.
- 먼저 읽기 지속 요청 검증을 수행하고, 그 다음 쓰기 지속 요청 검증을 수행한다.
- 목적은 실패 원인과 지표 변화를 더 단순하게 해석하는 데 있다.

---

# 11.1 범위 정의

## 포함(in scope)
- `k6` 시나리오를 현재 `X-API-Key` 계약 기준으로 정리
- 지속 요청 상태를 만드는 실험 조건 정의
- Blue/Green 또는 CodeDeploy 배포 수행 중 지표 관찰
- 배포 전/중/후 `5xx`, timeout, `p95/p99` 기록
- 실험 결과 문서화

## 제외(out of scope)
- 실제 사용자 트래픽 기반 검증
- 대규모 성능 테스트 또는 한계 부하 측정
- 알림 시스템(Slack/Discord) 최종 연동
- ALB / 멀티 EC2 / 고가용성 검증

# 11.2 완료 기준 (Done)

- `k6` 요청이 현재 API 계약(`X-API-Key`)에 맞게 동작한다.
- 지속 요청 상태에서 배포 또는 색상 전환을 1회 이상 수행한다.
- `k6` 실행 중 `http_req_failed = 0` 이다.
- 전환 중 timeout이 `0건`이다.
- 전환 중 `5xx ratio = 0%` 를 목표로 한다.
- `p95`, `p99`, `5xx ratio`, RPS를 기록한다.
- `p95`, `p99`는 전환 시점에 일시 변동이 있더라도 전환 후 짧은 시간 내 기준선 근처로 회복되는지 확인한다.
- 전환 후 공개 경로 응답, health, readiness가 모두 정상이다.
- 결과가 문서로 남고, 이후 운영 검증 기준으로 다시 참고할 수 있는 수준으로 정리된다.

# 11.3 검증 시나리오

## 시나리오 A. 지속 읽기 요청 + 배포 전환
- `GET /api/events/aggregates/paths`
- 목적:
  - 전환 중 읽기 요청이 끊기지 않는지 확인
  - `p95/p99`, `5xx ratio` 변화 관찰

## 시나리오 B. 지속 쓰기 요청 + 배포 전환
- `POST /api/events`
- 목적:
  - 전환 중 쓰기 요청이 실패 없이 처리되는지 확인
  - timeout 또는 `5xx`가 발생하지 않는지 확인
  - 테스트 데이터가 실험 단위로 식별 가능한지 확인

## 시나리오 C. 필요 시 읽기/쓰기 분리 반복
- read-heavy와 write-heavy를 한 번에 섞지 않고 각각 반복 수행
- 목적:
  - 문제 발생 시 원인을 단순하게 분리
  - 지표 비교를 쉽게 유지

# 11.4 실행 순서

## 1) 실험 조건 고정
1. 대상 환경 고정
   - local docker 또는 운영 EC2 중 하나를 선택
2. 대상 배포 방식 고정
   - `blue-green-prod-switch` 또는 CodeDeploy 배포
3. 대상 조직/API Key 준비
4. 실험 데이터 범위(테스트용 path prefix, org) 고정
5. 배포 대상 버전(커밋 SHA 또는 이미지 태그) 기록

## 2) 부하 도구 정리
1. `k6/smoke-read.js`를 현재 `X-API-Key` 기준으로 정리
2. `k6/smoke-write.js`를 현재 `X-API-Key` 기준으로 정리
3. 실행 환경 변수와 결과 기록 형식을 고정
4. `k6` 결과를 stdout 요약과 별도 파일 기준으로 남길 방법을 정리

## 3) 배포 전 기준값 확보
1. 배포 전 health / readiness 확인
2. Grafana에서 기준 지표 확인
3. `k6` 실행 시작
4. 요청이 안정적으로 유지되는지 확인
5. 배포 시작 전 기준 지표 스크린샷 또는 수치를 남긴다.

## 4) 전환 수행
1. 비활성 색상 기동 또는 CodeDeploy 배포 시작
2. readiness `UP` 확인
3. Nginx 전환 또는 CodeDeploy 완료 확인
4. 전환 직후 공개 경로와 health 재검증
5. 전환 관련 시각을 순서대로 기록한다.

## 5) 결과 기록
1. `k6` 실패율 / timeout 여부 기록
2. Grafana의 `p95`, `p99`, `5xx ratio`, RPS 기록
3. 전환 시각과 이상 징후 여부 기록
4. `k6` summary(stdout 또는 파일) 결과를 함께 남긴다.
5. 결과를 실행기록/운영 문서에 반영

# 11.5 체크포인트

## 요청 경로
- read/write 요청 모두 `X-API-Key` 기준으로 동작하는지
- 테스트 데이터가 운영 데이터와 식별 가능하게 분리되는지

## 테스트 데이터 식별 규칙
- write 시나리오는 테스트용 path prefix를 고정한다.
  - 예: `/loadtest/deploy-validation/*`
- 필요 시 `eventType` 또는 payload에 run id를 함께 넣는다.
- 테스트 organization 또는 테스트 API Key를 우선 사용한다.
- 목적은 실험 데이터와 기존 데이터를 분리해서 조회/정리/검증하기 위함이다.

## 배포 전환
- 새 색상 readiness `UP` 확인 전 트래픽을 넘기지 않는지
- 전환 후 루트 응답 `color`가 기대한 값인지
- health / readiness가 전환 후에도 정상인지

## 관측
- Grafana에서 전환 시점 전후로 지표를 비교할 수 있는지
- `5xx ratio`가 0 또는 허용 가능한 수준인지
- `p95/p99`가 급격히 튀는지 여부를 확인할 수 있는지

## 전환 시각 기록
- T0: `k6` 시작 시각
- T1: 새 색상 컨테이너 기동 시작 시각
- T2: readiness `UP` 확인 시각
- T3: Nginx upstream 전환 또는 CodeDeploy 전환 완료 시각
- T4: 공개 경로 재검증 완료 시각
- T5: 이전 색상 종료 시각

## 기록
- 실험 일시
- 대상 버전(커밋 SHA 또는 이미지 태그)
- 실행 명령
- 결과 수치
- 해석 / 다음 액션

# 11.6 실패 기준

- 전환 중 `5xx`가 반복적으로 발생한다.
- timeout 또는 연결 실패가 눈에 띄게 발생한다.
- `p95/p99`가 비정상적으로 급등하고 회복되지 않는다.
- readiness `UP` 이전에 트래픽이 넘어간 정황이 보인다.
- 전환 후 health 또는 공개 경로 응답이 불안정하다.

위 조건 중 하나라도 충족되면:
- 이번 검증은 실패로 기록한다.
- 다음 순서로 원인을 확인한다.
  1. `k6` summary에서 실패 유형 확인
  2. 배포 시각 기준으로 app-blue/app-green 로그 확인
  3. nginx access/error 로그 확인
  4. readiness endpoint 응답 상태 확인
  5. Sentry 이벤트 및 requestId 로그 확인
  6. old color stop 시점이 너무 빨랐는지 검토
- 해결 후 같은 조건으로 재실험한다.
