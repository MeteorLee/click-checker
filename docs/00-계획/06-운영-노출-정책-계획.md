# 06. 운영 노출 정책 계획 (v1.0)

## 6단계 목표
- 외부 공개 대상과 내부 전용 대상을 운영 기준으로 명확히 구분한다.
- 메인 서비스, 관리자 도구, 내부 포트의 노출 원칙을 문서로 고정한다.
- 이후 서브도메인 확장이나 접근 정책 변경 시 판단 기준으로 사용한다.

---

# 6.0 고정 원칙

## 1. 메인 서비스와 관리 도구를 같은 기준으로 다루지 않는다
- 메인 서비스는 사용자 접근 기준으로 본다.
- Grafana 같은 관리 도구는 관리자 접근 기준으로 별도 판단한다.

## 2. 외부 공개는 최소 범위만 허용한다
- 외부 공개 대상은 실제 서비스 사용과 운영 검증에 필요한 최소 범위로 제한한다.
- 기본은 비공개, 필요한 대상만 공개한다.

## 3. 내부 포트는 진입점 뒤로 숨긴다
- app `8080`, DB `5432` 같은 내부 포트는 직접 외부에 노출하지 않는다.
- 외부 접근은 Nginx 같은 진입점 뒤에서만 허용한다.

## 4. 운영 검증 경로도 최소 공개를 원칙으로 한다
- `/actuator/health`만 외부 공개 기준으로 사용한다.
- 내부 actuator와 관측 경로는 기본 비공개로 둔다.

## 5. 도메인 추가는 필요가 생길 때만 한다
- 서브도메인은 “가능하다”와 “지금 운영한다”를 구분한다.
- `www`, `grafana` 같은 서브도메인은 후속 판단 대상으로 둔다.

---

# 6.1 범위 정의

## 포함(in scope)
- 메인 앱 공개 대상 확정
- health 공개 범위 확정
- Grafana 현재 노출 정책 정리
- Prometheus 비공개 원칙 정리
- app / DB 내부 포트 비공개 원칙 정리
- 도메인 / 서브도메인 사용 원칙 정리

## 제외(out of scope)
- Grafana 도메인화 실제 적용
- Prometheus 외부 공개
- ALB / WAF / SSO 같은 상위 인프라 도입
- VPN / Bastion / 사내망 연계

---

# 6.2 완료 기준 (Done)
- 외부 공개 대상, 제한 공개 대상, 내부 전용 대상이 문서로 구분된다.
- `clickchecker.dev`의 역할이 메인 진입점으로 고정된다.
- Grafana 현재 정책이 “임시 허용”이 아니라 운영 판단으로 설명된다.
- Prometheus, DB, app 내부 포트의 비공개 원칙이 명시된다.
- `www`, `grafana` 같은 서브도메인 정책이 현재/후속으로 구분된다.

---

# 6.3 판단 순서

## 1) 메인 앱
- 외부 공개 대상인지
- 도메인 하나로 고정 가능한지
- health 경로를 어떻게 둘지

## 2) 관리자 도구
- 실제로 외부에서 자주 접근해야 하는지
- IP 제한 / 로그인만으로 충분한지
- 별도 도메인화 가치가 있는지

## 3) 내부 서비스
- 직접 포트를 열 이유가 있는지
- Nginx 또는 내부 네트워크 뒤로 숨길 수 있는지

---

# 6.4 체크포인트

## 공개 대상 점검
- 메인 서비스만 외부 공개돼 있는지
- health 외의 actuator가 열려 있지 않은지

## 제한 공개 점검
- Grafana가 현재 내 IP 제한 + 로그인 기준인지

## 내부 비공개 점검
- `8080`, `5432`가 외부 비공개인지
- Prometheus가 외부에서 직접 접근되지 않는지

---

# 6.5 후속 작업

- `www.clickchecker.dev` 지원 여부 검토
- 운영 노출 정책을 배포/운영 체크리스트와 더 강하게 연결
