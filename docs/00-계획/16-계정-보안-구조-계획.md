# 16. 계정 / 보안 구조 계획 (v1.2)

## 목표
- 로그인 가능한 사용자 계정 기반의 최소 인증 구조를 도입한다.
- 기존 `X-API-Key` 기반 수집/분석 API와 사용자 로그인 인증 경계를 분리한다.
- Spring Security + JWT 기반 최소 보안 구조를 먼저 고정하고, 조직 멤버십 및 RBAC은 다음 단계로 넘긴다.

---

# 16.0 고정 원칙

## 1. 이번 단계는 "계정 인증 뼈대 도입"에 집중한다
- 이번 단계의 핵심은 로그인 가능한 사용자 계정과 토큰 기반 인증 구조를 만드는 것이다.
- 기존 수집/분석 API의 조직 인증 방식(`X-API-Key`)은 유지한다.
- 즉 이번 단계는 "기존 인증 교체"가 아니라 "계정 로그인 인증 추가" 단계다.

## 2. 인증 수단을 목적별로 분리한다
- `X-API-Key`는 서버 간 수집/분석 호출용으로 유지한다.
- JWT는 로그인한 사용자 계정 기반 보호 API용으로 사용한다.
- 두 인증 수단은 같은 의미로 취급하지 않고, 경계와 보호 범위를 분리한다.

## 3. RBAC은 이번 단계에서 최소화한다
- 이번 단계에서는 OWNER / ADMIN / VIEWER 같은 세부 역할 모델까지 바로 도입하지 않는다.
- 먼저 "로그인 가능한 계정"과 "보호된 계정 기반 경로"를 만드는 데 집중한다.
- 조직과 계정의 관계, 권한 모델 세분화는 다음 단계(`OrganizationMember`, RBAC)로 넘긴다.

## 4. Access Token과 Refresh Token의 역할을 분리한다
- Access Token은 짧은 수명의 인증 토큰으로 사용한다.
- Refresh Token은 재발급 전용으로만 사용한다.
- 만료, 폐기, 재발급 정책은 구현 전에 문서에서 먼저 고정한다.
- 최소 버전은 refresh token rotation을 기본값으로 사용한다.

## 5. 보안 규칙은 구현 전에 먼저 고정한다
- 비밀번호 저장 방식
- JWT 만료 시간
- refresh token 저장/회전 정책
- `401` / `403` 기준
- 로그 민감정보 차단 기준
- 위 항목은 구현보다 먼저 문서에서 결정한다.

## 5.1 인증 분기는 경로 기준으로 고정한다
- `/api/admin/**`
  - JWT 인증 대상
- `/api/events/**`, `/api/v1/events/**`
  - 기존 `X-API-Key` 인증 대상
- `/actuator/health`
  - 비보호 경로
- 즉 두 인증 수단을 모든 요청에 동시에 적용하지 않고, 경로 기준으로 명확히 분기한다.

## 6. 운영 배포와 충돌하지 않는 방향으로 설계한다
- 현재 배포 구조(`GitHub Actions -> ECR -> S3 -> CodeDeploy -> EC2 -> Blue/Green`)는 유지한다.
- secret 주입, 프로파일 설정, 운영 health/smoke 구조를 깨지 않는 방식으로 설계한다.
- 인증 추가 때문에 기존 배포 검증 경로가 복잡해지지 않도록 한다.

# 16.2 완료 기준 (Done)

- 사용자 계정으로 로그인할 수 있다.
- 로그인 성공 시 access token / refresh token을 발급할 수 있다.
- access token으로 보호된 계정 기반 API에 접근할 수 있다.
- refresh token으로 access token 재발급이 가능하다.
- 로그아웃 또는 refresh token 폐기 후 재사용이 차단된다.
- `X-API-Key` 기반 이벤트 수집/분석 API와 JWT 보호 경로가 충돌하지 않는다.
- 보안 정책과 API 계약이 문서와 코드에 반영된다.

# 16.3 현재 상태

## 이미 있는 것
- `X-API-Key` 기반 organization 인증 구조
- `/api/events/**`, `/api/v1/events/**` 보호 경로
- API Key 평문 비노출 원칙
- `requestId` 기반 요청 추적
- 운영 배포 / 관측 / Blue/Green 구조

## 현재 구조의 한계
- 로그인 가능한 사용자 계정 주체가 없다.
- 계정 기반 보호 API 경계가 없다.
- 조직/운영 관리 API는 별도 사용자 인증 없이 열려 있거나, 예외 경로로 남아 있다.
- JWT / refresh / 만료 정책이 아직 없다.

즉 이번 단계는 "API Key 기반 서버 간 인증" 위에 "계정 로그인 기반 사용자 인증"을 올리는 단계다.

# 16.4 선행 확정 항목

## 로그인 주체
- 로그인 주체는 `Account`로 고정한다.
- 1차 버전에서는 계정 식별값을 `loginId` 기준으로 시작한다.
- email은 연락/확장 가능 필드로 둘 수 있으나, 로그인 필수값으로 강제하지 않는다.
- 이번 단계의 `Account`는 아직 특정 `Organization`과 직접 연결되지 않는 독립 로그인 주체로 본다.
- 계정과 조직의 관계는 다음 단계의 `OrganizationMember`에서 다룬다.

## 인증 경계
- `X-API-Key`
  - ingest / analytics 호출용
  - organization scope 결정용
- JWT
  - 계정 기반 보호 API 접근용
  - 사용자 로그인 상태 표현용
- 두 인증 수단은 같은 필터나 같은 컨텍스트 의미로 합치지 않는다.

## 비밀번호 정책
- 비밀번호는 평문 저장 금지
- `BCryptPasswordEncoder` 사용
- password hash만 저장
- 최소 버전에서는 강한 비밀번호 정책보다 "안전한 저장과 검증"을 우선한다

## JWT 정책
- access token은 짧은 만료 시간을 사용한다
  - 1차 권고: `15m`
- refresh token은 더 긴 만료 시간을 사용한다
  - 1차 권고: `14d`
- access token claim 최소 범위
  - `sub` (`accountId` 문자열)
  - 필요 시 `type=access`
- signing secret은 환경변수로 주입한다

## refresh token 정책
- refresh token은 서버에서 추적 가능한 저장 구조를 둔다
- 평문 저장 대신 hash 저장을 우선 검토한다
- 최소 버전에서는 refresh token을 request body로 전달한다
- 최소 버전에서는 "재발급 시 기존 refresh token 폐기 + 새 refresh token 발급" 기준으로 시작한다
- 이미 `revoked`된 refresh token으로 재발급을 다시 시도하면 `401`로 거부한다
- 재사용 감지 시 보안 로그는 남기되, 계정 전체 세션 강제 종료는 다음 단계로 미룬다
- 로그아웃 시 현재 refresh token을 폐기할 수 있어야 한다
- 이후 관리자 콘솔 UI 단계에서 HttpOnly cookie 방식 전환을 별도 검토할 수 있다

## 에러 정책
- 인증 실패: `401`
- 권한 부족: `403`
- 로그인 요청 형식 오류: `400`
- loginId/password 불일치: `401`
- 로그인 시 `DISABLED` 계정도 외부 응답은 `401`로 처리한다
- 이미 인증된 access token으로 보호 API 접근 시 account 상태가 `DISABLED`면 `403`으로 처리한다

## 최초 계정 생성 방식
- local 환경에서는 seed 또는 bootstrap 방식으로 기본 계정 생성을 허용한다
- prod 환경에서는 자동 계정 생성을 허용하지 않는다
- prod 최초 계정은 수동 bootstrap 절차 또는 수동 SQL로 준비한다

## secret / 운영 정책
- JWT secret은 `.env` 또는 운영 환경변수로만 주입한다
- local / prod secret을 분리한다
- 로그에 비밀번호, access token, refresh token 원문을 남기지 않는다

## 민감정보 마스킹 정책
- 최소 마스킹 대상:
  - `password`
  - `newPassword`
  - `accessToken`
  - `refreshToken`
  - `Authorization`
  - `X-API-Key`
- request / response logging이 있다면 위 필드는 기본 마스킹한다
- 인증/재발급 API에서 token 원문은 운영 로그와 예외 로그 모두에 남기지 않는다

# 16.5 도메인 설계

## 1) Account
- 역할:
  - 로그인 사용자 주체
- 최소 필드:
  - `id`
  - `loginId`
  - `passwordHash`
  - `status`
  - `createdAt`
  - `updatedAt`
- 상태 최소 버전:
  - `ACTIVE`
  - `DISABLED`
- 상태 규칙:
  - `ACTIVE`: 로그인 가능, 토큰 발급 가능
  - `DISABLED`: 로그인 불가, 보호 API 접근 불가

## 2) RefreshToken
- 역할:
  - access token 재발급 전용 상태 저장
- 최소 필드:
  - `id`
  - `account`
  - `tokenHash`
  - `expiresAt`
  - `revokedAt`
  - `createdAt`
  - `lastUsedAt`
- 목적:
  - 재발급 가능 여부 판단
  - 로그아웃/폐기 처리
  - 토큰 재사용 차단

## 3) Organization과의 관계
- 이번 단계에서는 `Account -> OrganizationMember -> Organization` 구조를 바로 만들지 않는다.
- 즉 계정과 조직 권한 연결은 17단계의 멤버십/RBAC 단계로 넘긴다.
- 이번 단계의 목적은 로그인 구조와 보호 경로 확보에 한정한다.

# 16.6 API 설계

## 1) 로그인
- `POST /api/admin/auth/login`
- 입력:
  - `loginId`
  - `password`
- 출력:
  - `accessToken`
  - `refreshToken`
  - `accessTokenExpiresIn`
  - `refreshTokenExpiresIn`

## 2) 토큰 재발급
- `POST /api/admin/auth/refresh`
- 입력:
  - body의 `refreshToken`
- 출력:
  - `accessToken`
  - `refreshToken`
  - `accessTokenExpiresIn`
  - `refreshTokenExpiresIn`
- 최소 버전 정책:
  - refresh 성공 시 기존 refresh token은 폐기하고 새 refresh token을 발급한다
  - 이미 폐기된 refresh token 재사용 시 `401`로 거부한다

## 3) 로그아웃
- `POST /api/admin/auth/logout`
- 입력:
  - body의 `refreshToken`
- 동작:
  - 현재 refresh token 폐기

## 4) 내 정보
- `GET /api/admin/me`
- 목적:
  - access token 검증
  - 현재 로그인 계정 식별
- 최소 응답:
  - `accountId`
  - `loginId`
  - `status`

# 16.7 Security 설계

## 보호 경로
- `/api/admin/**`는 JWT 보호 대상으로 본다.
- 단, 아래 예외는 비보호 경로로 둔다.
  - `POST /api/admin/auth/login`
  - `POST /api/admin/auth/refresh`
- 현재 `/api/admin/**`는 "조직 멤버십/역할이 필요한 관리자 기능"까지 뜻하지 않고, 계정 로그인 기반 보호 경로의 최소 prefix로 본다.

## 기존 보호 경로 유지
- `/api/events/**`, `/api/v1/events/**`는 `X-API-Key` 보호를 유지한다.
- 기존 API Key 필터의 의미와 조직 스코프는 변경하지 않는다.

## 비보호 경로
- `/actuator/health`
- 로그인
- refresh
- 기존 공개 대상은 현재 운영 정책 문서를 따른다

## 필터 체인
- Spring Security를 도입하되, 기존 API Key 필터와 충돌하지 않는 순서를 사용한다.
- JWT 인증은 `SecurityContext` 기준으로 정리한다.
- API Key 인증은 기존 request attribute / organization context 기준을 유지한다.
- 즉 경로 기준으로 인증 방식을 분기하고, 모든 요청에 두 인증 필터를 무조건 중복 적용하지 않는다.

## 운영 보안 기준
- secret 환경변수 미설정 시 fail-fast 여부를 검토한다
- prod에서는 debug 로그로 인증 민감정보를 남기지 않는다
- request / response logging이 있다면 비밀번호 / token 필드는 기본 마스킹한다

# 16.9 체크포인트

## 인증 경계
- JWT와 API Key 보호 범위가 섞이지 않는지
- 계정 로그인 인증이 이벤트 수집/분석 API에 영향을 주지 않는지
- `/api/admin/**`와 `/api/events/**` 계열이 서로 다른 인증 방식으로 안정적으로 동작하는지

## 토큰
- access token 만료 후 보호 경로 접근이 차단되는지
- refresh token으로만 재발급이 가능한지
- 폐기된 refresh token 재사용이 차단되는지
- refresh token rotation 이후 이전 token이 다시 통과하지 않는지

## 운영
- secret이 코드가 아니라 환경변수로만 주입되는지
- 로그에 비밀번호 / access token / refresh token 원문이 남지 않는지
- 기존 배포/운영 health 경로를 깨지 않는지
- local/prod 관리자 계정 생성 방식이 혼동되지 않는지

## 회귀
- 기존 `X-API-Key` 인증 테스트가 깨지지 않는지
- `test`, `postgresTest` 기준 회귀가 없는지
- 현재 배포 구조에서 Security 추가로 불필요한 운영 복잡도가 생기지 않는지
