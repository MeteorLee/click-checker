# ADR-002 Querydsl 도입

## 배경(왜 이 결정이 필요했는가)

- 기존 JPA 방식으로는 필터마다 새로운 메서드를 추가해야하는 구조였다.
- 집계/조회 기능이 확장될수록 조건의 조합이 증가해 Repository에 추가해야하는 메서드가 급격히 늘어나게 되었다.
- JPQL을 사용하는 경우 컴파일 과정에서 오류를 잡아낼 수 없어 개발 과정에서 불편함이 생겼다.

---

## 선택(무엇을 선택했는가)

- 동적 조건이 필요한 조회/집계 쿼리에 Querydsl을 도입한다.
- 적용 범위: Event 조회 / 집계 쿼리 / 전체 검색 API 등

---

## 이유(왜 이 선택이 적절하다고 판단했는가)

- 필터 조합이 늘어날수록 JPA Repository 메서드가 조합 폭발로 증가할 수 있다.
- Querydsl로 동적 where 조건을 조립하면 메서드 수를 줄이고 코드 중복을 줄일 수 있다.
- 집계 중심 프로젝트는 기간/organization 등 조건이 유동적이므로, 조회 조건 변경에 유연한 구조가 유지보수에 유리하다.
- Querydsl은 문자열 기반 JPQL과 달리 타입 안정성을 제공하여 컴파일 시점에 오류를 검출할 수 있다.
