# 코딩 규칙 (v1.0)

## 1. 목적
- 현재 프로젝트에서 코드 구조와 구현 형식을 일관되게 유지한다.
- 기능 추가보다 읽기 쉬움, 수정 용이성, 책임 분리를 우선한다.
- 포트폴리오 관점에서 "왜 이렇게 나눴는지" 설명 가능한 구조를 유지한다.

## 2. 적용 범위
- `src/main/java/com/clickchecker/**`
- `src/test/java/com/clickchecker/**`
- API 계약 문서와 구현 간 형식 일관성

## 3. 기본 원칙
- 패키지는 기능 기준으로 나눈다.
- 컨트롤러는 얇게 유지하고, 실제 규칙은 서비스로 보낸다.
- 읽기 전용 복잡 쿼리는 서비스에서 메모리 가공하지 않고 DB 조회 단계에서 좁힌다.
- API 응답 DTO와 DB 조회 projection은 역할을 섞지 않는다.
- "일단 동작"보다 "다음 단계에서 덜 꼬이는 구조"를 우선한다.

## 4. 패키지 구조 규칙
- 최상위 패키지는 기능 기준으로 둔다.
  - 예: `organization`, `event`, `route`, `eventtype`, `account`, `auth`, `organizationmember`
- 기능 패키지 내부는 역할 기준으로 나눈다.
  - `controller`
  - `entity`
  - `repository`
  - `service`
- 공통 인프라 성격 코드는 기능 패키지 밖으로 분리한다.
  - `config`
  - `web/filter`
  - `web/resolver`
  - `logging`

## 5. Controller 규칙
- 요청/응답 DTO는 `controller/request`, `controller/response`에 별도 파일로 둔다.
- 컨트롤러 내부 중첩 `record`는 지양한다.
- 컨트롤러는 아래 역할만 담당한다.
  - 요청 검증
  - 상태 코드 결정
  - 서비스 호출
  - 응답 매핑
- 인증/조직 컨텍스트 추출은 컨트롤러가 직접 파싱하지 않고 resolver 또는 명시된 request attribute 규칙을 사용한다.

## 6. Service 규칙
- 서비스는 비즈니스 규칙의 중심이다.
- 서비스는 컨트롤러 응답 DTO를 직접 알지 않는다.
- 여러 곳에서 재사용되는 서비스 결과는 별도 타입으로 분리한다.
- 서비스 내부에서만 잠깐 쓰는 로컬 값은 내부 `record`를 허용할 수 있다.
- 서비스에서 `findAll()` 후 메모리 필터링/정렬은 지양한다.

## 7. Repository / Query 규칙
- `JpaRepository`는 저장과 단순 조회를 담당한다.
- 조인, 정렬, 조건 조합이 있는 읽기 쿼리는 별도 `QueryRepository`로 분리한다.
- 프로젝트의 읽기 전용 복잡 쿼리는 Querydsl을 우선 사용한다.
- "특정 조직 데이터만 필요"한 경우 DB 조회 단계에서 바로 조직 스코프를 적용한다.

## 8. Projection 규칙
- DB 읽기 결과 전용 타입은 `repository/projection`에 둔다.
- projection은 DB 조회 결과 표현용이다.
- projection을 controller response로 직접 노출하지 않는다.
- controller response는 API 계약 중심으로 별도 정의한다.

## 9. record 사용 기준
- 허용:
  - `controller/request`
  - `controller/response`
  - `repository/projection`
  - 서비스 내부의 매우 로컬한 임시 결과
- 지양:
  - 컨트롤러 내부 중첩 `record`
  - 여러 서비스/컨트롤러에서 공유되는 내부 `record`
- 기준:
  - 파일 밖에서도 의미가 있으면 별도 파일로 뺀다.
  - 한 클래스 내부 구현 세부사항이면 내부 `record`를 고려한다.

## 10. Mapper 규칙
- 단순 DTO 변환에는 mapper 사용을 허용한다.
  - 예: `request -> entity`
  - 예: `entity -> response`
- 비즈니스 규칙이 들어가는 조립은 mapper가 아니라 service에서 처리한다.
  - 예: 조직 조회 후 연결
  - 예: role 검증
  - 예: 중복 검증 후 entity 생성
- repository projection과 mapper는 역할을 섞지 않는다.
  - projection은 DB 읽기 결과 표현
  - mapper는 객체 간 변환
- MapStruct를 사용할 경우 Spring bean 방식으로 통일한다.
- mapper는 "필드 복사"를 단순화하기 위한 도구로만 사용하고, 도메인 규칙을 숨기는 장소로 쓰지 않는다.

## 11. 예외 처리 규칙
- 현재 프로젝트에서는 `ResponseStatusException`을 우선 사용한다.
- 같은 종류의 실패는 같은 상태 코드와 비슷한 메시지 톤을 유지한다.
- `404`는 리소스 자체가 없을 때 사용한다.
- `403`은 인증은 되었지만 조직 membership 또는 role이 부족할 때 사용한다.
- `401`은 인증 실패 또는 무효 토큰/키일 때 사용한다.
- 서비스 계층에서 예외 의미를 먼저 고정하고, 컨트롤러는 그 의미를 그대로 따른다.

## 12. 테스트 규칙
- 새 기능 추가 시 최소 한 개 이상의 성공 경로 테스트를 둔다.
- 권한/인증 경계 추가 시 실패 경로 테스트를 함께 둔다.
- 설정, 필터, Flyway, 공통 인프라 변경은 전체 회귀 가능성을 고려한다.
- 단순 서비스/DTO 수정은 관련 테스트 위주로 검증한다.

## 13. 리팩터링 기준
- 기능 추가 중이라도 아래 경우는 즉시 정리한다.
  - `findAll()` 후 메모리 필터링
  - controller 내부 구현 형식이 기존 규칙과 크게 어긋나는 경우
  - projection / response / service result 역할이 섞인 경우
- 후속 단계에서 구조가 크게 바뀔 것이 명확하면, 현재 단계에서는 최소 동작만 구현하고 다음 계획 문서에 정리 대상을 남긴다.

## 14. 현재 프로젝트에 바로 적용할 기준
- `Admin` 계열 API의 request/response는 별도 DTO 파일로 정리한다.
- `OrganizationMember` 읽기 쿼리는 `QueryRepository`를 통해 조회한다.
- route / event type 목록처럼 조직별 조회가 필요한 경로는 repository 단계에서 조직 스코프를 적용한다.
- controller response와 repository projection은 분리한다.

## 15. 완료 기준
- 새 기능 패키지를 추가할 때 구조 선택 기준을 설명할 수 있다.
- controller / service / repository 책임이 섞이지 않는다.
- 읽기 쿼리와 API 응답 타입이 분리되어 있다.
- 팀원 없이 혼자 보더라도 "왜 이 파일이 여기 있는지" 이해할 수 있다.
