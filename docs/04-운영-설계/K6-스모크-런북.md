# k6 Smoke Runbook (2.0)

## 0. 목적
- `k6/smoke-read.js`, `k6/smoke-write.js`를 동일한 조건으로 반복 실행한다.
- p95/p99, 실패율, RPS를 기준으로 성능 기준선을 남긴다.
- 로컬 baseline 실행과 운영형 배포 검증에서 같은 스크립트를 일관된 기준으로 사용한다.

## 0.1 문서 적용 범위
- 1~7장은 로컬 baseline 또는 수동 회귀 실행 기준이다.
- 8장 이후 기록은 실제 실행 결과를 남기는 용도다.
- 운영형 Blue/Green 배포 검증에서는 같은 스크립트를 쓰되, 대상 앱과 검증 목적이 다르다.
  - 전환 전: target color direct smoke
  - 전환 후: public smoke

## 1. 사전 준비
1. 인프라 기동 (DB/Prometheus/Grafana)
```bash
docker compose up -d postgres prometheus grafana
```

2. 애플리케이션 로컬 실행 (개발 기본 방식)
```bash
SPRING_PROFILES_ACTIVE=local ./gradlew bootRun
```

메모:
- 로컬 기본 프로필은 `local` 기준으로 본다.
- 개발 편의상 IntelliJ + DevTools 실행을 함께 사용해도 되지만, smoke 기준 명령은 위 `bootRun`을 기준으로 기록한다.
- `bootRun`만으로는 코드 저장 시 자동 재기동이 항상 보장되지 않으므로, 변경 반영이 의심되면 앱 재기동 여부를 먼저 확인한다.
- Docker app 기준으로 검증할 때는 `docker-compose.local.yml` 오버레이를 같이 사용한다.
- Docker image가 최신 코드를 담으려면 `bootJar` 선행 후 재빌드가 필요하다.

3. 헬스 체크
```bash
curl -sS http://localhost:8080/actuator/health
```

## 2. 테스트 데이터 준비
- write 시나리오는 직접 데이터를 적재한다.
- read 시나리오는 조회 대상 조직의 `X-API-Key` 기준으로 데이터가 있어야 의미 있는 결과가 나온다.
- 관리자 콘솔 데모 확인이 필요하면 `bash scripts/data/seed-demo-organization.sh`로 `demo_web_shop` 데이터셋을 다시 생성할 수 있다.

메모:
- demo seed는 `최근 60일`, `300,000 events`, `10,000 identified users` 기준의 로컬/데모용 데이터셋이다.
- demo seed는 기존 demo organization 내부 데이터를 갈아엎고 다시 채우는 방식이므로, 운영 데이터가 있는 환경에서는 기본 실행 대상으로 보지 않는다.

## 3. Read-heavy 실행
```bash
BASE_URL=http://localhost:8080 \
API_KEY=<ORG_API_KEY> \
FROM=2026-02-13T00:00:00Z \
TO=2026-02-14T00:00:00Z \
TOP=5 \
k6 run k6/smoke-read.js
```

## 4. Write-heavy 실행
```bash
BASE_URL=http://localhost:8080 \
API_KEY=<ORG_API_KEY> \
EVENT_TYPE=click \
k6 run k6/smoke-write.js
```

## 5. 결과 확인 포인트
- k6 콘솔
  - `http_req_failed`
  - `http_req_duration p(95), p(99)`
  - `iterations`, `vus`

- Grafana (http://localhost:3000)
  - `Overview`: RPS, p95 / p99, 4xx / 5xx 비율
  - `Deploy / Smoke`: 단건 smoke와 최근 성공 흔적
  - `Logs`: app 로그 / nginx 로그 분리 확인

- Prometheus 타깃 상태
```bash
curl -sS http://localhost:9090/api/v1/targets
```
  - 로컬 단일 앱 기준: `click-checker-app` 수집 상태 확인
  - 운영형 Blue/Green 기준: `app-blue:8081`, `app-green:8082` 타깃 수집 상태 확인

## 6. 비교 시 고정 조건 (반드시 동일)
- 동일 RPS
- 동일 duration
- 동일 warm-up 정책
- 동일 API Key / 기간 조건
- 동일 코드 버전(커밋 해시)

## 7. 실패 시 빠른 점검
1. 앱 미기동
```bash
curl -sS http://localhost:8080/actuator/health
```
2. Prometheus 미수집
```bash
docker compose logs --tail=100 prometheus
```
3. DB 연결 문제
```bash
docker compose logs --tail=200 app
```

메모:
- 현재 이벤트 API는 `organizationId`가 아니라 `X-API-Key` 기준으로 조직을 확정한다.
- k6 실행 전 대상 조직의 API Key를 먼저 준비해야 한다.
- 필요하면 `EXPORT_API_KEY_FILE=... BASE_URL=http://localhost:8080 bash scripts/deploy/deploy-smoke.sh`로 임시 키를 준비할 수 있다.
- 로컬 baseline은 `BASE_URL=http://localhost:8080` 기준으로 실행한다.
- 운영형 배포 검증에서는 같은 스크립트를 direct/public smoke 용도로 사용한다.
  - direct smoke: target color 포트(`127.0.0.1:8081` 또는 `127.0.0.1:8082`)
  - public smoke: `https://clickchecker.dev`

## 8. 기준선 기록 (Baseline #1)
- 실행일: 2026-02-27
- 실행 환경: `docker network(click-checker_default) + grafana/k6 컨테이너`
- 대상 앱: `http://app:8080`
- 공통 조건:
  - executor: `constant-arrival-rate`
  - rate: `20 req/s`
  - duration: `2m`

- script: `k6/smoke-write.js`

- `http_req_duration p(95)`: `29.7ms`
- `http_req_duration p(99)`: `118.52ms`
- `http_reqs`: `2401` (`19.91 req/s`)

### 8.2 Read-heavy 결과
- script: `k6/smoke-read.js`

- `http_req_duration p(95)`: `22.44ms`
- `http_req_duration p(99)`: `42.25ms`
- `http_reqs`: `2400` (`19.91 req/s`)

### 8.3 비교 메모
- 현재 기준선에서는 read-heavy가 write-heavy보다 지연이 낮다.
- 두 시나리오 모두 threshold 통과:
  - `http_req_failed < 1%`
  - `p95 < 500ms`
  - `p99 < 1000ms`
- 이 기준선은 로컬 단일 app 기준 기록이며, 운영형 Blue/Green 배포 검증 수치와 직접 동일시하지 않는다.
