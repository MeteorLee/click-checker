# 운영 관측 정책

## 문서 목적

현재 Click-Checker 프로젝트의 운영 관측 기준을 정리한다.  
이번 문서는 배포 과정이나 트러블슈팅 기록이 아니라, 현재 운영 기준에서 무엇을 보고 어떤 순서로 판단하는지를 짧게 고정하는 데 목적이 있다.

## 현재 관측 구조

- 메트릭: Prometheus
- 로그: Loki
- 시각화: Grafana
- 알림: Alertmanager
- 예외 추적: Sentry

현재 운영 기준 관측 흐름은 아래와 같다.

```text
지표 확인
-> 로그 확인
-> 필요 시 예외 확인
```

## 운영 기준

### 1. 메트릭

Prometheus는 다음 메트릭을 기본 관측 대상으로 본다.

- RPS
- `p95`, `p99`
- `4xx / 5xx ratio`
- DB pool usage
- JVM heap usage

local에서는 `app:8080`을 scrape 대상으로 본다.  
prod에서는 `app-blue:8081`, `app-green:8082`를 scrape 대상으로 본다.

### 2. 로그

로그 조회는 Loki 기준으로 수행한다.

- app 로그
  - stdout 기반
- nginx 로그
  - access / error file log 기반

요청 추적 기준은 `X-Request-Id`다.  
Blue/Green 구분 기준은 app 로그의 `APP_COLOR` 출력이다.

### 3. 대시보드

Grafana는 다음 데이터소스를 기본으로 사용한다.

- Prometheus
- Loki

현재 기준 대시보드는 다음 세 가지다.

- `Click Checker Overview`
  - 추세, burst, 지연, 에러율, DB 상태 확인
- `Click Checker Logs`
  - app / nginx 로그 분리 확인
- `Click Checker Deploy / Smoke`
  - 단건 smoke, 최근 성공 흔적, 최근 API 5xx 확인

즉 현재 운영 기준에서는
- `Overview`는 추세 / 부하 확인용
- `Logs`는 로그 원문 확인용
- `Deploy / Smoke`는 배포 직후 / smoke 직후 확인용
으로 역할을 분리한다.

### 4. 알림

Prometheus rule + Alertmanager 기준 최소 알림은 다음과 같다.

- `ClickCheckerNoScrapeTargets`
- `ClickCheckerAppDown`
- `ClickCheckerHigh5xxRatio`
- `ClickCheckerHighP95Latency`
- `ClickCheckerHighDbPoolUsage`

현재 단계에서는 알림 채널 고도화보다, 어떤 상태를 이상 징후로 볼지 rule을 코드로 고정하는 데 우선순위를 둔다.
현재 알림 전달 채널은 Discord webhook 기준으로 연결한다.
현재 `ClickCheckerHigh5xxRatio`, `ClickCheckerHighP95Latency`는 전체 요청이 아니라 제품 API(`/api/...`) 중심으로 해석한다.

### 5. 예외

예외는 Sentry에서 확인한다.  
Grafana와 Loki에서 이상 징후나 특정 요청 흐름을 먼저 확인한 뒤, 필요 시 Sentry에서 예외 상세를 본다.

현재 Sentry에서 우선 확인하는 운영 문맥은 다음과 같다.

- `requestId`
- `appColor`
- `release`
- `authType`
- 필요 시 `accountId`, `organizationId`, `apiKeyKid`

다음 프레임워크성 4xx는 Sentry 전송 대상에서 제외한다.

- `NoResourceFoundException`
- `HttpRequestMethodNotSupportedException`
- `HttpMediaTypeNotAcceptableException`
- `HttpMediaTypeNotSupportedException`

다만 이 기준은 "예외 전송을 줄이는 규칙"일 뿐, 요청 자체를 줄이는 규칙은 아니다.  
따라서 운영 기준에서는 Sentry 필터링과 별도로, Nginx가 명백한 스캐너 경로를 앞단에서 404로 short-circuit 하도록 유지한다.

## 현재 판단 원칙

- 단건 smoke 여부는 `Deploy / Smoke`에서 먼저 본다.
- 추세, burst, 지연, 에러율은 `Overview`에서 본다.
- app / nginx 로그는 `Logs`에서 분리해서 본다.
- 지연, 에러 비율, DB pool 이상이 보이면 Loki에서 같은 시점 로그를 확인한다.
- 특정 요청 흐름 확인이 필요하면 `requestId` 기준으로 로그를 찾는다.
- 예외 원인 확인이 필요하면 Sentry를 본다.
- 배포 직후에는 `health -> smoke -> Grafana -> Loki -> Sentry` 순서로 확인한다.
- prod에서는 inactive color가 scrape target에 남아 일시적으로 `down`으로 보일 수 있으므로, `AppDown`은 `sum(up) == 0` 기준으로 해석한다.
- 명백한 스캐너 404 잡음은 `Sentry 필터링`과 `Nginx 앞단 404`를 분리해서 관리한다.

## 후속 검토 항목

- Alert 실제 발송 체인 반복 검증
- `Smoke Log Hits`를 더 좁은 의미의 지표로 바꿀지 검토
- 성능 개선 단계에서 현재 `Overview`를 기준선 대시보드로 계속 사용할지 검토
- Loki retention / compactor / 장기 보존 정책
- 초기 기동 시 과거 로그 경고 허용 범위
- Promtail deprecated/EOL 상태를 고려한 Grafana Alloy 전환
