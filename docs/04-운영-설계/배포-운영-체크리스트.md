# 배포 운영 체크리스트 (2.0)

## 1. 목적
- 배포 실수를 줄이기 위해 배포 전/후 확인 항목을 고정한다.
- 현재 브랜치/배포 운영 규칙과 알림 기준 초안을 한 문서로 관리한다.

---

## 2. 현재 운영 규칙
- 브랜치:
  - `develop`: 기능 통합
  - `main`: 강한 검증(`test + postgresTest`)
  - `prod`: 배포 전용
- 배포 트리거(현재):
  - `prod` push 시 자동 배포
  - 필요 시 `workflow_dispatch` 수동 실행

---

## 3. 배포 전 체크
- [ ] `main` 기준 CI 통과 확인(`test`, `postgresTest`)
- [ ] 운영 DB 백업 수행
- [ ] 이번 배포에 DB migration 포함 여부 확인
- [ ] `.env` 필수 값 확인(`DB_URL`, `DB_USERNAME`, `DB_PASSWORD`, `API_KEY_PEPPER`, `API_KEY_ENV`, `SENTRY_DSN`)
- [ ] 롤백 기준 커밋(이전 정상 SHA) 확인

---

## 4. 배포 후 체크
- [ ] `/actuator/health`가 `UP`인지 확인
- [ ] target color `/actuator/health/readiness`가 `UP`인지 확인
- [ ] direct smoke 쓰기 1건(Organization 생성 + Event 저장) 확인
- [ ] public smoke 읽기 1건(aggregate 조회) 확인
- [ ] active color(`app-blue` 또는 `app-green`)의 `DB_URL`이 운영 기준 DB(RDS)인지 확인
- [ ] old color가 `draining` 후 정상 종료됐는지 확인
- [ ] 외부 공개/비공개 대상이 [운영-노출-정책.md](운영-노출-정책.md) 기준과 일치하는지 확인
- [ ] 실패 시 기존 active color 유지/복구 우선으로 롤백 절차 실행

---

## 5. Flyway 운영 규칙
- 운영 DB 최초 전환 시에만 baseline 1회 적용
- baseline 적용 후 `baselineOnMigrate` 설정은 제거
- 이후 DB 변경은 `V{N}__*.sql` 추가 후 배포로만 반영
- migration은 `추가 -> 백필 -> 제약` 순서를 기본 원칙으로 사용

---

## 6. 알림 기준 초안 (초기값)
- 5xx 비율: 5분 평균 `>= 2%` 경보
- p95 응답시간: 5분 평균 `>= 800ms` 경보
- DB pool active connection: max 대비 `>= 80%` 5분 지속 시 경보

---

## 7. 조정 원칙
- 위 임계값은 초기값이며 실제 트래픽 기준으로 조정한다.
- 알림은 “장애 감지” 용도, 원인 분석은 requestId 로그 + Sentry로 수행한다.

## 8. 현재 배포 흐름 메모
- 현재 prod 배포는 `GitHub Actions -> ECR/S3 -> CodeDeploy -> EC2 Blue/Green` 흐름이다.
- 전환 순서는 `target color 기동 -> readiness -> direct smoke -> Nginx 전환 -> public smoke -> internal drain -> old color stop` 기준으로 본다.
