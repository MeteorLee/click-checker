# 도메인 모델 개요 (v0.1 - 현재 상태 기준)

## 현재 엔티티 구조
- Organization
- EventUser
- Event

## 엔티티 관계
- Organization 1:N EventUser
- Organization 1:N Event

## 설계 의도
- 조직 단위 데이터 격리
- 향후 API Key 기반 인증 전환 예정

## 격리 기준
- 조회/집계는 항상 organizationId 기준으로 제한한다.

## 확장 예정
- Plan / Subscription 도입
- 집계 캐싱 구조 고려

## 핵심 엔티티 (v0.2)
### Organization
- 필수 필드: `id`, `name`
- 용도: 데이터 격리의 최상위 단위

### Event
- 필수 필드: `id`, `organizationId`, `eventType`, `path`, `occurredAt`
- 선택 필드: `externalUserId`, `payload`
- 용도: 사용자 행동 로그 원본 저장

## API 계약 (현재 구현 기준)
### 이벤트 저장
- `POST /api/events`
- 필수 요청값: `organizationId`, `eventType`, `path`, `occurredAt`
- 에러 조건:
  - 필수값 누락/형식 오류 -> 400
  - 내부 처리 실패 -> 500

### 경로 집계 조회
- `GET /api/events/aggregates/paths`
- 필수 요청값: `organizationId`, `from`, `to`
- 선택 요청값: `top` (미입력 시 기본값 적용)
- 에러 조건:
  - 기간/파라미터 검증 실패 -> 400
  - 내부 처리 실패 -> 500

## 비즈니스 규칙 (기본)
- 기간 검증: `from <= to`
- `top` 규칙: 기본값 사용, 과도한 값은 상한으로 제한
- 집계 기준: `organizationId + path` 기준 count 내림차순

## 운영 규칙 (현재 결정)
- 스모크 데이터는 `eventType="__smoke"` 태그로 구분한다.
- 운영 집계에서 스모크 제외 여부는 후속 정책으로 확정한다.
- 장애 추적의 공통 식별자는 `requestId`를 사용한다.

## 운영 진입점 정의 (1.X)
- 수집 API(`POST /api/events`)는 서버 간 호출 전용으로 간주한다. (추후 API Key 강제)
- 조회 API(`GET /api/events/aggregates/*`)는 관리자/운영자 용도로 제한한다. (추후 인증 강제)
- 임시 `organizationId` 쿼리 기반 접근은 인증/권한 체계 도입 시 종료한다.
