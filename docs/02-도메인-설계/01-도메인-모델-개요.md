# 도메인 모델 개요 (v0.2 - 현재 상태 기준)

## 현재 엔티티 구조
- Organization
- EventUser
- Event

## 엔티티 관계
- Organization 1:N EventUser
- Organization 1:N Event

## 설계 의도
- 조직 단위 데이터 격리
- API Key 기반 organization 스코프 강제

## 격리 기준
- 조회/집계/수집은 `X-API-Key` 인증 결과(`authOrgId`) 기준으로 제한한다.

## 확장 예정
- Plan / Subscription 도입
- 집계 캐싱 구조 고려

## 핵심 엔티티 (v0.2)
### Organization
- 필수 필드: `id`, `name`, `apiKeyKid`, `apiKeyHash`, `apiKeyPrefix`, `apiKeyStatus`
- 용도: 데이터 격리의 최상위 단위

### Event
- 필수 필드: `id`, `organizationId(인증 컨텍스트 기반)`, `eventType`, `path`, `occurredAt`
- 선택 필드: `externalUserId`, `payload`
- 용도: 사용자 행동 로그 원본 저장

### EventUser
- 고유 규칙: `(organization_id, external_user_id)` 유니크
- 용도: 조직 내부 사용자 식별자 정규화
- 이벤트 저장 시 `externalUserId`가 있으면 조회 후 없을 때 자동 생성

## API 계약 (현재 구현 기준)
### 이벤트 저장
- `POST /api/events`
- 필수 요청값: `X-API-Key`(헤더), `eventType`, `path`, `occurredAt`
- 선택 요청값: `externalUserId`, `payload`
- 에러 조건:
  - `X-API-Key` 누락/무효 -> 401
  - 필수값 누락/형식 오류 -> 400
  - 내부 처리 실패 -> 500

### 경로 집계 조회
- `GET /api/events/aggregates/paths`
- 필수 요청값: `X-API-Key`(헤더), `from`, `to`
- 선택 요청값: `top` (미입력 시 기본값 적용)
- 에러 조건:
  - `X-API-Key` 누락/무효 -> 401
  - 기간/파라미터 검증 실패 -> 400
  - 내부 처리 실패 -> 500

## 비즈니스 규칙 (기본)
- 기간 검증: `from <= to`
- `top` 규칙: 기본값 사용, 과도한 값은 상한으로 제한
- 집계 기준: `authOrgId + path` 기준 count 내림차순

## 운영 규칙 (현재 결정)
- 스모크 데이터는 `eventType="__smoke"` 태그로 구분한다.
- 운영 집계에서 스모크 제외 여부는 후속 정책으로 확정한다.
- 장애 추적의 공통 식별자는 `requestId`를 사용한다.

## 운영 진입점 정의 (2단계 반영)
- 수집 API(`POST /api/events`)는 `X-API-Key` 기반 organization 스코프를 강제한다.
- 조회 API(`GET /api/events/aggregates/*`)도 `X-API-Key` 기반 organization 스코프를 강제한다.
- 외부 API 계약에서 `organizationId` 직접 전달 방식은 제거했다.
