# 프론트엔드 계획 (v1.0)

## 목표
- Click-Checker에 브라우저에서 직접 사용할 수 있는 최소 콘솔 화면을 추가한다.
- 1차 프런트는 `로그인 -> organization 선택 -> overview 대시보드` 흐름만 닫는다.
- 기존 백엔드의 JWT 인증과 admin overview API를 그대로 활용해, 제품 흐름을 실제로 보여줄 수 있는 데모 가능한 UI를 만든다.

---

## 1. 기본 방향

- 프런트는 현재 저장소 안에서 함께 관리한다.
- 코드 위치는 루트 아래 `frontend/` 디렉토리로 분리한다.
- 백엔드와 문서는 현재 저장소 체계를 유지하고, 프런트 문서는 `docs/07-front` 아래에 둔다.

설명:
- 지금 단계에서는 프런트와 백엔드가 하나의 제품 흐름으로 강하게 연결돼 있다.
- 저장소를 분리하기보다 같은 저장소 안에서 같이 관리하는 편이 더 단순하고 빠르다.

---

## 2. 기술 방향

- 프레임워크:
  - Next.js
- UI 시스템:
  - Mantine
- 상태 관리:
  - 1차는 React 기본 상태와 단순 API 호출 중심으로 간다.
- 인증:
  - `Authorization: Bearer <accessToken>`
- API 사용 범위:
  - `login`
  - `me`
  - admin `overview`

설명:
- 지금 단계의 목적은 프런트 기술 실험이 아니라, 백엔드 기능을 실제 화면으로 연결하는 것이다.
- Mantine은 로그인, AppShell, 카드, 목록, 선택 UI를 빠르게 안정적으로 만들기 좋다.
- 1차에서는 복잡한 전역 상태관리 도구를 넣지 않는다.

---

## 3. 화면 범위

## 포함(in scope)
- 로그인 화면
- organization 선택 화면
- overview 대시보드 화면

## 제외(out of scope)
- member 관리 화면
- API key 관리 화면
- routes / event-types / trends / users / activity / funnels / retention 화면
- 복잡한 차트 시각화
- 모바일 우선 최적화
- 다크 모드

설명:
- 지금 단계에서는 "보여지는 전체 흐름"을 먼저 닫는 것이 중요하다.
- overview 하나만으로도 JWT 인증, organization membership, admin analytics 흐름을 충분히 보여줄 수 있다.

---

## 4. 사용자 흐름

1. 사용자는 로그인 화면에서 `loginId`, `password`를 입력한다.
2. 로그인 성공 시 access token을 저장한다.
3. 프런트는 `/api/v1/admin/me`를 호출해 사용자가 속한 organization membership 목록을 가져온다.
4. 사용자는 organization 선택 화면에서 볼 organization을 고른다.
5. 프런트는 선택된 `organizationId`로 admin overview API를 호출한다.
6. overview 대시보드에서 핵심 지표를 보여준다.

즉 1차 흐름은 아래처럼 고정한다.

```text
/login
-> /organizations
-> /dashboard/[organizationId]
```

---

## 5. 화면별 역할

## 5.1 로그인 화면
- 역할:
  - 계정 로그인
  - access token 확보
- 구성:
  - 서비스명
  - loginId 입력
  - password 입력
  - 로그인 버튼
- 성공 후:
  - `/organizations`로 이동

## 5.2 organization 선택 화면
- 역할:
  - 현재 계정이 접근 가능한 organization 목록 표시
  - 대시보드 진입 organization 선택
- 구성:
  - organization 이름
  - 현재 role(`OWNER / ADMIN / VIEWER`)
  - `대시보드 열기` 버튼

## 5.3 overview 대시보드 화면
- 역할:
  - 선택된 organization의 overview 조회
- 구성:
  - 현재 organization 정보
  - 기간 선택
  - KPI 카드
  - 비교 정보
  - top routes
  - top event types

## 6. 디자인 방향

- 전체 분위기는 "운영 콘솔형 분석 대시보드"로 간다.
- 마케팅 랜딩페이지처럼 과하게 장식하지 않는다.
- 숫자와 상태를 빠르게 읽을 수 있는 밝은 관리 화면을 우선한다.

## 디자인 원칙
- 배경은 밝고 차분하게 유지한다.
- 카드 중심 구조로 정보를 분리한다.
- 포인트 색은 제한적으로 사용한다.
- 숫자는 크고 선명하게 보여야 한다.
- 차트보다 카드/요약 리스트를 먼저 보여준다.

## 1차 톤
- 밝은 회색 계열 배경
- 흰 카드
- 진한 텍스트
- 파랑/청록 계열 포인트 색

---

## 7. 1차 데이터/API 계약

## 로그인
- `POST /api/v1/admin/auth/login`

## 현재 사용자/organization 목록
- `GET /api/v1/admin/me`

## overview
- `GET /api/v1/admin/organizations/{organizationId}/analytics/overview`
- 파라미터:
  - `from`
  - `to`

## overview API 사용 원칙
- 프런트는 `timezone` 파라미터를 보내지 않는다.
- overview 날짜 해석은 백엔드 기준 `Asia/Seoul` 고정 정책을 따른다.
- 프런트는 organization 선택 후 path의 `organizationId`를 그대로 사용한다.

---

## 8. 1차 상태 관리 원칙

- access token
- 현재 사용자 정보(`/me`)
- 선택된 `organizationId`
- overview 결과
- 각 화면의 loading / error 상태

원칙:
- 1차는 복잡한 전역 상태관리 도구를 도입하지 않는다.
- 페이지 단위 state와 단순 공용 유틸로 시작한다.
- 인증 실패 시 로그인 화면으로 돌려보내는 최소 흐름만 먼저 닫는다.

---

## 9. 디렉토리 방향

예상 구조:

```text
frontend/
  src/
    app/
      login/
        page.tsx
      organizations/
        page.tsx
      dashboard/
        [organizationId]/
          page.tsx
    components/
      auth/
      organization/
      dashboard/
      common/
    lib/
      api/
      session/
      utils/
    types/
```

원칙:
- 페이지는 얇게 유지한다.
- API 호출 코드는 `lib/api`로 분리한다.
- UI 조각은 `components`로 분리한다.
